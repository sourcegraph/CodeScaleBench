It operates under a **consumer group model** where multiple consumers can coordinate to collectively consume partitions of topics.\n\n### Key Characteristics\n\n- **Single-threaded design**: The KafkaConsumer is NOT thread-safe except for `wakeup()`. All calls (except `wakeup()` and `close()` in certain contexts) must come from a single thread.\n- **Consumer group coordination**: Automatic partition assignment and rebalancing via a group coordinator broker.\n- **Flexible offset management**: Both automatic and manual offset commit strategies are supported.\n- **Stateful API**: The consumer maintains state across poll() calls, including fetched records buffer, position tracking, and rebalance state.\n- **Blocking poll semantics**: The `poll()` method blocks waiting for broker responses, but can be woken up via `wakeup()`.\n\n### Threading Model\n\n```\nConsumer Thread\n├── poll() - Main event loop\n│   ├── Heartbeat sending (background via heartbeat thread when needed)\n│   ├── Metadata fetching\n│   ├── Record fetching\n│   ├── Rebalance callbacks (onPartitionsRevoked, onPartitionsAssigned)\n│   └── Record buffering and returning\n├── Other API calls (seek, assign, subscribe, etc.)\n└── wakeup() - Can be called from OTHER threads\n\nNote: poll() is the ONLY entry point for heartbeats and rebalance events.\nAll rebalance callbacks execute DURING poll(), not asynchronously.\n```\n\n---\n\n## Core Concepts\n\n### Consumer Groups\n\nA **consumer group** is identified by `group.id` configuration. Multiple KafkaConsumer instances with the same group.id form a group that collectively consumes topic partitions.\n\n- **Partition Assignment**: Partitions are automatically distributed among group members\n- **Rebalancing**: When members join/leave, a rebalance redistributes partitions\n- **Coordination**: One group member becomes the \"generation leader\" and computes assignments\n\n### Offsets and Position Management\n\nSeveral offset concepts are critical to understand:\n\n| Concept | Meaning | Method | Notes |\n|---------|---------|--------|-------|\n| **Position** | Next offset to be fetched in current session | `position(TopicPartition)` | Changes when records fetched; All network I/O, metadata updates, and callbacks occur within poll().\n\n### `poll(Duration timeout)`\n\n```java\npublic ConsumerRecords<K, V> poll(Duration timeout);\n```\n\nFetch records within the specified timeout.\n\n**Behavioral Notes:**\n\n1. **Blocking Semantics:**\n   - If records are available in client buffer, returns immediately\n   - Otherwise, blocks waiting for broker response (up to timeout duration)\n   - If rebalance occurs during poll, timeout may be exceeded\n   - **GOTCHA**: poll() can block longer than timeout if rebalance callbacks are slow\n\n2. **What Happens During poll():**\n   - Sends heartbeats to group coordinator (keep-alive)\n   - Fetches new records from brokers\n   - Handles metadata updates\n   - Executes rebalance callbacks (onPartitionsRevoked, onPartitionsAssigned)\n   - Commits offsets if auto-commit is enabled\n   - Filters messages based on isolation level\n\n3. **Return Value:**\n   - `ConsumerRecords<K, V>` collection (never null, may be empty)\n   - Records are ordered within each partition\n   - No ordering guarantees across partitions\n   - Records are buffered locally; multiple polls may return buffered records\n\n4. **Exceptions:**\n   - `WakeupException` - if `wakeup()` was called from another thread\n   - `InterruptException` - if thread was interrupted\n   - `SerializationException` - if key/value deserialization fails\n   - `OffsetOutOfRangeException` - if committed offset no longer exists\n   - `NoOffsetForPartitionException` - if auto.offset.reset=none and no committed offset\n\n**Example:**\n```java\nwhile (true) {\n    try {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n        for (ConsumerRecord<String, String> record : records) {\n            // Process record\n            System.out.println(\"Topic: \" + record.topic() +\n                             \", Partition: \" + record.partition() +\n                             \", Offset: \" + record.offset() +\n                             \", Key: \" + record.key() +\n                             \", Value: \" + record.value());\n        }\n\n        // Explicit offset commit after batch processing\n        consumer.commitSync();\n    } catch (WakeupException e) {\n        // Consumer was woken up, exit gracefully\n        break;\n    }\n}\n```\n\n### Timeout Configuration\n\nThe `timeout` parameter controls how long poll() waits for broker responses:\n\n```java\n// Short timeout - returns quickly if no data available\nconsumer.poll(Duration.ofMillis(100));\n\n// Long timeout - waits longer for records\nconsumer.poll(Duration.ofMillis(1000));\n\n// Zero timeout - return immediately if no buffered records\nconsumer.poll(Duration.ZERO);\n```\n\n**Real-world Tuning:**\n- High-latency networks: Longer timeouts to accommodate network delays\n- Low-latency requirements: Shorter timeouts for responsiveness\n- **Default (300ms)**: Balances latency and throughput\n\n### `pollForRecords(long timeoutMs)` [Deprecated]\n\nLegacy method; Management\n\n### Understanding Rebalancing\n\nA **rebalance** redistributes partition ownership among group members. Triggered by:\n1. New member joins (via subscribe() on first poll())\n2. Member leaves (close() or timeout)\n3. Topic subscribed via pattern gains new matching topics\n4. Partition count changes for subscribed topics\n\n### Rebalance Lifecycle\n\n```\nPoll detects rebalance needed\n    ↓\nMember joins group with current assignment\n    ↓\nGroup enters \"Rebalancing\" state\n    ↓\nCoordinator assigns new leader\n    ↓\nLeader computes new assignment (based on configured strategy)\n    ↓\nAll members receive new assignment\n    ↓\nonPartitionsRevoked(old_partitions) executed\n    ↓\nMember revokes partitions, sends commit offsets\n    ↓\nonPartitionsAssigned(new_partitions) executed\n    ↓\nMember can seek to custom positions\n    ↓\nGroup is \"Stable\", polling resumes\n```\n\n### ConsumerRebalanceListener Callbacks\n\n#### `onPartitionsRevoked(Collection<TopicPartition> partitions)`\n\nCalled DURING poll() when partitions are being revoked.\n\n**Key Points:**\n- **This is your last chance to commit offsets** for revoked partitions\n- Called with partitions being removed from this consumer\n- Blocks poll() until callback returns\n- Exceptions thrown here cause rebalance failure\n\n```java\nconsumer.subscribe(Arrays.asList(\"my-topic\"),\n    new ConsumerRebalanceListener() {\n        @Override\n        public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n            System.out.println(\"Revoking: \" + partitions);\n\n            // CRITICAL: Commit offsets before losing partitions\n            try {\n                consumer.commitSync();\n                System.out.println(\"Offsets committed for revoked partitions\");\n            } catch (Exception e) {\n                logger.error(\"Failed to commit during rebalance\", e);\n                // Don't throw - let rebalance continue\n                // But log it so you know you may have lost progress\n            }\n        }\n\n        @Override\n        public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n            System.out.println(\"Assigned: \" + partitions);\n\n            // Optionally seek to custom positions\n            for (TopicPartition tp : partitions) {\n                // Seek to beginning of newly assigned partitions\n                consumer.seekToBeginning(Arrays.asList(tp));\n            }\n        }\n    }\n);\n```\n\n#### `onPartitionsAssigned(Collection<TopicPartition> partitions)`\n\nCalled DURING poll() when new partitions are assigned.\n\n**Key Points:**\n- First chance to set initial position for new partitions\n- If not called, position defaults to committed offset or auto.offset.reset\n- Blocks poll() until callback returns\n- Called AFTER partitions have been revoked from other members\n\n**Default Behavior (if no callback or callback does nothing):**\n```\nFor each assigned partition:\n  if committed offset exists:\n    seek to committed offset\n  else if auto.offset.reset == \"earliest\":\n    seek to beginning\n  else if auto.offset.reset == \"latest\":\n    seek to end\n  else:\n    next poll() throws NoOffsetForPartitionException\n```\n\n#### Rebalance Callback Exceptions\n\nIf callback throws exception:\n- Rebalance fails\n- Consumer remains unassigned\n- Next poll() attempts rebalance again\n- **Avoid throwing exceptions** from callbacks; log and handle gracefully\n\n### Group Management Methods\n\n#### `groupMetadata()`\n\n```java\npublic ConsumerGroupMetadata groupMetadata();\n```\n\nReturns metadata about this consumer's group membership.\n\n```java\nConsumerGroupMetadata metadata = consumer.groupMetadata();\nSystem.out.println(\"Generation ID: \" + metadata.generationId());\nSystem.out.println(\"Member ID: \" + metadata.memberId());\n```\n\n#### `leave()`\n\n```java\npublic void leave();\n```\n\nProactively leave the consumer group.\n\n**Behavioral Notes:**\n- Revokes all assigned partitions\n- Triggers `onPartitionsRevoked()` callbacks\n- Consumer becomes unassigned\n- Next poll() on same consumer rejoins the group\n- Thread-unsafe; call from consumer thread\n\n```java\n// Later when shutting down\nconsumer.leave();\nconsumer.close();\n```\n\n#### Assignment Strategies\n\nConfigured via `partition.assignment.strategy`:\n\n1. **RangeAssignor** (default)\n   - Assigns contiguous ranges of partitions to consumers\n   - Topic: [0, 1, 2, 3, 4, 5], 2 consumers\n   - Consumer 1: [0, 1, 2], Consumer 2: [3, 4, 5]\n   - Issue: Imbalanced load distribution in multi-topic scenarios\n\n2. **RoundRobinAssignor**\n   - Assigns partitions in round-robin across consumers\n   - Topic: [0, 1, 2, 3, 4, 5], 2 consumers\n   - Consumer 1: [0, 2, 4], Consumer 2: [1, 3, 5]\n   - Better load balancing across topics\n\n3. **StickyAssignor** (Kafka 0.11+)\n   - Minimizes partition movement during rebalance\n   - Assigns new partitions while keeping existing assignments stable\n   - Best for minimizing state transfer on rebalance\n\n4. **CooperativeStickyAssignor** (Kafka 2.4+)\n   - Like StickyAssignor but uses cooperative rebalance protocol\n   - Reduces stop-the-world pause time during rebalance\n\n---\n\n## Metadata and Monitoring\n\n### Topic and Partition Metadata\n\n#### `close()`\n\n```java\npublic void close();\npublic void close(Duration timeout);\n```\n\nClose consumer and release resources.\n\n**Behavioral Notes:**\n- Revokes all assigned partitions\n- Triggers `onPartitionsRevoked()` callbacks\n- Commits offsets if auto-commit enabled\n- Waits for pending operations (heartbeats, commits)\n- Timeout specifies how long to wait before forcefully closing\n- Thread-unsafe except in specific contexts (see wakeup())\n\n```java\n// Graceful shutdown\nconsumer.close();\n\n// With timeout\nconsumer.close(Duration.ofSeconds(5));\n\n// In try-with-resources (Java 7+)\ntry (KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props)) {\n    consumer.subscribe(Arrays.asList(\"my-topic\"));\n\n    while (!Thread.currentThread().isInterrupted()) {\n        ConsumerRecords<String, String> records =\n            consumer.poll(Duration.ofMillis(100));\n        // Process records\n    }\n} catch (Exception e) {\n    logger.error(\"Consumer error\", e);\n}\n// close() called automatically\n```\n\n#### `wakeup()`\n\n```java\npublic void wakeup();\n```\n\nWake up consumer from blocking poll(). Can be called from another thread.\n\n**Behavioral Notes:**\n- ONLY thread-safe method besides close()\n- Causes poll() to throw WakeupException immediately\n- Useful for graceful shutdown from signal handler or scheduler thread\n- Does not close consumer; next poll() can be called\n- Successive calls to wakeup() queue exceptions for multiple poll() calls\n\n```java\n// Main consumer thread\nThread consumerThread = new Thread(() -> {\n    try {\n        while (true) {\n            ConsumerRecords<String, String> records =\n                consumer.poll(Duration.ofSeconds(1));\n            processRecords(records);\n        }\n    } catch (WakeupException e) {\n        logger.info(\"Consumer woken up, shutting down\");\n    } finally {\n        consumer.close();\n    }\n});\n\nconsumerThread.start();\n\n// Signal handler or shutdown hook - runs in different thread\nRuntime.getRuntime().addShutdownHook(new Thread(() -> {\n    logger.info(\"Shutdown signal received\");\n    consumer.wakeup();  // Wake up poll()\n\n    try {\n        consumerThread.join(5000);  // Wait for graceful shutdown\n    } catch (InterruptedException e) {\n        consumerThread.interrupt();\n    }\n}));\n```\n\n---\n\n## Exception Handling\n\n### Exception Hierarchy\n\n```\nConsumerException\n├── RecordDeserializationException\n│   └── SerializationException (key/value deserialization fails)\n├── OffsetOutOfRangeException (committed offset no longer exists)\n├── NoOffsetForPartitionException (no committed offset, auto.offset.reset=none)\n├── CommitFailedException (lost group membership during commit)\n├── IllegalStateException (subscription/assignment errors)\n├── WakeupException (wakeup() called)\n├── InterruptException (thread interrupted)\n├── TimeoutException (operation exceeded deadline)\n├── NotCoordinatorException (broker is not group coordinator)\n├── NotLeaderForPartitionException (broker doesn't lead partition)\n├── TopicAuthorizationException (unauthorized topic access)\n├── GroupAuthorizationException (unauthorized group access)\n├── AuthenticationException (authentication failure)\n└── ... (other networking and broker errors)\n```\n\n### Common Exception Scenarios\n\n#### SerializationException\n\n```\nWhen: Key or value deserialization fails during poll()\nWhy: Custom deserializer threw exception\nAction: Check deserializer logic, validate message format\n```\n\n```java\ntry {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n} catch (SerializationException e) {\n    logger.error(\"Deserialization failed for record: \" + e.getMessage(), e);\n    // May need to skip corrupted messages\n    // Option 1: Seek to next offset and continue\n    consumer.seek(new TopicPartition(\"my-topic\", 0),\n                  consumer.position(new TopicPartition(\"my-topic\", 0)));\n    // Option 2: Stop and alert ops team\n}\n```\n\n#### OffsetOutOfRangeException\n\n```\nWhen: Committed offset no longer exists (retention policy deleted it)\nWhy: Consumer didn't poll for days, retention.ms expired\nAction: Seek to beginning or end, handle gracefully\n```\n\n```java\ntry {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n} catch (OffsetOutOfRangeException e) {\n    logger.warn(\"Offset out of range for partition: \" + e.getMessage());\n\n    // Reset to beginning\n    Map<TopicPartition, Long> beginnings =\n        consumer.beginningOffsets(consumer.assignment());\n    for (TopicPartition tp : consumer.assignment()) {\n        consumer.seek(tp, beginnings.get(tp));\n    }\n\n    logger.info(\"Reset position to beginning offset\");\n}\n```\n\n#### NoOffsetForPartitionException\n\n```\nWhen: No committed offset and auto.offset.reset=none\nWhy: First-time consumer with strict offset reset policy\nAction: Set auto.offset.reset or manually seek before poll()\n```\n\n```java\n// Config problem - fix in configuration\nprops.put(\"auto.offset.reset\", \"earliest\");  // or \"latest\"\n\n// Or handle in code\ntry {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n} catch (NoOffsetForPartitionException e) {\n    logger.info(\"No offset found; seeking to beginning\");\n    consumer.seekToBeginning(consumer.assignment());\n}\n```\n\n#### CommitFailedException\n\n```\nWhen: commitSync() fails due to lost group membership\nWhy: Rebalance occurred between poll() and commit(), or session timeout\nAction: Stop processing, next poll() rejoins group\n```\n\n```java\ntry {\n    consumer.commitSync();\n} catch (CommitFailedException e) {\n    logger.error(\"Commit failed; lost group membership\", e);\n    // Do NOT retry - will fail again\n    // Consumer will automatically rejoin on next poll()\n    // Stop processing current batch to avoid duplicate processing\n    break;  // Exit batch processing loop\n}\n```\n\n#### WakeupException\n\n```\nWhen: poll() called after wakeup()\nWhy: Another thread called wakeup()\nAction: Exit loop, commit offsets if needed, close consumer\n```\n\n```java\ntry {\n    while (true) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n        processRecords(records);\n        consumer.commitSync();\n    }\n} catch (WakeupException e) {\n    logger.info(\"Consumer woken up, shutting down gracefully\");\n} finally {\n    consumer.close();\n}\n```\n\n#### TimeoutException\n\n```\nWhen: Operation doesn't complete within timeout\nWhy: Network latency, broker unavailable, or too many concurrent requests\nAction: Retry with longer timeout or investigate broker health\n```\n\n```java\ntry {\n    consumer.commitSync(Duration.ofSeconds(5));\n} catch (TimeoutException e) {\n    logger.warn(\"Commit timed out; will retry next batch\", e);\n    // Retry at next batch - offset will be re-committed\n}\n```\n\n---\n\n## Configuration-Driven Behaviors\n\nKey configuration properties affecting API behavior:\n\n### Consumer Group and Coordination\n\n| Multiple KafkaConsumer instances with the same group.id form a group that collectively consumes topic partitions.\n\n- **Partition Assignment**: Partitions are automatically distributed among group members\n- **Rebalancing**: When members join/leave, a rebalance redistributes partitions\n- **Coordination**: One group member becomes the \"generation leader\" and computes assignments\n\n### Offsets and Position Management\n\nSeveral offset concepts are critical to understand:\n\n| Concept | Meaning | Method | Notes |\n|---------|---------|--------|-------|\n| **Position** | Next offset to be fetched in current session | `position(TopicPartition)` | Changes when records fetched; multiple polls may return buffered records\n\n4. **Exceptions:**\n   - `WakeupException` - if `wakeup()` was called from another thread\n   - `InterruptException` - if thread was interrupted\n   - `SerializationException` - if key/value deserialization fails\n   - `OffsetOutOfRangeException` - if committed offset no longer exists\n   - `NoOffsetForPartitionException` - if auto.offset.reset=none and no committed offset\n\n**Example:**\n```java\nwhile (true) {\n    try {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n        for (ConsumerRecord<String, String> record : records) {\n            // Process record\n            System.out.println(\"Topic: \" + record.topic() +\n                             \", Partition: \" + record.partition() +\n                             \", Offset: \" + record.offset() +\n                             \", Key: \" + record.key() +\n                             \", Value: \" + record.value());\n        }\n\n        // Explicit offset commit after batch processing\n        consumer.commitSync();\n    } catch (WakeupException e) {\n        // Consumer was woken up, exit gracefully\n        break;\n    }\n}\n```\n\n### Timeout Configuration\n\nThe `timeout` parameter controls how long poll() waits for broker responses:\n\n```java\n// Short timeout - returns quickly if no data available\nconsumer.poll(Duration.ofMillis(100));\n\n// Long timeout - waits longer for records\nconsumer.poll(Duration.ofMillis(1000));\n\n// Zero timeout - return immediately if no buffered records\nconsumer.poll(Duration.ZERO);\n```\n\n**Real-world Tuning:**\n- High-latency networks: Longer timeouts to accommodate network delays\n- Low-latency requirements: Shorter timeouts for responsiveness\n- **Default (300ms)**: Balances latency and throughput\n\n### `pollForRecords(long timeoutMs)` [Deprecated]\n\nLegacy method;  // Commit every 5 seconds\n```\n\nOffsets are committed automatically during poll().\n\n**Behavioral Notes:**\n- Commit occurs approximately every `auto.commit.interval.ms` milliseconds during poll()\n- If poll() is not called, offsets are not committed\n- Commit fails silently (logged but not surfaced to application)\n- **Issue**: Application may crash between poll and background commit\n  - Solution: Use manual commit strategy for critical applications\n\n### Manual Commit - Synchronous\n\n```java\npublic void commitSync();\npublic void commitSync(Duration timeout);\npublic void commitSync(Map<TopicPartition, OffsetAndMetadata> offsets);\npublic void commitSync(Map<TopicPartition, OffsetAndMetadata> offsets, Duration timeout);\n```\n\nExplicitly commit current position (or specified offsets). Blocks until commit completes.\n\n**Method Details:**\n\n#### `commitSync()`\nCommits current position for all assigned partitions.\n\n```java\n// Disable auto-commit\nprops.put(\"enable.auto.commit\", \"false\");\n\nKafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);\nconsumer.subscribe(Arrays.asList(\"my-topic\"));\n\nwhile (true) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n    for (ConsumerRecord<String, String> record : records) {\n        processRecord(record);\n    }\n\n    // Commit all offsets synchronously\n    try {\n        consumer.commitSync();\n    } catch (CommitFailedException e) {\n        // Lost consumer group membership during commit\n        logger.error(\"Commit failed; rebalance likely occurred\", e);\n        // Consumer is no longer in group, next poll() will rejoin\n    }\n}\n```\n\n**Exception Behavior:**\n- `CommitFailedException` - Lost group membership (group fencing or rebalance)\n  - **Action**: Next poll() will trigger rejoin and rebalance\n  - **Do not retry** the commit; it will fail again\n- `OffsetMetadataException` - Offsets topic misconfigured\n- `TimeoutException` - Broker didn't respond within timeout\n\n#### `commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)`\nCommits specific offsets for specific partitions.\n\n```java\n// Commit specific offset for a partition\nMap<TopicPartition, OffsetAndMetadata> offsets = new HashMap<>();\noffsets.put(\n    new TopicPartition(\"my-topic\", 0),\n    new OffsetAndMetadata(42, \"processed up to offset 42\")\n);\nconsumer.commitSync(offsets);\n```\n\n**Use Cases:**\n- Processing some partitions faster than others\n- Batching commits across multiple batches\n- Manual offset tracking in custom stores\n\n#### `commitSync(Duration timeout)`\nCommits with explicit timeout.\n\n```java\n// Commit with 5-second timeout\ntry {\n    consumer.commitSync(Duration.ofSeconds(5));\n} catch (TimeoutException e) {\n    logger.warn(\"Commit timed out; retrying next batch\", e);\n}\n```\n\n### Manual Commit - Asynchronous\n\n```java\npublic void commitAsync();\npublic void commitAsync(OffsetCommitCallback callback);\npublic void commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets);\npublic void commitAsync(Map<TopicPartition, OffsetAndMetadata> offsets,\n                       OffsetCommitCallback callback);\n```\n\nNon-blocking commit. Management\n\n### Understanding Rebalancing\n\nA **rebalance** redistributes partition ownership among group members. Triggered by:\n1. New member joins (via subscribe() on first poll())\n2. Member leaves (close() or timeout)\n3. Topic subscribed via pattern gains new matching topics\n4. Partition count changes for subscribed topics\n\n### Rebalance Lifecycle\n\n```\nPoll detects rebalance needed\n    ↓\nMember joins group with current assignment\n    ↓\nGroup enters \"Rebalancing\" state\n    ↓\nCoordinator assigns new leader\n    ↓\nLeader computes new assignment (based on configured strategy)\n    ↓\nAll members receive new assignment\n    ↓\nonPartitionsRevoked(old_partitions) executed\n    ↓\nMember revokes partitions, sends commit offsets\n    ↓\nonPartitionsAssigned(new_partitions) executed\n    ↓\nMember can seek to custom positions\n    ↓\nGroup is \"Stable\", polling resumes\n```\n\n### ConsumerRebalanceListener Callbacks\n\n#### `onPartitionsRevoked(Collection<TopicPartition> partitions)`\n\nCalled DURING poll() when partitions are being revoked.\n\n**Key Points:**\n- **This is your last chance to commit offsets** for revoked partitions\n- Called with partitions being removed from this consumer\n- Blocks poll() until callback returns\n- Exceptions thrown here cause rebalance failure\n\n```java\nconsumer.subscribe(Arrays.asList(\"my-topic\"),\n    new ConsumerRebalanceListener() {\n        @Override\n        public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n            System.out.println(\"Revoking: \" + partitions);\n\n            // CRITICAL: Commit offsets before losing partitions\n            try {\n                consumer.commitSync();\n                System.out.println(\"Offsets committed for revoked partitions\");\n            } catch (Exception e) {\n                logger.error(\"Failed to commit during rebalance\", e);\n                // Don't throw - let rebalance continue\n                // But log it so you know you may have lost progress\n            }\n        }\n\n        @Override\n        public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n            System.out.println(\"Assigned: \" + partitions);\n\n            // Optionally seek to custom positions\n            for (TopicPartition tp : partitions) {\n                // Seek to beginning of newly assigned partitions\n                consumer.seekToBeginning(Arrays.asList(tp));\n            }\n        }\n    }\n);\n```\n\n#### `onPartitionsAssigned(Collection<TopicPartition> partitions)`\n\nCalled DURING poll() when new partitions are assigned.\n\n**Key Points:**\n- First chance to set initial position for new partitions\n- If not called, position defaults to committed offset or auto.offset.reset\n- Blocks poll() until callback returns\n- Called AFTER partitions have been revoked from other members\n\n**Default Behavior (if no callback or callback does nothing):**\n```\nFor each assigned partition:\n  if committed offset exists:\n    seek to committed offset\n  else if auto.offset.reset == \"earliest\":\n    seek to beginning\n  else if auto.offset.reset == \"latest\":\n    seek to end\n  else:\n    next poll() throws NoOffsetForPartitionException\n```\n\n#### Rebalance Callback Exceptions\n\nIf callback throws exception:\n- Rebalance fails\n- Consumer remains unassigned\n- Next poll() attempts rebalance again\n- **Avoid throwing exceptions** from callbacks; log and handle gracefully\n\n### Group Management Methods\n\n#### `groupMetadata()`\n\n```java\npublic ConsumerGroupMetadata groupMetadata();\n```\n\nReturns metadata about this consumer's group membership.\n\n```java\nConsumerGroupMetadata metadata = consumer.groupMetadata();\nSystem.out.println(\"Generation ID: \" + metadata.generationId());\nSystem.out.println(\"Member ID: \" + metadata.memberId());\n```\n\n#### `leave()`\n\n```java\npublic void leave();\n```\n\nProactively leave the consumer group.\n\n**Behavioral Notes:**\n- Revokes all assigned partitions\n- Triggers `onPartitionsRevoked()` callbacks\n- Consumer becomes unassigned\n- Next poll() on same consumer rejoins the group\n- Thread-unsafe; call from consumer thread\n\n```java\n// Later when shutting down\nconsumer.leave();\nconsumer.close();\n```\n\n#### Assignment Strategies\n\nConfigured via `partition.assignment.strategy`:\n\n1. **RangeAssignor** (default)\n   - Assigns contiguous ranges of partitions to consumers\n   - Topic: [0, 1, 2, 3, 4, 5], 2 consumers\n   - Consumer 1: [0, 1, 2], Consumer 2: [3, 4, 5]\n   - Issue: Imbalanced load distribution in multi-topic scenarios\n\n2. **RoundRobinAssignor**\n   - Assigns partitions in round-robin across consumers\n   - Topic: [0, 1, 2, 3, 4, 5], 2 consumers\n   - Consumer 1: [0, 2, 4], Consumer 2: [1, 3, 5]\n   - Better load balancing across topics\n\n3. **StickyAssignor** (Kafka 0.11+)\n   - Minimizes partition movement during rebalance\n   - Assigns new partitions while keeping existing assignments stable\n   - Best for minimizing state transfer on rebalance\n\n4. **CooperativeStickyAssignor** (Kafka 2.4+)\n   - Like StickyAssignor but uses cooperative rebalance protocol\n   - Reduces stop-the-world pause time during rebalance\n\n---\n\n## Metadata and Monitoring\n\n### Topic and Partition Metadata\n\n#### OffsetOutOfRangeException\n\n```\nWhen: Committed offset no longer exists (retention policy deleted it)\nWhy: Consumer didn't poll for days, retention.ms expired\nAction: Seek to beginning or end, to timeout for broker response\n2. **During rebalance**: Blocks indefinitely in callbacks (timeout ignored)\n3. **Metadata update**: May block fetching metadata from brokers\n4. **Network latency**: Added on top of timeout\n\n**Key Implication**:\n```\nTimeout parameter is NOT a hard guarantee.\npoll() can block longer than timeout if rebalance occurs.\n\nExample:\npoll(Duration.ofMillis(100))  // Timeout: 100ms\n← rebalance triggered during poll\n← onPartitionsRevoked() called (blocks indefinitely)\n← onPartitionsAssigned() called (blocks indefinitely)\n← poll() can block for many seconds\n```\n\n**Recommendation:**\n```\nUse reasonable timeouts (100-1000ms) but don't rely on exact timing.\nIf you need precise timing, implement a deadline check outside poll():\n\nlong deadline = System.currentTimeMillis() + 60000;\nwhile (System.currentTimeMillis() < deadline) {\n    ConsumerRecords<String, String> records =\n        consumer.poll(Duration.ofMillis(100));\n    // Process...\n    if (shouldShutdown) break;\n}\n```\n\n### Offset Commit Semantics\n\n#### Synchronous Commit (`commitSync()`)\n\n```\nApplication calls commitSync()\n    ↓\nConsumer builds commit request with current position for all partitions\n    ↓\nSends to group coordinator\n    ↓\nCoordinator stores offsets in __consumer_offsets topic\n    ↓\nReturns success or CommitFailedException\n    ↓\nApplication resumes (offset is now durable)\n```\n\n**Exception During Commit:**\n- Offset NOT committed\n- Group membership may be lost\n- Next poll() attempts rejoin\n- **Do not retry the same commit** - wait for rejoin\n\n#### Asynchronous Commit (`commitAsync()`)\n\n```\nApplication calls commitAsync(callback)\n    ↓\nConsumer queues commit request\n    ↓\ncommitAsync() returns immediately\n    ↓\nDuring next poll(): commit request sent\n    ↓\nCoordinator processes and stores offsets\n    ↓\nDuring later poll(): callback invoked with success/failure\n    ↓\nApplication handles response in callback\n```\n\n**Key Difference**:\n- Application doesn't block\n- Callback may be invoked many polls later\n- Multiple outstanding commits can cause ordering issues\n\n### Group Membership and Rebalancing\n\nWhen does a consumer stay in the group?\n\n```\nConsumer stays in group IF:\n1. poll() is called frequently enough (at least every max.poll.interval.ms)\n   AND\n2. Heartbeats are sent (automatically during poll())\n```\n\nIf consumer stops calling poll() for max.poll.interval.ms:\n```\nCoordinator removes consumer from group\n    ↓\nRebalance triggered for remaining consumers\n    ↓\nEvicted consumer's partitions reassigned\n    ↓\nWhen evicted consumer calls poll() again:\n    ↓\nConsumer rejoins group (potentially with different partitions)\n    ↓\nAnother rebalance occurs\n```\n\n### Static Group Membership\n\nEnable with `group.instance.id` to preserve partition assignment across restarts:\n\n```\nConsumer 1 (instance-1) - assigned [partition-0, partition-1]\nConsumer 1 restarts\n    ↓\nConsumer rejoins with same instance-1 ID\n    ↓\nPartitions are reassigned to same instance without rebalance\n```\n\nUse cases:\n- Minimizing rebalance time during rolling restarts\n- Preserving local caches per instance\n\n### Message Filtering with read_committed\n\nWith `isolation.level=read_committed`:\n\n```\nProducer writes messages in transaction:\n- Offset 0: msg1\n- Offset 1: msg2 (uncommitted)\n- Offset 2: msg3 (uncommitted)\n\nConsumer sees Last Stable Offset (LSO) = 0 (before uncommitted messages)\n\nConsumer poll() only returns msg at offset 0\nNext poll() may return more if transaction commits\n\nThis prevents consuming partially committed transactions\n```\n\n---\n\n## Troubleshooting Guide\n\n### Consumer Not Receiving Messages\n\n**Symptom**: Consumer calls poll() but receives no records\n\n**Potential Causes**:\n\n1. **Partition assignment incomplete**\n   ```java\n   // Check assignment\n   System.out.println(\"Assigned: \" + consumer.assignment());\n\n   // Solution: Ensure subscription/assign and initial poll completed\n   consumer.subscribe(Arrays.asList(\"my-topic\"));\n   consumer.poll(Duration.ZERO);
