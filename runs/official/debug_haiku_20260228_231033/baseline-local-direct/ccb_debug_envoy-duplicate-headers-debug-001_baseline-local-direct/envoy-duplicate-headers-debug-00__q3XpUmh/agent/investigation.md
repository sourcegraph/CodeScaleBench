# Investigation Report: Duplicate Response Headers in Envoy Filter Pipeline

## Summary

Response headers configured via `response_headers_to_add` in route configuration are duplicated in local replies generated by the router filter. This occurs because `finalizeResponseHeaders()` is called twice in the local reply code path—once explicitly in the filter manager and again implicitly when the `modify_headers_` callback (which itself contains a `finalizeResponseHeaders()` call) is invoked.

## Root Cause

The root cause is the **double invocation of `finalizeResponseHeaders()`** in the local reply handling path. This happens because:

1. **In router.cc (lines 444-461)**: The router filter creates a `modify_headers_` callback that includes an explicit call to `finalizeResponseHeaders()`:
   ```cpp
   modify_headers_ = [this](Http::ResponseHeaderMap& headers) {
     // ... other code ...
     route_entry_->finalizeResponseHeaders(headers, callbacks_->streamInfo());
   };
   ```

2. **In filter_manager.cc (lines 1123-1129, 1072-1078, 1156-1162)**: When constructing the local reply, the filter manager creates a lambda that calls `finalizeResponseHeaders()` directly AND then calls the `modify_headers_` callback:
   ```cpp
   [this, modify_headers](ResponseHeaderMap& headers) -> void {
     if (streamInfo().route() && streamInfo().route()->routeEntry()) {
       streamInfo().route()->routeEntry()->finalizeResponseHeaders(headers, streamInfo());  // FIRST
     }
     if (modify_headers) {
       modify_headers(headers);  // SECOND (contains finalizeResponseHeaders internally)
     }
   }
   ```

3. **In utility.cc (line 719)**: This lambda is invoked during the local reply preparation, causing both calls to `finalizeResponseHeaders()` to execute on the same response headers.

### Header Parser Behavior

**header_parser.cc:182-200** - Shows how headers are added based on `append_action`:
```cpp
switch (entry->append_action_) {
  PANIC_ON_PROTO_ENUM_SENTINEL_VALUES;
case HeaderValueOption::APPEND_IF_EXISTS_OR_ADD:
  headers_to_add.emplace_back(key, value);
  break;
case HeaderValueOption::ADD_IF_ABSENT:
  if (auto header_entry = headers.get(key); header_entry.empty()) {
    headers_to_add.emplace_back(key, value);
  }
  break;
case HeaderValueOption::OVERWRITE_IF_EXISTS:
  if (headers.get(key).empty()) {
    break;
  }
  FALLTHRU;
case HeaderValueOption::OVERWRITE_IF_EXISTS_OR_ADD:
  headers_to_overwrite.emplace_back(key, value);
  break;
}
```

When `finalizeResponseHeaders()` is called twice with `APPEND_IF_EXISTS_OR_ADD`:
- First call: Header is added
- Second call: Header is appended again (since it now exists), creating a duplicate

With `OVERWRITE_IF_EXISTS_OR_ADD`, the header is overwritten in both calls, but this still represents inefficient double-processing.

## Causal Chain

1. **Symptom**: Response headers configured via `response_headers_to_add` appear twice in HTTP responses for local replies (e.g., `x-custom-trace: abc123\r\nx-custom-trace: abc123`)

2. **Access Log Evidence**: The `%RESPONSE_CODE_DETAILS%` formatter shows responses like `upstream_response_timeout` with duplicate headers present in the actual HTTP response

3. **Router Filter Initialization** (router.cc:444-461):
   - Router filter creates `modify_headers_` callback during `decodeHeaders()`
   - This callback includes a call to `route_entry_->finalizeResponseHeaders()` on line 453
   - Designed to apply all route-level response header transformations

4. **Local Error Detection** (router.cc:520-525, etc.):
   - Router filter detects a local error condition (e.g., cluster not found)
   - Calls `callbacks_->sendLocalReply(code, "", modify_headers_, ...)`
   - Passes the `modify_headers_` callback to filter manager

5. **Filter Manager Local Reply Path** (filter_manager.cc:1109-1140):
   - `DownstreamFilterManager::sendLocalReplyViaFilterChain()` receives the `modify_headers_` callback
   - Creates a new lambda that:
     - Calls `finalizeResponseHeaders()` directly (line 1125)
     - Calls the passed `modify_headers_` callback (line 1128)

6. **Lack of Awareness of Redundancy** (filter_manager.cc:1072-1078, 1156-1162):
   - Filter manager code pattern repeated identically in three methods: `sendLocalReplyViaFilterChain()`, `prepareLocalReplyViaFilterChain()`, `sendDirectLocalReply()`
   - Code does not account for the fact that `modify_headers_` already contains a `finalizeResponseHeaders()` call
   - No guard to prevent double-finalization

7. **Lambda Invocation** (utility.cc:719):
   - `Utility::prepareLocalReply()` invokes the filter manager's lambda on response headers
   - Both `finalizeResponseHeaders()` calls execute:
     - First explicit call modifies headers
     - Second implicit call (via `modify_headers_`) modifies headers again

8. **Header Addition Mechanism** (header_parser.cc:182-212):
   - For each configured header to add, `evaluateHeaders()` processes it according to its `append_action`
   - First call: Header is added to response
   - Second call: Header is processed again:
     - If `APPEND_IF_EXISTS_OR_ADD`: Header is APPENDED (creating duplicate)
     - If `OVERWRITE_IF_EXISTS_OR_ADD`: Header is OVERWRITTEN (inefficient but no visible duplication if value is identical)
     - If `ADD_IF_ABSENT`: Header is NOT added (no duplication)
     - If `OVERWRITE_IF_EXISTS`: Header is OVERWRITTEN (inefficient but no visible duplication)

9. **Root Cause**: The filter manager's local reply construction unconditionally calls `finalizeResponseHeaders()` AND then calls `modify_headers_`, without recognizing that the router filter has already embedded `finalizeResponseHeaders()` within the `modify_headers_` callback

10. **Contrast with Upstream Path** (router.cc:1794):
    - For upstream responses, router calls `modify_headers_(*headers)` directly
    - Upstream connection code does NOT wrap this with another `finalizeResponseHeaders()` call
    - Result: Upstream responses receive headers exactly once

## Recommendation

### Root Cause Fix Strategy

The fix should eliminate the double `finalizeResponseHeaders()` invocation. Two approaches:

**Option A: Remove duplicate call in filter_manager.cc**
- Modify the lambda creation in `sendLocalReplyViaFilterChain()`, `prepareLocalReplyViaFilterChain()`, and `sendDirectLocalReply()`
- Remove the direct `finalizeResponseHeaders()` call since `modify_headers_` already contains it
- Relies on router filter always passing a `modify_headers_` callback that includes finalization
- Risk: Other callers of `sendLocalReply()` that don't pass `modify_headers_` would skip finalization

**Option B: Remove finalization from router.cc's modify_headers_**
- Move the `finalizeResponseHeaders()` call out of the `modify_headers_` callback in router.cc
- Call `finalizeResponseHeaders()` explicitly for upstream responses (after line 1794)
- Allow filter manager to handle finalization for all cases
- More centralized, but requires changes in router filter's upstream path handling

**Option C: Add guard in filter_manager.cc**
- Check if `modify_headers_` already performs finalization
- Only call `finalizeResponseHeaders()` if needed
- Most defensive but harder to implement and maintain

### Diagnostic Steps

1. Enable debug logging in router.cc to track when `modify_headers_` is created and called
2. Add instrumentation in `evaluateHeaders()` (header_parser.cc) to log each call to `finalizeResponseHeaders()`
3. Check response headers in access logs before and after each finalization step
4. Verify that upstream responses (non-local replies) only show headers once

### Testing Requirements

- Add integration tests for local replies (timeout, cluster not found, overloaded) with `response_headers_to_add` configured
- Verify headers appear exactly once with all `append_action` values: APPEND_IF_EXISTS_OR_ADD, ADD_IF_ABSENT, OVERWRITE_IF_EXISTS_OR_ADD, OVERWRITE_IF_EXISTS
- Verify upstream responses still work correctly after fix
- Verify gRPC local replies don't break
- Verify direct local replies (when response headers already started) don't duplicate headers
