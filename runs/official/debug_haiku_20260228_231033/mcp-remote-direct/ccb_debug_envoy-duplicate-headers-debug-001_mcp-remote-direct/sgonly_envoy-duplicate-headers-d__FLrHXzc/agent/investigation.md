# Investigation Report: Duplicate Response Headers in Envoy Filter Pipeline

## Summary

Response headers configured via route-level `response_headers_to_add` are being duplicated in local replies generated by the router filter because `finalizeResponseHeaders()` is invoked twice through different call paths: once during local reply preparation in the filter manager's header mutation lambda, and again through the router's callback mechanism or filter chain processing. The filter manager creates a lambda that applies `finalizeResponseHeaders()`, but the router's callback (particularly for direct responses and error replies) may also trigger header finalization, resulting in duplicate header additions when using the default `APPEND_IF_EXISTS_OR_ADD` behavior for `HeaderValueOption`.

## Root Cause

**Files and Functions:**
- **source/common/http/filter_manager.cc**: `DownstreamFilterManager::sendLocalReplyViaFilterChain()` (lines 1109-1141) and `DownstreamFilterManager::prepareLocalReplyViaFilterChain()` (lines 1055-1094)
- **source/common/router/router.cc**: `Filter::decodeHeaders()` (lines 440-530) and `Filter::onUpstreamAbort()` (lines 1387-1413)
- **source/common/router/header_parser.cc**: `HeaderParser::evaluateHeaders()` (lines 139-213)
- **api/envoy/config/core/v3/base.proto**: `HeaderValueOption` message definition (lines 428-481)

**The Mechanism:**

The filter manager wraps the router's `modify_headers_` callback in an additional lambda during local reply processing:

```cpp
// source/common/http/filter_manager.cc:1123-1129
[this, modify_headers](ResponseHeaderMap& headers) -> void {
  if (streamInfo().route() && streamInfo().route()->routeEntry()) {
    streamInfo().route()->routeEntry()->finalizeResponseHeaders(headers, streamInfo());  // FINALIZE CALL #1
  }
  if (modify_headers) {
    modify_headers(headers);  // Router's callback - potentially contains another finalization
  }
}
```

This wrapper lambda is passed as `EncodeFunctions.modify_headers_` to `Utility::prepareLocalReply()` (line 1070-1093), where it's invoked at lines 718-720 of `source/common/http/utility.cc`:

```cpp
// source/common/http/utility.cc:718-720
if (encode_functions.modify_headers_) {
  encode_functions.modify_headers_(*response_headers);
}
```

For direct responses, the router's callback (lines 488-502 of router.cc) explicitly calls:

```cpp
direct_response->finalizeResponseHeaders(response_headers, callbacks_->streamInfo());  // FINALIZE CALL #2
```

This creates a scenario where:
1. **First finalization**: The filter manager's lambda calls `route_entry_->finalizeResponseHeaders()`, applying route-level `response_headers_to_add`
2. **Second finalization**: The router's callback then calls `direct_response->finalizeResponseHeaders()` (for direct responses) or `modify_headers_()` which could trigger additional processing
3. **Result**: Response headers are added twice through overlapping or sequential evaluation paths


## Causal Chain

1. **Symptom**: Response headers appear twice in HTTP response (e.g., `x-custom-trace: abc123` appears twice)
2. **Immediate Cause**: Router generates local reply (upstream timeout, connection failure, etc.)
3. **Root Mechanism**:
   - Filter manager's `sendLocalReplyViaFilterChain()` creates a lambda that wraps the router's callback
   - This lambda calls `route_entry_->finalizeResponseHeaders()` which evaluates route-level `response_headers_to_add`
   - The router's callback may also trigger `finalizeResponseHeaders()` (for direct responses) or rely on default proto values
4. **Header Processing**:
   - `HeaderParser::evaluateHeaders()` processes headers based on `append_action_`
   - For APPEND_IF_EXISTS_OR_ADD (default), headers are added via `addReferenceKey()` which creates duplicates
   - For OVERWRITE_IF_EXISTS_OR_ADD, headers should use `setReferenceKey()` but if called twice, both operations succeed
5. **Filter Chain Encoding**:
   - Prepared local reply headers are passed to `FilterManager::encodeHeaders()`
   - Encoder filters run in reverse order and potentially re-process headers
   - Filter manager's `encodeHeaders()` does NOT call `finalizeResponseHeaders()` again (lines 1244-1330)
6. **Access Log Observation**: Header appears twice with identical values, visible in `%RESPONSE_HEADERS%` logging

## Recommendation

### Fix Strategy

The root cause is the double wrapping of `finalizeResponseHeaders()` calls during local reply processing:

**Option 1: Deduplicate in Filter Manager** (Recommended)
- Modify `DownstreamFilterManager::sendLocalReplyViaFilterChain()` to NOT wrap the `modify_headers` callback with an additional `finalizeResponseHeaders()` call
- Instead, let the router's callback handle header finalization directly
- Update code at filter_manager.cc:1123-1129 and filter_manager.cc:1072-1078 to conditionally apply `finalizeResponseHeaders()` only if the `modify_headers` callback doesn't already handle it

**Option 2: Track Finalization State**
- Add a flag to indicate that `finalizeResponseHeaders()` has been called
- Check this flag before calling it again to prevent duplicate processing

**Option 3: Consolidate Header Mutations**
- Move all header finalization to occur exactly once in `Utility::prepareLocalReply()` or `Utility::sendLocalReply()`
- Ensure the router and filter manager don't both invoke finalization

### Diagnostic Steps

1. **Verify the issue**: Enable debug logging in `HeaderParser::evaluateHeaders()` to see when headers are added
2. **Trace the call stack**: Add instrumentation at `finalizeResponseHeaders()` calls to count invocations per request
3. **Check append_action values**: Log the actual `append_action_` value being used for each header
4. **Monitor for default values**: Search for cases where `append_action` is not explicitly set (relying on proto default)
5. **Test with explicit OVERWRITE_IF_EXISTS_OR_ADD**: Verify that headers are still duplicated even with overwrite action

## Key Insights

1. **Proto Default Values Matter**: The default `APPEND_IF_EXISTS_OR_ADD` for `append_action` means headers append on default, but the code expects explicit values
2. **Callback Wrapping**: The filter manager wraps the router's callback in another lambda, creating potential for double-processing
3. **Local Reply Path**: Local replies take a different code path than upstream responses, specifically through `sendLocalReplyViaFilterChain()` which adds the extra wrapping
4. **Filter Chain Iteration**: Encoder filters run in reverse order, but `encodeHeaders()` itself doesn't re-apply route-level headers (good)
5. **Deprecated Fields**: The deprecated `append` BoolValue field complicates proto parsing logic and can cause unexpected behavior
