after actual API operations\n- `Private`: Opaque binary data for next operation\n- `Diagnostics`: Errors or warnings from API calls\n\n**Read Operations:**\n\nThe `ReadResource` RPC is called:\n- During `apply` after resource creation/update to verify actual state\n- During `refresh` to synchronize state with infrastructure\n- With current state and provider metadata\n\n### Error Handling in Provider Operations\n\n**Categorized Errors:**\n\n1. **Validation Errors**: Schema violations, missing required attributes\n2. **Permission Errors**: Authentication/authorization failures\n3. **API Errors**: Infrastructure API rejections (quota exceeded, invalid combination)\n4. **Transient Errors**: Network timeouts, temporary unavailability\n5. **Fatal Errors**: Unrecoverable provider issues (plugin crash, protocol error)\n\n**Error Propagation:**\n- Provider diagnostics include severity (error, warning)\n- Errors during plan cause plan abortion with diagnostic output\n- Errors during apply cause apply abortion with partial state rollback (if possible)\n\n### Provider Configuration Dependencies\n\n**Graph Integration:**\n- Each resource or data source creates implicit dependency on its provider node\n- Provider node depends on variable nodes it references in configuration\n- Provider initialization is sequenced before any resource operations\n\n---\n\n## State Management\n\n### Purpose and Responsibilities\n\nState Management is the system for:\n- Persisting resource metadata to enable incremental updates\n- Tracking resource identity across runs\n- Detecting infrastructure drift\n- Enabling safe concurrent and remote execution\n\n### State Structure\n\n**State File Format (JSON):**\n\n```json\n{\n  \"version\": 4,\n  \"terraform_version\": \"1.5.0\",\n  \"serial\": 42,\n  \"lineage\": \"unique-id\",\n  \"outputs\": {\n    \"instance_id\": {\n      \"value\": \"i-1234567890abcdef0\",\n      \"type\": \"string\"\n    }\n  },\n  \"resources\": [\n    {\n      \"type\": \"aws_instance\",\n      \"name\": \"example\",\n      \"instances\": [\n        {\n          \"index_key\": 0,\n          \"schema_version\": 1,\n          \"attributes\": {\n            \"id\": \"i-1234567890abcdef0\",\n            \"ami\": \"ami-0c55b159cbfafe1f0\",\n            \"instance_type\": \"t2.micro\",\n            \"tags\": {\n              \"Name\": \"HelloWorld\"\n            }\n          },\n          \"sensitive_attributes\": [],\n          \"private\": \"...\"\n        }\n      ]\n    }\n  ]\n}\n```\n\n**Key Components:**\n- `version`: State file schema version (supports backward compatibility)\n- `serial`: Monotonically increasing counter for change detection\n- `lineage`: Unique identifier tracking state identity across migrations\n- `resources`: Array of resource instances with full state\n- `outputs`: Root module output values\n\n### State Managers\n\n**Manager Responsibilities:**\n- Load state from backend (local file, remote storage, secrets manager)\n- Coordinate state mutations during graph evaluation\n- Persist modified state back to backend\n- Manage state locking for concurrent execution prevention\n- Handle state encryption and secrets\n\n**State Manager Hierarchy:**\n\n```\nBackend Interface (abstract storage)\nв”ңв”Җв”Җ Local Backend\nв”Ӯ   в””в”Җв”Җ File-based state with optional locking\nв”ңв”Җв”Җ Remote Backends\nв”Ӯ   в”ңв”Җв”Җ Terraform Cloud / Terraform Enterprise\nв”Ӯ   в”ңв”Җв”Җ AWS S3 + DynamoDB locking\nв”Ӯ   в”ңв”Җв”Җ Azure Storage Blob + Lease locking\nв”Ӯ   в”ңв”Җв”Җ Google Cloud Storage + Bucket Lock\nв”Ӯ   в”ңв”Җв”Җ Consul\nв”Ӯ   в””в”Җв”Җ etcd\nв””в”Җв”Җ Special Backends\n    в”ңв”Җв”Җ Terraform Testing Backend (in-memory)\n    в””в”Җв”Җ Multi-State Backend (for workspaces)\n```\n\n**Backend Operations:**\n- `StateFn()`: Load current state\n- `DeleteState()`: Remove state (dangerous)\n- `Lock(info)`: Acquire exclusive lock\n- `Unlock(id)`: Release lock\n- Operations fail if lock cannot be acquired (prevents concurrent modifications)\n\n### State Synchronization and Snapshots\n\n**Immutable Snapshot Pattern:**\n\nDuring graph evaluation, state operates in \"frozen snapshot\" mode:\n\n```\n1. Load Initial State вҶ’ Snapshot (read-only reference)\n2. Plan Graph Evaluation\n   в”ңв”Җв”Җ Node reads from snapshot\n   в”ңв”Җв”Җ Node calculates changes\n   в”ңв”Җв”Җ Node records new state (not yet persisted)\n   в””в”Җв”Җ Continue with next node\n3. Collect All Planned State Changes\n4. Persist Combined New State\n```\n\n**Benefits:**\n- Graph nodes don't see partial intermediate states from other evaluations\n- Enables safe concurrent node evaluation (different nodes can run in parallel)\n- Provides consistent view of infrastructure across the entire graph\n- If evaluation fails, original state remains unchanged\n\n**State Refresh Pattern (before planning):**\n\n```\n1. For each resource in current state\n   в”ңв”Җв”Җ Call ReadResource() via provider\n   в”ңв”Җв”Җ Detect drift (configuration vs actual)\n   в””в”Җв”Җ Update state with actual values\n2. Output refresh summary\n3. Proceed with planning based on refreshed state\n```\n\n### State Locking\n\n**Pessimistic Locking:**\n- Acquired immediately when state operation begins\n- Held for entire duration of plan/apply/refresh operation\n- Released only after operation completes\n- Prevents concurrent modifications in shared state backends\n\n**Lock Metadata:**\n```json\n{\n  \"ID\": \"unique-uuid\",\n  \"Path\": \"path/in/backend\",\n  \"Operation\": \"OperationTypePlan\",\n  \"Info\": \"user@hostname:pid\",\n  \"Who\": \"user@hostname\",\n  \"Version\": \"1.5.0\",\n  \"Created\": \"2024-02-28T10:30:00Z\"\n}\n```\n\n**Lock Timeout and Conflict Handling:**\n- Lock acquisition has configurable timeout\n- If lock exists but is stale (from crashed process), force unlock with warning\n- Prevents deadlock from interrupted operations\n\n### Remote State Backends\n\n**Terraform Cloud / Terraform Enterprise:**\n- Manages all state through authenticated REST API\n- Provides versioning, state history, and run tracking\n- Automatically handles locking and encryption\n- Integrates with VCS for automation\n\n**Self-Hosted Remote Backends (S3, GCS, Azure):**\n- State stored in managed object storage (durability)\n- Locking via distributed coordination service\n  - S3: Uses DynamoDB table for lease-based locking\n  - GCS: Uses metadata-based optimistic locking\n  - Azure: Uses blob lease mechanism\n- Encryption at rest and in transit\n\n**State Access Flow (Remote Backend):**\n\n```\nTerraform CLI\n    вҶ“\nBackend Interface\n    вҶ“\nHTTP API / Object Storage API\n    вҶ“\n(Optional) Locking Service\n    вҶ“\nTerraform Cloud / S3 / GCS / Azure\n```\n\n### State Drift Detection\n\n**Drift Detection Process:**\n\n```\n1. Load State (prior knowledge)\n2. Execute ReadResource() for each resource\n3. Compare actual attributes with state attributes\n4. Categorize changes:\n   в”ңв”Җв”Җ No Drift: Actual matches state\n   в”ңв”Җв”Җ In-Band Drift: State needs updating, no HCL change\n   в””в”Җв”Җ Out-of-Band Drift: Actual diverges from config (indicates external modification)\n5. Report drift summary\n```\n\n**Handling Out-of-Band Drift:**\n- Detected when actual attribute differs from configuration\n- Plan shows update from actual вҶ’ config (replaying configuration intent)\n- User can review drift before apply\n- Option to overwrite actual with config or preserve actual (refresh-only mode)\n\n### State Secrets and Encryption\n\n**Sensitive Attribute Handling:**\n- Marked in provider schema as `Sensitive: true`\n- Recorded in state with `sensitive_attributes` array\n- Redacted from console output and logs\n- Encrypted if backend supports encryption\n\n**Backend Encryption:**\n- Terraform Cloud: Automatic AES-256-GCM encryption\n- Remote backends (S3): Server-side encryption (user-managed keys or AWS-managed)\n- Local state: Unencrypted by default (can use `aws_kms_encrypt` functions for sensitive blocks)\n\n---\n\n## Execution Flow and Hook System\n\n### Overall Execution Flow\n\n#### Complete Plan/Apply Workflow\n\n```\nterraform plan/apply invocation\n    вҶ“\n1. INITIALIZATION PHASE\n    в”ңв”Җв”Җ Load and parse HCL configuration\n    в”ңв”Җв”Җ Load backend configuration\n    в”ңв”Җв”Җ Initialize working directory\n    в”ңв”Җв”Җ Acquire state lock\n    в””в”Җв”Җ Load current state from backend\n    вҶ“\n2. CONFIGURATION VALIDATION PHASE\n    в”ңв”Җв”Җ Validate module structure\n    в”ңв”Җв”Җ Validate resource references\n    в”ңв”Җв”Җ Validate variable types and defaults\n    в””в”Җв”Җ Call provider ValidateResourceConfig() for each resource\n    вҶ“\n3. GRAPH CONSTRUCTION PHASE (Plan)\n    в”ңв”Җв”Җ Build initial resource/provider/variable vertices\n    в”ңв”Җв”Җ Apply graph transformers\n    в”Ӯ   в”ңв”Җв”Җ Expand count/for_each\n    в”Ӯ   в”ңв”Җв”Җ Expand modules\n    в”Ӯ   в”ңв”Җв”Җ Resolve dynamic blocks\n    в”Ӯ   в””в”Җв”Җ Add dependency edges\n    в””в”Җв”Җ Validate acyclic graph\n    вҶ“\n4. STATE REFRESH PHASE (optional, enabled by default)\n    в”ңв”Җв”Җ For each resource in current state\n    в”Ӯ   в”ңв”Җв”Җ Call provider ReadResource()\n    в”Ӯ   в”ңв”Җв”Җ Detect and report drift\n    в”Ӯ   в””в”Җв”Җ Update state snapshot\n    в””в”Җв”Җ Output refresh summary\n    вҶ“\n5. PLAN GENERATION PHASE\n    в”ңв”Җв”Җ Create plan context (immutable state snapshot)\n    в”ңв”Җв”Җ Initialize Walker with concurrency limit\n    в”ңв”Җв”Җ Walk graph in dependency order\n    в”Ӯ   в”ңв”Җв”Җ For each resource node\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider PlanResourceChange()\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Determine action (create/update/delete/no-op)\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Record plan changes\n    в”Ӯ   в”Ӯ   в””в”Җв”Җ Emit hook callbacks\n    в”Ӯ   в””в”Җв”Җ Continue until all nodes evaluated\n    в”ңв”Җв”Җ Collect all planned changes\n    в””в”Җв”Җ Serialize plan to file\n    вҶ“\n6. APPROVAL PHASE (interactive mode)\n    в”ңв”Җв”Җ Display plan summary\n    в”ңв”Җв”Җ Prompt user for approval\n    в””в”Җв”Җ Abort if user declines (Plan only)\n    вҶ“\n7. GRAPH CONSTRUCTION PHASE (Apply)\n    в”ңв”Җв”Җ Rebuild graph with apply-specific transformers\n    в”ңв”Җв”Җ Filter to affected resources (from plan)\n    в””в”Җв”Җ Validate acyclic graph\n    вҶ“\n8. APPLY EXECUTION PHASE\n    в”ңв”Җв”Җ Create apply context (plan reference)\n    в”ңв”Җв”Җ Initialize Walker with concurrency limit\n    в”ңв”Җв”Җ Walk graph in dependency order\n    в”Ӯ   в”ңв”Җв”Җ For each affected resource node\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider ApplyResourceChange()\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Update state with new values\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider ReadResource() to verify\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Record actual state\n    в”Ӯ   в”Ӯ   в””в”Җв”Җ Emit hook callbacks\n    в”Ӯ   в””в”Җв”Җ Continue until all nodes evaluated\n    в”ңв”Җв”Җ Collect all actual changes\n    в””в”Җв”Җ Build final state\n    вҶ“\n9. STATE PERSISTENCE PHASE\n    в”ңв”Җв”Җ Serialize final state to JSON\n    в”ңв”Җв”Җ Write to backend\n    в”ңв”Җв”Җ Increment serial number\n    в”ңв”Җв”Җ Release state lock\n    в””в”Җв”Җ Persist state file metadata\n    вҶ“\n10. OUTPUT EVALUATION PHASE\n    в”ңв”Җв”Җ Evaluate all output value expressions\n    в”ңв”Җв”Җ Resolve references to resource attributes\n    в””в”Җв”Җ Display outputs to user\n    вҶ“\nCompletion: Return success/failure status\n```\n\n### Graph Node Execution\n\n#### Plan Phase Node Execution (`NodeResourceInstancePlanner`)\n\n```\nNode.EvalTree() for Resource \"aws_instance\" \"web\":\n\n1. VALIDATE PHASE\n   в”ңв”Җв”Җ Ensure provider is configured\n   в”ңв”Җв”Җ Validate resource schema\n   в””в”Җв”Җ Return if invalid\n\n2. READ STATE PHASE\n   в”ңв”Җв”Җ Retrieve resource from state snapshot\n   в”ңв”Җв”Җ Deserialize JSON state\n   в””в”Җв”Җ Parse as structured data\n\n3. EVAL CONFIG PHASE\n   в”ңв”Җв”Җ Evaluate HCL expressions\n   в”ңв”Җв”Җ Resolve variable references\n   в”ңв”Җв”Җ Resolve resource references (depends_on)\n   в”ңв”Җв”Җ Interpolate functions\n   в””в”Җв”Җ Validate config against provider schema\n\n4. PLAN PHASE\n   в”ңв”Җв”Җ Call provider.PlanResourceChange()\n   в”Ӯ   в”ңв”Җв”Җ Send: prior state, config, provider metadata\n   в”Ӯ   в””в”Җв”Җ Receive: planned state, requires_replace, diagnostics\n   в”ңв”Җв”Җ Analyze change:\n   в”Ӯ   в”ңв”Җв”Җ Create: prior_state empty, planned_state not empty\n   в”Ӯ   в”ңв”Җв”Җ Update: both have values, differs\n   в”Ӯ   в”ңв”Җв”Җ Delete: config says destroy, planned_state empty\n   в”Ӯ   в””в”Җв”Җ No-op: both identical\n   в”ңв”Җв”Җ Determine if replace needed\n   в”Ӯ   в”ңв”Җв”Җ Check requires_replace from provider\n   в”Ӯ   в”ңв”Җв”Җ Check for attribute changes forcing replacement\n   в”Ӯ   в””в”Җв”Җ Update planned action if needed\n   в””в”Җв”Җ Record plan change in graph state\n\n5. OUTPUT RESULTS\n   в”ңв”Җв”Җ Store planned change (action + new state)\n   в”ңв”Җв”Җ Prepare change for display\n   в””в”Җв”Җ Return success or error\n```\n\n#### Apply Phase Node Execution (`NodeResourceInstanceApply`)\n\n```\nNode.EvalTree() for Resource \"aws_instance\" \"web\":\n\n1. VALIDATE PHASE\n   в”ңв”Җв”Җ Ensure provider is configured\n   в”ңв”Җв”Җ Check plan exists for this resource\n   в””в”Җв”Җ Return if invalid\n\n2. PLAN RETRIEVAL PHASE\n   в”ңв”Җв”Җ Read planned change from graph\n   в”ңв”Җв”Җ Extract planned state\n   в””в”Җв”Җ Validate plan matches current state\n\n3. PRE-APPLY HOOKS\n   в”ңв”Җв”Җ Emit PreApply hook\n   в”ңв”Җв”Җ Allow external systems to prepare\n   в””в”Җв”Җ Abort if hook returns error\n\n4. APPLY PHASE\n   в”ңв”Җв”Җ Based on change action:\n   в”Ӯ   в”ңв”Җв”Җ CREATE:\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Generate ID if auto-generated\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider.ApplyResourceChange()\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Send: nil prior, planned state, config\n   в”Ӯ   в”Ӯ   в””в”Җв”Җ Receive: new state, diagnostics\n   в”Ӯ   в”ңв”Җв”Җ UPDATE:\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider.ApplyResourceChange()\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Send: prior state, planned state, config\n   в”Ӯ   в”Ӯ   в””в”Җв”Җ Receive: updated state, diagnostics\n   в”Ӯ   в”ңв”Җв”Җ DELETE:\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider.ApplyResourceChange()\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Send: prior state, nil planned state, config\n   в”Ӯ   в”Ӯ   в””в”Җв”Җ Receive: empty state, diagnostics\n   в”Ӯ   в””в”Җв”Җ NO-OP:\n   в”Ӯ       в””в”Җв”Җ Skip provider call\n   в”ңв”Җв”Җ Handle errors\n   в”Ӯ   в”ңв”Җв”Җ If error, may mark resource as tainted\n   в”Ӯ   в”ңв”Җв”Җ Tainted resource will be recreated on next apply\n   в”Ӯ   в””в”Җв”Җ Continue with remaining resources if applicable\n   в””в”Җв”Җ Update state with actual response\n\n5. POST-APPLY HOOKS\n   в”ңв”Җв”Җ Emit PostApply hook\n   в”ңв”Җв”Җ Allow external logging/notification\n   в””в”Җв”Җ Continue regardless of hook errors\n\n6. READ VERIFICATION PHASE\n   в”ңв”Җв”Җ Call provider.ReadResource() to verify state\n   в”ңв”Җв”Җ Detect any drift from what we just created\n   в”ңв”Җв”Җ Use actual state for final record\n   в””в”Җв”Җ Log if discrepancies detected\n\n7. OUTPUT RESULTS\n   в”ңв”Җв”Җ Store final state change\n   в”ңв”Җв”Җ Prepare change for display\n   в””в”Җв”Җ Return success or error\n```\n\n### Walker and Graph Traversal\n\n**Walker Pattern** (`Graph.Walk()`):\n\nThe Walker implements topological sort with concurrency:\n\n```\n1. INITIALIZATION\n   в”ңв”Җв”Җ Build dependency map (vertex вҶ’ dependencies)\n   в”ңв”Җв”Җ Identify zero-in-degree vertices (no dependencies)\n   в”ңв”Җв”Җ Initialize worker goroutine pool (parallelism limit)\n   в”ңв”Җв”Җ Create result channel for completion signals\n   в””в”Җв”Җ Create vertex input channel\n\n2. EXECUTION LOOP\n   в”ңв”Җв”Җ While vertices remain:\n   в”Ӯ   в”ңв”Җв”Җ Send next available vertex to worker\n   в”Ӯ   в”ңв”Җв”Җ Worker executes vertex.Eval()\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call node's EvalTree()\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Handle panics (convert to errors)\n   в”Ӯ   в”Ӯ   в””в”Җв”Җ Return result (state + errors)\n   в”Ӯ   в”ңв”Җв”Җ Collect result from channel\n   в”Ӯ   в”ңв”Җв”Җ Update state with vertex result\n   в”Ӯ   в”ңв”Җв”Җ Identify newly available vertices\n   в”Ӯ   в”Ӯ   (dependents of just-completed vertex)\n   в”Ӯ   в””в”Җв”Җ Send newly available to workers\n   в”Ӯ\n   в”ңв”Җв”Җ If vertex errors:\n   в”Ӯ   в”ңв”Җв”Җ Mark as failed\n   в”Ӯ   в”ңв”Җв”Җ Determine if dependents should be skipped\n   в”Ӯ   в”ңв”Җв”Җ Emit error hook\n   в”Ӯ   в””в”Җв”Җ Continue or abort based on error handling mode\n   в”Ӯ\n   в””в”Җв”Җ Repeat until all vertices processed\n\n3. COMPLETION\n   в”ңв”Җв”Җ Collect all results\n   в”ңв”Җв”Җ Report success or failure\n   в””в”Җв”Җ Return final state\n```\n\n**Parallelism Control:**\n- Default parallelism: 10 concurrent graph operations\n- Configurable via `TF_PARALLELISM` environment variable or `parallelism` in terraform block\n- Applies per `terraform apply/plan/destroy` invocation\n- Does NOT apply across multiple unrelated resources in same graph (already serialized by dependencies)\n\n**Example Parallelism:**\n\nIn this graph (numbers are dependencies):\n```\nA (0 deps) вҶ’ C (dep on A)\nB (0 deps) вҶ’ D (dep on B, C)\nE (0 deps)\n\nWith parallelism=3:\nTime 0: Execute A, B, E (all have 0 dependencies, parallel)\nTime 1: Execute C, D (both dependencies satisfied, parallel when A+B+E done)\nDone: All vertices completed\n```\n\n### Hook System\n\n**Hook Interface** (`ProvisionerHook`):\n\nHooks are callback points during execution allowing external monitoring and customization:\n\n```go\ntype Hook interface {\n    // PreApply is called before ApplyResourceChange\n    PreApply(context.Context, *InstanceState, *InstanceDiff) (HookAction, error)\n\n    // PostApply is called after ApplyResourceChange\n    PostApply(context.Context, *InstanceState, *InstanceDiff) (HookAction, error)\n\n    // PrePlan is called before PlanResourceChange\n    PrePlan(context.Context, *InstanceDiff) (HookAction, error)\n\n    // PostPlan is called after PlanResourceChange\n    PostPlan(context.Context, *InstanceDiff) (HookAction, error)\n\n    // PreProvisioner is called before running provisioners\n    PreProvisioner(context.Context, *InstanceState, string) (HookAction, error)\n\n    // PostProvisioner is called after provisioner execution\n    PostProvisioner(context.Context, *InstanceState, string) (HookAction, error)\n\n    // ProvisionOutput outputs provisioner output\n    ProvisionOutput(context.Context, *InstanceState, string, string)\n}\n```\n\n**Hook Actions:**\n\n```\nContinue: Proceed normally (default)\nSkip: Skip this resource/operation\nAbort: Stop entire graph evaluation immediately\n```\n\n**Built-in Hooks:**\n\n1. **UI/Output Hook**: Formats and displays progress to console\n2. **State Hook**: Tracks all state mutations during execution\n3. **Logging Hook**: Records all operations to Terraform logs\n4. **Validation Hook**: Performs post-operation assertions\n\n**Custom Hook Integration:**\n\nExternal systems can register custom hooks:\n- Monitoring systems: Record all state changes to audit logs\n- Policy enforcement: Abort apply if policy violations detected\n- Integration systems: Notify external systems of infrastructure changes\n- Testing frameworks: Verify expected operations occurred\n\n### Dynamic Expansion (count and for_each)\n\n**Count Expansion:**\n\nConfiguration:\n```hcl\nresource \"aws_instance\" \"web\" {\n  count = 3\n  ami   = \"ami-123456\"\n}\n```\n\nExpansion process:\n```\n1. During graph construction, count.value expression is detected\n2. For each resource with count:\n   в”ңв”Җв”Җ Evaluate count expression (must be known at plan time)\n   в”ңв”Җв”Җ For each index (0 to count-1):\n   в”Ӯ   в”ңв”Җв”Җ Create resource node with index\n   в”Ӯ   в”ңв”Җв”Җ Name becomes aws_instance.web[0], aws_instance.web[1], etc.\n   в”Ӯ   в”ңв”Җв”Җ Create state instance placeholder\n   в”Ӯ   в””в”Җв”Җ Add to dependency graph\n   в””в”Җв”Җ Add count metadata to graph\n3. During plan/apply, each indexed instance is treated as separate node\n4. Each node has independent state, provider call, and plan/apply\n```\n\n**For-Each Expansion:**\n\nConfiguration:\n```hcl\nresource \"aws_security_group\" \"groups\" {\n  for_each = var.security_groups\n  name     = each.value.name\n}\n```\n\nExpansion process:\n```\n1. Evaluate for_each expression (keys must be strings)\n2. For each key-value pair:\n   в”ңв”Җв”Җ Create resource node with string key\n   в”ңв”Җв”Җ Name becomes aws_security_group.groups[\"production\"], etc.\n   в”ңв”Җв”Җ Provide each.key and each.value in evaluation context\n   в”ңв”Җв”Җ Create state instance placeholder\n   в””в”Җв”Җ Add to dependency graph\n3. For_each metadata tracked separately from count\n4. State stored with key-based lookup (not index-based)\n5. The pipeline operates in two distinct phases:\n\n- **Plan Phase**: Analyzes the desired state (configuration) against the current state to determine what changes are needed\n- **Apply Phase**: Executes those changes against real infrastructure through provider plugins\n\nThe architecture is built around a **directed acyclic graph (DAG)** that models resource dependencies, which enables Terraform to execute operations in the correct order and identify parallelization opportunities. The system is designed for extensibility, allowing custom components to integrate at well-defined points.\n\n### Key Design Principles\n\n- **Immutable State Snapshots**: State is read once and frozen for the duration of graph evaluation, enabling safe concurrent evaluation\n- **Plugin-Based Providers**: Infrastructure operations are delegated to external provider plugins communicating via RPC\n- **Declarative Execution**: The graph structure itself defines the execution plan rather than imperative control flow\n- **Hook-Based Extensibility**: Well-defined hooks allow monitoring and customization of execution behavior\n\n---\n\n## Graph Builder System\n\n### Purpose and Responsibilities\n\nThe Graph Builder System constructs a dependency graph that represents:\n- All resources, data sources, variables, and outputs in the configuration\n- Dependencies between these elements\n- Provider requirements and configurations\n- Lifecycle and ordering constraints\n\nThis graph serves as the execution blueprint for both plan and apply operations.\n\n### Core Components\n\n#### 1. **Graph Structure**\n\nThe core graph data structure (`dag.AcyclicGraph`) consists of:\n\n```\nVertices: Resources, data sources, providers, modules, variables, outputs\nEdges: Dependencies representing \"must execute after\" relationships\n```\n\nEach vertex is a `dag.Vertex` interface implementation that can:\n- Generate dependencies that must be satisfied before its execution\n- Execute its operation (validate, plan, apply)\n- Accept result transformations\n\n#### 2. **Graph Builder (`GraphBuilder` Interface)**\n\nThe primary graph builder orchestrates the graph construction process:\n\n**Responsibilities:**\n- Load and parse Terraform configuration\n- Instantiate all infrastructure components from the configuration\n- Build initial resource, data source, and provider vertices\n- Coordinate graph transformers to refine the graph\n\n**Process Flow:**\n1. Initialize root module parsing\n2. Create vertices for all configuration blocks (resources, data sources, locals, outputs)\n3. Create vertices for provider requirements\n4. Build initial edges based on literal resource references\n5. Apply a series of transformers to enhance the graph\n\n#### 3. **Graph Nodes**\n\nEach vertex in the execution graph implements the `dag.Vertex` interface and represents a unit of work:\n\n**Resource Node (`NodeResourceAbstract` and subclasses)**\n- Represents a single resource or data source\n- Variants exist for different lifecycle stages: validate, plan, apply, destroy\n- Contains the resource's configuration, type, and name\n- Generates dependencies based on resource references in configuration\n\n```\nNodeResourceInstanceAbstract (base)\nв”ңв”Җв”Җ NodeResourceInstancePreCondition\nв”ңв”Җв”Җ NodeResourceInstanceValidator\nв”ңв”Җв”Җ NodeResourceInstancePlannerReferencer\nв”ңв”Җв”Җ NodeResourceInstanceDestroyValidator\nв”ңв”Җв”Җ NodeResourceInstancePlanner\nв”ңв”Җв”Җ NodeResourceInstanceApply\nв””в”Җв”Җ NodeResourceInstanceDestroy\n```\n\n**Provider Node (`NodeProviderAbstract`)**\n- Represents provider configuration and initialization\n- One node per unique provider instance\n- Manages provider plugin lifecycle (configure, close)\n- Creates dependency edges to all resources using that provider\n\n**Module Node (`NodeModuleMeta`, `NodeModuleInstance`)**\n- Represents module instantiation\n- Wraps a sub-graph (the module's own graph)\n- Dynamically expands for count/for_each module instances\n- Manages module output and input variable connections\n\n**Variable and Output Nodes**\n- `NodeLocalValue`: Represents local values\n- `NodeOutput`: Represents root module outputs\n- `NodeRootVariable`: Represents input variables\n- Used for reference resolution and dependency tracking\n\n**Destroy Edge Nodes**\n- `NodeDestroyEdge`: Creates implicit dependencies for destruction ordering\n- Ensures resources are destroyed in reverse dependency order\n\n#### 4. **Graph Transformers**\n\nTransformers are composable functions that modify the graph after initial construction. Each transformer implements `GraphTransformer` interface:\n\n**Key Transformers (Applied in Sequence):**\n\n1. **PlanGraphBuilder Transformers**\n   - `TransformProviders`: Ensures all providers are configured before resources using them\n   - `TransformResourceCount`: Expands resources with `count` into multiple instances\n   - `TransformForEachResource`: Expands resources with `for_each` into multiple instances\n   - `TransformDynamicBlock`: Resolves `dynamic` blocks within resource configurations\n   - `TransformRemovalInGraph`: Removes resources marked for deletion\n   - `TransformRootVariables`: Adds variable evaluation nodes\n   - `TransformModuleInstance`: Expands module count/for_each instances\n   - `TransformOutputs`: Adds output evaluation nodes\n   - `TransformDestroyEdge`: Adds implicit destroy ordering edges\n   - `TransformProviderValidation`: Ensures provider configuration is validated\n   - `TransformManagedResourceStrategy`: Determines operation strategy (create, update, delete)\n   - `TransformBuildModuleDestroyGraph`: Builds destruction graphs for nested modules\n\n2. **DestroyPlanGraphBuilder Transformers**\n   - Similar to plan transformers but optimized for destruction order\n   - Reverses dependency direction for destroy-only graphs\n\n3. **ApplyGraphBuilder Transformers**\n   - `TransformProviderValidation`: Re-validates providers before apply\n   - `TransformApplyNodeExpanders`: Handles dynamic expansion during apply\n   - `TransformResourceCount`: Handles count changes from plan to apply\n   - `TransformRemovalInGraph`: Filters to only affected resources\n\n#### 5. **Dependency Discovery**\n\nDependencies are discovered and encoded as graph edges through:\n\n**Reference Tracking:**\n- Configuration AST is parsed to identify resource references (e.g., `aws_instance.web.id`)\n- References extract resource types, names, and attributes\n- These form the basis of graph edges\n\n**Implicit Dependencies:**\n- Dependencies between provider and resources\n- Dependencies between modules and their variable sources\n- Dependencies in variable interpolation expressions\n\n**Explicit Dependencies:**\n- `depends_on` meta-arguments create explicit edges\n- Lifecycle edges (provider depends on provider configuration)\n\n**Dynamic Dependencies:**\n- `count` and `for_each` create conditional edges\n- Edges only exist if the resource is instantiated\n- The graph builder must handle dynamic expansion before edge creation\n\n### Graph Transformation Pipeline Example\n\n```\nInitial Configuration вҶ’ Parse вҶ’ Create Core Vertices\n    вҶ“\nAdd Resource Vertices вҶ’ Add Provider Vertices вҶ’ Add Variable Vertices\n    вҶ“\nApply Transformer 1 (Count/ForEach Expansion)\n    вҶ“\nApply Transformer 2 (Provider Dependencies)\n    вҶ“\nApply Transformer 3 (Module Expansion)\n    вҶ“\nApply Transformer N (Final Cleanup)\n    вҶ“\nValidated Execution Graph вҶ’ Ready for Traversal\n```\n\n### Error Cases and Recovery\n\n- **Cycle Detection**: The graph builder validates that the result is acyclic; cycles in dependencies cause immediate error\n- **Invalid References**: References to non-existent resources are detected during dependency discovery\n- **Missing Providers**: Required providers without configuration block trigger provider initialization errors\n- **Module Errors**: Invalid module sources or missing child modules fail during module expansion\n\n---\n\n## Provider Interface and Lifecycle\n\n### Purpose and Responsibilities\n\nThe Provider Interface acts as the abstraction layer between Terraform core and infrastructure-specific implementations. Providers handle:\n- Authentication and connection to infrastructure platforms\n- Resource CRUD operations (Create, Read, Update, Delete)\n- Data source queries\n- Resource schema definition and validation\n\n### Provider Plugin Architecture\n\n#### 1. **Plugin Communication Protocol**\n\nTerraform communicates with provider plugins via **gRPC** with Protocol Buffers:\n\n**Two-Way Communication:**\n- Core вҶ’ Plugin: Resource operations (create, read, update, delete), data source read, validation\n- Plugin вҶ’ Core: Resource state, diagnostic messages, results\n\n**Protocol Definition:**\n- Providers expose a gRPC service implementing Terraform provider protocol\n- Messages include: resource type, operation, state, config, plan\n- Responses include: new state, diagnostic messages, error codes\n\n#### 2. **Provider Manager and Lifecycle**\n\n**`ResourceProvider` Interface** (abstracts provider behavior):\n\n```\nMethods:\nв”ңв”Җв”Җ Configure(context.Context, ConfigureRequest) ConfigureResponse\nв”ңв”Җв”Җ GetMetadata(context.Context) GetMetadataResponse\nв”ңв”Җв”Җ GetSchema(context.Context) GetSchemaResponse\nв”ңв”Җв”Җ ValidateResourceConfig(context.Context, ValidateResourceConfigRequest)\nв”ңв”Җв”Җ ValidateDataSourceConfig(context.Context, ValidateDataSourceConfigRequest)\nв”ңв”Җв”Җ ReadResource(context.Context, ReadResourceRequest) ReadResourceResponse\nв”ңв”Җв”Җ PlanResourceChange(context.Context, PlanResourceChangeRequest) PlanResourceChangeResponse\nв”ңв”Җв”Җ ApplyResourceChange(context.Context, ApplyResourceChangeRequest) ApplyResourceChangeResponse\nв”ңв”Җв”Җ ImportResourceState(context.Context, ImportResourceStateRequest)\nв”ңв”Җв”Җ ReadDataSource(context.Context, ReadDataSourceRequest) ReadDataSourceResponse\nв””в”Җв”Җ CallFunction(context.Context, CallFunctionRequest) CallFunctionResponse\n```\n\n**Provider Initialization Flow:**\n\n```\n1. Provider Discovery\n   в”ңв”Җв”Җ Locate provider binary (plugin cache or registry)\n   в”ңв”Җв”Җ Launch provider process as subprocess\n   в””в”Җв”Җ Establish gRPC connection\n\n2. Schema Loading\n   в”ңв”Җв”Җ Call GetSchema() RPC\n   в”ңв”Җв”Җ Cache schema in memory\n   в””в”Җв”Җ Validate against configuration\n\n3. Configuration Validation\n   в”ңв”Җв”Җ Call ValidateResourceConfig() for each resource type\n   в””в”Җв”Җ Call ValidateDataSourceConfig() for each data source\n\n4. Provider Configuration\n   в”ңв”Җв”Җ Build provider config from HCL configuration block\n   в”ңв”Җв”Җ Call Configure() RPC with credentials and settings\n   в””в”Җв”Җ Provider establishes connection to infrastructure\n\n5. Ready for Operations\n   в””в”Җв”Җ Provider instance ready to handle resource operations\n```\n\n#### 3. **Provider Instances and Caching**\n\n**Instance Management:**\n- One provider instance per unique provider configuration\n- Provider instances are cached and reused across the graph evaluation\n- Instance identity is based on provider type and configuration hash\n\n**Multi-Region/Multi-Account Pattern:**\n```hcl\nprovider \"aws\" {\n  alias  = \"us-east-1\"\n  region = \"us-east-1\"\n}\n\nprovider \"aws\" {\n  alias  = \"eu-west-1\"\n  region = \"eu-west-1\"\n}\n\nresource \"aws_instance\" \"east\" {\n  provider = aws.us-east-1\n  # ...\n}\n\nresource \"aws_instance\" \"eu\" {\n  provider = aws.eu-west-1\n  # ...\n}\n```\n\nEach provider reference creates a separate instance.\n\n#### 4. **Resource Operation Execution**\n\n**Plan Phase Resource Operations:**\n\nThe `PlanResourceChange` RPC is called with:\n- `TypeName`: Resource type (e.g., \"aws_instance\")\n- `PriorState`: Current resource state (JSON-encoded)\n- `Config`: Desired configuration from HCL (JSON-encoded)\n- `ProposedNewState`: Terraform's calculated new state\n- `ProviderMeta`: Provider-specific metadata\n\nProvider responds with:\n- `PlannedState`: Final planned state after applying provider logic\n- `RequiresReplace`: Attributes that require resource replacement\n- `Diagnostics`: Validation or planning issues\n\n**Apply Phase Resource Operations:**\n\nThe `ApplyResourceChange` RPC is called with:\n- `TypeName`: Resource type\n- `PriorState`: Current state\n- `PlannedState`: State from plan phase\n- `Config`: HCL configuration\n- `PlannedPrivateData`: Provider-specific binary data from plan\n\nProvider responds with:\n- `NewState`: State after actual API operations\n- `Private`: Opaque binary data for next operation\n- `Diagnostics`: Errors or warnings from API calls\n\n**Read Operations:**\n\nThe `ReadResource` RPC is called:\n- During `apply` after resource creation/update to verify actual state\n- During `refresh` to synchronize state with infrastructure\n- With current state and provider metadata\n\n### Error Handling in Provider Operations\n\n**Categorized Errors:**\n\n1. **Validation Errors**: Schema violations, missing required attributes\n2. **Permission Errors**: Authentication/authorization failures\n3. **API Errors**: Infrastructure API rejections (quota exceeded, invalid combination)\n4. **Transient Errors**: Network timeouts, temporary unavailability\n5. **Fatal Errors**: Unrecoverable provider issues (plugin crash, protocol error)\n\n**Error Propagation:**\n- Provider diagnostics include severity (error, warning)\n- Errors during plan cause plan abortion with diagnostic output\n- Errors during apply cause apply abortion with partial state rollback (if possible)\n\n### Provider Configuration Dependencies\n\n**Graph Integration:**\n- Each resource or data source creates implicit dependency on its provider node\n- Provider node depends on variable nodes it references in configuration\n- Provider initialization is sequenced before any resource operations\n\n---\n\n## State Management\n\n### Purpose and Responsibilities\n\nState Management is the system for:\n- Persisting resource metadata to enable incremental updates\n- Tracking resource identity across runs\n- Detecting infrastructure drift\n- Enabling safe concurrent and remote execution\n\n### State Structure\n\n**State File Format (JSON):**\n\n```json\n{\n  \"version\": 4,\n  \"terraform_version\": \"1.5.0\",\n  \"serial\": 42,\n  \"lineage\": \"unique-id\",\n  \"outputs\": {\n    \"instance_id\": {\n      \"value\": \"i-1234567890abcdef0\",\n      \"type\": \"string\"\n    }\n  },\n  \"resources\": [\n    {\n      \"type\": \"aws_instance\",\n      \"name\": \"example\",\n      \"instances\": [\n        {\n          \"index_key\": 0,\n          \"schema_version\": 1,\n          \"attributes\": {\n            \"id\": \"i-1234567890abcdef0\",\n            \"ami\": \"ami-0c55b159cbfafe1f0\",\n            \"instance_type\": \"t2.micro\",\n            \"tags\": {\n              \"Name\": \"HelloWorld\"\n            }\n          },\n          \"sensitive_attributes\": [],\n          \"private\": \"...\"\n        }\n      ]\n    }\n  ]\n}\n```\n\n**Key Components:**\n- `version`: State file schema version (supports backward compatibility)\n- `serial`: Monotonically increasing counter for change detection\n- `lineage`: Unique identifier tracking state identity across migrations\n- `resources`: Array of resource instances with full state\n- `outputs`: Root module output values\n\n### State Managers\n\n**Manager Responsibilities:**\n- Load state from backend (local file, remote storage, secrets manager)\n- Coordinate state mutations during graph evaluation\n- Persist modified state back to backend\n- Manage state locking for concurrent execution prevention\n- Handle state encryption and secrets\n\n**State Manager Hierarchy:**\n\n```\nBackend Interface (abstract storage)\nв”ңв”Җв”Җ Local Backend\nв”Ӯ   в””в”Җв”Җ File-based state with optional locking\nв”ңв”Җв”Җ Remote Backends\nв”Ӯ   в”ңв”Җв”Җ Terraform Cloud / Terraform Enterprise\nв”Ӯ   в”ңв”Җв”Җ AWS S3 + DynamoDB locking\nв”Ӯ   в”ңв”Җв”Җ Azure Storage Blob + Lease locking\nв”Ӯ   в”ңв”Җв”Җ Google Cloud Storage + Bucket Lock\nв”Ӯ   в”ңв”Җв”Җ Consul\nв”Ӯ   в””в”Җв”Җ etcd\nв””в”Җв”Җ Special Backends\n    в”ңв”Җв”Җ Terraform Testing Backend (in-memory)\n    в””в”Җв”Җ Multi-State Backend (for workspaces)\n```\n\n**Backend Operations:**\n- `StateFn()`: Load current state\n- `DeleteState()`: Remove state (dangerous)\n- `Lock(info)`: Acquire exclusive lock\n- `Unlock(id)`: Release lock\n- Operations fail if lock cannot be acquired (prevents concurrent modifications)\n\n### State Synchronization and Snapshots\n\n**Immutable Snapshot Pattern:**\n\nDuring graph evaluation, state operates in \"frozen snapshot\" mode:\n\n```\n1. Load Initial State вҶ’ Snapshot (read-only reference)\n2. Plan Graph Evaluation\n   в”ңв”Җв”Җ Node reads from snapshot\n   в”ңв”Җв”Җ Node calculates changes\n   в”ңв”Җв”Җ Node records new state (not yet persisted)\n   в””в”Җв”Җ Continue with next node\n3. Collect All Planned State Changes\n4. Persist Combined New State\n```\n\n**Benefits:**\n- Graph nodes don't see partial intermediate states from other evaluations\n- Enables safe concurrent node evaluation (different nodes can run in parallel)\n- Provides consistent view of infrastructure across the entire graph\n- If evaluation fails, original state remains unchanged\n\n**State Refresh Pattern (before planning):**\n\n```\n1. For each resource in current state\n   в”ңв”Җв”Җ Call ReadResource() via provider\n   в”ңв”Җв”Җ Detect drift (configuration vs actual)\n   в””в”Җв”Җ Update state with actual values\n2. Output refresh summary\n3. Proceed with planning based on refreshed state\n```\n\n### State Locking\n\n**Pessimistic Locking:**\n- Acquired immediately when state operation begins\n- Held for entire duration of plan/apply/refresh operation\n- Released only after operation completes\n- Prevents concurrent modifications in shared state backends\n\n**Lock Metadata:**\n```json\n{\n  \"ID\": \"unique-uuid\",\n  \"Path\": \"path/in/backend\",\n  \"Operation\": \"OperationTypePlan\",\n  \"Info\": \"user@hostname:pid\",\n  \"Who\": \"user@hostname\",\n  \"Version\": \"1.5.0\",\n  \"Created\": \"2024-02-28T10:30:00Z\"\n}\n```\n\n**Lock Timeout and Conflict Handling:**\n- Lock acquisition has configurable timeout\n- If lock exists but is stale (from crashed process), force unlock with warning\n- Prevents deadlock from interrupted operations\n\n### Remote State Backends\n\n**Terraform Cloud / Terraform Enterprise:**\n- Manages all state through authenticated REST API\n- Provides versioning, state history, and run tracking\n- Automatically handles locking and encryption\n- Integrates with VCS for automation\n\n**Self-Hosted Remote Backends (S3, GCS, Azure):**\n- State stored in managed object storage (durability)\n- Locking via distributed coordination service\n  - S3: Uses DynamoDB table for lease-based locking\n  - GCS: Uses metadata-based optimistic locking\n  - Azure: Uses blob lease mechanism\n- Encryption at rest and in transit\n\n**State Access Flow (Remote Backend):**\n\n```\nTerraform CLI\n    вҶ“\nBackend Interface\n    вҶ“\nHTTP API / Object Storage API\n    вҶ“\n(Optional) Locking Service\n    вҶ“\nTerraform Cloud / S3 / GCS / Azure\n```\n\n### State Drift Detection\n\n**Drift Detection Process:**\n\n```\n1. Load State (prior knowledge)\n2. Execute ReadResource() for each resource\n3. Compare actual attributes with state attributes\n4. Categorize changes:\n   в”ңв”Җв”Җ No Drift: Actual matches state\n   в”ңв”Җв”Җ In-Band Drift: State needs updating, no HCL change\n   в””в”Җв”Җ Out-of-Band Drift: Actual diverges from config (indicates external modification)\n5. Report drift summary\n```\n\n**Handling Out-of-Band Drift:**\n- Detected when actual attribute differs from configuration\n- Plan shows update from actual вҶ’ config (replaying configuration intent)\n- User can review drift before apply\n- Option to overwrite actual with config or preserve actual (refresh-only mode)\n\n### State Secrets and Encryption\n\n**Sensitive Attribute Handling:**\n- Marked in provider schema as `Sensitive: true`\n- Recorded in state with `sensitive_attributes` array\n- Redacted from console output and logs\n- Encrypted if backend supports encryption\n\n**Backend Encryption:**\n- Terraform Cloud: Automatic AES-256-GCM encryption\n- Remote backends (S3): Server-side encryption (user-managed keys or AWS-managed)\n- Local state: Unencrypted by default (can use `aws_kms_encrypt` functions for sensitive blocks)\n\n---\n\n## Execution Flow and Hook System\n\n### Overall Execution Flow\n\n#### Complete Plan/Apply Workflow\n\n```\nterraform plan/apply invocation\n    вҶ“\n1. INITIALIZATION PHASE\n    в”ңв”Җв”Җ Load and parse HCL configuration\n    в”ңв”Җв”Җ Load backend configuration\n    в”ңв”Җв”Җ Initialize working directory\n    в”ңв”Җв”Җ Acquire state lock\n    в””в”Җв”Җ Load current state from backend\n    вҶ“\n2. CONFIGURATION VALIDATION PHASE\n    в”ңв”Җв”Җ Validate module structure\n    в”ңв”Җв”Җ Validate resource references\n    в”ңв”Җв”Җ Validate variable types and defaults\n    в””в”Җв”Җ Call provider ValidateResourceConfig() for each resource\n    вҶ“\n3. GRAPH CONSTRUCTION PHASE (Plan)\n    в”ңв”Җв”Җ Build initial resource/provider/variable vertices\n    в”ңв”Җв”Җ Apply graph transformers\n    в”Ӯ   в”ңв”Җв”Җ Expand count/for_each\n    в”Ӯ   в”ңв”Җв”Җ Expand modules\n    в”Ӯ   в”ңв”Җв”Җ Resolve dynamic blocks\n    в”Ӯ   в””в”Җв”Җ Add dependency edges\n    в””в”Җв”Җ Validate acyclic graph\n    вҶ“\n4. STATE REFRESH PHASE (optional, enabled by default)\n    в”ңв”Җв”Җ For each resource in current state\n    в”Ӯ   в”ңв”Җв”Җ Call provider ReadResource()\n    в”Ӯ   в”ңв”Җв”Җ Detect and report drift\n    в”Ӯ   в””в”Җв”Җ Update state snapshot\n    в””в”Җв”Җ Output refresh summary\n    вҶ“\n5. PLAN GENERATION PHASE\n    в”ңв”Җв”Җ Create plan context (immutable state snapshot)\n    в”ңв”Җв”Җ Initialize Walker with concurrency limit\n    в”ңв”Җв”Җ Walk graph in dependency order\n    в”Ӯ   в”ңв”Җв”Җ For each resource node\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider PlanResourceChange()\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Determine action (create/update/delete/no-op)\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Record plan changes\n    в”Ӯ   в”Ӯ   в””в”Җв”Җ Emit hook callbacks\n    в”Ӯ   в””в”Җв”Җ Continue until all nodes evaluated\n    в”ңв”Җв”Җ Collect all planned changes\n    в””в”Җв”Җ Serialize plan to file\n    вҶ“\n6. APPROVAL PHASE (interactive mode)\n    в”ңв”Җв”Җ Display plan summary\n    в”ңв”Җв”Җ Prompt user for approval\n    в””в”Җв”Җ Abort if user declines (Plan only)\n    вҶ“\n7. GRAPH CONSTRUCTION PHASE (Apply)\n    в”ңв”Җв”Җ Rebuild graph with apply-specific transformers\n    в”ңв”Җв”Җ Filter to affected resources (from plan)\n    в””в”Җв”Җ Validate acyclic graph\n    вҶ“\n8. APPLY EXECUTION PHASE\n    в”ңв”Җв”Җ Create apply context (plan reference)\n    в”ңв”Җв”Җ Initialize Walker with concurrency limit\n    в”ңв”Җв”Җ Walk graph in dependency order\n    в”Ӯ   в”ңв”Җв”Җ For each affected resource node\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider ApplyResourceChange()\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Update state with new values\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider ReadResource() to verify\n    в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Record actual state\n    в”Ӯ   в”Ӯ   в””в”Җв”Җ Emit hook callbacks\n    в”Ӯ   в””в”Җв”Җ Continue until all nodes evaluated\n    в”ңв”Җв”Җ Collect all actual changes\n    в””в”Җв”Җ Build final state\n    вҶ“\n9. STATE PERSISTENCE PHASE\n    в”ңв”Җв”Җ Serialize final state to JSON\n    в”ңв”Җв”Җ Write to backend\n    в”ңв”Җв”Җ Increment serial number\n    в”ңв”Җв”Җ Release state lock\n    в””в”Җв”Җ Persist state file metadata\n    вҶ“\n10. OUTPUT EVALUATION PHASE\n    в”ңв”Җв”Җ Evaluate all output value expressions\n    в”ңв”Җв”Җ Resolve references to resource attributes\n    в””в”Җв”Җ Display outputs to user\n    вҶ“\nCompletion: Return success/failure status\n```\n\n### Graph Node Execution\n\n#### Plan Phase Node Execution (`NodeResourceInstancePlanner`)\n\n```\nNode.EvalTree() for Resource \"aws_instance\" \"web\":\n\n1. VALIDATE PHASE\n   в”ңв”Җв”Җ Ensure provider is configured\n   в”ңв”Җв”Җ Validate resource schema\n   в””в”Җв”Җ Return if invalid\n\n2. READ STATE PHASE\n   в”ңв”Җв”Җ Retrieve resource from state snapshot\n   в”ңв”Җв”Җ Deserialize JSON state\n   в””в”Җв”Җ Parse as structured data\n\n3. EVAL CONFIG PHASE\n   в”ңв”Җв”Җ Evaluate HCL expressions\n   в”ңв”Җв”Җ Resolve variable references\n   в”ңв”Җв”Җ Resolve resource references (depends_on)\n   в”ңв”Җв”Җ Interpolate functions\n   в””в”Җв”Җ Validate config against provider schema\n\n4. PLAN PHASE\n   в”ңв”Җв”Җ Call provider.PlanResourceChange()\n   в”Ӯ   в”ңв”Җв”Җ Send: prior state, config, provider metadata\n   в”Ӯ   в””в”Җв”Җ Receive: planned state, requires_replace, diagnostics\n   в”ңв”Җв”Җ Analyze change:\n   в”Ӯ   в”ңв”Җв”Җ Create: prior_state empty, planned_state not empty\n   в”Ӯ   в”ңв”Җв”Җ Update: both have values, differs\n   в”Ӯ   в”ңв”Җв”Җ Delete: config says destroy, planned_state empty\n   в”Ӯ   в””в”Җв”Җ No-op: both identical\n   в”ңв”Җв”Җ Determine if replace needed\n   в”Ӯ   в”ңв”Җв”Җ Check requires_replace from provider\n   в”Ӯ   в”ңв”Җв”Җ Check for attribute changes forcing replacement\n   в”Ӯ   в””в”Җв”Җ Update planned action if needed\n   в””в”Җв”Җ Record plan change in graph state\n\n5. OUTPUT RESULTS\n   в”ңв”Җв”Җ Store planned change (action + new state)\n   в”ңв”Җв”Җ Prepare change for display\n   в””в”Җв”Җ Return success or error\n```\n\n#### Apply Phase Node Execution (`NodeResourceInstanceApply`)\n\n```\nNode.EvalTree() for Resource \"aws_instance\" \"web\":\n\n1. VALIDATE PHASE\n   в”ңв”Җв”Җ Ensure provider is configured\n   в”ңв”Җв”Җ Check plan exists for this resource\n   в””в”Җв”Җ Return if invalid\n\n2. PLAN RETRIEVAL PHASE\n   в”ңв”Җв”Җ Read planned change from graph\n   в”ңв”Җв”Җ Extract planned state\n   в””в”Җв”Җ Validate plan matches current state\n\n3. PRE-APPLY HOOKS\n   в”ңв”Җв”Җ Emit PreApply hook\n   в”ңв”Җв”Җ Allow external systems to prepare\n   в””в”Җв”Җ Abort if hook returns error\n\n4. APPLY PHASE\n   в”ңв”Җв”Җ Based on change action:\n   в”Ӯ   в”ңв”Җв”Җ CREATE:\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Generate ID if auto-generated\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider.ApplyResourceChange()\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Send: nil prior, planned state, config\n   в”Ӯ   в”Ӯ   в””в”Җв”Җ Receive: new state, diagnostics\n   в”Ӯ   в”ңв”Җв”Җ UPDATE:\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider.ApplyResourceChange()\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Send: prior state, planned state, config\n   в”Ӯ   в”Ӯ   в””в”Җв”Җ Receive: updated state, diagnostics\n   в”Ӯ   в”ңв”Җв”Җ DELETE:\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call provider.ApplyResourceChange()\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Send: prior state, nil planned state, config\n   в”Ӯ   в”Ӯ   в””в”Җв”Җ Receive: empty state, diagnostics\n   в”Ӯ   в””в”Җв”Җ NO-OP:\n   в”Ӯ       в””в”Җв”Җ Skip provider call\n   в”ңв”Җв”Җ Handle errors\n   в”Ӯ   в”ңв”Җв”Җ If error, may mark resource as tainted\n   в”Ӯ   в”ңв”Җв”Җ Tainted resource will be recreated on next apply\n   в”Ӯ   в””в”Җв”Җ Continue with remaining resources if applicable\n   в””в”Җв”Җ Update state with actual response\n\n5. POST-APPLY HOOKS\n   в”ңв”Җв”Җ Emit PostApply hook\n   в”ңв”Җв”Җ Allow external logging/notification\n   в””в”Җв”Җ Continue regardless of hook errors\n\n6. READ VERIFICATION PHASE\n   в”ңв”Җв”Җ Call provider.ReadResource() to verify state\n   в”ңв”Җв”Җ Detect any drift from what we just created\n   в”ңв”Җв”Җ Use actual state for final record\n   в””в”Җв”Җ Log if discrepancies detected\n\n7. OUTPUT RESULTS\n   в”ңв”Җв”Җ Store final state change\n   в”ңв”Җв”Җ Prepare change for display\n   в””в”Җв”Җ Return success or error\n```\n\n### Walker and Graph Traversal\n\n**Walker Pattern** (`Graph.Walk()`):\n\nThe Walker implements topological sort with concurrency:\n\n```\n1. INITIALIZATION\n   в”ңв”Җв”Җ Build dependency map (vertex вҶ’ dependencies)\n   в”ңв”Җв”Җ Identify zero-in-degree vertices (no dependencies)\n   в”ңв”Җв”Җ Initialize worker goroutine pool (parallelism limit)\n   в”ңв”Җв”Җ Create result channel for completion signals\n   в””в”Җв”Җ Create vertex input channel\n\n2. EXECUTION LOOP\n   в”ңв”Җв”Җ While vertices remain:\n   в”Ӯ   в”ңв”Җв”Җ Send next available vertex to worker\n   в”Ӯ   в”ңв”Җв”Җ Worker executes vertex.Eval()\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Call node's EvalTree()\n   в”Ӯ   в”Ӯ   в”ңв”Җв”Җ Handle panics (convert to errors)\n   в”Ӯ   в”Ӯ   в””в”Җв”Җ Return result (state + errors)\n   в”Ӯ   в”ңв”Җв”Җ Collect result from channel\n   в”Ӯ   в”ңв”Җв”Җ Update state with vertex result\n   в”Ӯ   в”ңв”Җв”Җ Identify newly available vertices\n   в”Ӯ   в”Ӯ   (dependents of just-completed vertex)\n   в”Ӯ   в””в”Җв”Җ Send newly available to workers\n   в”Ӯ\n   в”ңв”Җв”Җ If vertex errors:\n   в”Ӯ   в”ңв”Җв”Җ Mark as failed\n   в”Ӯ   в”ңв”Җв”Җ Determine if dependents should be skipped\n   в”Ӯ   в”ңв”Җв”Җ Emit error hook\n   в”Ӯ   в””в”Җв”Җ Continue or abort based on error handling mode\n   в”Ӯ\n   в””в”Җв”Җ Repeat until all vertices processed\n\n3. COMPLETION\n   в”ңв”Җв”Җ Collect all results\n   в”ңв”Җв”Җ Report success or failure\n   в””в”Җв”Җ Return final state\n```\n\n**Parallelism Control:**\n- Default parallelism: 10 concurrent graph operations\n- Configurable via `TF_PARALLELISM` environment variable or `parallelism` in terraform block\n- Applies per `terraform apply/plan/destroy` invocation\n- Does NOT apply across multiple unrelated resources in same graph (already serialized by dependencies)\n\n**Example Parallelism:**\n\nIn this graph (numbers are dependencies):\n```\nA (0 deps) вҶ’ C (dep on A)\nB (0 deps) вҶ’ D (dep on B, C)\nE (0 deps)\n\nWith parallelism=3:\nTime 0: Execute A, B, E (all have 0 dependencies, parallel)\nTime 1: Execute C, D (both dependencies satisfied, parallel when A+B+E done)\nDone: All vertices completed\n```\n\n### Hook System\n\n**Hook Interface** (`ProvisionerHook`):\n\nHooks are callback points during execution allowing external monitoring and customization:\n\n```go\ntype Hook interface {\n    // PreApply is called before ApplyResourceChange\n    PreApply(context.Context, *InstanceState, *InstanceDiff) (HookAction, error)\n\n    // PostApply is called after ApplyResourceChange\n    PostApply(context.Context, *InstanceState, *InstanceDiff) (HookAction, error)\n\n    // PrePlan is called before PlanResourceChange\n    PrePlan(context.Context, *InstanceDiff) (HookAction, error)\n\n    // PostPlan is called after PlanResourceChange\n    PostPlan(context.Context, *InstanceDiff) (HookAction, error)\n\n    // PreProvisioner is called before running provisioners\n    PreProvisioner(context.Context, *InstanceState, string) (HookAction, error)\n\n    // PostProvisioner is called after provisioner execution\n    PostProvisioner(context.Context, *InstanceState, string) (HookAction, error)\n\n    // ProvisionOutput outputs provisioner output\n    ProvisionOutput(context.Context, *InstanceState, string, string)\n}\n```\n\n**Hook Actions:**\n\n```\nContinue: Proceed normally (default)\nSkip: Skip this resource/operation\nAbort: Stop entire graph evaluation immediately\n```\n\n**Built-in Hooks:**\n\n1. **UI/Output Hook**: Formats and displays progress to console\n2. **State Hook**: Tracks all state mutations during execution\n3. **Logging Hook**: Records all operations to Terraform logs\n4. **Validation Hook**: Performs post-operation assertions\n\n**Custom Hook Integration:**\n\nExternal systems can register custom hooks:\n- Monitoring systems: Record all state changes to audit logs\n- Policy enforcement: Abort apply if policy violations detected\n- Integration systems: Notify external systems of infrastructure changes\n- Testing frameworks: Verify expected operations occurred\n\n### Dynamic Expansion (count and for_each)\n\n**Count Expansion:**\n\nConfiguration:\n```hcl\nresource \"aws_instance\" \"web\" {\n  count = 3\n  ami   = \"ami-123456\"\n}\n```\n\nExpansion process:\n```\n1. During graph construction, count.value expression is detected\n2. For each resource with count:\n   в”ңв”Җв”Җ Evaluate count expression (must be known at plan time)\n   в”ңв”Җв”Җ For each index (0 to count-1):\n   в”Ӯ   в”ңв”Җв”Җ Create resource node with index\n   в”Ӯ   в”ңв”Җв”Җ Name becomes aws_instance.web[0], aws_instance.web[1], etc.\n   в”Ӯ   в”ңв”Җв”Җ Create state instance placeholder\n   в”Ӯ   в””в”Җв”Җ Add to dependency graph\n   в””в”Җв”Җ Add count metadata to graph\n3. During plan/apply, each indexed instance is treated as separate node\n4. Each node has independent state, provider call, and plan/apply\n```\n\n**For-Each Expansion:**\n\nConfiguration:\n```hcl\nresource \"aws_security_group\" \"groups\" {\n  for_each = var.security_groups\n  name     = each.value.name\n}\n```\n\nExpansion process:\n```\n1. Evaluate for_each expression (keys must be strings)\n2. For each key-value pair:\n   в”ңв”Җв”Җ Create resource node with string key\n   в”ңв”Җв”Җ Name becomes aws_security_group.groups[\"production\"], etc.\n   в”ңв”Җв”Җ Provide each.key and each.value in evaluation context\n   в”ңв”Җв”Җ Create state instance placeholder\n   в””в”Җв”Җ Add to dependency graph\n3. For_each metadata tracked separately from count\n4. State stored with key-based lookup (not index-based)\n5. Allows references like aws_security_group.groups[\"production\"].id\n```\n\n**Dynamic Block Expansion:**\n\nConfiguration:\n```hcl\nresource \"aws_security_group\" \"example\" {\n  dynamic \"ingress\" {\n    for_each = var.ports\n    content {\n      from_port   = ingress.value.from\n      to_port     = ingress.value.to\n      cidr_blocks = ingress.value.cidrs\n    }\n  }\n}\n```\n\nExpansion process:\n```\n1. During config evaluation (before provider schema validation)\n2. For each dynamic block:\n   в”ңв”Җв”Җ Evaluate for_each expression\n   в”ңв”Җв”Җ For each iteration:\n   в”Ӯ   в”ңв”Җв”Җ Create context with iterator variable\n   в”Ӯ   в”ңв”Җв”Җ Evaluate content block with context\n   в”Ӯ   в””в”Җв”Җ Inject evaluated content into resource config\n   в””в”Җв”Җ Results in flattened resource config with multiple block instances\n3. Final config sent to provider already expanded (provider doesn't see dynamic)\n```\n\n**State Instance Tracking:**\n\nFor resources with count/for_each, state tracks instances:\n```json\n{\n {...}\n    },\n    {\n      \"index_key\": \"production\",\n      \"attributes\": {...}\n    }\n  ]\n}\n```\n\n---\n\n## Integration and Data Flow\n\n### End-to-End Data Flow: Creating a
