# Investigation Report: Duplicate Response Headers in Envoy Local Replies

## Summary

Response headers configured via `response_headers_to_add` in route configuration are being duplicated in local replies generated by the router filter (e.g., upstream timeouts, direct responses). The issue stems from `finalizeResponseHeaders()` being called by both the router's lambda callback AND the filter manager's header modification callback, causing response headers to be added twice despite `OVERWRITE_IF_EXISTS_OR_ADD` semantics.

## Root Cause

The duplication occurs due to a mismatch in responsibility between how the router handles upstream responses vs. local replies:

1. **For upstream responses** (`onUpstreamHeaders`): Router calls `finalizeResponseHeaders()` directly at line 1792, then `modify_headers_` at line 1793, then `encodeHeaders()` at line 1799
2. **For local replies** (`onUpstreamAbort`, direct responses): Router calls `sendLocalReply()` with a lambda that calls `modify_headers_`, and the filter manager INDEPENDENTLY calls `finalizeResponseHeaders()` in its header modification callback

This creates a scenario where:
- **Direct responses**: The router's lambda invokes `direct_response->finalizeResponseHeaders()` (line 501 in router.cc)
- **Filter manager callback**: The filter manager's callback invokes `route_entry_->finalizeResponseHeaders()` (line 1074 in filter_manager.cc)
- Result: Headers are added twice if both the direct response entry and the route entry have the same headers configured

### Call Chain for Direct Responses

1. Router's `onResponseDirect()` calls `sendLocalReply()` with lambda containing `direct_response->finalizeResponseHeaders()`
2. Filter manager's `sendLocalReply()` → `sendLocalReplyViaFilterChain()`
3. `sendLocalReplyViaFilterChain()` calls `Utility::sendLocalReply()`
4. `Utility::sendLocalReply()` calls `prepareLocalReply()`
5. `prepareLocalReply()` invokes `encode_functions.modify_headers_()` callback from filter manager
6. Filter manager's callback calls `route_entry_->finalizeResponseHeaders()` (already called by router's lambda!)
7. Result: **Headers added twice**

## Causal Chain

1. **Symptom**: Duplicate response headers in access logs for local replies (e.g., two `x-custom-trace` headers)
2. **Router behavior**: For direct responses, router's lambda calls `direct_response->finalizeResponseHeaders()`
3. **Filter manager intercept**: Filter manager's `sendLocalReply()` creates its own callback
4. **Callback creation**: Filter manager's callback includes BOTH `route_entry_->finalizeResponseHeaders()` AND router's lambda
5. **Header evaluation**: `finalizeResponseHeaders()` calls `HeaderParser::evaluateHeaders()`
6. **Header parser logic**: Parser uses `setReferenceKey()` for `OVERWRITE_IF_EXISTS_OR_ADD`, but since called twice, both instances add headers
7. **Root cause**: `finalizeResponseHeaders()` invoked twice in sequence:
   - First by router's lambda (for direct responses)
   - Second by filter manager's callback
8. **Symptom manifestation**: Headers appear twice in response because `evaluateHeaders()` was called twice on the same header map

## Recommendation

**Fix Strategy**: The filter manager should NOT call `finalizeResponseHeaders()` in its local reply callback if the router's lambda has already invoked it.

**Implementation Options**:

1. **Option A** (Recommended): Router should NOT invoke `finalizeResponseHeaders()` from the lambda passed to `sendLocalReply()`. Instead, rely solely on the filter manager to call it once. Modify:
   - `Filter::onResponseDirect()` - remove `finalizeResponseHeaders()` call from lambda
   - Let filter manager's callback handle all response header finalization

2. **Option B**: Filter manager should track if `finalizeResponseHeaders()` has already been called in the router's lambda and skip its own invocation. This requires explicit coordination between router and filter manager.

3. **Option C**: Create a separate code path for direct responses that doesn't use the filter manager's header modification callback for `finalizeResponseHeaders()`.

**Diagnostic Steps**:

1. Check if headers are duplicated specifically for direct responses (redirect routes) vs. timeout-generated local replies
2. Verify the `append_action` proto field values in the route configuration
3. Trace `HeaderParser::evaluateHeaders()` calls using debug logging at lines 947 and 1074
4. Confirm if `route_entry_->finalizeResponseHeaders()` and `direct_response->finalizeResponseHeaders()` are both being invoked
5. Check if similar duplication occurs with `request_headers_to_add` for early header mutations in the reverse direction

## Proto Field Considerations

**HeaderValueOption.append_action** (api/envoy/config/core/v3/base.proto:476):
- `APPEND_IF_EXISTS_OR_ADD`: Uses `addReferenceKey()` - always appends
- `OVERWRITE_IF_EXISTS_OR_ADD`: Uses `setReferenceKey()` - replaces if exists, adds if absent
- `ADD_IF_ABSENT`: Uses `addReferenceKey()` conditionally - only adds if missing
- `OVERWRITE_IF_EXISTS`: Uses `setReferenceKey()` conditionally - only if exists

Legacy **append** BoolValue field is deprecated in favor of `append_action` enum (source/common/router/header_parser.cc:58-63), but both may be present in configs requiring compatibility handling.
