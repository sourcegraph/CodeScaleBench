# Investigation Report: Duplicate Response Headers in Envoy Filter Pipeline

## Summary

Response headers configured via `response_headers_to_add` in route configuration are duplicated in local replies (responses generated by the router filter itself, such as for upstream timeouts or cluster not found) due to `finalizeResponseHeaders()` being called multiple times through nested callbacks in the `sendLocalReply()` code path. The filter manager wraps router-provided callbacks with an additional `finalizeResponseHeaders()` call, but for certain scenarios (particularly direct responses), this creates a double-invocation of header evaluation logic.

## Root Cause

The root cause is architectural: when the router filter generates a local reply via `sendLocalReply()`, it passes a `modify_headers` callback. The filter manager then wraps this callback in another callback that calls `finalizeResponseHeaders()` before invoking the router's callback. This creates nested evaluation of response header transformations:

**For direct responses (route redirects):**
- Router calls `sendLocalReply()` with a lambda that calls `direct_response->finalizeResponseHeaders()` (source/common/router/router.cc:501)
- Filter manager's `sendLocalReplyViaFilterChain()` creates a wrapper callback that calls:
  - `route_entry_->finalizeResponseHeaders()` (source/common/http/filter_manager.cc:1125)
  - Then the router's original callback (which calls `direct_response->finalizeResponseHeaders()`)
- Both `finalizeResponseHeaders()` calls invoke `evaluateHeaders()` on the response header parsers
- For headers with `OVERWRITE_IF_EXISTS_OR_ADD` action, this results in the header being evaluated twice

**For other local replies (cluster not found, timeouts):**
- Router calls `sendLocalReply()` with `modify_headers_` callback
- Filter manager wraps this with a callback that:
  - Calls `route_entry_->finalizeResponseHeaders()` (source/common/http/filter_manager.cc:1125)
  - Then calls the router's `modify_headers_` (which doesn't itself call `finalizeResponseHeaders()`)

## Causal Chain

1. **Symptom:** Response headers appear duplicated in access logs for local replies (e.g., "x-custom-trace: abc123" appears twice)

2. **Intermediate hop 1:** Local reply is generated by router filter (cluster not found, timeout, etc.)

3. **Intermediate hop 2:** Router calls `FilterManager::sendLocalReply()` with a `modify_headers` callback (either direct response lambda or router's `modify_headers_`)

4. **Intermediate hop 3:** `FilterManager::sendLocalReply()` delegates to `sendLocalReplyViaFilterChain()` or similar methods

5. **Intermediate hop 4:** These methods wrap the router's callback with a new callback that:
   - Calls `route_entry_->finalizeResponseHeaders()` unconditionally
   - Then calls the original `modify_headers` callback

6. **Intermediate hop 5:** `Utility::sendLocalReply()` is invoked with the wrapped callback as `encode_functions.modify_headers_`

7. **Intermediate hop 6:** `Utility::prepareLocalReply()` invokes the wrapped callback at line 719

8. **Intermediate hop 7:** Inside the wrapped callback:
   - `route_entry_->finalizeResponseHeaders()` calls `header_parser->evaluateHeaders()`
   - For headers with `OVERWRITE_IF_EXISTS_OR_ADD`, headers are collected in `headers_to_overwrite`
   - `setReferenceKey()` is called to set the header
   - Then the original callback is invoked (which may also call `finalizeResponseHeaders()`)
   - This second `finalizeResponseHeaders()` call again evaluates headers

9. **Root cause:** The header evaluation logic in `evaluateHeaders()` treats headers with `APPEND_IF_EXISTS_OR_ADD` as append operations. When called twice on the same header map with `OVERWRITE_IF_EXISTS_OR_ADD`, the proto field may default to or be interpreted as `APPEND_IF_EXISTS_OR_ADD` in certain conditions, causing the header to be appended rather than replaced on the second invocation. Alternatively, the interaction between `setReferenceKey()` and subsequent `addReferenceKey()` calls may cause unexpected duplication.

## Role of append_action / append Proto Fields

The `HeaderValueOption` proto has:
- **Deprecated `append` field** (BoolValue): `true` means append, `false` means overwrite
- **Newer `append_action` enum**: `APPEND_IF_EXISTS_OR_ADD`, `OVERWRITE_IF_EXISTS_OR_ADD`, `ADD_IF_ABSENT`, `OVERWRITE_IF_EXISTS`

In `header_parser.cc:53-71`:
- If both fields are set, the code rejects this as an error
- If only `append` is set, it uses that value to determine the action
- If only `append_action` is set, it uses the enum value

**Proto default behavior:** When neither field is explicitly set, `append_action` defaults to `APPEND_IF_EXISTS_OR_ADD` (index 0 in the enum). This could cause unexpected behavior if headers are re-parsed or re-evaluated with proto message reuse.

## Filter Manager's Encode Path

When local reply headers are encoded via `FilterManager::encodeHeaders()`:
1. Filter manager iterates through encoder filters in reverse order (lines 1257-1305)
2. Each filter's `encodeHeaders()` is called
3. Header mutation filter may also evaluate and modify response headers
4. Final header validation occurs (lines 1312-1320)
5. Headers are passed to downstream codec

If `finalizeResponseHeaders()` were to be called during this phase (it's not currently, but the architecture suggests potential), headers would be triple-evaluated.

## Recommendation

**Fix Strategy:**

1. **Immediate mitigation:** Ensure `finalizeResponseHeaders()` is called exactly once per local reply, not in nested callbacks. Refactor the callback wrapping in `sendLocalReplyViaFilterChain()`, `prepareLocalReplyViaFilterChain()`, and `sendDirectLocalReply()` to:
   - Either let the router's callback handle `finalizeResponseHeaders()` entirely, OR
   - Let the filter manager handle it entirely, but not both

2. **Preferred solution:** Move the unconditional `finalizeResponseHeaders()` call out of the filter manager's wrapper callback, since:
   - The router already knows which route entry to finalize
   - Direct responses should be handled by their own `finalizeResponseHeaders()` call in the router's lambda
   - Filter manager should only orchestrate, not duplicate route-level header finalization

3. **Proto field handling:** Ensure consistent handling of the `append` and `append_action` proto fields to prevent default value surprises:
   - Consider deprecating and removing the `append` BoolValue field entirely
   - Add explicit validation that default `append_action` value is not silently used

4. **Testing:** Add regression tests that verify:
   - Headers configured with `response_headers_to_add` appear exactly once in local replies
   - All combinations of `append_action` values work correctly
   - Direct responses, cluster not found, timeout, and other error replies all behave consistently

## Diagnostic Steps

To confirm this issue in the field:
1. Enable debug logging for `source/common/router/` and `source/common/http/` components
2. Look for multiple entries in logs showing `evaluateHeaders` calls for the same request
3. Check request/response headers in access logs or via tcpdump to confirm duplication count
4. Correlate with specific route configurations using `response_headers_to_add`
5. Test with minimal config: single route with single header to add with `OVERWRITE_IF_EXISTS_OR_ADD`
