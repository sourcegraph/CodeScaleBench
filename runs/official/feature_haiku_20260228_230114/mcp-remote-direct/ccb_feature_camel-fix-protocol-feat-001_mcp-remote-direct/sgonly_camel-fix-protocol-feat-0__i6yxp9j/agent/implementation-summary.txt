COMPONENT IMPLEMENTATION DETAILS
==============================================================================

Component Scheme: fix
URI Syntax: fix:sessionID?option1=value1&option2=value2

Example Usage:
  from("fix:TRADER1?configFile=/etc/fix.cfg&senderCompID=TRADER")
    .to("log:fixmessages");

Configuration Parameters:
  - configFile (required) — Path to FIX configuration file
  - senderCompID — Sender identification
  - targetCompID — Target identification  
  - fixVersion — FIX protocol version (default: FIX.4.4)
  - heartBeatInterval — Heartbeat interval in seconds (default: 30)
  - socketConnectHost — Socket connection hostname
  - socketConnectPort — Socket connection port (default: 9898)

Message Headers:
  - CamelFixMessageType — Message type identifier
  - CamelFixSessionID — FIX session identifier
  - CamelFixSenderCompID — Sender CompID
  - CamelFixTargetCompID — Target CompID

==============================================================================
ARCHITECTURAL DECISIONS
==============================================================================

1. Engine Abstraction
   - FixEngine interface allows pluggable implementations
   - Supports: QuickFIX/J, custom implementations, mocking for tests
   - Decouples: Camel routing from FIX protocol handling

2. Async Producer
   - Extends DefaultAsyncProducer for non-blocking sends
   - Implements: boolean process(Exchange, AsyncCallback)
   - Aligns: With Camel's async model

3. Listener Pattern
   - FixMessageListener enables clean callback from engine
   - FixConsumer implements listener interface
   - Benefits: Loose coupling between engine and consumer

4. Configuration Inheritance
   - Component-level defaults copied to each endpoint
   - URI parameters override defaults
   - Pattern: Standard Camel configuration flow

5. Service Discovery
   - META-INF/services descriptor enables auto-discovery
   - Allows: Dynamic component loading without registration code
   - Aligns: With Camel's plugin architecture
