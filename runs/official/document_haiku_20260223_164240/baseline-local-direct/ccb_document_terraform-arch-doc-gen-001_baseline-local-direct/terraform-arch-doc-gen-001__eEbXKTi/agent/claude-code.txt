State Management\n\n### 3.1 State Structure and Semantics\n\nTerraform state is a JSON document containing:\n\n```json\n{\n  \"version\": 4,\n  \"terraform_version\": \"1.0.0\",\n  \"serial\": 42,\n  \"lineage\": \"abc-123\",\n  \"outputs\": {},\n  \"resources\": [\n    {\n      \"type\": \"aws_instance\",\n      \"name\": \"web\",\n      \"provider\": \"provider[\\\"aws\\\"]\",\n      \"instances\": [\n        {\n          \"index_key\": 0,\n          \"schema_version\": 2,\n          \"attributes\": {\n            \"id\": \"i-1234567890\",\n            \"ami\": \"ami-0c55b159cbfafe1f0\",\n            ...\n          },\n          \"sensitive_attributes\": [],\n          \"private\": {},\n          \"dependencies\": []\n        }\n      ]\n    }\n  ]\n}\n```\n\n**Key Semantics:**\n\n- **Serial Number**: Incremented on each write, used for remote state conflict detection\n- **Lineage ID**: Identifies state progression; Unlock State (plan doesn't modify state)\n```\n\nThe snapshot is crucial: it provides a consistent view of state during concurrent graph evaluation. Unlock State\n```\n\n**State Locking:**\n\nRemote state backends provide locking to prevent concurrent applies:\n\n- **Lock Acquisition**: At plan/apply start, attempt to acquire lock with timeout\n- **Lock Information**: Stored with:\n  - Lock ID (unique per operation)\n  - Operation ID (plan vs apply)\n  - Who locked it (user/workspace)\n  - When it was locked (timestamp)\n  - Whoami (hostname, PID)\n- **Lock Release**: On completion (success or error)\n- **Lock Timeout**: Typically 30 seconds; exceeded locks can be force-released\n- **Deadlock Prevention**: Different operations (plan vs apply) have separate lock paths\n\n### 3.4 State Synchronization During Concurrent Execution\n\n**Isolation Strategy:**\n\nDuring graph walk with concurrency, isolated state copies prevent race conditions:\n\n1. **State Snapshot**: Immutable view created at walk start\n2. **Isolated Handles**: Each node gets its own state copy\n3. **Change Accumulation**: Node writes accumulate in isolated handle\n4. **Merge on Completion**: Node completion merges changes back to mutable state\n\n**Merge Conflicts:**\n\nIf two nodes modify the same resource:\n- Later node sees version skew\n- Merge detects conflict\n- Operation either retries or reports conflict error\n\n### 3.5 Remote State Backends\n\nRemote backends provide:\n\n- **Centralized State**: Team access via HTTP API or cloud service\n- **Consistency**: Serialize writes to prevent conflicts\n- **Durability**: Backups and versioning\n- **Audit Trail**: Track who changed what and when\n\n**Common Backends:**\n\n- **S3 Backend**: State in S3, locking via DynamoDB\n- **Terraform Cloud**: Full-featured state management with runs API\n- **Azure Storage**: State in Azure blob storage with container-level locking\n- **HTTP Backend**: Generic HTTP endpoint for custom backends\n\n---\n\n## 4. Execution Flow and Hook System\n\n### 4.1 High-Level Command Flow\n\n**Command Invocation to Completion:**\n\n```\nterraform plan\n    ↓\nParse CLI Arguments\n    ↓\nLoad Configuration Files\n    ↓\nParse Configuration (HCL → AST)\n    ↓\nEvaluate Local Values and Variables\n    ↓\nBuild Resource Graph\n    ↓\nApply Graph Transformers\n    ↓\nInitialize Providers and Load Schemas\n    ↓\nWalk Graph (Plan Mode)\n    ├─ ReadResource for each resource\n    ├─ PlanResourceChange for each resource\n    ├─ Invoke Hooks (Pre/Post Node Evaluation)\n    └─ Generate Diff Output\n    ↓\nUnlock State\n    ↓\nOutput Plan Summary\n```\n\n**Apply Flow is Similar, but Walk Phase Executes Changes:**\n\n```\nWalk Graph (Apply Mode)\n    ├─ ApplyResourceChange for each resource\n    ├─ Update State\n    ├─ Invoke Hooks (Pre/Post Node Evaluation, State Updates)\n    └─ Accumulate Results\n```\n\n### 4.2 Graph Execution Engine (Walker)\n\nThe graph walker is the core execution engine. It traverses the DAG respecting dependencies while maximizing parallelism.\n\n**Walker Responsibilities:**\n\n1. **Dependency Tracking**: Maintains node completion state\n2. **Ready Node Detection**: Identifies nodes with all dependencies met\n3. **Concurrent Scheduling**: Submits ready nodes to worker pool\n4. **Completion Detection**: Signals when node finishes\n5. **Error Propagation**: Stops dependent nodes on upstream error\n\n**Walker Algorithm:**\n\n```\nInitialize:\n  - Create dependency map\n  - Mark all nodes as pending\n  - Create worker pool (default 10 workers)\n\nMain Loop:\n  While any node pending or running:\n    1. Identify ready nodes (all dependencies complete)\n    2. Submit ready nodes to worker pool (up to max concurrency)\n    3. Wait for worker completion signal\n    4. Graph Builder System\n\n### 1.1 Core Graph Architecture\n\nThe dependency graph is Terraform's fundamental execution model. It represents the entire infrastructure as a directed acyclic graph (DAG) where:\n- **Nodes** represent resources, data sources, modules, variables, outputs, and providers\n- **Edges** represent dependencies between nodes\n- **Graph properties** enable parallel execution of independent resources\n\nThe graph builder constructs this DAG in multiple phases:\n\n```\nConfiguration → Resource Graph Construction → Graph Transformations → Optimized Execution Graph\n```\n\n### 1.2 Graph Construction Phase\n\n**Primary Node Types:**\n\n- **Resource Nodes**: Represent managed resources (e.g., `aws_instance.web`). Each resource node encapsulates:\n  - The resource type and name\n  - The provider instance it depends on\n  - The configuration schema\n  - State data for diff computation\n\n- **Data Source Nodes**: Similar to resource nodes but read-only. They fetch external data without managing resources.\n\n- **Module Nodes**: Represent module invocations. They contain sub-graphs for their contained resources and recursively apply transformations.\n\n- **Provider Nodes**: Represent provider configuration blocks. They establish dependencies on provider resources and variable references.\n\n- **Root Variable/Local Nodes**: Represent input variables, locals, and their computed values.\n\n- **Output Nodes**: Represent outputs that depend on resource attributes.\n\n**Dependency Discovery:**\n\nDependencies are discovered through:\n\n1. **Explicit References**: Direct attribute references like `aws_instance.web.id` create edges from referencing nodes to referenced resources.\n\n2. **Provider Dependencies**: Every resource node implicitly depends on its provider node, ensuring providers are initialized before resources are operated on.\n\n3. **Transitive Dependencies**: Module outputs that reference resources create transitive edges through the module hierarchy.\n\n4. **Implicit Ordering**: Some dependencies are implicit (e.g., destroy operations reverse implicit creation order).\n\n### 1.3 Graph Transformers\n\nGraph transformers are middleware components that modify and enhance the graph before execution. They run sequentially and each transformer sees the output of the previous one. Key transformers include:\n\n**Resource Transformers:**\n\n- **Expand Transform**: Expands `count` and `for_each` resources into multiple concrete nodes. For example:\n  ```hcl\n  resource \"aws_instance\" \"server\" {\n    count = 3\n  }\n  ```\n  becomes three separate graph nodes: `aws_instance.server[0]`, `aws_instance.server[1]`, `aws_instance.server[2]`. The expansion is lazy—it occurs once count/for_each values are known.\n\n- **Orphan Resource Transform**: Detects resources that existed in prior state but are no longer in configuration (removed from code). These nodes are added to the graph with special handling to ensure they're destroyed.\n\n- **Provisioner Transform**: Wraps provisioner configurations around resource nodes, creating sub-graphs for provisioner execution.\n\n**Dependency Transformers:**\n\n- **Provider Config Transform**: Ensures provider configuration resources (especially `terraform_data` and similar) are available before regular resources depend on them.\n\n- **Destroy Edge Transform**: For destroy operations, reverses edges so resources are destroyed in reverse dependency order.\n\n- **Targeted Resource Transform**: When using `terraform plan -target`, the graph is pruned to only include the targeted resource and its dependents.\n\n**Validation Transformers:**\n\n- **Cycle Detection**: Detects circular dependencies and reports them as errors before execution begins.\n\n- **Schema Validation**: Ensures resource and provider schemas are available and valid.\n\n### 1.4 Module Graph Expansion\n\nModules create nested graph structures:\n\n1. **Module Call Node**: When a module is invoked, a module call node is created with edges to:\n   - All variable nodes referenced in module arguments\n   - All resources/outputs accessed from the calling context\n\n2. **Internal Module Graph**: Each module call contains a complete sub-graph of:\n   - Module variables (with edges to call arguments)\n   - Module resources\n   - Module outputs\n   - Module locals\n\n3. **Output Referencing**: When code references a module output (e.g., `module.vpc.vpc_id`), an edge is created from the referencing node to the module output node.\n\nThis hierarchical structure maintains logical separation while enabling dependency tracking across module boundaries.\n\n---\n\n## 2. Provider Interface and Lifecycle\n\n### 2.1 Provider Plugin Architecture\n\nTerraform uses a plugin-based architecture where providers are external processes communicating via gRPC. The provider configuration block (e.g., `provider \"aws\" { region = \"us-east-1\" }`) is interpolated\n   - The `Configure` method receives the provider config data\n   - Provider stores configuration internally for subsequent resource operations\n   - Takes: current state and planned change\n  - Returns: new state after operation\n  - Returns: diagnostic errors if operation fails\n\n- **ImportResourceState**: Imports existing infrastructure (when using `terraform import`)\n\n**Read-Only Operations:**\n\n- **ReadDataSource**: For data sources, fetches external data without state management\n\n### 2.4 Provider Concurrency and Instance Management\n\n**Provider Multiplexing:**\n\n- Multiple resource operations can be performed on the same provider concurrently\n- The provider plugin process handles concurrency internally\n- Terraform manages request queuing to respect provider concurrency limits\n\n**Provider Instance Reuse:**\n\n- Each unique provider configuration creates one provider instance\n- Multiple resources can share the same provider instance\n- Configuration changes create new provider instances\n\n**Error Handling in Provider Operations:**\n\nProvider operations can fail with:\n\n1. **Diagnostic Errors**: Non-fatal errors (invalid argument, resource not found)\n   - Walk Graph Nodes (potentially concurrently)\n   - Each node receives isolated state handle\n   - Writes occur to isolated handle, not live state\n   - Nodes must not see writes from other nodes\n5. Unlock State\n```\n\n**State Locking:**\n\nRemote state backends provide locking to prevent concurrent applies:\n\n- **Lock Acquisition**: At plan/apply start, attempt to acquire lock with timeout\n- **Lock Information**: Stored with:\n  - Lock ID (unique per operation)\n  - Operation ID (plan vs apply)\n  - Who locked it (user/workspace)\n  - When it was locked (timestamp)\n  - Whoami (hostname, PID)\n- **Lock Release**: On completion (success or error)\n- **Lock Timeout**: Typically 30 seconds; exceeded locks can be force-released\n- **Deadlock Prevention**: Different operations (plan vs apply) have separate lock paths\n\n### 3.4 State Synchronization During Concurrent Execution\n\n**Isolation Strategy:**\n\nDuring graph walk with concurrency, isolated state copies prevent race conditions:\n\n1. **State Snapshot**: Immutable view created at walk start\n2. **Isolated Handles**: Each node gets its own state copy\n3. **Change Accumulation**: Node writes accumulate in isolated handle\n4. **Merge on Completion**: Node completion merges changes back to mutable state\n\n**Merge Conflicts:**\n\nIf two nodes modify the same resource:\n- Later node sees version skew\n- Merge detects conflict\n- Operation either retries or reports conflict error\n\n### 3.5 Remote State Backends\n\nRemote backends provide:\n\n- **Centralized State**: Team access via HTTP API or cloud service\n- **Consistency**: Serialize writes to prevent conflicts\n- **Durability**: Backups and versioning\n- **Audit Trail**: Track who changed what and when\n\n**Common Backends:**\n\n- **S3 Backend**: State in S3, locking via DynamoDB\n- **Terraform Cloud**: Full-featured state management with runs API\n- **Azure Storage**: State in Azure blob storage with container-level locking\n- **HTTP Backend**: Generic HTTP endpoint for custom backends\n\n---\n\n## 4. Execution Flow and Hook System\n\n### 4.1 High-Level Command Flow\n\n**Command Invocation to Completion:**\n\n```\nterraform plan\n    ↓\nParse CLI Arguments\n    ↓\nLoad Configuration Files\n    ↓\nParse Configuration (HCL → AST)\n    ↓\nEvaluate Local Values and Variables\n    ↓\nBuild Resource Graph\n    ↓\nApply Graph Transformers\n    ↓\nInitialize Providers and Load Schemas\n    ↓\nWalk Graph (Plan Mode)\n    ├─ ReadResource for each resource\n    ├─ PlanResourceChange for each resource\n    ├─ Invoke Hooks (Pre/Post Node Evaluation)\n    └─ Generate Diff Output\n    ↓\nUnlock State\n    ↓\nOutput Plan Summary\n```\n\n**Apply Flow is Similar, but Walk Phase Executes Changes:**\n\n```\nWalk Graph (Apply Mode)\n    ├─ ApplyResourceChange for each resource\n    ├─ Update State\n    ├─ Invoke Hooks (Pre/Post Node Evaluation, State Updates)\n    └─ Accumulate Results\n```\n\n### 4.2 Graph Execution Engine (Walker)\n\nThe graph walker is the core execution engine. It traverses the DAG respecting dependencies while maximizing parallelism.\n\n**Walker Responsibilities:**\n\n1. **Dependency Tracking**: Maintains node completion state\n2. **Ready Node Detection**: Identifies nodes with all dependencies met\n3. **Concurrent Scheduling**: Submits ready nodes to worker pool\n4. **Completion Detection**: Signals when node finishes\n5. **Error Propagation**: Stops dependent nodes on upstream error\n\n**Walker Algorithm:**\n\n```\nInitialize:\n  - Create dependency map\n  - Mark all nodes as pending\n  - Create worker pool (default 10 workers)\n\nMain Loop:\n  While any node pending or running:\n    1. Identify ready nodes (all dependencies complete)\n    2. Submit ready nodes to worker pool (up to max concurrency)\n    3. Wait for worker completion signal\n    4. Mark node complete, update state\n    5. If node failed:\n       - Mark dependent nodes as failed\n       - Continue on other branches if not fatal\n    6. Invoke post-node hooks\n\nCompletion:\n  - All nodes processed\n  - Aggregate errors\n  - Report final status\n```\n\n**Concurrency Control:**\n\n- **Default Parallelism**: 10 concurrent operations\n- **Configurable**: `-parallelism=N` flag\n- **Per-Provider Limits**: Some providers have connection limits\n- **State Lock**: Remote backends may serialize writes despite parallelism\n\n### 4.3 Plan vs Apply Execution\n\n**Plan Mode Behavior:**\n\nDuring plan mode, each resource node:\n\n1. **Calls ReadResource**: Gets current state via provider\n2. **Compares Current vs Desired**: Evaluates configuration against current state\n3. **Calls PlanResourceChange**: Provider computes what would change\n4. **Records Diff**: Captures planned change for display and apply verification\n\nPlan never modifies state or actual resources. It only computes and displays what would change.\n\n**Apply Mode Behavior:**\n\nDuring apply mode, each resource node:\n\n1. **Reads from Prior State**: Gets state from plan (or refreshes if needed)\n2. **Calls ApplyResourceChange**: Provider executes the planned operation\n3. **Updates State**: Records new resource state\n4. **Executes Provisioners** (if any): Runs local or remote provisioners\n5. **Handles Errors**: On failure, marks resource state as tainted (partially applied)\n\n### 4.4 Dynamic Resource Expansion\n\n**Count Expansion:**\n\nWhen `count` is used:\n\n1. **Count Expression Evaluation**: `count` value is evaluated first\n2. **Instance Creation**: For each count value, create instance nodes\n3. **Dependency Tracking**: Instance `[i]` depends on count evaluation\n4. **State Management**: State maps instance indices to resource instances\n\nExample:\n```hcl\nresource \"aws_instance\" \"server\" {\n  count = var.instance_count\n}\n```\n\nGraph nodes created: `aws_instance.server[0]`, `aws_instance.server[1]`, ..., `aws_instance.server[var.instance_count-1]`\n\n**For_Each Expansion:**\n\nSimilar to count but uses map keys:\n\n```hcl\nresource \"aws_instance\" \"server\" {\n  for_each = var.instances\n}\n```\n\nGraph nodes: `aws_instance.server[\"web\"]`, `aws_instance.server[\"db\"]`, etc.\n\n**Conditional Creation:**\n\nUsing `count` or `for_each` with conditionals:\n\n```hcl\nresource \"aws_instance\" \"server\" {\n  count = var.create_server ? 1 : 0\n}\n```\n\nIf condition is false, count is 0, so no instances are created in the graph.\n\n### 4.5 Hook System\n\nHooks provide extension points throughout execution. Used for: logging, metrics, custom validation\n\n2. **PostApply**: After ApplyResourceChange completes\n   - Used for: recording changes, cleanup, notifications\n\n3. **PrePlanChange**: Before PlanResourceChange is called\n   - Used for: logging, caching invalidation\n\n4. **PostPlanChange**: After PlanResourceChange completes\n   - Used for: diff recording, policy validation\n\n5. **PreProvisionResource**: Before provisioners run\n   - Used for: setup before provisioning\n\n6. **PostProvisionResource**: After provisioners complete\n   - Used for: cleanup, validation\n\n**Hook Implementation Patterns:**\n\n- **UI Hooks**: Update terminal with progress, resource status\n- **State Hooks**: Validate state transitions, record audit trail\n- **Error Hooks**: Custom error handling, recovery attempts\n- **Policy Hooks**: Enforce policies, prevent disallowed operations\n\n### 4.6 Error Handling and Recovery\n\n**Error Classification:**\n\n1. **Configuration Errors**: Caught during parsing\n   - Invalid HCL syntax\n   - Missing required blocks\n   - Type mismatches\n\n2. **Validation Errors**: Caught before execution\n   - Missing providers\n   - Circular dependencies\n   - Schema violations\n\n3. **Provider Errors**: Caught during execution\n   - Invalid arguments to provider\n   - API failures\n   - Resource not found\n\n4. **State Errors**: Caught during state operations\n   - State lock conflicts\n   - Corruption detection\n   - Lineage mismatches\n\n**Error Propagation:**\n\n```\nProvider Error on Node A\n    ↓\nMark Node A as Failed\n    ↓\nMark Dependent Nodes (B, C, D) as Failed\n    ↓\nContinue other branches\n    ↓\nAt walk completion, aggregate all errors\n    ↓\nReport error summary with affected resources\n```\n\n**Tainted Resources:**\n\nIf a resource partially applies (e.g., creates successfully but provisioning fails):\n\n1.
