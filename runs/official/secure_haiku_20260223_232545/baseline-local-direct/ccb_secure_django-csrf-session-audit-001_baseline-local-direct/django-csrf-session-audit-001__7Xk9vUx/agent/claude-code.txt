CSRF Token Masking Scheme (Security & Cryptography)\n\n#### Design\n- **Mask generation:** Random 32-char alphanumeric string from `secrets.choice()` (cryptographically secure)\n- **Cipher operation:** XOR-like modular addition on character indices in CSRF_ALLOWED_CHARS\n- **Token structure:** 64 chars = mask (32) + cipher (32)\n  - Mask = random value\n  - Cipher[i] = CSRF_ALLOWED_CHARS[(secret[i] + mask[i]) % len(CSRF_ALLOWED_CHARS)]\n\n#### Security Properties\n- **Prevents BREACH attack:** Token appears random in HTTPS responses (different every page load due to random mask)\n- **Entropy:** 62^32 possibilities for mask alone (≈190 bits), secret independently random (≈190 bits)\n- **Pattern concealment:** Repeated secrets not visible in responses\n- **Stateless:** No server state needed for masking/unmasking\n\n#### Cryptographic Analysis\n- **Strength:** Effective against BREACH-style attacks on HTTPS with stream compression\n- **Weakness:** XOR is not cryptographically strong, but only used for masking, not encryption\n- **Modern context:** Token unmasking happens server-side only; masking mainly prevents response compression attacks\n- **Token generation:** Both secret and mask use `secrets.choice()` (✓ cryptographically secure)\n\n#### Implementation Details (django/middleware/csrf.py)\n- `_get_new_csrf_string()` (line 55-56): Uses `get_random_string(CSRF_SECRET_LENGTH, CSRF_ALLOWED_CHARS)`\n- `_mask_cipher_secret()` (line 59-68): Generates mask, performs XOR masking\n- `_unmask_cipher_token()` (line 71-81): Reverses masking via modular subtraction\n\n---\n\n### 2. Cryptographic Signing (Session Integrity)\n\n#### Design\n```\nsigning.dumps(session_dict, salt=\"django.contrib.sessions.db\", compress=True)\n  → JSON serialize → optional zlib compress → base64 encode\n  → salted_hmac() for signature → Signer.sign() appends signature\n  Result: \"base64_data:timestamp:signature\"\n```\n\n#### Cryptographic Properties\n- **HMAC algorithm:** SHA-256 (default in `signing.py` line 192)\n- **Key derivation:** `salted_hmac(key_salt, value, secret, algorithm=\"sha1\")`\n  - Key = SHA1(key_salt + SECRET_KEY) for HMAC\n  - Rationale: Derives a key deterministically from salt + master secret\n  - Prevents key reuse across different salts\n- **Message authentication:** HMAC-SHA256(derived_key, value) prevents tampering\n- **Signature verification:** `constant_time_compare()` using `secrets.compare_digest()`\n  - Prevents timing attacks on signature comparison\n\n#### Entry Point Validation\n```python\n# Session decode (django/contrib/sessions/backends/base.py:131-143)\ndef decode(self, session_data):\n    try:\n        return signing.loads(session_data, salt=self.key_salt, ...)\n    except signing.BadSignature:\n        logger.warning(\"Session data corrupted\")\n    except Exception:\n        pass\n    return {}\n```\n- **Silent failure on corruption:** Returns empty dict if signature invalid (important for robustness)\n- **Salt uniqueness:** Uses `\"django.contrib.sessions.\" + class_qualname` as salt\n- **Max age validation:** Timestamp in signature checked against max_age (session cookie age)\n\n#### Signature Format (django/core/signing.py:177-213)\n```\nSigner.sign(value):\n  value = \"data\" (base64 encoded session)\n  signature = base64_hmac(salt + \"signer\", value, key, algorithm=\"sha256\")\n  return \"data:signature\"\n\nTimestampSigner.sign(value):\n  timestamp = b62_encode(int(time.time()))  # Base62-encoded Unix timestamp\n  value = \"data:timestamp\"\n  return Signer.sign(\"data:timestamp\") → \"data:timestamp:signature\"\n```\n\n#### Constant-Time Comparison (django/utils/crypto.py:65-67)\n```python\ndef constant_time_compare(val1, val2):\n    return secrets.compare_digest(force_bytes(val1), force_bytes(val2))\n```\n- Uses `secrets.compare_digest()` (guaranteed constant-time in Python 3.3+)\n- Prevents timing-based signature forgery attacks\n- All signature comparisons in signing.py use this (line 211)\n\n---\n\n### 3. Potential Attack Scenarios and Mitigations\n\n#### A. CSRF Token Prediction Attack\n**Scenario:** Attacker tries to predict CSRF token for victim user\n\n**Mitigation Chain:**\n1. Token depends on unpredictable random 32-char secret (secrets.choice, ~190 bits entropy)\n2. Secret generated independently per-request (CSRF_COOKIE_NEEDS_UPDATE flag)\n3. Attacker cannot access secret (not in response due to masking, HttpOnly flag)\n4. Even with token, validation requires matching stored secret\n5. HTTPS+SameSite prevents cookie leakage across sites\n\n**Residual Risk:** Low. Token generation uses cryptographically secure RNG, no predictable patterns exposed.\n\n#### B. Session Fixation via Cookie Injection\n**Scenario:** Attacker sets session cookie to known value (via HTTPOnly bypass, form submission, etc.)\n\n**Mitigation Chain:**\n1. Session data signed with HMAC-SHA256 (attacker cannot forge valid signature)\n2. Signature verified on load (BadSignature caught, empty dict returned)\n3. Server-side session expiry checked (expire_date__gt=now in DB query)\n4. Session key rotation via `rotate_token()` on login\n\n**Residual Risk:** Low if:\n- SECRET_KEY kept confidential (shared with no other services)\n- Session expiry properly configured\n- Login triggers session rotation\n\n**Potential Gap:** If attacker compromises SECRET_KEY, all sessions forgeable (but this is catastrophic anyway).\n\n#### C. BREACH Attack on CSRF Token Response\n**Scenario:** Attacker compresses HTTPS response to infer token via response size\n\n**Mitigation Chain:**\n1. CSRF token masked with random 32-char mask on every render\n2. Unmasking happens server-side only (client cannot expose plaintext)\n3. Mask changes on every page load (different compressed size)\n4. SameSite cookie prevents token from being sent in attacker's request\n\n**Residual Risk:** Very low. Token masking specifically designed against this. Note: BREACH mitigation has diminished importance with modern TLS (no compression by default).\n\n#### D. Man-in-the-Middle (MITM) Downgrade on HTTP\n**Scenario:** Attacker forces HTTP connection instead of HTTPS\n\n**Mitigation Chain:**\n1. CSRF cookie: `secure=True` prevents HTTP transmission\n2. Session cookie: `secure=True` prevents HTTP transmission\n3. Origin/Referer validation: HTTPS-only sites reject HTTP Referer (line 312)\n4. Browsers: HSTS, Upgrade-Insecure-Requests directives\n\n**Residual Risk:** Medium on non-HSTS sites. Initial visit vulnerable to downgrade if no HSTS header.\n\n#### E. Cross-Site Session Hijacking via Signed Cookie Forgery\n**Scenario:** Attacker tries to forge signed session cookie\n\n**Mitigation Chain:**\n1. Session signed with HMAC-SHA256 using salted key\n2. Signature verification happens before deserialization (line 133)\n3. BadSignature exception caught, empty dict returned (no crash or data leak)\n4. Constant-time comparison prevents timing attacks\n\n**Residual Risk:** Very low if:\n- SECRET_KEY is unique per deployment\n- No side-channel attacks on timing comparison (secrets module handles this)\n\n#### F. Session Data Tampering (if using DB backend)\n**Scenario:** Attacker modifies session_data in database\n\n**Mitigation Chain:**\n1. Session_data is signed string (attacker cannot modify without signature)\n2. Signature verification (HMAC-SHA256) detects any modification\n3. BadSignature caught, session reset to empty\n\n**Residual Risk:** Low if database access controlled. If DB is compromised, all sessions forgeable anyway.\n\n#### G. CSRF Token Reuse Across Requests\n**Scenario:** Attacker captures masked token, uses it in multiple requests\n\n**Mitigation Chain:**\n1. Token validation checks request.META[\"CSRF_COOKIE\"] against POST/header token\n2. Server regenerates token on each page load (CSRF_COOKIE_NEEDS_UPDATE flag)\n3. Each token is valid only if it matches current cookie value\n4. Attacker cannot use old token after cookie rotates\n\n**Residual Risk:** Very low. Validation is per-request against current cookie.\n\n---\n\n### 6.
