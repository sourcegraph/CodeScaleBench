# Investigation Report: Duplicate Response Headers in Envoy Filter Pipeline

## Summary

Response headers configured via `response_headers_to_add` are being duplicated in certain local replies generated by the router filter. The root cause is a double invocation of `finalizeResponseHeaders()` — once by the filter manager's wrapping lambda (added in v1.35.0 to support response headers in local replies) and once by the router filter's own lambda for direct response entries.

## Root Cause

**Location**: Interaction between `source/common/router/router.cc` and `source/common/http/filter_manager.cc`

**Mechanism**:
1. The router filter's direct response path (`router.cc:486-503`) passes a lambda to `sendLocalReply()` that explicitly calls `direct_response->finalizeResponseHeaders()` (line 501)
2. The filter manager's `sendLocalReplyViaFilterChain()` method (`filter_manager.cc:1109-1130`) wraps this lambda in a new lambda that:
   - First calls `streamInfo().route()->routeEntry()->finalizeResponseHeaders()` (lines 1073-1074)
   - Then calls the router's passed-in lambda (line 1077), which internally calls `finalizeResponseHeaders()` again
3. For certain routing configurations, both calls execute successfully, causing headers from `response_headers_to_add` to be added twice

## Evidence

### Header Parser - How Headers Are Added
**File**: `source/common/router/header_parser.cc:139-213`

The `evaluateHeaders()` method in `HeaderParser` processes each configured header based on its `append_action`:
- `APPEND_IF_EXISTS_OR_ADD`: Appends to existing header or adds if absent
- `OVERWRITE_IF_EXISTS_OR_ADD`: Overwrites or adds if absent
- `ADD_IF_ABSENT`: Only adds if header doesn't exist

When `finalizeResponseHeaders()` calls `evaluateHeaders()`, it iterates through all configured headers and applies them. If called twice with the same headers map, headers with `APPEND_IF_EXISTS_OR_ADD` will be appended twice.

### Route Entry Implementation
**File**: `source/common/router/config_impl.h:628-630`

`RouteEntryImplBase` inherits from both `RouteEntryAndRoute` and `DirectResponseEntry`, making it simultaneously a route entry and direct response entry. This dual inheritance is central to the issue.

## Causal Chain

1. **Symptom**: Access logs show duplicate response headers in local replies
   - Example: `x-custom-trace: abc123` appears twice despite `OVERWRITE_IF_EXISTS_OR_ADD` config

2. **Path to Root Cause**:
   - Local reply is generated by router (timeout/error condition)
   - Router calls `callbacks_->sendLocalReply(code, body, modify_headers_lambda, ...)`
   - For direct response paths, the lambda includes `direct_response->finalizeResponseHeaders()`
   - `sendLocalReply()` delegates to filter manager's `sendLocalReplyViaFilterChain()`
   - Filter manager creates a wrapping lambda that adds another `finalizeResponseHeaders()` call
   - Both calls execute, evaluating header parsers twice
   - Headers with `APPEND_IF_EXISTS_OR_ADD` (default) get appended twice

3. **Enabling Change**: v1.35.0 added response header support for local replies
   - Added `finalizeResponseHeaders()` call to filter manager's wrapping lambda (line 1074, 1125, 1158)
   - This was intended to apply `response_headers_to_add` from route config to local replies
   - Did not account for cases where the route's lambda already calls `finalizeResponseHeaders()`

## Interaction Between Components

1. **Router Filter's Role**:
   - Determines if route has direct response entry
   - If yes, passes lambda to `sendLocalReply()` that calls `direct_response->finalizeResponseHeaders()`
   - If no (normal route), passes `modify_headers_` lambda that only adds attempt count header

2. **Filter Manager's Wrapping**:
   - Receives the router's lambda as `modify_headers` parameter
   - Creates a NEW lambda that calls `finalizeResponseHeaders()` first (if route entry exists)
   - Then calls the passed-in lambda
   - This wrapping happens for both `prepareLocalReplyViaFilterChain()` and `sendLocalReplyViaFilterChain()`

3. **Header Parser Behavior**:
   - Each call to `evaluateHeaders()` iterates through all configured headers
   - Headers with `APPEND_IF_EXISTS_OR_ADD` are appended via `headers.addReferenceKey()`
   - When called twice on same headers map, appends happen twice
   - Result: duplicated headers in the final response

## Recommendation

**Fix Strategy**: Prevent double invocation of `finalizeResponseHeaders()` by ensuring it's called exactly once per local reply response, regardless of whether the route is a direct response or regular route entry.

**Options**:
1. **Move responsibility to filter manager**: Remove `finalizeResponseHeaders()` calls from route lambdas and let filter manager handle all cases (safest approach)
2. **Route-specific handling**: Check route type in filter manager before calling `finalizeResponseHeaders()` to avoid duplication
3. **Callback deduplication**: Mark when `finalizeResponseHeaders()` has been called and skip second invocation
4. **Validate append_action semantics**: Ensure `append_action` correctly enforces `OVERWRITE_IF_EXISTS_OR_ADD` behavior even on second call

**Diagnostic Steps**:
1. Enable debug logging in `HeaderParser::evaluateHeaders()` to track each invocation
2. Add counters for `finalizeResponseHeaders()` calls per request in `StreamInfo`
3. Monitor response header counts for local replies vs upstream responses
4. Audit all code paths that call `sendLocalReply()` to identify which pass lambdas with embedded `finalizeResponseHeaders()` calls
