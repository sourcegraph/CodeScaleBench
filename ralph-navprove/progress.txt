## Codebase Patterns

### Task Structure
- Benchmark tasks live in `benchmarks/ccb_{benchmark}/{task_name}/`
- Each task has: `task.toml`, `instruction.md`, `environment/Dockerfile`, `tests/test.sh`
- Build context is `environment/` dir (NOT task root) — COPY paths in Dockerfile relative to environment/
- Harbor uploads `tests/` to `/tests/` at runtime; test.sh runs from `/tests/test.sh`
- task.toml verification: use `/tests/test.sh`, NOT `/workspace/tests/test.sh`
- Reference patches: stored in `tests/reference_fix.patch`, uploaded to `/tests/reference_fix.patch` at runtime

### Task Registration
- Register in `configs/selected_benchmark_tasks.json` with fields: benchmark, task_name, difficulty, language, sdlc_phase, category, mcp_benefit_score, mcp_breakdown, repo
- Add to relevant `configs/*_2config.sh` script's TASKS array (or ALL_TASK_IDS)
- For SG_full: add repo mapping to TASK_SG_REPO_NAMES associative array in run script
- DIR_PREFIX_TO_SUITE mapping in BOTH `scripts/aggregate_status.py` (line 51) AND `scripts/generate_manifest.py` (line 26) — MUST update both

### Run Script Pattern (2config)
- Source `_common.sh` for shared infra (token refresh, parallel execution, multi-account)
- `run_paired_configs TASK_IDS _run_single_func "$JOBS_BASE"` for paired baseline + SG_full
- `_run_single_func` signature: `task_id task_home config mcp_type jobs_base`
- All tasks use `--path` mode (not `--dataset`) for custom benchmark tasks
- `harbor run --path <task_dir> --agent-import-path <agent> --model <model> --jobs-dir <dir> -n <concurrency> --timeout-multiplier <mult>`
- SOURCEGRAPH_REPO_NAME env var set per-task for SG_full mode

### Verifier Pattern
- test.sh runs inside Docker container at `/tests/test.sh`
- Must write reward (float 0.0-1.0) to stdout OR to `/logs/verifier/reward.txt`
- Harbor reads reward from verifier stdout; some scripts also write to file for debugging
- Pattern: source shared utils, check agent output exists, score components, output final score

### Suite CLAUDE.md Pattern
- Lives at `benchmarks/ccb_{suite}/CLAUDE.md`
- Provides search strategy guidance for MCP tools (SG_full mode)
- Lists relevant MCP tools with descriptions
- Specifies output requirements and file paths
- Examples: ccb_investigation, ccb_docgen, ccb_nlqa, ccb_onboarding

### Navprove-Specific
- "Find and Prove" verification: 2-phase (test fails on buggy code, test passes after fix)
- Majority-of-3 voting: run test 3 times per phase, pass if 2/3 agree (navprove only — other suites use single-shot)
- Agent writes regression test to `/workspace/regression_test.{py,go,test.ts}`
- Reference patch applied by verifier, NEVER visible to agent
- Shared verifier template at `benchmarks/ccb_navprove/_shared/find_and_prove_verifier.sh`
- Task names follow pattern: `navprove-{repo_short}-{bug_short}-NNN`
