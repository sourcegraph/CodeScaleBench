## Codebase Patterns
- Cross-repo tasks: benchmarks/ccb_crossrepo/{task_id}/ — existing suite, add new tasks alongside existing ones
- Existing crossrepo tasks: crossrepo-etcd-*, crossrepo-numpy-* (migration/reasoning focused)
- New tasks use F1 scorer (NOT weighted checklist): agent outputs JSON, scorer computes precision+recall
- F1 scorer template at benchmarks/templates/f1_json_scorer.sh (created by infra Ralph)
- Multi-repo Dockerfile: clone all repos into single container (/workspace/repo1/, /workspace/repo2/, etc.)
- Agent output format: /workspace/callers.json or /workspace/implementors.json or /workspace/chain.json
- ground_truth.json for F1 tasks: {entries: [{repo, file, line, function, ...}], match_keys: ["repo", "file", "function"]}
- task.toml: category=symbol_resolution, language may be multi (go,cpp)
- Config: add to configs/crossrepo_2config.sh TASKS array
- MCP advantage: find_references and go_to_definition work cross-repo — this is the killer feature
- Use Sourcegraph find_references to build ground truth caller/implementor lists
- Pin all repos to specific commit SHAs — all must be indexed in Sourcegraph (sg-benchmarks org)
- IMPORTANT: istio/istio is NOT indexed on Sourcegraph — use grpc/grpc-go as alternative xDS consumer
- IMPORTANT: Sourcegraph find_references on proto files returns empty — use keyword_search + read_file to verify callers instead
- Existing crossrepo tasks use base Docker image (harbor-ccb_crossrepo:base); new F1 tasks use per-task Dockerfiles that clone repos directly
- TASK_SG_REPO_NAMES in crossrepo_2config.sh: new symbol tasks use upstream repo names (github.com/...) not sg-benchmarks forks
- ground_truth.json key_fields is "key_fields" (not "match_keys" as noted above — template uses key_fields)
