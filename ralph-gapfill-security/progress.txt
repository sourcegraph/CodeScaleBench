## Codebase Patterns
- Security tasks: benchmarks/ccb_security/{task_id}/ — new suite
- Three task subtypes: cve_triage (find vulnerable code), reachability (trace attack path), transitive_dependency (cross-repo vuln)
- CVE tasks: use ONLY CVEs with public patches (easier ground truth verification)
- Dockerfile: pin to PRE-FIX commit so vulnerable code is present
- Output format: /logs/agent/triage.md with sections: Summary, Affected Code, Attack Path, Severity Assessment, Remediation
- Scoring: weighted checklist (same as investigation tasks) via ground_truth.json
- ground_truth.json required_findings: vulnerable function(s), file path(s), vulnerability type
- ground_truth.json causal_chain: for reachability tasks, the complete call path from entry point to vulnerable function
- ground_truth.json negative_checks: must NOT blame wrong component; for NOT-reachable tasks, must NOT claim exploitable
- Transitive dependency tasks: Dockerfile clones 2 repos (vulnerable dep + consuming project)
- Pair affected/not-affected tasks for calibration (at least 1 each)
- Use web search to find CVE details, fix commits, advisory text
- Use Sourcegraph find_references to trace call paths and build ground truth
- Config: configs/security_2config.sh (created by infra Ralph)
- Fix commit lookup: use Sourcegraph commit_search with message terms (e.g., ["socks", "hostname", "remote resolve"])
- Pre-fix commit: use compare_revisions with base=fix_commit~1 to get exact parent SHA
- Verifier: test.sh is identical to investigation tasks but reads from /logs/agent/triage.md instead of /logs/agent/investigation.md
- Dockerfile base image: use debian:bookworm-slim for C projects (no Go/Python runtime needed for analysis-only tasks)
- Always chmod +x tests/test.sh

- **Learnings for future iterations:**
  - CVE-2023-39325 (HTTP/2 rapid reset) is also known as CVE-2023-44487 and was a major vulnerability affecting many HTTP/2 implementations
  - Found via web search: fixed in golang.org/x/net v0.17.0, vulnerable in v0.16.0
  - Vulnerable commit SHA verified via Sourcegraph: 88194ad8ab44a02ea952c169883c3f57db6cf9f4
  - Fix commit: b225e7ca6dde1ef5a5ae5ce922861bda011cfabd introduced scheduleHandler(), curHandlers counter, and unstartedHandlers queue
  - Root cause: processHeaders() directly spawned handler goroutines with `go sc.runHandler()` without enforcing MaxConcurrentStreams limit
  - Attack vector: rapid RST_STREAM frames free stream slots while handlers keep running, allowing unbounded goroutine accumulation
  - Ground truth includes 6 required findings (similar complexity to sec-cve-002) covering function names, vulnerability patterns, and attack mechanics
  - Dockerfile requires jq and bc for the bash-based weighted checklist verifier
  - Short SHA in git log is "88194ad" (7 chars), not "88194ad8" (8 chars) — important for Dockerfile verification grep
  - The Sourcegraph repo name for golang.org/x/net is go.googlesource.com/net (not github.com/golang/net)
  - This task provides language diversity (Go) compared to sec-cve-001 (C) and sec-cve-002 (C++)
  - Preflight validation passed for all 3 security tasks

- **Learnings for future iterations:**
  - Reachability tasks are structurally similar to CVE tasks but with different focus: given a function, trace how it's called
  - ground_truth.json format uses "patterns" not "keywords" and requires a top-level "weights" object
  - The causal_chain section in ground_truth.json represents the call stack: Curl_connect → Curl_connecthost → socks_proxy_connect → do_SOCKS5
  - MCP find_references is the key tool agents will use to trace call paths backwards from the vulnerable function
  - Negative checks are critical: prevent claiming "unreachable" when it IS reachable, prevent blaming wrong components
  - Same Dockerfile pinning pattern as CVE tasks (pre-fix commit)
  - Reachability analysis is slightly harder than CVE location (higher MCP benefit: 0.89 vs 0.85-0.88) because it requires deeper code understanding
