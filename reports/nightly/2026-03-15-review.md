## 1. Code & Architecture Review

### 1.1 HIGH: Skills Directory Uses Wrong Repository Path (54 Occurrences)

All 25 skill files in `skills/` reference `~/CodeScaleBench` as the
working directory. The actual on-disk repository path is
`~/CodeContextBench` (the GitHub remote redirects `CodeContextBench.git`
to `CodeScaleBench.git`, but the local clone uses the original name).

Examples:

**`skills/check-infra/SKILL.md`** line 25:
```bash
cd ~/CodeScaleBench && python3 scripts/check_infra.py
```

**`skills/validate-tasks/SKILL.md`** line 27:
```bash
cd ~/CodeScaleBench && python3 scripts/validate_tasks_preflight.py
```

54 total occurrences across 25 files. Every skill that includes
executable commands will fail with `cd: no such file or directory`
unless a symlink exists from `~/CodeScaleBench` to `~/CodeContextBench`.

**Impact**: Any agent (Claude Code, Cursor, Copilot) that executes
skill commands verbatim will fail on every single skill invocation.
Skills are the project's primary interface for AI agent automation --
having 100% of them reference a non-existent path renders the entire
skills system non-functional for new environments.

**Fix**: Bulk replace across all 25 files:
```bash
cd ~/CodeScaleBench -> cd ~/CodeContextBench
```
Or, more robustly, use a variable:
```bash
CSB_ROOT="${CSB_ROOT:-$(git rev-parse --show-toplevel)}"
cd "$CSB_ROOT"
```

### 1.2 HIGH: Skills Reference Legacy Config Name `sourcegraph_full` (21 Occurrences)

14 skill files reference the legacy config name `sourcegraph_full`
which was replaced by `mcp-remote-direct` during the 4-config
normalization. The README, `CONFIGS.md`, and `export_official_results.py`
all use the current naming (`baseline-local-direct`, `mcp-remote-direct`,
etc.), but skills were never updated.

Examples:

**`skills/run-benchmark/SKILL.md`** lines 33-35:
```markdown
- Default execution is paired by task: `baseline` + `sourcegraph_full`.
- Single-lane runs are gap-fill only:
  - `--baseline-only` requires valid existing `sourcegraph_full` counterpart runs.
```

**`skills/quick-rerun/SKILL.md`** line 73:
```bash
BASELINE_MCP_TYPE=sourcegraph_full harbor run \
```

This is additionally wrong because `BASELINE_MCP_TYPE` accepts
`none`, `sourcegraph`, or `deepsearch` -- not `sourcegraph_full`
(which is a config label, not an MCP type value).

**`skills/csb/analysis.md`** line 51:
```markdown
2. Pairs baseline vs sourcegraph_full tasks
```

**Impact**: Operators following skill instructions use incorrect
config names for filtering (`--config sourcegraph_full`) which
silently returns zero results since no runs use that config name
anymore. The `BASELINE_MCP_TYPE=sourcegraph_full` command sets an
invalid MCP type, causing the agent to run without MCP tools (falling
back to the `none` default) while the operator believes MCP is enabled.

**Fix**: Replace all 21 occurrences:
- `sourcegraph_full` -> `mcp-remote-direct` (in config filter contexts)
- `BASELINE_MCP_TYPE=sourcegraph_full` -> `BASELINE_MCP_TYPE=sourcegraph`
  (in harbor run contexts)
- `baseline` -> `baseline-local-direct` (in config filter contexts
  where specificity matters)

### 1.3 HIGH: Zero Dependency Management Files

The repository has no `requirements.txt`, `pyproject.toml`, `setup.py`,
or `setup.cfg`. The `scripts/` directory contains 200+ Python files
(200 in `scripts/`, 12 in `scripts/csb_metrics/`) plus 9 test files.

The README states "Stdlib only. Compatible with Python 3.10+" for the
core pipeline, but several scripts import non-stdlib packages:

- `tests/test_judge_engine.py` imports from `csb_metrics.judge` which
  uses the `anthropic` SDK
- `scripts/plot_*.py` files use `matplotlib` and `pandas`
- `.github/workflows/roam.yml` line 18 installs `roam-code` with no
  version pin

Without dependency management:
1. Contributors cannot reproduce the development environment
2. CI workflows rely on scripts being stdlib-only but have no mechanism
   to enforce this constraint
3. The `csb_metrics` package requires `sys.path.insert(0, ...)` hacks
   in every consumer instead of a standard `pip install -e .`
4. No mechanism exists to pin transitive dependency versions

**Impact**: New contributors attempting to run tests or analysis scripts
have no way to install dependencies. The `csb_metrics` package is
consumed by 200+ scripts via path manipulation rather than standard
Python packaging, making the codebase fragile to directory restructuring.

**Fix**: Create a minimal `pyproject.toml`:
```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "csb-metrics"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = []  # stdlib only for core

[project.optional-dependencies]
judge = ["anthropic>=0.40"]
plots = ["matplotlib", "pandas"]
dev = ["pytest>=8.0"]
```

### 1.4 MEDIUM: CI Workflows Use Three Different Python Versions

Four CI workflows each specify a different Python version:

| Workflow | Python Version | Purpose |
|----------|---------------|---------|
| `repo_health.yml` | 3.10 | CI gate (repo health checks) |
| `docs-consistency.yml` | 3.11 | CI gate (docs drift detection) |
| `roam.yml` | 3.12 | PR analysis (roam-code) |
| `task_smoke_matrix.yml` | 3.10 | Benchmark smoke tests |

The project claims "Compatible with Python 3.10+" but tests under
three different runtimes. If any script uses syntax introduced in 3.11+
(e.g., `except*` syntax, `tomllib`), it will pass in `docs-consistency`
CI but fail in `repo_health` CI. Conversely, 3.10-specific behavior
tested in `repo_health` may not be tested at all in `docs-consistency`.

**Impact**: Inconsistent CI results depending on which workflow catches
an issue first. A PR could pass `repo_health.yml` (Python 3.10) but
fail `docs-consistency.yml` (Python 3.11) or vice versa if there are
version-sensitive behaviors. The lack of a single declared Python
version means the project's compatibility claim is untested.

**Fix**: Standardize all workflows to Python 3.10 (the minimum
supported version) to ensure compatibility is actually tested at the
floor. If 3.12 features are needed for `roam-code`, document that
as a separate requirement.

### 2.2 Test Infrastructure Lacks Standard Python Packaging

All 9 test files use the same `sys.path.insert` hack:

```python
sys.path.insert(0, str(Path(__file__).resolve().parent.parent / "scripts"))
```

This pattern:
- Depends on directory layout rather than package installation
- Breaks if tests are run from a different working directory
- Cannot be replaced by a `conftest.py` because imports happen at
  module level before pytest fixtures are initialized
- Makes IDE navigation and type checking unreliable

There is also no `conftest.py` for shared test fixtures, no
`pytest.ini` or `pyproject.toml` `[tool.pytest]` section for test
configuration, and no test markers for categorizing fast vs slow tests.

**Improvement**: Create a `pyproject.toml` with installable package
(Section 1.3) and replace `sys.path.insert` with standard imports.
Add `conftest.py` with shared fixtures for common patterns (temp
directories, mock task data, etc.).

### 2.3 Skills Should Use Relative Paths or Git Root Detection

The `~/CodeScaleBench` hardcoded path (Section 1.1) is fragile even
after a bulk rename -- it assumes a specific home directory location.
Skills should detect the repository root dynamically.

**Improvement**: Add a preamble pattern for all skills:

```bash
# At the top of each skill's command blocks:
CSB_ROOT="$(git rev-parse --show-toplevel 2>/dev/null || echo ~/CodeContextBench)"
cd "$CSB_ROOT"
```

Or create a `skills/_common.sh` that skills source for shared setup.

### 2.4 Unscored Tasks Conflated with Zero-Reward Tasks

In `scripts/aggregate_status.py` lines 373-376:

```python
if reward is not None and reward > 0:
    record["status"] = "completed_pass"
else:
    record["status"] = "completed_fail"
```

When `reward is None` (task not scored -- e.g., verifier didn't run),
the status is set to `completed_fail`. This is indistinguishable from
`reward == 0.0` (task scored, agent produced no correct output). The
same conflation occurs in `generate_manifest.py` line 679 and
`mcp_audit.py` line 346.

**Improvement**: Add a `completed_unscored` status for tasks where
`reward is None` and no exception occurred:

```python
if reward is not None and reward > 0:
    record["status"] = "completed_pass"
elif reward is not None:
    record["status"] = "completed_fail"
else:
    record["status"] = "completed_unscored"
```

This allows operators to distinguish "task failed to score" (needs
investigation) from "task scored zero" (agent didn't solve it).

## 3. Research Recommendations

### 3.1 Introduce `pyproject.toml` and Proper Python Packaging

The absence of any dependency management file is the most significant
infrastructure gap remaining. A `pyproject.toml` provides:

1. **Reproducible environments**: `pip install -e ".[dev]"` sets up
   everything a contributor needs
2. **Standard imports**: Eliminates `sys.path.insert` hacks in all
   200+ scripts and 9 test files
3. **Dependency pinning**: Prevents unpinned packages (like `roam-code`)
   from breaking CI
4. **Test configuration**: Centralizes pytest settings, markers, and
   coverage requirements

Recommended structure:
```toml
[project]
name = "codescalebench"
version = "0.1.0"
requires-python = ">=3.10"
dependencies = []

[project.optional-dependencies]
judge = ["anthropic>=0.40"]
plots = ["matplotlib>=3.8", "pandas>=2.0"]
dev = ["pytest>=8.0", "pytest-cov"]

[tool.pytest.ini_options]
testpaths = ["tests"]
addopts = "--tb=short -q"
```

### 3.2 Skills Linting and Validation

With 25 skill files containing 54 stale path references, 21 stale
config names, and 3 stale model IDs, the skills system needs automated
validation. Add a `skills_integrity` check to `repo_health.py`:

```python
def check_skills_integrity(repo_root: Path) -> int:
    errors = 0
    for skill_file in (repo_root / "skills").rglob("*.md"):
        text = skill_file.read_text()

        # Check for stale repository paths
        if "~/CodeScaleBench" in text:
            print(f"  {skill_file}: uses stale path ~/CodeScaleBench")
            errors += 1

        # Check for legacy config names
        if "sourcegraph_full" in text:
            print(f"  {skill_file}: uses legacy config 'sourcegraph_full'")
            errors += 1

        # Check for deprecated model IDs
        for model in ["claude-opus-4-5-20251101", "claude-sonnet-4-5"]:
            if model in text:
                print(f"  {skill_file}: uses deprecated model {model}")
                errors += 1

    return errors
```

This catches drift automatically on every push, preventing the
skills system from silently falling behind the codebase.

### 3.3 Standardize CI Python Version via Matrix or Reusable Workflow

Instead of each workflow independently declaring a Python version,
use a reusable workflow or shared configuration:

```yaml
# .github/workflows/_setup.yml (reusable)
on:
  workflow_call:
    inputs:
      python-version:
        default: "3.10"
        type: string

jobs:
  setup:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ inputs.python-version }}
```

This ensures all CI workflows use the same Python version unless
explicitly overridden, and makes the minimum supported version visible
in one place.

### 3.4 Schema Consumer Registry

The March 14 report recommended schema synchronization via code
generation. This report adds a complementary recommendation: maintain
a consumer registry for each schema. Currently, 8 schema files exist
with zero automated consumers. Adding a comment block to each schema:

```json
{
  "$comment": "Consumers: generate_eval_report.py, export_official_results.py",
  ...
}
```

Combined with the `schema_integrity` check proposed in the March 14
report, this creates accountability: if a schema has no consumers,
it should be either connected to code or archived.
