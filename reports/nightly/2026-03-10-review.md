## 1. Code & Architecture Review

### 1.1 CRITICAL: All 275 Unified CSB Tasks Have Mismatched TARGET_SUITE

The unified benchmark (`benchmarks/csb/`) merges 20 legacy suites into
9 thematic suites (e.g., `csb_crossrepo`, `csb_security`, `csb_debug`).
However, the `test.sh` and `eval.sh` files inside each task still use
the **old un-merged suite names** as `TARGET_SUITE`:

| CSB Suite | Expected TARGET_SUITE | Actual TARGET_SUITE in test.sh |
|-----------|----------------------|-------------------------------|
| `csb_crossrepo` (47 tasks) | `csb_crossrepo` | `csb_sdlc_design` (12), `csb_sdlc_test` (7), not set (28) |
| `csb_security` (39 tasks) | `csb_security` | `csb_sdlc_secure` (10), not set (29) |
| `csb_refactor` (43 tasks) | `csb_refactor` | `csb_sdlc_refactor` (9), not set (34) |
| `csb_understand` (44 tasks) | `csb_understand` | `csb_sdlc_design` (5), not set (39) |
| `csb_feature` (34 tasks) | `csb_feature` | not set (29), various (5) |
| `csb_debug` (26 tasks) | `csb_debug` | `csb_sdlc_debug` (7), not set (19) |
| `csb_fix` (19 tasks) | `csb_fix` | not set (19) |
| `csb_document` (11 tasks) | `csb_document` | not set (11) |
| `csb_test` (12 tasks) | `csb_test` | not set (12) |

**Impact**: `promoted_verifier.py` uses `SUITE_WEIGHTS` keyed by
`TARGET_SUITE` to compute weighted oracle scores. Mismatched suite names
cause weight lookup failures for 55 tasks with stale values, and the 220
tasks without `TARGET_SUITE` defined bypass promoted verification
entirely, falling through to equal-weight `oracle_checks.py`. Scoring
across the unified benchmark is systematically inconsistent.

**Root cause**: `scripts/scaffold_csb_unified.py` copied task files from
legacy suites verbatim without updating the hardcoded `TARGET_SUITE`
values in test/eval scripts.

### 1.2 CRITICAL: `dual_score_lib.sh` Has Dead Code and Data Loss Bug

`benchmarks/csb/*/tests/dual_score_lib.sh` (identical across all 275
tasks) lines 205-211:

```python
vr.setdefault("dual_score", {})          # Line 205: dead code
vr["dual_score"] = {                     # Line 206: overwrites above
    "reward_direct": round(reward_direct, 4),
    "reward_artifact": round(reward_artifact, 4),
    "scorer_direct": "primary_verifier",
    "scorer_artifact": vr.get("dual_score", {}).get("scorer_artifact", "auto"),
}
```

Three issues:
1. Line 205 `setdefault()` is immediately overwritten by line 206's
   unconditional assignment -- pure dead code
2. Line 206 replaces the dict that line 210 tries to read from -- the
   `.get("dual_score", {}).get("scorer_artifact")` on line 210 reads
   from the **new empty dict** (because line 206 already assigned a new
   dict to `vr["dual_score"]`), so `scorer_artifact` is always `"auto"`
3. Any `scorer_artifact` value set by the artifact scoring pipeline
   (e.g., `"promoted_verifier"` or `"oracle_checks"`) is silently lost

**Impact**: The `scorer_artifact` field in `validation_result.json`
never records which scorer actually produced the artifact score. All 275
tasks report `"auto"` regardless of the actual scoring path used. This
makes it impossible to audit which scoring strategy produced each result.

**Fix**: Read `scorer_artifact` before overwriting:
```python
prev_scorer = vr.get("dual_score", {}).get("scorer_artifact", "auto")
vr["dual_score"] = {
    ...
    "scorer_artifact": prev_scorer,
}
```

### 1.3 HIGH: Judge Rubric `max_score=0` Falsy Check Produces Wrong Weights

`scripts/csb_metrics/judge/engine.py` line 115:

```python
max_score = float(c.get("max_score", 1) or 1)
```

If a criterion has `max_score: 0` (a valid value meaning "informational
only, not scored"), the `or 1` fallback activates because `0` is falsy.
The criterion gets `max_score=1` instead of `0`, and line 124-125's
normalization (`raw_score / max_score`) produces a non-zero normalized
score for what should be an unscored criterion.

**Impact**: Criteria intended as informational inflate rubric scores.
While no current `criteria.json` files use `max_score: 0` (the hybrid
scoring feature was noted as inert in the March 7 report), any future
activation of hybrid scoring will hit this bug.

**Fix**: `max_score = float(c.get("max_score") if c.get("max_score") is not None else 1)`

### 1.4 HIGH: Quarantine Script Has Inverted None Logic

`scripts/quarantine_invalid_tasks.py` line 43:

```python
if (t.get("mcp_total_calls") or 0) != 0:
    continue
```

The intent is: skip tasks that have non-zero MCP calls (they're valid).
But when `mcp_total_calls` is `None` (metric extraction failed), `None
or 0` evaluates to `0`, and `0 != 0` is `False`, so the task is NOT
skipped. It proceeds to quarantine. This means tasks where MCP metrics
were never extracted (e.g., crashed before transcript post-processing)
are incorrectly quarantined as "rate-limited."

**Impact**: Legitimate failures (crashes, sandbox timeouts) with missing
MCP metrics get quarantined alongside genuinely rate-limited results.
The March 10 commit `8eb1ff7ff` quarantined 31 results -- some may be
false positives from this bug.

**Fix**: Explicitly handle None:
```python
mcp_calls = t.get("mcp_total_calls")
if mcp_calls is not None and mcp_calls != 0:
    continue
```

### 1.6 HIGH: `promote_run.py` Crashes on Non-Dict Environment Config

`scripts/promote_run.py` line 312:

```python
env_kwargs = ((data.get("environment") or {}).get("kwargs") or {})
```

If `data["environment"]` is a non-dict truthy value (e.g., a string
like `"daytona"` from a simplified config), `.get("kwargs")` raises
`AttributeError`. The `or {}` fallback only triggers for falsy values,
not type mismatches.

**Impact**: Run promotion crashes with an unhandled `AttributeError`
when Harbor config.json uses a simplified environment format. This
blocks the staging-to-official promotion pipeline.

**Fix**: Add type guard:
```python
env = data.get("environment")
env_kwargs = env.get("kwargs", {}) if isinstance(env, dict) else {}
```

### 1.9 MEDIUM: Silent Error Suppression in Oracle Discovery

`scripts/csb_metrics/judge/oracle.py` lines 564, 612, 624, 642 all
use bare exception handlers:

```python
except (OSError, json.JSONDecodeError):
    pass
```

When oracle files fail to parse (malformed JSON, encoding errors), the
failure is completely silent -- no logging, no counters. If 50% of
oracle files had syntax errors, the judge would silently degrade to
lower-confidence evaluation for those tasks with no observable signal.

**Impact**: Oracle data quality issues (like the 66 incomplete oracle
files from the March 9 report) cannot be detected during judge runs.
Add `logging.warning()` calls to these handlers.

### 1.10 MEDIUM: Judge Discovery Loads Results Without Schema Validation

`scripts/csb_metrics/discovery.py` lines 250-256 load judge result JSON

## 2. Feature & UX Improvements

### 2.1 No Runner or Config for the Unified CSB Benchmark

The unified CSB benchmark (`benchmarks/csb/`) has:
- Task definitions: 275 tasks across 9 suites
- Task selection: `configs/selected_csb_tasks.json`
- Score library: `dual_score_lib.sh` in every task

But it is missing:
- A runner config script (equivalent to `configs/run_selected_tasks.sh`)
- Integration with `aggregate_status.py` (which doesn't recognize
  `csb_*` suite names)
- Integration with `generate_eval_report.py` (which expects the legacy
  20-suite taxonomy)
- Integration with `compare_configs.py`

The benchmark data layer is complete but the entire operational pipeline
is disconnected. Running the unified benchmark currently requires
manually adapting the legacy runner scripts.

**Improvement**: Create `configs/run_csb_unified.sh` that reads
`selected_csb_tasks.json` and maps to the `benchmarks/csb/` task tree.
Update `aggregate_status.py` and `generate_eval_report.py` to recognize
the 9-suite taxonomy alongside the legacy 20-suite names.

### 2.2 Oracle Confidence Decoupled from Discovery Quality

`scripts/csb_metrics/judge/engine.py` lines 243 and 312:

```python
oracle_confidence="high" if (judge_input.oracle_ground_truth or "").strip() else "low",
```

Oracle confidence is set to `"high"` whenever ground truth text exists,
regardless of how it was discovered. The oracle discovery pipeline in
`oracle.py` computes actual confidence levels (`"high"`, `"medium"`,
`"low"`) based on the discovery strategy (structured GT file vs.
instruction.md extraction vs. heuristic). But `engine.py` ignores this
field entirely.

**Impact**: A `"high"` confidence judge score may be based on ground
truth extracted from unstructured text (genuinely `"low"` confidence).
The `oracle_confidence` field in judge results is misleading.

**Improvement**: Pass `oracle_bundle.confidence` through `JudgeInput`
to `JudgeResult` instead of recomputing it from text presence.

### 2.3 No Unified Test Configuration

The test suite (212 tests across 9 files) has:
- No `pyproject.toml` with pytest configuration (first noted March 6)
- Mixed test frameworks: `pytest` style (abc_audit, criteria, score) and
  `unittest.TestCase` style (oracle_curation, oracle_checks, judge)
- No test markers for categorization (unit vs integration)
- 210 passing, 2 failing (Section 1.11)

Beyond the missing config file (March 6), the framework mismatch is a
new finding. The two styles have different assertion patterns, fixture
mechanisms, and discovery rules. Standardizing on pytest (which can run
unittest tests) with markers (`@pytest.mark.unit`,
`@pytest.mark.integration`) would improve CI ergonomics.

### 2.4 Judge Prompt Template Naming Inconsistency

`scripts/csb_metrics/judge/prompts.py` uses `{reference_answer}` as the
placeholder name, while the data model (`JudgeInput`) calls the same
field `oracle_ground_truth`. The mapping happens in `engine.py`:

```python
reference_answer=judge_input.oracle_ground_truth or "(none provided)"
```

This naming split makes it unclear which variable represents what.
Maintainers updating oracle_ground_truth may not realize it feeds the
reference_answer placeholder.

**Improvement**: Rename the prompt placeholder to `{oracle_ground_truth}`
for consistency with the data model.

## 3. Research Recommendations

### 3.1 Parameterize TARGET_SUITE Instead of Hardcoding

The root cause of Section 1.1 is that `TARGET_SUITE` is hardcoded in
each task's test.sh/eval.sh. When suites are reorganized (legacy -> CSB
unified), every task file must be manually updated.

Instead, derive `TARGET_SUITE` from the task's filesystem location at
runtime:

```bash
# In dual_score_lib.sh:
TASK_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
TARGET_SUITE="csb_$(basename "$(dirname "$TASK_DIR")")"
```

This eliminates the hardcoded value entirely and makes the benchmark
robust to future suite reorganizations. Update `dual_score_lib.sh` once
and all 275 tasks benefit.

### 3.2 Add Suite Taxonomy Registry

The project has three competing sources of suite taxonomy:
1. `configs/selected_benchmark_tasks.json` (20 legacy suites)
2. `configs/selected_csb_tasks.json` (9 unified suites)
3. Hardcoded lists in `discovery.py`, `task_selection.py`,
   `aggregate_status.py`, etc. (8+ copies, each incomplete)

Create a single `configs/suite_taxonomy.json`:

```json
{
  "legacy": {
    "csb_sdlc_fix": {"category": "sdlc", "phase": "fix"},
    "csb_org_security": {"category": "org", "phase": "security"},
    ...
  },
  "unified": {
    "csb_fix": {"merges": ["csb_sdlc_fix"]},
    "csb_security": {"merges": ["csb_sdlc_secure", "csb_org_security", "csb_org_compliance"]},
    ...
  }
}
```

All scripts import from this registry. Adding a new suite or
reorganizing suites becomes a single-file change instead of an 8+ file
update.

### 3.3 Add Scorer Attribution to Validation Results

The `dual_score_lib.sh` bug (Section 1.2) exposes a design gap: there
is no reliable way to audit which scoring strategy produced each task's
artifact score. Even after fixing the bug, the attribution chain is
fragile because it depends on Python dict state within a bash heredoc.

Add explicit scorer logging:

```python
# In each scoring script, write a scorer_log.json:
{"scorer": "promoted_verifier", "suite_weights": {...}, "timestamp": "..."}
```

Then `dual_score_lib.sh` reads this file instead of relying on dict
state:

```python
scorer_log = json.loads(open("/logs/verifier/scorer_log.json").read())
scorer_artifact = scorer_log.get("scorer", "unknown")
```

### 3.4 Add Oracle Discovery Metrics

The silent exception suppression in oracle.py (Section 1.9) should be
replaced with structured metrics. After a judge run, emit:

```json
{
  "oracle_discovery_stats": {
    "total_tasks": 275,
    "strategy_used": {
      "structured_gt": 145,
      "instruction_extraction": 89,
      "heuristic_fallback": 41
    },
    "parse_failures": 12,
    "missing_fields": {"text": 66, "symbols": 7}
  }
}
```

This makes oracle data quality observable without manually scanning
oracle files. The same metrics can feed a dashboard or CI check.

### 3.5 Validate Unified Benchmark Before First Production Run

Before using the CSB unified benchmark for official results, run a
validation pass:

```bash
# 1. Verify all 275 tasks are discoverable
python3 scripts/validate_tasks_preflight.py --suite-dir benchmarks/csb

# 2. Verify dual_score_lib.sh is present and identical across all tasks
find benchmarks/csb -name dual_score_lib.sh -exec sha256sum {} \;

# 3. Verify TARGET_SUITE alignment (after fix)
grep -rn "TARGET_SUITE=" benchmarks/csb/*/tests/*.sh | \
  awk -F= '{print $2}' | sort | uniq -c | sort -rn

# 4. Dry-run artifact scoring on sample tasks
for task in benchmarks/csb/crossrepo/ccx-config-trace-010 \
            benchmarks/csb/fix/ansible-abc-imports-fix-001 \
            benchmarks/csb/feature/bustub-hyperloglog-impl-001; do
  echo "--- $task ---"
  bash $task/tests/test.sh --dry-run 2>&1 | head -5
done
```

benchmark data integrity (fixing oracle completeness). This report
recommends **unified benchmark operational readiness** because the CSB
