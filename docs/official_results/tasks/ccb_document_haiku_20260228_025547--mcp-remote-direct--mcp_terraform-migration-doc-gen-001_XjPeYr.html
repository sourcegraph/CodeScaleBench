Marks are now combined from all values within the expression to ensure consistency in results and avoid losing sensitive information.\n\n**Key behavior change**: In v1.9, sensitive marks could be lost in certain combinations of conditional expressions and unknown values. In v1.10, marks are preserved more accurately.\n\n#### Impact\n- **Improved accuracy**: Your configuration validation may now correctly identify more sensitive values\n- **Potential visibility changes**: You may see outputs marked as sensitive where they weren't in v1.9\n- **Breaking for some workflows**: If you relied on conditional expressions that \"hid\" sensitive marks, you'll see different behavior\n\n#### Before (v1.9.0)\n```hcl\nvariable \"database_password\" {\n  type      = string\n  sensitive = true\n}\n\nvariable \"include_password\" {\n  type    = bool\n  default = false\n}\n\n# In v1.9, this could lose the sensitive mark in certain conditions\nlocals {\n  config = var.include_password ? {\n    password = var.database_password\n    user     = \"admin\"\n  } : {\n    password = null\n    user     = \"admin\"\n  }\n}\n\n# Output could incorrectly show as non-sensitive\noutput \"config\" {\n  value     = local.config\n  sensitive = false  # This contradicts the actual sensitivity\n}\n```\n\n#### After (v1.10.0)\n```hcl\nvariable \"database_password\" {\n  type      = string\n  sensitive = true\n}\n\nvariable \"include_password\" {\n  type    = bool\n  default = false\n}\n\n# In v1.10, the sensitive mark is properly preserved through the conditional\nlocals {\n  config = var.include_password ? {\n    password = var.database_password\n    user     = \"admin\"\n  } : {\n    password = null\n    user     = \"admin\"\n  }\n}\n\n# Output must now declare it as sensitive OR use nonsensitive() to explicitly remove marks\noutput \"config\" {\n  value = var.include_password ? local.config : {\n    password = null\n    user     = \"admin\"\n  }\n  sensitive = var.include_password  # Now correctly marks as sensitive when needed\n}\n```\n\n#### Migration Steps\n1. **Run `terraform plan`** and check for warnings about sensitive values in outputs\n2. **Review outputs** that should have marked sensitive data:\n   - Add `sensitive = true` to the output declaration if the output may contain sensitive data\n   - OR use `nonsensitive()` function if you explicitly want to expose non-sensitive parts\n3. **Use the `nonsensitive()` function** to selectively remove sensitive marks:\n```hcl\n# This removes the sensitive mark while preserving non-sensitive parts\noutput \"non_sensitive_config\" {\n  value = nonsensitive(local.config.user)\n}\n```\n\n4. **For complex expressions**, ensure sensitive marks are handled explicitly:\n```hcl\n# When mixing sensitive and non-sensitive in conditionals\nlocals {\n  # Result will be sensitive if any branch uses sensitive data\n  value = var.use_secret ? var.secret_value : \"default_public_value\"\n}\n```\n\n5. **Run `terraform plan` again** to verify mark handling is correct\n\n#### Using `nonsensitive()` to Override Marks\n```hcl\nvariable \"api_key\" {\n  type      = string\n  sensitive = true\n}\n\n# You can remove the sensitive mark explicitly if needed\nlocals {\n  config = {\n    key = nonsensitive(var.api_key)  # Explicitly strip sensitive mark\n  }\n}\n```\n\n#### Common Pitfalls\n- \u274c Not updating output blocks that now show warnings about missing `sensitive` declaration\n- \u274c Using `nonsensitive()` to hide sensitive values (they're still sensitive in state)\n- \u274c Assuming marks automatically propagate without declaring `sensitive = true` on outputs\n- \u274c Mixing sensitive and non-sensitive values without explicitly handling the result\n\n#### When Sensitive Marks Are Preserved\nMarks are combined from ALL values in a conditional expression:\n```hcl\n# Result is always sensitive (combines both branches)\nresult = condition ?
