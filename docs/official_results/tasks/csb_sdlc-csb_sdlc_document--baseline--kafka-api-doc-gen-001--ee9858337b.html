# Task: Generate API Reference Documentation for Kafka KafkaConsumer

## Objective

Generate comprehensive API reference documentation for the Apache Kafka `KafkaConsumer` Java API. The documentation should cover the complete API surface with emphasis on behavioral semantics, offset management strategies, rebalancing mechanics, and error handling patterns.

## Scope

Your documentation should cover the **KafkaConsumer API** from the `org.apache.kafka.clients.consumer` package, including:

1. **Core Consumer Lifecycle**
   - Constructor variants and configuration
   - Subscription methods (dynamic and manual assignment)
   - Polling and data fetching semantics
   - Resource cleanup and closing

2. **Offset Management**
   - Synchronous and asynchronous commit strategies
   - Offset queries and position control
   - Seek operations and offset discovery
   - Committed offset semantics

3. **Consumer Group Mechanics**
   - ConsumerRebalanceListener interface and callbacks
   - Rebalance triggers and timing
   - Group membership and heartbeat behavior
   - Partition assignment vs subscription models

4. **Flow Control and Position Management**
   - Pause and resume functionality
   - Position queries and manipulation
   - Offset-to-timestamp lookups
   - Beginning and end offset discovery

5. **Error Handling**
   - Exception types and recovery strategies
   - CommitFailedException and group fencing
   - WakeupException and thread interruption
   - Timeout and authentication errors

## Requirements

### API Methods Documentation (40%)

Document all public methods of the `KafkaConsumer` class with:
- Method signatures including all overloads
- Parameter semantics and validation rules
- Return types and their meanings
- Exception types thrown and conditions

### Behavioral Notes (30%)

- **poll() blocking behavior**: when it returns immediately vs when it blocks, timeout handling, rebalance callback execution during poll
- **Offset commit semantics**: difference between sync/async commits, retry behavior, commit failure handling

- **Thread safety**: which methods are thread-safe, wakeup() special case, event loop model
- **Group membership**: max.poll.interval.ms enforcement, proactive leave behavior, session timeout vs poll timeout
- **Manual vs dynamic assignment**: mutually exclusive nature, use cases for each model
- **Position vs committed offset**: the off-by-one relationship ("committed should be next offset to read")
- **Transactional semantics**: read_committed isolation level, LSO boundary, filtered messages

- **Multi-threaded processing pattern**: single consumer thread with pause/resume coordination and worker pool

## Overview

### Threading Model

The Kafka KafkaConsumer is **NOT thread-safe**. All consumer operations must occur within a single consumer thread. The only exception is `wakeup()`, which is explicitly designed to be called from external threads to interrupt blocking operations.

- `KafkaConsumer` maintains an internal event loop that processes rebalances, heartbeats, and data fetches

timeout)\n```\n\n**Semantics:**\n- Fetches records from assigned partitions\n- Returns immediately if records are available\n- Blocks for up to `timeout` duration if no records available\n- **May block longer than timeout** if a rebalance is triggered\n- Executes rebalance callbacks if rebalancing occurs\n- Sends heartbeats to maintain group membership\n- Automatically refreshes metadata when needed\n- Can throw `WakeupException` if `wakeup()` called\n\n**Return Value:**\n- `ConsumerRecords` containing fetched records partitioned by `TopicPartition`\n- Empty `ConsumerRecords` if no data available (timeout expired, or empty assignment)\n- Records from paused partitions are excluded\n\n**Important Behaviors:**\n\n1. **Rebalance Callbacks During Poll**\n   - If a rebalance is detected, callbacks execute **within the `poll()` call**\n   - Rebalance can occur at any point during poll processing\n   - Callbacks may block arbitrary durations\n   - `poll()` only returns after rebalance completes\n\n2. **Max Records Limiting**\n   - `max.poll.records` configuration limits the batch size\n   - Useful for predictable processing time windows\n   - Default is 500 records\n\n3. **Offset Advancement**\n   - Position advances as records are returned\n   - Position = offset of next record to be returned\n   - NOT advanced if record is returned but not processed\n   - Automatic commits occur relative to position\n\n### poll() Loop Requirements\n\nTo maintain group membership, you **MUST call `poll()` frequently**:\n- At least once every `max.poll.interval.ms` (default: 5 minutes)\n- If not called within this interval, the consumer proactively leaves the group\n- Subsequent `commitSync()` calls fail with `CommitFailedException`\n\n```java\n// RECOMMENDED: Keep poll interval tight for responsiveness\ntry {\n    while (true) {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n        for (ConsumerRecord<String, String> record : records) {\n            // Fast processing only\n            processRecord(record);\n        }\n        consumer.commitSync();\n    }\n} catch (WakeupException e) {\n    // Expected on shutdown\n} finally {\n    consumer.close();\n}\n```\n\n---\n\n## Offset Management\n\n### Offset Concepts\n\n**Position** (consumer's current position):\n- Offset of the next record to be fetched\n- Automatically advanced by `poll()`\n- Can be queried with `position(TopicPartition)`\n- Can be moved with `seek(TopicPartition, long)`\n\n**Committed Offset**:\n- Last offset stored in the broker for the partition\n- Used as the starting position on consumer restart\n- Committed via `commitSync()` or `commitAsync()`\n- Queried with `committed(Set<TopicPartition>)`\n\n**Relationship:**\n- Committed offset should be **next offset to read** (one past the last processed message)\n- If position > committed: records haven't been committed yet\n- If position < committed: some records were skipped\n\n### Offset Commit Strategies\n\n#### Automatic Committing\n\n**Configuration:**\n```properties\nenable.auto.commit=true\nauto.commit.interval.ms=1000\n```\n\n**Behavior:**\n- Offsets committed periodically (every `auto.commit.interval.ms`)\n- Committed offset = current position at commit time\n- Non-blocking (background operation)\n- Risk: Can commit offsets before records are processed\n\n**Suitable For:**\n- Simple use cases with fast, reliable processing\n- \"At-least-once\" delivery with automatic retry\n\n#### Synchronous Commits\n\n```java\nconsumer.commitSync()\nconsumer.commitSync(Duration.ofSeconds(10))\nconsumer.commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)\nconsumer.commitSync(Map<TopicPartition, OffsetAndMetadata> offsets, Duration timeout)\n```\n\n**Behavior:**\n- Blocks until commit succeeds or fails\n- Retries transient failures\n- Throws `CommitFailedException` if consumer lost group membership\n- Throws `TimeoutException` if operation exceeds timeout\n- May fail with `RebalanceInProgressException` during rebalance\n\n**Suitable For:**\n- Exact-once or at-least-once processing with external state\n- Batched processing where records are buffered before commit\n- Custom offset stores\n\n**Best Practice:**\n```java\nwhile (running) {\n    ConsumerRecords<String, String> callback)\n```\n\n**Behavior:**\n- Returns immediately without waiting for broker response\n- Callback invoked asynchronously when request completes\n- Callback may execute in any thread (typically Mechanics\n\n### Group Membership and Coordination\n\n**Group Concept:**\n- Logical aggregation of consumers with the same `group.id`\n- One consumer group can have multiple members\n- Each partition assigned to exactly one member (for load balancing)\n\n**Lifecycle:**\n1. **Init**: Consumer created but not in group\n2. **Join**: First `poll()` triggers join via JoinGroup request\n3. **Sync**: Broker coordinates partition assignment\n4. **Stable**: Member receives partitions and starts fetching\n5. **Rebalancing**: Triggered by membership or subscription changes\n6. **Revocation**: Member gives up partitions (calls `onPartitionsRevoked`)\n7. **Assignment**: Member receives new partitions (calls `onPartitionsAssigned`)\n\n### Rebalancing Details\n\n**Rebalancing is triggered when:**\n- New member joins group\n- Existing member leaves/crashes\n- Subscription changes (new topics match pattern)\n- Topic partitions change\n- Consumer calls `enforceRebalance()`\n\n**Detection and Execution:**\n- Rebalancing is **ONLY** detected during `poll()` calls\n- Once detected, callbacks execute immediately within `poll()`\n- No `poll()` called = no rebalance detection = potential stale assignment\n\n**Rebalance Protocols:**\n\n**Classic Protocol (Eager):**\n- All members revoke partitions simultaneously\n- `onPartitionsRevoked()` called with full assignment\n- All members wait for synchronization\n- Then all receive new assignments\n- Higher stop-the-world pause time\n- Higher latency during rebalance\n\n**Classic Protocol (Cooperative):**\n- Members retain partitions where possible\n- Only revoked partitions are given up\n- `onPartitionsRevoked()` called with minimal set\n- Reduces stop-the-world pause\n- Requires compatible assignors\n\n**Consumer Protocol (New):**\n- Always cooperative rebalancing\n- Incremental partition migration\n- Lowest downtime during rebalance\n\n**Rebalance Callback Guarantees:**\n- `onPartitionsRevoked()` is called before partition move\n- `onPartitionsAssigned()` is called after partition is acquired\n- During normal rebalance: revoked \u2192 new_owner_assigned guaranteed ordering\n- Exception scenarios: partition may be lost without revoke \u2192 `onPartitionsLost()` called\n- `onPartitionsAssigned()` always called (even if empty list)\n\n### Heartbeating and Session Management\n\n**Session Timeout** (`session.timeout.ms`, default: 10 seconds):\n- If broker receives no heartbeat for this duration, consumer is removed\n- Other members detect removal and rebalance\n\n**Max Poll Interval** (`max.poll.interval.ms`, default: 5 minutes):\n- If `poll()` not called for this duration, consumer proactively leaves\n- Prevents livelock: consumer continues heartbeating but makes no progress\n- Subsequent commits fail with `CommitFailedException`\n\n**Heartbeat Thread:**\n- Background thread sends heartbeats periodically\n- Interval controlled by `heartbeat.interval.ms` (default: 3 seconds)\n- Runs independently of main consumer thread\n\n**Practical Implications:**\n```java\n// GOOD: Tight poll loop maintains membership\nwhile (running) {\n    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));  // \u2713 Called frequently\n    for (ConsumerRecord record : records) {\n        processRecord(record);  // Fast processing\n    }\n}\n\n// BAD: Slow processing causes member removal\nwhile (running) {\n    ConsumerRecords records = consumer.poll(Duration.ofSeconds(30));\n    for (ConsumerRecord record : records) {\n        slowProcess(record);  // \u2717 Can exceed max.poll.interval.ms\n    }\n}\n\n// BETTER: Offload processing to thread pool\nwhile (running) {\n    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));\n    for (ConsumerRecord record : records) {\n        executor.submit(() -> processRecord(record));  // Non-blocking\n    }\n}\n```\n\n### Enforcing Rebalance\n\n```java\nvoid enforceRebalance()\nvoid enforceRebalance(String reason)\n```\n\nManually trigger a rebalance on next `poll()`. Useful for:\n- Graceful shutdown with immediate state flush\n- Redistribution of partitions\n- Internal consumer state reconciliation\n\n---\n\n## Flow Control and Position Management\n\n### Pause and Resume\n\n**Methods:**\n```java\nvoid pause(Collection<TopicPartition> partitions)\nvoid resume(Collection<TopicPartition> partitions)\nSet<TopicPartition> paused()\n```\n\n**Behavior:**\n- `pause()`: Excludes partitions from future `poll()` calls\n- Position is NOT reset (unchanged when resumed)\n- Useful for flow control or batching\n- Multiple pauses for different partitions are independent\n- `paused()` returns the current set of paused partitions\n\n**Use Case - Processing Coordination:**\n```java\n// Pause fast partition to let slow partition catch up in join\nfor (TopicPartition tp : consumer.assignment()) {\n    long lag = endOffset - position(tp);\n    if (lag > 100000) {\n        consumer.resume(Collections.singletonList(tp));\n    } else {\n        consumer.pause(Collections.singletonList(tp));\n    }\n}\n```\n\n### Current Lag\n\n```java\nOptionalLong currentLag(TopicPartition partition)\n```\n\nReturns the difference between high water mark and consumer position. part of the group\n- Consumer didn't call `poll()` within `max.poll.interval.ms`\n- Consumer was fenced by another member\n- Group coordinator changed (rare)\n\n**Recovery:**\n- Stop committing and exit consumer loop\n- Optionally notify external state manager\n- Close consumer and rejoin group\n\n```java\ntry {\n    consumer.commitSync();\n} catch (CommitFailedException e) {\n    logger.error(\"Lost group membership, exiting\", e);\n    break;  // Exit loop\n}\n```\n\n### RetriableCommitFailedException\n\n**Thrown by:** `commitSync()` callback with automatic retry\n\n**Cause:**\n- Broker not available\n- Coordinator not stable\n- Other transient failures\n\n**Behavior:**\n- `commitSync()` automatically retries\n- `commitAsync()` only notifies callback, no retry\n- Not fatal - can retry manually after wait\n\n### WakeupException\n\n**Thrown by:** `poll()`, `commitSync()`, `commitAsync()` if `wakeup()` called\n\n**Use:**\n- Signal to interrupt blocking operations\n- Allows shutdown from another thread\n- Not an error condition (expected on shutdown)\n\n**Handling:**\n```java\ntry {\n    while (true) {\n        consumer.poll(Duration.ofMillis(100));\n    }\n} catch (WakeupException e) {\n    logger.info(\"Consumer woken up, shutting down\");\n    // Expected on shutdown\n} finally {\n    consumer.close();\n}\n```\n\n### InterruptException\n\n**Thrown by:** Operations if consumer thread's interrupt flag is set\n\n**Cause:**\n- Another thread called `Thread.interrupt()` on consumer thread\n- Not preferred method (use `wakeup()` instead)\n\n### Offset Exceptions\n\n**NoOffsetForPartitionException:**\n- No committed offset found and `auto.offset.reset` is \"none\"\n\n**OffsetOutOfRangeException:**\n- Seek position is before earliest or after latest offset\n- Consumer resets position based on `auto.offset.reset` strategy\n\n**InvalidOffsetException:**\n- Offset is invalid (negative, non-existent)\n\n**LogTruncationException:**\n- Partition was truncated and committed offset was lost\n- Consumer must reset needed\n- Can throw `WakeupException` if `wakeup()` called\n\n**Return Value:**\n- `ConsumerRecords` containing fetched records partitioned by `TopicPartition`\n- Empty `ConsumerRecords` if no data occur relative to position\n\n### poll() Loop Requirements\n\nTo maintain group membership, you **MUST call `poll()` frequently**:\n- At least once every `max.poll.interval.ms` (default: 5 minutes)\n- Mechanics\n\n### Group Membership and Coordination\n\n**Group Concept:**\n- Logical aggregation of consumers with the same `group.id`\n- One consumer group can have multiple members\n- Each partition assigned to exactly one member (for load balancing)\n\n**Lifecycle:**\n1. **Init**: Consumer created but not in group\n2. **Join**: First `poll()` triggers join via JoinGroup request\n3. **Sync**: Broker coordinates partition assignment\n4. **Stable**: Member receives partitions and starts fetching\n5. **Rebalancing**: Triggered by membership or subscription changes\n6. **Revocation**: Member gives up partitions (calls `onPartitionsRevoked`)\n7. **Assignment**: Member receives new partitions (calls `onPartitionsAssigned`)\n\n### Rebalancing Details\n\n**Rebalancing is triggered when:**\n- New member joins group\n- Existing member leaves/crashes\n- Subscription changes (new topics match pattern)\n- Topic partitions change\n- Consumer calls `enforceRebalance()`\n\n**Detection and Execution:**\n- Rebalancing is **ONLY** detected during `poll()` calls\n- Once detected, callbacks execute immediately within `poll()`\n- No `poll()` called = no rebalance detection = potential stale assignment\n\n**Rebalance Protocols:**\n\n**Classic Protocol (Eager):**\n- All members revoke partitions simultaneously\n- `onPartitionsRevoked()` called with full assignment\n- All members wait for synchronization\n- Then all receive new assignments\n- Higher stop-the-world pause time\n- Higher latency during rebalance\n\n**Classic Protocol (Cooperative):**\n- Members retain partitions where possible\n- Only revoked partitions are given up\n- `onPartitionsRevoked()` called with minimal set\n- Reduces stop-the-world pause\n- Requires compatible assignors\n\n**Consumer Protocol (New):**\n- Always cooperative rebalancing\n- Incremental partition migration\n- Lowest downtime during rebalance\n\n**Rebalance Callback Guarantees:**\n- `onPartitionsRevoked()` is called before partition move\n- `onPartitionsAssigned()` is called after partition is acquired\n- During normal rebalance: revoked \u2192 new_owner_assigned guaranteed ordering\n- Exception scenarios: partition may be lost without revoke \u2192 `onPartitionsLost()` called\n- `onPartitionsAssigned()` always called (even if empty list)\n\n### Heartbeating and Session Management\n\n**Session Timeout** (`session.timeout.ms`, default: 10 seconds):\n- If broker receives no heartbeat for this duration, consumer is removed\n- Other members detect removal and rebalance\n\n**Max Poll Interval** (`max.poll.interval.ms`, default: 5 minutes):\n- If `poll()` not called for this duration, consumer proactively leaves\n- Prevents livelock: consumer continues heartbeating but makes no progress\n- Subsequent commits fail with `CommitFailedException`\n\n**Heartbeat Thread:**\n- Background thread sends heartbeats periodically\n- Interval controlled by `heartbeat.interval.ms` (default: 3 seconds)\n- Runs independently of main consumer thread\n\n**Practical Implications:**\n```java\n// GOOD: Tight poll loop maintains membership\nwhile (running) {\n    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));  // \u2713 Called frequently\n    for (ConsumerRecord record : records) {\n        processRecord(record);  // Fast processing\n    }\n}\n\n// BAD: Slow processing causes member removal\nwhile (running) {\n    ConsumerRecords records = consumer.poll(Duration.ofSeconds(30));\n    for (ConsumerRecord record : records) {\n        slowProcess(record);  // \u2717 Can exceed max.poll.interval.ms\n    }\n}\n\n// BETTER: Offload processing to thread pool\nwhile (running) {\n    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));\n    for (ConsumerRecord record : records) {\n        executor.submit(() -> processRecord(record));  // Non-blocking\n    }\n}\n```\n\n### Enforcing Rebalance\n\n```java\nvoid enforceRebalance()\nvoid enforceRebalance(String reason)\n```\n\nManually trigger a rebalance on next `poll()`. Useful for:\n- Graceful shutdown with immediate state flush\n- Redistribution of partitions\n- Internal consumer state reconciliation\n\n---\n\n## Flow Control and Position Management\n\n### Pause and Resume\n\n**Methods:**\n```java\nvoid pause(Collection<TopicPartition> partitions)\nvoid resume(Collection<TopicPartition> partitions)\nSet<TopicPartition> paused()\n```\n\n**Behavior:**\n- `pause()`: Excludes partitions from future `poll()` calls\n- Position is NOT reset (unchanged when resumed)\n- Useful for flow control or batching\n- Multiple pauses for different partitions are independent\n- `paused()` returns the current set of paused partitions\n\n**Use Case - Processing Coordination:**\n```java\n// Pause fast partition to let slow partition catch up in join\nfor (TopicPartition tp : consumer.assignment()) {\n    long lag = endOffset - position(tp);\n    if (lag > 100000) {\n        consumer.resume(Collections.singletonList(tp));\n    } else {\n        consumer.pause(Collections.singletonList(tp));\n    }\n}\n```\n\n### Current Lag\n\n```java\nOptionalLong currentLag(TopicPartition partition)\n```\n\nReturns the difference between high water mark and consumer position. part of the group\n- Consumer didn't call `poll()` within `max.poll.interval.ms`\n- Consumer was fenced by another member\n- Group coordinator changed (rare)\n\n**Recovery:**\n- Stop committing and exit consumer loop\n- Optionally notify external state manager\n- Close consumer and rejoin group\n\n```java\ntry {\n    consumer.commitSync();\n} catch (CommitFailedException e) {\n retry\n\n**Cause:**\n- Broker not available\n- Coordinator not stable\n- Other transient failures\n\n**Behavior:**\n- `commitSync()` automatically retries\n- `commitAsync()` only notifies callback, no retry\n- Not fatal - can retry manually after wait\n\n### WakeupException\n\n**Thrown by:** `poll()`, `commitSync()`, `commitAsync()` if `wakeup()` called\n\n**Use:**\n- Signal to interrupt blocking operations\n- Allows shutdown from another thread\n- Not an error condition (expected on shutdown)\n\n**Handling:**\n```java\ntry {\n    while (true) {\n ## Overview

### Threading Model

The Kafka KafkaConsumer is **NOT thread-safe**. All consumer operations must occur within a single consumer thread. The only exception is `wakeup()`, which is explicitly designed to be called from external threads to interrupt blocking operations.

**Key Threading Points:**
- `KafkaConsumer` maintains an internal event loop that processes rebalances, heartbeats, and data fetches
- All callbacks (ConsumerRebalanceListener, OffsetCommitCallback) execute within the user's calling thread
- Concurrent access without synchronization results in `ConcurrentModificationException`
- The `wakeup()` method is the only safe cross-thread operation
