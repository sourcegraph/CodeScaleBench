# Team Handoff: Cilium eBPF Datapath Subsystem

## Scenario

You are taking over ownership of Cilium's eBPF datapath subsystem from a departing team member. The eBPF datapath is the core of Cilium's networking and security capabilities, responsible for loading eBPF programs into the kernel, enforcing network policies, and implementing service mesh functionality.

Your team member has left limited documentation. Your task is to explore the codebase and produce a comprehensive handoff document that will help you and future team members understand, maintain, and extend the eBPF datapath system.

## Your Task

Explore the cilium/cilium codebase and create a structured handoff document covering the eBPF datapath subsystem. Your document should address the following sections:

### 1. Purpose
- What problem does the eBPF datapath solve in Cilium's architecture?
- Why does Cilium use eBPF instead of traditional iptables or userspace networking?
- What are the key responsibilities of the datapath subsystem?
- How does the datapath integrate with Kubernetes networking?

### 2. Dependencies
- What other Cilium subsystems does the datapath interact with?
- What are the upstream dependencies (what calls into the datapath)?
- What are the downstream dependencies (what does the datapath call)?
- How does the Go code interact with the C eBPF programs?
- What kernel APIs does the datapath rely on?

### 3. Relevant Components
- What are the main source files and directories for the datapath subsystem?
- Where is the eBPF loader implemented?
- Where are the eBPF C programs located?
- Where is the policy enforcement logic implemented?
- Where are the eBPF maps managed?
- What files are critical for understanding how eBPF programs are compiled, loaded, and updated?

### 4. Failure Modes
- What can go wrong with eBPF program loading?
- How does the system handle eBPF verifier failures?
- What happens when eBPF maps are full or can't be updated?
- How are kernel compatibility issues detected and handled?
- What are common configuration errors and how are they detected?
- How does the system recover from eBPF program failures or kernel panics?

### 5. Testing
- How is the eBPF datapath tested?
- What test patterns are used for eBPF programs?
- Where are the datapath tests located?
- How do you test policy enforcement without a full cluster?
- What integration tests exist for the datapath?
- How are eBPF programs validated before loading?

### 6. Debugging
- How do you troubleshoot eBPF program loading failures?
- How do you inspect eBPF maps at runtime?
- What tools are available for debugging eBPF programs (bpftool, cilium monitor, etc.)?
- How do you verify that policies are being enforced correctly?
- What logs or metrics help diagnose datapath issues?
- How do you investigate performance problems in the datapath?

### 7. Adding a New Hook
- If you needed to add a new eBPF hook point (e.g., for a new protocol or policy type), what would be the step-by-step process?
- What eBPF program types can be used?
- What files need to be created or modified (C programs, Go loader code)?
- How do you integrate a new hook with the existing policy engine?
- How is a new hook registered and attached to network interfaces?

## Deliverable

Create your handoff document as a markdown file at `/logs/agent/onboarding.md`.

Deliver a clear, well-structured document that covers all requested sections. Include:
- Specific file paths and directory names
- Key function/type names
- Code flow descriptions
- Concrete examples where helpful
- Architecture diagrams in text/ASCII if helpful

## Evaluation

Your handoff document will be evaluated on:
- **Completeness**: All 7 sections addressed with substantive content
- **Accuracy**: Correct identification of relevant components, interfaces, and architectural patterns
- **Specificity**: Concrete file paths, type names, and code references (not generic descriptions)
- **Understanding**: Demonstrates comprehension of how the eBPF datapath works, including the Go+C boundary
- **Depth**: Goes beyond surface-level file listing to explain how components interact and why design decisions were made

Good luck with your exploration!

Failure Modes\n\n### eBPF Program Loading Failures\n\n#### **4.1 eBPF Verifier Failures**\n\n**Symptoms:** Program fails to load, error message shows verifier rejection\n\n**Root Causes:**\n1. **Unbounded loops:** eBPF programs must be provably terminating (kernel 5.3+ allows bounded loops)\n2. **Out-of-bounds memory access:** Verifier can't prove all memory accesses are within bounds\n3. **Unreachable code:** Dead code paths confuse the verifier\n4. **Stack overflow:** Local variables exceed 512-byte limit\n5. **Helper function misuse:** Wrong argument types, or calling unsupported helpers on older kernels\n\n**Handling:**\n- Located in `pkg/bpf/collection.go:256-349`\n- Verifier error log starts at 4 MiB buffer (line ~260)\n- If truncated (`VerifierError.Truncated == true`), quadruples buffer and retries\n- Retries up to 5 times before giving up with truncated output\n- Error output sent to stderr for debugging (line ~349)\n\n**Recovery:**\n- Compilation fails, program not loaded\n- Falls back to previous version of program (if previously loaded)\n- Cilium continues with old policies until fix is applied\n- New code must be verified offline and committed\n\n**Prevention:**\n- Use `__always_inline` for helper functions (reduces stack depth)\n- Keep loops bounded (`#pragma unroll`)\n- Use BPF `if` guards to prove memory safety\n- Test with kernel BPF verifier locally (`llvm-objdump`, `bpftool prog dump`)\n\n#### **4.2 Map Incompatibility Errors**\n\n**Symptoms:** Program loads but fails to use maps, or hangs on map operations\n\n**Root Causes:**\n1. **Map definition change:** Changed map size/type without cleanup (line ~160 in `collection.go`)\n2. **Pinned map with different layout:** Old map persists with incompatible schema\n3. **Map in use:** Kernel prevents deletion/recreation while in use\n\n**Handling:**\n- Detected in `LoadCollectionSpec()` \u2192 `mapIncompatible := ...` (line ~160)\n- Incompatible maps are unpinned via `UnpinLink()` and recreated\n- New maps created with correct schema\n- Automatic recovery with `CILIUM_PIN_REPLACE` flag (line ~180)\n\n**Recovery:**\n- Maps are automatically recreated with new schema\n- Existing data is lost (acceptable for most maps - ephemeral state)\n- Policy maps rebuilt from control plane state\n- NAT/CT maps cleared (new connections will re-establish state)\n\n**Prevention:**\n- Never change map sizes without coordination\n- Use versioning in pinned map names\n- Test map schema changes in staging first\n- Document all map layout changes in commit messages\n\n#### **4.3 Program Type Mismatches**\n\n**Symptoms:** Program fails to attach: \"Invalid argument\" or \"Unsupported\"\n\n**Root Causes:**\n1. **Wrong program type:** Trying to attach XDP program as TC filter\n2. **Missing kernel feature:** Kernel doesn't support TCX, falling back to legacy TC\n3. **Incompatible instruction set:** ISA doesn't support CPU (e.g., 32-bit on 64-bit kernel)\n\n**Handling:**\n- TC/TCX fallback logic (line ~150-200 in `tc.go`)\n- Tries TCX first (kernel 5.11+)\n- Falls back to legacy TC if TCX unavailable\n- XDP: tries bpf_link (kernel 5.7+), falls back to netlink (lines ~230-275 in `xdp.go`)\n\n**Recovery:**\n- Automatic fallback to older mechanisms\n- Program still functions, possibly with different performance\n- User notified via logs of fallback mechanism used\n- No manual intervention required\n\n---\n\n### Map-Related Failures\n\n#### **4.4 Map Size Exhaustion**\n\n**Symptoms:** Map updates fail, \"Map full\" errors in logs\n\n**Root Causes:**\n1. **Connection tracking explosion:** Millions of connections in CT maps\n2. **Endpoint explosion:** More endpoints than map capacity\n3. **Service explosion:** Too many load-balanced services\n4. **SNAT port exhaustion:** More unique source ports needed than available\n\n**Handling:**\n- LRU maps (CT, NAT) automatically evict least-recently-used entries\n- Policy maps per-endpoint (distributed across many maps, not global)\n- Metrics show map pressure via `bpf_map_update_element()` return codes\n- Errors logged with context (which endpoint, which map)\n\n**Recovery:**\n- **LRU maps:** Automatic eviction of old entries (typically succeeds)\n- **Per-endpoint policy maps:** Can resize dynamically via endpoint regeneration\n- **Manual:** Increase kernel memory limits, increase map sizes, scale horizontally (more nodes)\n\n**Prevention:**\n- Monitor map pressure via Cilium metrics\n- Set appropriate map sizes at startup (`BPF_CT4_GLOBAL_TCP_MAX`, etc.)\n- Use ClusterIP load-balancing to reduce endpoint-to-endpoint connections\n- Implement connection limits via security policies\n\n#### **4.5 Map Update Race Conditions**\n\n**Symptoms:** Inconsistent policy enforcement, stale NAT translations, connection leaks\n\n**Root Causes:**\n1. **Policy update during packet processing:** Packet sees partial policy\n2. **Map replacement without synchronization:** Old programs still running with new map IDs\n3. **Concurrent map writers:** Multiple Go routines writing to same map entry\n\n**Handling:**\n- Tail call maps updated AFTER kernel load (line ~120 in `netlink.go`)\n- Link update atomicity: `bpf_link.Update()` is atomic (TCX, kernel 5.8+)\n- Per-CPU maps isolate writer conflicts\n- Policy programs are per-endpoint (no cross-endpoint contention)\n\n**Recovery:**\n- Packets with partial policy state drop with visible error (metrics)\n- Policy revalidated on retry\n- Long-lived connections may see transient policy changes but remain open\n\n**Prevention:**\n- Always attach all tail calls before activating programs\n- Test concurrent policy updates in benchmarks\n- Monitor for unexpected drops during policy rollout\n\n---\n\n### Kernel Compatibility Failures\n\n#### **4.6 Kernel Verifier Regression**\n\n**Symptoms:** Code that compiled on one kernel fails on another\n\n**Root Causes:**\n1. **Stricter verifier:** Newer kernel rejects previously-valid code\n2. **Removed helpers:** Kernel version doesn't have required BPF helper\n3. **Breaking BPF ABI:** Kernel changed map/program structure layout\n\n**Handling:**\n- Probed at startup via `ProbeSystemConfig()` (line ~50 in `probes.go`)\n- `bpftool -j feature probe` run to detect capability gaps\n- Compilation flags adjust based on kernel features (e.g., `ENABLE_JUMBO_FRAMES` if kernel supports it)\n- Configuration warns if required features missing\n\n**Recovery:**\n- Cilium refuses to start if required features absent\n- Upgrade kernel to supported version (documented in Cilium prerequisites)\n- Disable features via configuration to match older kernel\n- No runtime recovery - requires kernel upgrade or configuration change\n\n**Prevention:**\n- Test on target kernel versions before deployment\n- Document minimum kernel version for each feature\n- Use conditional compilation for optional features\n- Run feature probes in CI/CD\n\n#### **4.7 Instruction Set Incompatibility**\n\n**Symptoms:** \"Instruction not supported by kernel\" during program load\n\n**Root Causes:**\n1. **64-bit BPF on 32-bit kernel:** Kernel doesn't support 64-bit instructions\n2. **32-bit ALU on old kernel:** Kernel requires compatibility mode\n3. **ISA extensions:** BPF code uses unreleased instruction sets\n\n**Handling:**\n- Compilation target hardcoded to `-target bpf` (clang chooses correct ISA)\n- Code reviewed to avoid newer ISA features\n- ISA version constraints in Cilium CI\n\n**Recovery:**\n- Program load fails, requires kernel upgrade or code reversion\n- No runtime recovery available\n\n**Prevention:**\n- Test on minimum kernel version regularly\n- Use `llvm-objdump -d` to inspect generated assembly\n- Avoid pointer arithmetic that requires 64-bit operations on 32-bit targets\n\n---\n\n### Configuration and Runtime Failures\n\n#### **4.8 Configuration Header Corruption**\n\n**Symptoms:** Program compiles but exhibits wrong behavior (wrong endpoint ID, wrong policies)\n\n**Root Causes:**\n1. **Template variable not replaced:** Placeholder like `SECLABEL` not substituted\n2. **Config generation race:** Multiple processes writing config simultaneously\n3. **File system sync failure:** Config written but not flushed before compilation\n\n**Handling:**\n- Config generation in `template.go` uses atomic writes (write-to-temp, rename)\n- Compilation lock prevents concurrent compilation (line ~50 in `cache.go`)\n- Verification: Compiled object checked for expected symbols\n\n**Recovery:**\n- Verifier typically rejects programs with missing macros (undefined symbols)\n- Manual: Check generated config files in `/var/run/cilium/` or `/sys/fs/bpf/cilium/`\n- Logs show exact config used for compilation\n- Manual config cleanup if corrupted\n\n**Prevention:**\n- Template generation tests in unit tests\n- Config files read-only after generation (no external modification)\n- Atomic file operations throughout\n\n#### **4.9 Device Not Found**\n\n**Symptoms:** Program compiles but doesn't attach: \"device not found\"\n\n**Root Causes:**\n1. **Device removed after endpoint creation:** Interface deleted before program attachment\n2. **Device name changed:** Endpoint created on `eth0`, later renamed to `eth1`\n3. **Device in wrong namespace:** Device in container namespace, not host namespace\n\n**Handling:**\n- Device enumeration just before attachment (line ~150 in `loader.go`)\n- Graceful skip if device not found (line ~200 in `tc.go`)\n- Retry mechanism for transient device unavailability\n- Logs indicate which devices received programs\n\n**Recovery:**\n- Program attachment skipped for unavailable device\n- Next endpoint regeneration will retry\n- Pod continues running with policy enforcement on available devices\n- If all devices unavailable, endpoint marked unhealthy\n\n**Prevention:**\n- Ensure network setup completes before datapath reload\n- Test device lifecycle handling in integration tests\n- Monitor for repeated device-not-found errors (indicates infrastructure issue)\n\n#### **4.10 Insufficient Kernel Memory**\n\n**Symptoms:** Map creation fails: \"Cannot allocate memory\"\n\n**Root Causes:**\n1. **Too many maps created:** LRU, per-endpoint maps multiply quickly\n2. **Maps too large:** Per-endpoint policy map sized for millions of entries\n3. **System memory pressure:** Kernel memory already exhausted\n\n**Handling:**\n- Map creation wrapped in error handler (line ~220 in `map_linux.go`)\n- LRU maps use kernel memory efficiently (auto-eviction)\n- Map sizes configurable via `CILIUM_MAP_*` environment variables\n- Metrics track memory usage per map\n\n**Recovery:**\n- Map creation failure returns error, endpoint stays with old maps\n- Next garbage collection attempt may free memory\n- Manual: Reduce map sizes via configuration, restart Cilium agent\n- Horizontal scaling: Move workloads to other nodes\n\n**Prevention:**\n- Pre-size maps based on expected scale\n- Monitor kernel memory usage via `/proc/meminfo`\n- Set memory limits for BPF subsystem\n- Test scaling with expected endpoint/service counts\n\n---\n\n### Debugging Failure Modes\n\n#### **4.11 Missed Tail Calls**\n\n**Symptoms:** Policy not enforced, drops with \"missed tail call\" error\n\n**Root Causes:**\n1. **Program not installed in tail call map:** `POLICY_CALL_MAP[endpoint_id]` empty\n2. **Tail call map full:** Can't add more programs\n3. **Wrong endpoint ID:** Program tries to call non-existent entry\n4. **Race condition:** Program loaded but policy program not yet inserted\n\n**Handling:**\n- Tail call validation during load (line ~170 in `collection.go`)\n- Programs inserted in POLICY_CALL_MAP after all tail calls resolved\n- Ordering: all programs loaded before any activated (line ~120 in `netlink.go`)\n- Metrics: `cilium_bpf_program_runtime_micro_seconds{reason=\"missed_tail_call\"}`\n\n**Recovery:**\n- Packet dropped with identifiable reason code\n- Policy regeneration retry will repopulate tail call maps\n- Cilium logs show failed endpoint, user can investigate why\n\n**Prevention:**\n- Unit tests validate tail call insertion\n- Integration tests verify policy enforcement\n- Monitor missed tail call metrics in production\n\n---\n\n## 5. ## 1. Purpose

### What Problem Does the eBPF Datapath Solve?

The eBPF datapath is the **core execution engine** that implements Cilium's networking and security policies at the kernel level. It solves several critical problems:

1. **High-Performance Policy Enforcement:** Traditional iptables-based firewalls have linear lookup performance and can't efficiently express complex policy logic. The eBPF datapath executes policy checks in kernel space with O(1) or O(log n) map lookups, avoiding expensive context switches between kernel and userspace.

2. **Dynamic Network Policy:** Cilium uses identity-based (rather than IP-based) policies. The eBPF datapath performs real-time policy lookups against dynamically updated maps as endpoints join/leave the cluster, without requiring userspace packet interception.

3. **Transparent Service Mesh:** eBPF socket-level hooks (`cgroup/connect4`, `cgroup/sendmsg4`, etc. in `bpf_sock.c`) enable transparent load-balancing and service mesh without requiring sidecar proxies, reducing latency and resource overhead.

4. **In-Kernel NAT and Load-Balancing:** Rather than delegating to userspace proxies, the datapath implements SNAT, DNAT, and load-balancing directly in eBPF programs (via `lib/nat.h` and `lib/lb.h`).
