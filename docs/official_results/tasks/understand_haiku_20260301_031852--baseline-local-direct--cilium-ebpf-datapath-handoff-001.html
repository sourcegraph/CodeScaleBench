Failure Modes\n\n### A. eBPF Program Compilation Failures\n\n#### 1.1 Clang Compilation Error\n**Symptom**: `failed to compile eBPF programs: clang command failed`\n\n**Root Causes**:\n- C syntax errors in eBPF program source\n- Undefined macros (missing node_config.h or ep_config.h)\n- Incompatible helper function usage\n- Non-portable C constructs\n\n**Detection**:\n```go\n// loader/compile.go: compileDatapathProgramSubcommand()\ncmd := exec.CommandContext(ctx, compiler, ...)\nif err := cmd.Run(); err != nil {\n    // Clang returned error; stderr contains compiler messages\n}\n```\n\n**Recovery**:\n- Fix C code syntax\n- Check configuration header generation (`base.go:writeNetdevHeader()`)\n- Verify Clang version supports required BPF instruction set\n- Check `node_config.h` is properly generated before compilation\n\n#### 1.2 eBPF Verifier Rejection\n**Symptom**: `error verifying eBPF program: invalid instruction sequence`\n\n**Root Causes**:\n- Program uses unverifiable memory access patterns\n- Loop unrolling insufficient (verifier cannot prove termination)\n- Dead code with unreachable paths\n- Stack overflow (BPF stack is 512 bytes)\n- Unbounded loops\n\n**Common Patterns**:\n```c\n// BAD: Infinite loop (verifier rejects)\n#pragma unroll\nfor (int i = 0; i < 1000; i++) { ... }\n\n// GOOD: Bounded unrolling\n#pragma unroll\nfor (int i = 0; i < 10; i++) { ... }  // Known constant\n\n// BAD: Variable loop bound\nfor (int i = 0; i < n; i++) { ... }  // Verifier can't prove termination\n\n// GOOD: Pragma unroll without loop\n#pragma unroll\nfor (int i = 0; i < max_iterations; i++) {\n    if (i >= actual_count) break;  // Explicit bound check\n}\n```\n\n**Detection**:\n```go\n// pkg/datapath/linux/probes/probes.go\n// Verifier failures caught when loading collection:\ncoll, err := spec.Load(&ebpf.CollectionOptions{})\n// err contains verifier error message with instruction number\n```\n\n**Recovery**:\n- Reduce loop iterations\n- Unroll loops manually with explicit iteration bounds\n- Simplify stack usage (split into multiple functions)\n- Use local helper functions instead of inlining everything\n- Profile with `llvm-objdump` to see code size and complexity\n\n#### 1.3 Map Size Too Small\n**Symptom**: `BPF_MAP_TYPE_HASH: max_entries too small for hash`\n\n**Root Causes**:\n- Policy maps need entries for all policy decisions\n- CT maps need entries for all concurrent connections\n- LB maps insufficient for all services/endpoints\n\n**Configuration**:\nMap sizes are defined in:\n- `lib/maps.h` (eBPF program side)\n- `pkg/maps/*/map.go` (Go side)\n\n**Recovery**:\n- Increase `MaxEntries` in map definition\n- For policy maps: `POLICY_MAP_SIZE` in `lib/maps.h`\n- For CT maps: `CONNTRACK_TABLE_SIZE` in node config\n- Monitor map utilization with `bpftool`\n\n### B. eBPF Program Loading Failures\n\n#### 2.1 BPF Map Already Exists Error\n**Symptom**: `map already exists` or `pin already exists`\n\n**Root Causes**:\n- Previous program instance not cleaned up\n- Reloading with mismatched map definitions\n- Multiple loader instances running\n\n**Detection**:\n```go\n// loader/loader.go: ReloadDatapath()\n// When map already pinned in bpffs\nspec.Maps[mapName].Pinning = ebpf.PinByName\ncoll, err := spec.Load(...)  // Fails if pinned map has different type/size\n```\n\n**Recovery**:\n- Unload previous programs: `loader.Unload(endpoint)`\n- Remove stale pinned maps: `/sys/fs/bpf/cilium/`\n- Restart Cilium agent (cleans up on startup)\n- Check for concurrent Cilium instances (should only be one)\n\n#### 2.2 Insufficient Kernel Memory\n**Symptom**: `Cannot allocate memory` when loading maps\n\n**Root Causes**:\n- Not enough kernel memory for eBPF maps\n- System out of memory\n- ulimit restrictions on locked memory\n\n**Detection**:\n```go\n// First time eBPF programs loaded\n// bpf() syscall fails with ENOMEM\n```\n\n**Recovery**:\n```bash\n# Remove memory locks for eBPF programs\nulimit -l unlimited\n\n# Or increase system memory limits\nsysctl -w vm.max_map_count=262144\n```\n\n#### 2.3 Program Type Mismatch\n**Symptom**: `program type mismatch: expected BPF_PROG_TYPE_SCHED_CLS, got BPF_PROG_TYPE_XDP`\n\n**Root Causes**:\n- Attaching program with wrong section to hook\n- Program compiled with wrong entrypoint symbol\n- Kernel doesn't support program type\n\n**Detection**:\n```go\n// loader/tc.go or loader/xdp.go\n// When attaching program\nprog := coll.Programs[\"cil_from_container\"]\n// Must be BPF_PROG_TYPE_SCHED_CLS for TC\n```\n\n**Recovery**:\n- Verify program section in ELF: `llvm-objdump -d bpf_lxc.o`\n- Check kernel supports required program types: `bpftool feature probe`\n- Recompile with correct libbpf sections\n\n### C. Program Attachment Failures\n\n#### 3.1 TC/TCX Not Supported\n**Symptom**: `attaching tcx program: operation not supported`\n\n**Root Causes**:\n- Kernel too old (TCX added in Linux 6.6)\n- Interface doesn't support TC_EGRESS_BLOCK/TC_INGRESS_BLOCK\n- Netkit interfaces on kernel without netkit support\n\n**Graceful Fallback**:\n```go\n// loader/tc.go: attachSKBProgram()\n// Tries TCX first\nif errors.Is(err, link.ErrNotSupported) {\n    // Falls back to legacy TC\n    return attachTCProgram(device, prog, ...)\n}\n```\n\n#### 3.2 XDP Load Failure\n**Symptom**: `XDP attach failed: device does not support XDP`\n\n**Root Causes**:\n- Device driver doesn't support XDP (e.g., veth when not in driver mode)\n- XDP mode conflict (already attached in different mode)\n- Virtual devices (veth) only support generic XDP without netkit\n\n**Detection**:\n```go\n// loader/xdp.go: ReloadXDP()\n// Tries to attach; falls back gracefully for unsupported devices\n```\n\n**Recovery**:\n- Use `option.XDPModeGeneric` instead of driver mode for virtual devices\n- Disable XDP if not needed\n- Update device drivers to support XDP\n\n#### 3.3 Interface Not Found\n**Symptom**: `link with name eth0 not found`\n\n**Root Causes**:\n- Device deleted before attachment\n- Interface name changed\n- Network namespace not accessible\n\n**Recovery**:\n- Wait for endpoint setup before attachment\n- Use device index instead of name if possible\n- Verify network namespace\n\n### D. ## 1. Purpose

### Problem Statement
The eBPF datapath is the core execution engine of Cilium's networking and security capabilities. It solves the fundamental problem of enforcing network policies and implementing sophisticated packet processing at kernel level without requiring userspace context switches, making it extremely performant and efficient.

### Why eBPF Instead of iptables or Userspace Networking?

**Performance Benefits:**
- **Zero-copy processing**: eBPF programs run in kernel space, eliminating userspace/kernel boundary crossings
- **Atomic operations**: All packet processing decisions are made atomically without lock contention
- **Early packet filtering**: XDP allows dropping packets at the NIC driver level before allocation to kernel networking stack
- **In-kernel maps**: Shared state (policies, connections, load balancing) is maintained in kernel memory with O(1) access

**Capability Benefits:**
- **Stateful enforcement**: Connection tracking and stateful firewalling without iptables conntrack overhead
- **L7 policy integration**: Can mark packets for user-space proxy without full kernel L7 support
- **Service mesh**: Integrated load balancing, traffic management, and service discovery
- **Observability**: Deep packet introspection and monitoring without mirror ports or packet copying
