# Envoy Connection Pool Management Troubleshooting Runbook

## 1. Architecture Overview

### Connection Pool Organization

Envoy's connection pools are the critical infrastructure layer managing upstream connections. Understanding their architecture is essential for diagnosing production issues.

#### Pool Hierarchy

- **Per-Cluster Pools**: Each upstream cluster has its own pool manager
- **Per-Host Pools**: Within each cluster, a separate connection pool exists for each upstream host
- **Per-Thread Pools**: Connection pools are NOT shared across worker threads—each thread maintains its own independent pools
- **Per-Priority Pools**: Separate capacity tracking for DEFAULT and HIGH priority streams (separate circuit breaker limits)

#### Connection State Machine

Each connection progresses through well-defined states:

```
Connecting → Ready/ReadyForEarlyData → Busy → Draining → Closed
```

**State Definitions:**

- **Connecting**: Connection is being established, not yet available for requests
  - File: `source/common/conn_pool/conn_pool_base.h:90`
  - Timer active: `connect_timeout` (default varies by cluster)

- **Ready**: Connection is established, has capacity for additional concurrent streams
  - HTTP/2: Has unused concurrent stream slots
  - HTTP/1.1: Only one stream active at a time
  - Transitions to Busy when capacity is exhausted

- **ReadyForEarlyData** (HTTP/3 only): Connection established but handshake in progress
  - Can accept requests before full TLS handshake
  - Transitions to Ready or Busy on handshake completion

- **Busy**: Connection is at its concurrent stream limit
  - No new streams can be attached
  - Pending requests queue and wait for capacity
  - Transitions to Ready when a stream completes
