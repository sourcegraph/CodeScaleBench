## Key Findings Summary

**Root Cause:** A race condition at `/workspace/storage/remote/queue_manager.go:1241` where the `shards.start()` function unconditionally resets the `pendingSamples` metric counter to zero during shard creation, without accounting for samples still in-flight from concurrent append operations.

**The Problem:**
1. When resharding is triggered (e.g., 4→6 shards), `stop()` closes old shards and waits for flush completion
2. Concurrent `Append()` calls that fail to enqueue retry with exponential backoff
3. `start()` is called immediately after `stop()` and resets pending counters to 0
4. Backed-off samples eventually enqueue to new shards, but the counter was already reset
5. This causes the `prometheus_remote_storage_samples_pending` metric to become inconsistent and stall

**Why It's Intermittent:**
The race only manifests when samples are appended during the narrow timing window between old shards closing and new shards starting. The exact probability depends on workload frequency and retry backoff timing.

**Key Code Locations:**
- **Primary issue:** Line 1241: `s.qm.metrics.pendingSamples.Set(0)`
- **Reshard trigger:** Lines 1189-1194 in `reshardLoop()`
- **Stop/Start sequence:** Lines 1269-1266 and 1237-1266
- **Append retry loop:** Lines 702-758

The report includes specific diagnostic steps, metrics to monitor, and recommendations for fixing the synchronization issue between metric lifecycle and shard lifecycle transitions.
