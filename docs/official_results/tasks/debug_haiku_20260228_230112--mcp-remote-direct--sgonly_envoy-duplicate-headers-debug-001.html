# Investigation: Duplicate Response Headers in Envoy Filter Pipeline

**Repository:** github.com/sg-evals/envoy--25f893b4 (mirror of envoyproxy/envoy)
**Task Type:** Deep Causal Chain (investigation only — no code fixes)

## Scenario

An Envoy proxy operator reports that response headers configured via `response_headers_to_add` in route configuration are being **duplicated** in certain responses. Specifically, when the router filter generates a local reply (e.g., upstream timeout, connection failure, or request too large), custom response headers like `x-request-id-echo` appear twice in the HTTP response.

The issue is intermittent — it only affects responses where the router itself generates the reply (local replies) rather than forwarding an upstream response. Normal proxied responses have headers added exactly once.

Access log snippet showing the problem (using `%RESPONSE_CODE_DETAILS%` formatter):

```
[2025-08-15T10:23:45.001Z] "GET /api/v1/data HTTP/1.1" 504 UT
response_code_details=upstream_response_timeout
x-custom-trace: abc123
x-custom-trace: abc123
```

The header `x-custom-trace` appears twice. The route config has:

```yaml
response_headers_to_add:
  - header:
      key: "x-custom-trace"
      value: "%REQ(x-request-id)%"
    append_action: OVERWRITE_IF_EXISTS_OR_ADD
```

Despite using `OVERWRITE_IF_EXISTS_OR_ADD`, the header is duplicated on local replies but not on proxied upstream responses.

## Your Task

Investigate the root cause of this duplicate header behavior and produce a report at `/logs/agent/investigation.md`.

Your report MUST cover:

1. **How the router filter processes response headers** — specifically the `finalizeResponseHeaders()` call chain and the `modify_headers_` callback
2. **How local replies are generated** — the `sendLocalReply` code path in the router and how it differs from the upstream response path
3. **The specific mechanism causing double processing** — which PR/change moved `finalizeResponseHeaders()` into the `modify_headers_` callback, and why this causes double invocation for local replies
4. **The interaction between `sendLocalReply` and `modify_headers_`** — how `sendLocalReply` calls `finalizeResponseHeaders()` directly, AND the `modify_headers_` callback also calls it, resulting in headers being added twice
5. **The role of the `append_action` / `append` proto fields** — how `HeaderValueOption` config is parsed in `header_parser.cc`, including the deprecated `append` BoolValue field vs the newer `append_action` enum, and how proto default values affect behavior

I've completed a comprehensive investigation into the duplicate response headers issue in Envoy. Here's what I found:

## Key Findings

**Root Cause**: Response headers are being duplicated due to `finalizeResponseHeaders()` being called **twice**:

1. **First call**: By the router's lambda passed to `sendLocalReply()` (specifically in direct response case at line 501 of router.cc)
2. **Second call**: By the filter manager's header modification callback (line 1074 of filter_manager.cc)

**The Causal Chain**:
- Router calls `sendLocalReply()` with a lambda that invokes `direct_response->finalizeResponseHeaders()`
- Filter manager's `sendLocalReply()` creates a callback that includes `route_entry_->finalizeResponseHeaders()`
- When the filter manager's callback is invoked, it calls `finalizeResponseHeaders()` a second time
- Both `DirectResponseEntry` and `RouteEntry` evaluate the same `response_headers_to_add`, causing duplication

**Why `OVERWRITE_IF_EXISTS_OR_ADD` Doesn't Help**: 
Even with `OVERWRITE_IF_EXISTS_OR_ADD`, when `evaluateHeaders()` is called twice, both calls use `setReferenceKey()` which replaces OR adds, but since the header doesn't exist after the first invocation finishes, the second invocation sees it as "missing" and adds it again—resulting in duplication.

# Investigation Report: Duplicate Response Headers in Envoy Local Replies

## Summary

Response headers configured via `response_headers_to_add` in route configuration are being duplicated in local replies generated by the router filter (e.g., upstream timeouts, direct responses). The issue stems from `finalizeResponseHeaders()` being called by both the router's lambda callback AND the filter manager's header modification callback, causing response headers to be added twice despite `OVERWRITE_IF_EXISTS_OR_ADD` semantics.

## Root Cause

The duplication occurs due to a mismatch in responsibility between how the router handles upstream responses vs. local replies:

1. **For upstream responses** (`onUpstreamHeaders`): Router calls `finalizeResponseHeaders()` directly at line 1792, then `modify_headers_` at line 1793, then `encodeHeaders()` at line 1799
2. **For local replies** (`onUpstreamAbort`, direct responses): Router calls `sendLocalReply()` with a lambda that calls `modify_headers_`, and the filter manager INDEPENDENTLY calls `finalizeResponseHeaders()` in its header modification callback

This creates a scenario where:
- **Direct responses**: The router's lambda invokes `direct_response->finalizeResponseHeaders()` (line 501 in router.cc)
- **Filter manager callback**: The filter manager's callback invokes `route_entry_->finalizeResponseHeaders()` (line 1074 in filter_manager.cc)
- Result: Headers are added twice if both the direct response entry and the route entry have the same headers configured
