## Key Findings

### **Defect 1: config.ts - Null reference error in `isFeatureAllowedForScope`**

At line 76, the function tries to call `.includes()` on `config.scope`, but the interface (line 10) defines `scope` as optional: `scope?: OptInFeatureScope[]`. When a feature doesn't specify a scope, it should be allowed for all scopes (as stated in the comment on lines 70-71), but the code will crash with a TypeError.

All other functions in the file (`getOptInFeaturesForScope`, `HAS_USER_OPT_IN_FEATURES`, etc.) correctly handle the undefined case with `!f.scope ||`, but this function doesn't.

### **Defect 2: FeatureOptInService.ts - Inverted filter logic in `listFeaturesForUser`**

At line 205, the code filters features with `!state.globalEnabled`, but the comment (line 187) says it should return "globally enabled" features. This is backwards. The filter should be `state.globalEnabled` (without the `!`), which matches the behavior of the analogous `listFeaturesForTeam` method on line 240.

### **Defect 3: _router.ts - Missing validation in `setUserState`**

The `setUserState` endpoint (lines 88-104) doesn't validate the slug with `isOptInFeature()` before passing it to the service. However, the equivalent `setTeamState` (line 117) and `setOrganizationState` (line 146) endpoints both include this validation. This is a compliance violation—inconsistent validation pattern across mutation endpoints.
