# Envoy Connection Pool Management Troubleshooting Runbook

## 1. Architecture Overview

### Connection Pool Organization

Envoy organizes connection pools hierarchically to manage upstream connections efficiently:

- **Per-Cluster Pools**: Each cluster has connection pools segregated by:
  - Protocol (HTTP/1.1, HTTP/2, TCP)
  - Thread (each worker thread has independent pools)
  - Resource Priority (DEFAULT or HIGH priority requests)

- **Per-Host Pools**: Within a cluster, pools exist per upstream endpoint to ensure load distribution

- **Thread-Local Architecture**: Pools are thread-local to avoid synchronization overhead. Each worker thread maintains its own set of pools.

**Code References**:
- `source/common/http/conn_pool_base.h:56-110` — `HttpConnPoolImplBase` class
- `source/common/tcp/conn_pool.h:144-187` — TCP `ConnPoolImpl` class
- `source/common/upstream/cluster_manager_impl.h:66-82` — Pool allocation methods

### Connection Lifecycle

Each connection in the pool transitions through defined states:

1. **Connecting** — TCP handshake or TLS negotiation in progress
2. **Ready** — Connection established, can accept new streams
3. **Busy** — All stream slots are occupied (HTTP/2 multiplexed streams)
4. **Draining** — No new streams accepted, waiting for active streams to complete
5. **Closed** — Connection destroyed

**Transition Logic**:
- `Connecting` → `Ready` once connection succeeds (or fails)
- `Ready` → `Busy` when max concurrent streams reached (HTTP/2 only)
- `Busy` → `Ready` when a stream completes and capacity becomes available
- `Ready`/`Busy` → `Draining` on graceful shutdown or connection drain
- Any state → `Closed` on connection failure or timeout
