## Summary of Findings

**Kubernetes Leader Election** (lease-based via kubernetes/client-go):
- Core files: `tools/leaderelection/leaderelection.go`, `resourcelock/` interfaces, and `leaselock.go`
- Key mechanism: Optimistic lease renewal with configurable timeout parameters (LeaseDuration: 60s, RenewDeadline: 15s, RetryPeriod: 5s)
- Failover: Slower (waits for lease expiration) but tolerant of cluster partitions
- State transitions via LeaderCallbacks (OnStartedLeading, OnStoppedLeading, OnNewLeader)

**etcd Raft-Based Election** (etcd-io/raft):
- Core files: `raft.go` (state machine), `node.go` (public interface), `types.go` (configuration)
- Key mechanism: Distributed consensus voting with randomized election timeouts
- Failover: Faster (immediate timeout-triggered elections) with advanced mechanisms like PreVote and CheckQuorum
- State transitions via becomeLeader/becomeFollower/becomeCandidate functions

**Key Differences**:
- Kubernetes relies on a centralized API server authority vs etcd's peer-to-peer quorum model
- Kubernetes has slower failover latency vs etcd's immediate election triggering
- Both support graceful leadership transfer but through different mechanisms
