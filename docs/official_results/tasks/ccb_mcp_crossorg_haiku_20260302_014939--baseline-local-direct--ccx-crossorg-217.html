## Key Findings:

**Middleware Stack Construction** (`django/core/handlers/base.py:BaseHandler.load_middleware`):
- The `settings.MIDDLEWARE` list is processed in **reversed order** (line 41)
- Each middleware is instantiated with the next handler in the chain, creating nested wrappers
- Exception handling is automatically applied to each middleware

**Process Request/Response Call Order**:
- **process_request**: Executes in **declaration order** (first middleware first)
- **process_response**: Executes in **reverse order** (last middleware first)
- This follows Django's MiddlewareMixin pattern in `django/utils/deprecation.py`

**Special Middleware Hooks**:
- **process_view**: Executes in **reverse order** (inserted at position 0)
- **process_template_response**: Executes in **declaration order** (appended)
- **process_exception**: Executes in **declaration order** (appended)

**Django vs WSGI Middleware**:
Unlike raw WSGI middleware that requires explicit nesting, Django:
1. Uses a centralized `settings.MIDDLEWARE` list
2. Automatically constructs the wrapper chain
3. Provides special hooks for request/response/view/exception handling
4. Wraps each middleware with exception handling to prevent exception leaks
