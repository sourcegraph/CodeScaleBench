## Scope

2. **DiagnosticCollection interface**: Container for managing diagnostics across documents

4. **DiagnosticRelatedInformation class**: Additional context for diagnostics
5. **DiagnosticTag enum**: Tags for categorizing diagnostics (Unnecessary, Deprecated)

### Key Functions

1. **createDiagnosticCollection()**: Factory function for creating diagnostic collections

### Events

1. **onDidChangeDiagnostics**: Event fired when diagnostics change globally

The API provides methods to create, update, retrieve, and listen for changes in diagnostics across the editor.\n\nKey features:\n- Report multiple severity levels (Error, Warning, Information, Hint)\n- Organize diagnostics by language or tool source\n- Include related information to help users understand issues\n- Tag diagnostics as deprecated or unnecessary\n- Listen to global diagnostic changes\n- Manage diagnostic lifecycles with proper cleanup\n\n---\n\n## Core Types\n\n### DiagnosticSeverity\n\nEnum representing the severity level of a diagnostic.\n\n**Signature:**\n```typescript\nexport enum DiagnosticSeverity {\n    Error = 0,\n    Warning = 1,\n    Information = 2,\n    Hint = 3\n}\n```\n\n**Values:**\n\n| Value | Numeric | Description |\n|-------|---------|-------------|\n| `Error` | 0 | Something not allowed by the rules of a language or other means. Displayed with red squiggle underline. |\n| `Warning` | 1 | Something suspicious but allowed. Displayed with orange squiggle underline. |\n| `Information` | 2 | Something to inform about but not a problem. Displayed with blue squiggle underline. |\n| `Hint` | 3 | Something to hint to a better way of doing it, like proposing a refactoring. Displayed as subtle dots. |\n\n**Behavior Notes:**\n- `Error` is the default severity when creating a Diagnostic without specifying severity\n- Severity levels are ordered by importance: Error > Warning > Information > Hint\n- The Problems panel filters and sorts diagnostics by severity\n- The severity level affects UI rendering, gutter icon colors, and minimap coloring\n\n---\n\n### DiagnosticTag\n\nEnum representing additional metadata about the type of a diagnostic.\n\n**Signature:**\n```typescript\nexport enum DiagnosticTag {\n    Unnecessary = 1,\n    Deprecated = 2\n}\n```\n\n**Values:**\n\n| Value | Numeric | Description |\n|-------|---------|-------------|\n| `Unnecessary` | 1 | Unused or unnecessary code. Rendered with reduced opacity (controlled by `editorUnnecessaryCode.opacity` theme color, e.g., 75% opacity). In high contrast themes, uses `editorUnnecessaryCode.border` to underline the code instead of fading. |\n| `Deprecated` | 2 | Deprecated or obsolete code. Rendered with a strike-through line. |\n\n**Behavior Notes:**\n- Tags are optional and can be combined (multiple tags per diagnostic)\n- Tags provide visual feedback without requiring code action integration\n- `Unnecessary` is commonly used for unused variables, imports, or dead code\n- `Deprecated` is commonly used for obsoleted API usage\n- Tags affect rendering only; they don't change severity or classification\n\n**Example:**\n```typescript\nconst diagnostic = new vscode.Diagnostic(\n    range,\n    'Unused variable',\n    vscode.DiagnosticSeverity.Information\n);\ndiagnostic.tags = [vscode.DiagnosticTag.Unnecessary];\n```\n\n---\n\n### DiagnosticRelatedInformation\n\nClass representing additional context for a diagnostic, such as related code locations that contribute to the issue.\n\n**Signature:**\n```typescript\nexport class DiagnosticRelatedInformation {\n    location: Location;\n    message: string;\n\n    constructor(location: Location, message: string);\n}\n```\n\n**Properties:**\n\n| Property | Type | Description |\n|----------|------|-------------|\n| `location` | `Location` | The location of this related diagnostic information. A `Location` consists of a `Uri` and a `Range`. |\n| `message` | `string` | The message describing the related information. Should be concise and contextual. |\n\n**Constructor:**\n- **Parameters:**\n  - `location: Location` - The location to reference (required)\n  - `message: string` - Description of the related information (required)\n\n**Behavior Notes:**\n- Used to point to code locations that cause or relate to a diagnostic\n- Commonly used when multiple symbols have the same name (name collision in scope)\n- All related information appears in the Problems panel when you click on a diagnostic\n- Helpful for understanding the context and root cause of issues\n\n**Example:**\n```typescript\nconst diagnostic = new vscode.Diagnostic(\n    range,\n    'Duplicate symbol definition',\n    vscode.DiagnosticSeverity.Error\n);\n\ndiagnostic.relatedInformation = [\n    new vscode.DiagnosticRelatedInformation(\n        new vscode.Location(firstDefUri, firstDefRange),\n        'First definition here'\n    ),\n    new vscode.DiagnosticRelatedInformation(\n        new vscode.Location(secondDefUri, secondDefRange),\n        'Conflicting definition here'\n    )\n];\n```\n\n---\n\n### Diagnostic\n\nClass representing a single diagnostic (error, warning, info, or hint).\n\n**Signature:**\n```typescript\nexport class Diagnostic {\n    range: Range;\n    message: string;\n    severity: DiagnosticSeverity;\n    source?: string;\n    code?: string | number | { value: string | number; target: Uri };\n    relatedInformation?: DiagnosticRelatedInformation[];\n    tags?: DiagnosticTag[];\n\n    constructor(\n        range: Range,\n        message: string,\n        severity?: DiagnosticSeverity\n    );\n}\n```\n\n**Properties:**\n\n| Property | Type | Required | Description |\n|----------|------|----------|-------------|\n| `range` | `Range` | Yes | The range in the document to which this diagnostic applies. A `Range` consists of a start and end `Position`. |\n| `message` | `string` | Yes | The human-readable message describing the issue. Should be clear and actionable. |\n| `severity` | `DiagnosticSeverity` | No | The severity level (Error, Warning, Information, Hint). Defaults to `DiagnosticSeverity.Error`. |\n| `source` | `string` | No | The source identifier (e.g., 'typescript', 'eslint', 'pylint'). Displayed in the Problems panel to identify the diagnostic source. |\n| `code` | `string \\| number \\| object` | No | A code or identifier for this diagnostic. Can be a simple string/number or an object with `value` and `target` properties. Used for code actions and quick fixes. |\n| `relatedInformation` | `DiagnosticRelatedInformation[]` | No | Array of related diagnostic information pointing to related code locations. Optional but valuable for complex issues. |\n| `tags` | `DiagnosticTag[]` | No | Array of tags providing additional metadata (Unnecessary, Deprecated). Multiple tags can be applied to a single diagnostic. |\n\n**Constructor:**\n- **Parameters:**\n  - `range: Range` - The affected range (required)\n  - `message: string` - The human-readable message (required)\n  - `severity?: DiagnosticSeverity` - The severity level (optional, defaults to Error)\n\n**Behavior Notes:**\n- Diagnostics are immutable once published in a DiagnosticCollection\n- The range defines where the diagnostic is displayed in the editor (squiggle underline, gutter icon, etc.)\n- Diagnostics are scoped to a specific file URI and collection\n- Only one range per diagnostic; if multiple ranges need highlighting, create multiple diagnostics\n- The `code` property is essential for code action providers to reference diagnostics\n- Messages should be localized for international users\n- Diagnostics are sent to the Problems panel and displayed with minimap icons\n\n**Example:**\n```typescript\nconst range = new vscode.Range(\n    new vscode.Position(5, 0),\n    new vscode.Position(5, 10)\n);\n\nconst diagnostic = new vscode.Diagnostic(\n    range,\n    'Variable is never used',\n    vscode.DiagnosticSeverity.Warning\n);\n\ndiagnostic.source = 'eslint';\ndiagnostic.code = 'no-unused-vars';\ndiagnostic.tags = [vscode.DiagnosticTag.Unnecessary];\n```\n\n---\n\n### DiagnosticChangeEvent\n\nInterface representing the event data when diagnostics change globally.\n\n**Signature:**\n```typescript\nexport interface DiagnosticChangeEvent {\n    readonly uris: readonly Uri[];\n}\n```\n\n**Properties:**\n\n| Property | Type | Description |\n|----------|------|-------------|\n| `uris` | `readonly Uri[]` | An array of resource URIs for which diagnostics have changed. These are file paths where diagnostics were added, modified, or removed. |\n\n**Behavior Notes:**\n- Contains only the URIs that changed (not all URIs in the workspace)\n- Fired when any extension changes diagnostics, not just the current extension\n- Used to refresh UI components that display diagnostics\n- The array is immutable and read-only\n\n**Example:**\n```typescript\nvscode.languages.onDidChangeDiagnostics(event => {\n    console.log('Diagnostics changed for:', event.uris);\n    // Refresh custom diagnostic display or perform cleanup\n});\n```\n\n---\n\n### DiagnosticCollection\n\nInterface representing a container for managing diagnostics across documents.\n\n**Signature:**\n```typescript\nexport interface DiagnosticCollection extends Iterable<[uri: Uri, diagnostics: readonly Diagnostic[]]> {\n    readonly name: string;\n\n    set(uri: Uri, diagnostics: readonly Diagnostic[] | undefined): void;\n    set(entries: ReadonlyArray<[Uri, readonly Diagnostic[] | undefined]>): void;\n\n    delete(uri: Uri): void;\n    clear(): void;\n\n    forEach(\n        callback: (uri: Uri, diagnostics: readonly Diagnostic[], collection: DiagnosticCollection) => any,\n        thisArg?: any\n    ): void;\n\n    get(uri: Uri): readonly Diagnostic[] | undefined;\n    has(uri: Uri): boolean;\n\n    dispose(): void;\n}\n```\n\n**Properties:**\n\n| Property | Type | Description |\n|----------|------|-------------|\n| `name` | `string` (readonly) | The name of this diagnostic collection (e.g., 'typescript', 'eslint'). Displayed in the Problems panel and used by task framework for problem matchers. |\n\n**Methods:**\n\n#### set(uri: Uri, diagnostics: readonly Diagnostic[] | undefined): void\n\nAssigns or updates diagnostics for a given resource. Replaces existing diagnostics for that resource.\n\n**Parameters:**\n- `uri: Uri` - The file URI for which to set diagnostics\n- `diagnostics: readonly Diagnostic[] | undefined` - Array of diagnostics, or `undefined` to clear diagnostics for this URI\n\n**Behavior Notes:**\n- Passing `undefined` clears all diagnostics for the URI (equivalent to `delete()`)\n- Replaces all previous diagnostics for the URI with the new array\n- Empty array `[]` also clears diagnostics for the URI\n- Fires the `onDidChangeDiagnostics` event asynchronously\n- Multiple diagnostics can target the same range\n\n**Example:**\n```typescript\nconst uri = vscode.window.activeTextEditor?.document.uri;\nif (uri) {\n    const diagnostics = [\n        new vscode.Diagnostic(range1, 'Error 1', vscode.DiagnosticSeverity.Error),\n        new vscode.Diagnostic(range2, 'Error 2', vscode.DiagnosticSeverity.Warning)\n    ];\n    collection.set(uri, diagnostics);\n}\n```\n\n#### set(entries: ReadonlyArray<[Uri, readonly Diagnostic[] | undefined]>): void\n\nBatch update for multiple resources in a single call.\n\n**Parameters:**\n- `entries: ReadonlyArray<[Uri, readonly Diagnostic[] | undefined]>` - Array of tuples mapping URIs to diagnostic arrays\n\n**Behavior Notes:**\n- Merges multiple tuples with the same URI (e.g., `[[file1, [d1]], [file1, [d2]]]` becomes `[[file1, [d1, d2]]]`)\n- If the same URI appears with `undefined` after diagnostics, all previous diagnostics are removed before adding subsequent ones\n- More efficient than calling `set()` multiple times\n- Fires a single `onDidChangeDiagnostics` event for all changes\n- Allows atomic updates across multiple files\n\n**Example:**\n```typescript\ncollection.set([\n    [uri1, [diagnostic1, diagnostic2]],\n    [uri2, [diagnostic3]],\n    [uri3, undefined]  // Clears diagnostics for uri3\n]);\n```\n\n#### delete(uri: Uri): void\n\nRemoves all diagnostics for the specified URI.\n\n**Parameters:**\n- `uri: Uri` - The file URI for which to remove diagnostics\n\n**Behavior Notes:**\n- Equivalent to `set(uri, undefined)`\n- Fires the `onDidChangeDiagnostics` event\n- No-op if the URI has no diagnostics\n\n**Example:**\n```typescript\ncollection.delete(uri);\n```\n\n#### clear(): void\n\nRemoves all diagnostics from this collection.\n\n**Behavior Notes:**\n- Equivalent to calling `delete()` on every URI that has diagnostics\n- Fires the `onDidChangeDiagnostics` event once\n- Used during extension cleanup or when reloading diagnostics\n\n**Example:**\n```typescript\ncollection.clear();\n```\n\n#### forEach(callback, thisArg?): void\n\nIterates over each URI and its diagnostics in the collection.\n\n**Parameters:**\n- `callback: (uri: Uri, diagnostics: readonly Diagnostic[], collection: DiagnosticCollection) => any` - Function to execute for each entry\n- `thisArg?: any` - Optional context for the callback function\n\n**Behavior Notes:**\n- Iteration order is implementation-dependent\n- Diagnostics array is immutable (read-only)\n- Modifying the collection during iteration is safe; iteration continues with the snapshot taken at call time\n- Useful for custom reporting or cleanup\n\n**Example:**\n```typescript\ncollection.forEach((uri, diagnostics, collection) => {\n    console.log(`File: ${uri.fsPath}, Issues: ${diagnostics.length}`);\n});\n```\n\n#### get(uri: Uri): readonly Diagnostic[] | undefined\n\nRetrieves diagnostics for a given resource.\n\n**Parameters:**\n- `uri: Uri` - The file URI\n\n**Returns:**\n- `readonly Diagnostic[] | undefined` - Immutable array of diagnostics, or `undefined` if no diagnostics exist for the URI\n\n**Behavior Notes:**\n- Returns `undefined` if the URI has no diagnostics (not an empty array)\n- The returned array is immutable; modifications will not affect the collection\n- Safe to cache the reference; it won't change between calls for the same URI\n- Used to check for existing diagnostics before updating\n\n**Example:**\n```typescript\nconst diagnostics = collection.get(uri);\nif (diagnostics) {\n    console.log(`Found ${diagnostics.length} issues`);\n}\n```\n\n#### has(uri: Uri): boolean\n\nChecks if the collection contains diagnostics for a given resource.\n\n**Parameters:**\n- `uri: Uri` - The file URI\n\n**Returns:**\n- `boolean` - `true` if the collection has diagnostics for the URI, `false` otherwise\n\n**Behavior Notes:**\n- Faster than checking if `get(uri)` returns `undefined`\n- Useful for conditional logic\n\n**Example:**\n```typescript\nif (collection.has(uri)) {\n    // URI has diagnostics\n}\n```\n\n#### dispose(): void\n\nCleans up the collection and releases associated resources.\n\n**Behavior Notes:**\n- Calls `clear()` before disposing\n- After disposal, the collection should not be used\n- Prevents memory leaks when extensions unload\n- Should be called in the extension's deactivation hook\n\n**Example:**\n```typescript\ncontext.subscriptions.push(diagnosticsCollection);\n// or manually:\n// extension.deactivate() { diagnosticsCollection.dispose(); }\n```\n\n**Iterable Protocol:**\n\nDiagnosticCollection implements the Iterable protocol, allowing it to be used with `for...of` loops.\n\n```typescript\nfor (const [uri, diagnostics] of collection) {\n    console.log(`${uri.fsPath}: ${diagnostics.length} issues`);\n}\n```\n\n---\n\n## Functions\n\n### createDiagnosticCollection(name?: string)\n\nFactory function for creating a new diagnostic collection.\n\n**Signature:**\n```typescript\nexport function createDiagnosticCollection(name?: string): DiagnosticCollection\n```\n\n**Parameters:**\n- `name?: string` - # VS Code Diagnostic API Reference

## Overview

The Diagnostic API allows VS Code extensions to report code issues (errors, warnings, information, hints) to users within the editor. Diagnostics are language-agnostic markers that represent problems in code, such as compilation errors, linting warnings, or code style suggestions.

Diagnostics are managed through **DiagnosticCollection**, which serves as a container for organizing diagnostics by resource (file URI). The API provides methods to create, update, retrieve, and listen for changes in diagnostics across the editor.

Key features:
- Report multiple severity levels (Error, Warning, Information, Hint)
- Organize diagnostics by language or tool source
- Include related information to help users understand issues
- Tag diagnostics as deprecated or unnecessary
- Listen to global diagnostic changes
- Manage diagnostic lifecycles with proper cleanup
