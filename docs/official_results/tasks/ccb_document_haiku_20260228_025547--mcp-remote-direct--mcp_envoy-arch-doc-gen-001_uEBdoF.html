Your document must cover these four components and how they work together:

The upstream connection pool and cluster management layer. Explain:
- How `ClusterManagerImpl` provides connection pools per cluster

# Envoy HTTP Connection Manager Architecture

## Overview

The HTTP Connection Manager (HCM) is Envoy's central subsystem for processing HTTP requests from downstream clients through to upstream services. This document describes the design and data flow of the four major components that work together to implement request routing and forwarding.

**Repository:** `github.com/envoyproxy/envoy`

---

## 1. ConnectionManagerImpl: The Network Filter

### Component Responsibility

`ConnectionManagerImpl` (source/common/http/conn_manager_impl.h/cc) is a `Network::ReadFilter` that owns the lifecycle of a single downstream connection. It:

- Receives raw TCP/UDP bytes from the network layer via `onData()`
- Instantiates and manages the appropriate codec (HTTP/1.1, HTTP/2, or HTTP/3)
- Creates new `ActiveStream` objects for each HTTP request
- Manages connection-level concerns: drain decisions, overload management, flow control
- Implements `ServerConnectionCallbacks` to receive streams from the codec

### Key Data Structures

**ActiveStream** (nested struct in conn_manager_impl.h, lines 141-438):
- Represents a single HTTP request/response pair on a connection
- Implements `RequestDecoder` to receive decoded request data from the codec
- Holds the `FilterManager` which executes the per-stream filter chain
- Tracks stream state including headers, trailers, response encoder
- Manages local replies and stream-level timeouts

**ConnectionManagerStats**:
- Metrics for connection counts, request counts, codec events
- Generated via `generateStats()` static method

### Codec Creation

Codecs are created lazily:
- For **HTTP/1.1 and HTTP/2**: created on first `onData()` call (lines 84-86)
- For **HTTP/3**: created on `onConnection()` or when explicitly created via `createCodec()`
