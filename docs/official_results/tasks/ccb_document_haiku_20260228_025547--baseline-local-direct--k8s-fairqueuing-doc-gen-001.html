\u2514\u2500 Other queues see this queue recede in virtual time\n\u2502     \u2502\n\u2502     \u2514\u2500 Return true (dispatch succeeded)\n\u2502\n\u2514\u2500 Return false (if decision.Set failed or no queue candidate)\n   \u2502  Caller will retry this request once another finishes\n   \u2502\n   \u2514\u2500 (Can only happen if request's context was canceled before dispatch)\n```\n\n### 3.3 Virtual Time Progression: `syncTimeLocked()` (queueset.go:465\u2013522)\n\nVirtual time advances proportionally to concurrency limits and active queue load:\n\n```\n\u250c\u2500 syncTimeLocked(ctx)\n\u2502\n\u251c\u2500 Measure elapsed real time:\n\u2502  \u2502  realNow := clock.Now()\n\u2502  \u2502  timeSinceLast := realNow - lastRealTime\n\u2502  \u2502\n\u2502  \u2514\u2500 lastRealTime \u2190 realNow\n\u2502\n\u251c\u2500 Compute virtual time advancement rate:\n\u2502  \u2502\n\u2502  \u2514\u2500 getVirtualTimeRatioLocked()\n\u2502     \u2502\n\u2502     \u251c\u2500 activeQueues \u2190 count of non-empty queues\n\u2502     \u2502\n\u2502     \u251c\u2500 seatsRequested \u2190 \u2211(queue.seatsInUse + queue.requestsWaiting.MaxSeatsSum)\n\u2502     \u2502\n\u2502     \u251c\u2500 If activeQueues == 0:\n\u2502     \u2502  \u2502  return 0 (no advancement while idle)\n\u2502     \u2502  \u2502\n\u2502     \u2502  \u2514\u2500 No virtual time consumption\n\u2502     \u2502\n\u2502     \u2514\u2500 Else:\n\u2502        return min(seatsRequested, ConcurrencyLimit) / activeQueues\n\u2502        \u2502\n\u2502        \u2502  Each active queue \"consumes\" an equal share of concurrency\n\u2502        \u2502  Unmet demand (seatsRequested > limit) means queues share fairly\n\u2502        \u2502  Unmet demand (seatsRequested < limit) means virtual time = real time\n\u2502\n\u251c\u2500 Advance global progress meter:\n\u2502  \u2502\n\u2502  \u2514\u2500 currentR += SeatsTimesDuration(ratio, timeSinceLast)\n\u2502     \u2502  Converts: (seats/queue) \u00d7 (elapsed time) \u2192 seat-seconds\n\u2502     \u2502  Uses fixed-point SeatSeconds (scale = 1e8)\n\u2502\n\u2514\u2500 If currentR >= highR (overflow threshold):\n   \u2502\n   \u2514\u2500 advanceEpoch(ctx, realNow, incrR)\n      \u2502  Decrement currentR and all queue.nextDispatchR by rDecrement (2^63)\n      \u2502  Prevents uint64 overflow\n      \u2502  Idempotent for queues that become empty\n      \u2502\n      \u2514\u2500 [See Section 3.5 for anti-windup implications]\n```\n\n**Key insight:** Virtual time rate is `(fair share of concurrency) / (queue count)`. If 4 queues compete for 100-seat limit, each queue \"time\" advances at 25 seats/sec in real time, regardless of actual demand. This provides proportional fairness.\n\n### 3.4 Request Completion: `finishRequestLocked()` (queueset.go:863\u2013943)\n\n```\n\u250c\u2500 finishRequestLocked(req)\n\u2502\n\u251c\u2500 Update counters:\n\u2502  \u2502  \u251c\u2500 totRequestsExecuting--\n\u2502  \u2502  \u251c\u2500 Metrics: AddRequestsExecuting(-1)\n\u2502  \u2502  \u2502\n\u2502  \u2502  \u2514\u2500 actualServiceDuration := now - req.startTime\n\u2502\n\u251c\u2500 Determine seat release timing:\n\u2502  \u2502\n\u2502  \u2514\u2500 If AdditionalLatency <= 0:\n\u2502     \u2502  Release all seats immediately\n\u2502     \u2502\n\u2502     \u2514\u2500 releaseSeatsLocked():\n\u2502        \u251c\u2500 totSeatsInUse -= req.MaxSeats()\n\u2502        \u251c\u2500 req.queue.seatsInUse -= req.MaxSeats()\n\u2502        \u251c\u2500 Metrics: AddSeatConcurrencyInUse(-seats)\n\u2502        \u2514\u2500 Call removeQueueIfEmptyLocked() [queue housekeeping]\n\u2502\n\u2502  Else:\n\u2502     \u2502\n\u2502     \u251c\u2500 Release InitialSeats immediately\n\u2502     \u2502\n\u2502     \u251c\u2500 Schedule deferred release of FinalSeats after AdditionalLatency\n\u2502     \u2502  \u2502  via clock.EventAfterDuration(func() { releaseSeatsLocked() }, AdditionalLatency)\n\u2502     \u2502  \u2502\n\u2502     \u2502  \u2514\u2500 Client perceives no latency; seats held invisibly in background\n\u2502     \u2502\n\u2502     \u2514\u2500 Post-release: dispatchAsMuchAsPossibleLocked()\n\u2502\n\u251c\u2500 Adjust queue's virtual start time (anti-windup):\n\u2502  \u2502\n\u2502  \u2514\u2500 If req was dispatched from a queue:\n\u2502     \u2502\n\u2502     \u2514\u2500 queue.nextDispatchR -= SeatsTimesDuration(req.InitialSeats,\n\u2502        EstimatedServiceDuration - actualServiceDuration)\n\u2502        \u2502\n\u2502        \u2502  Rationale:\n\u2502        \u2502  - Estimated cost: initialSeats \u00d7 estimatedServiceDuration\n\u2502        \u2502  - Actual cost: initialSeats \u00d7 actualServiceDuration\n\u2502        \u2502  - Difference: credit back to queue if actual < estimated\n\u2502        \u2502\n\u2502        \u2514\u2500 Reduces queue's virtual start time \u2192 increases dispatch priority\n\u2502\n\u2514\u2500 Remove request from queue:\n   \u2502\n   \u251c\u2500 If queued: queue.requestsExecuting.Delete(req)\n   \u2502\n   \u2514\u2500 If not queued (concurrency-only mode):\n      \u2514\u2500 requestsExecutingSet.Delete(req)\n```\n\n### 3.5 Anti-Windup: `boundNextDispatchLocked()` (queueset.go:952\u2013964)\n\n**Problem:** A queue that cannot use all its allocated concurrency (e.g., bursty single-stream) accumulates a decreasing virtual start time (its `nextDispatchR`), gaining unfair priority over busier queues. The anti-windup bounds this:\n\n```\n\u250c\u2500 boundNextDispatchLocked(queue)\n\u2502\n\u251c\u2500 If queue is now empty: return (no action needed)\n\u2502\n\u251c\u2500 Peek oldest waiting request:\n\u2502  \u2502  oldestReqFromMinQueue := queue.requestsWaiting.Peek()\n\u2502  \u2502\n\u2502  \u2514\u2500 virtualStartBound := oldestReqFromMinQueue.arrivalR\n\u2502     \u2502  (request cannot start before it arrives)\n\u2502\n\u2514\u2500 If queue.nextDispatchR < virtualStartBound:\n   \u2502\n   \u251c\u2500 queue.nextDispatchR \u2190 virtualStartBound\n   \u2502\n   \u2514\u2500 Log: \"AntiWindup tweaked queue\" with delta\n      \u2502\n      \u2514\u2500 Prevents dispatch before arrival\n```\n\n**Rationale:** If a queue is idle (no requests), its virtual time would advance unbounded. When a new request arrives, we clamp the queue's virtual start \u2265 arrival time. This prevents \"catching up\" too much.\n\n### 3.6 Request Waiting and Decision: `request.wait()` (queueset.go:397\u2013441)\n\n```\n\u250c\u2500 req.wait() [called by req.Finish() to get dispatch decision]\n\u2502\n\u251c\u2500 Fetch decision promise:\n\u2502  \u2502  decisionAny := req.decision.Get()\n\u2502  \u2502  \u2502  Blocks until decision is fulfilled (by dispatchLocked or timeout)\n\u2502  \u2502  \u2502\n\u2502  \u2502  \u2514\u2500 Lock queueset and sync virtual time\n\u2502\n\u251c\u2500 If decision == decisionExecute:\n\u2502  \u2502  return (true, false)  \u2190 execute the request\n\u2502  \u2502\n\u2502  \u2514\u2500 Caller's execFn() will run immediately\n\u2502\n\u251c\u2500 Else (decision == decisionCancel, due to context timeout/cancel):\n\u2502  \u2502\n\u2502  \u251c\u2500 Remove request from queue:\n\u2502  \u2502  \u2502  queue.requestsWaiting.Length()--\n\u2502  \u2502  \u2502  totSeatsWaiting -= req.MaxSeats()\n\u2502  \u2502  \u2502\n\u2502  \u2502  \u2514\u2500 Metrics: AddReject(..., \"time-out\")\n\u2502  \u2502\n\u2502  \u2514\u2500 return (false, isIdleLocked())  \u2190 request was rejected (timed out)\n\u2502\n\u2514\u2500 Note: Cannot call wait() twice on same request (enforced)\n```\n\n---\n\n## 4. Fairness Algorithm Details\n\n### 4.1 Fair Queuing Foundations\n\nThe QueueSet implements a variant of **Weighted Fair Queuing (WFQ)**. The key idea:\n\n1. **Virtual Time (R):** A monotonically increasing counter measuring work allocated\n2. **Virtual Start/Finish:** Each request has a virtual arrival time (`arrivalR`) and virtual finish time (`arrivalR + totalWork`)\n3. **Selection Rule:** Always dispatch the request with minimum virtual finish time\n4. **Guarantee:** Over long timescales, each flow receives service proportional to its demand (up to fair share)\n\n**Example:**\n- Request A: work = 10 seat-seconds, arrives at R=0 \u2192 finishes at R=10\n- Request B: work = 20 seat-seconds, arrives at R=0 \u2192 finishes at R=20\n- Dispatch order: A (finish@10), then B (finish@20)\n- If another A arrives at R=10: A2 finishes at R=20 (same as B)\n- Fair: both flows receive equal service time\n\n### 4.2 Virtual Time Rate Adaptation\n\nInstead of fixed-weight fair queuing, the QueueSet uses **dynamic virtual time progression** based on concurrency limits and load:\n\n```\nvirtual_time_rate = min(seatsRequested, concurrencyLimit) / activeQueues\n```\n\n**Implications:**\n\n| Scenario | seatsRequested | activeQueues | Rate per Queue |\n|----------|----------------|--------------|----------------|\n| 4 queues, 100 seats | 100 | 4 | 25 seats/sec |\n| 4 queues, 40 seats | 40 | 4 | 10 seats/sec |\n| 4 queues, 200 seats | 200 | 4 | 25 seats/sec (clamped) |\n| 2 queues, 100 seats | 100 | 2 | 50 seats/sec |\n\n**Effect:** Queues with low demand don't slow down others. Queues sharing the limit get equal virtual time.\n\n### 4.3 Work Estimation and Seat-Seconds\n\nWork is measured in **seat-seconds**, a two-phase model:\n\n```\ntotalWork = InitialSeats \u00d7 EstimatedServiceDuration\n          + FinalSeats \u00d7 AdditionalLatency\n\nExample:\n  ListRequest with 1000 objects\n  InitialSeats=3, EstimatedServiceDuration=3ms \u2192 9ms-seats\n  FinalSeats=1, AdditionalLatency=0 \u2192 0ms-seats\n  Total work = 9 ms-seats\n```\n\n**Why two phases?**\n- **Initial phase:** Request actively uses concurrency (network send, compute)\n- **Final phase:** Minimal active use; seats held for flow completion guarantees (e.g., TCP RST handling)\n\n### 4.4 Shuffle Sharding: Load Balancing Among Queues\n\nShuffle sharding solves a crucial problem: **head-of-line blocking**. Without it, a single slow request from flow X blocks all other requests from X.\n\n**Algorithm** (shuffleShardLocked, queueset.go:583\u2013612):\n\n```\n1. dealer.DealIntoHand(hashValue, handSize)\n   \u2502  Input: flow's hash, desired hand size (e.g., 8)\n   \u2502  Output: array of queue indices (e.g., [3, 7, 1, 5, 2, 8, 0, 4])\n   \u2502\n   \u2502  Ensures: different flows get different hands, but hands overlap\n   \u2502           (reduces collision probability vs. simple hash mod)\n\n2. Iterate through hand, selecting queue with minimum TotalWorkSum\n   \u2502  offset := enqueues % handSize  (round-robin offset for fairness)\n   \u2502  For i = 0 to handSize-1:\n   \u2502    queueIdx := hand[(offset + i) % handSize]\n   \u2502    if queueSum[queueIdx].TotalWorkSum < minQueueWorkSum:\n   \u2502      bestQueueIdx \u2190 queueIdx\n\n3. Enqueue into bestQueueIdx\n\nOutcome:\n  - Same flow always gets queued to one of its hand's queues\n  - Requests from different flows spread across queues\n  - Load balancing: pick least-loaded queue from hand\n  - Starvation prevention: offset ensures round-robin fairness among hand members\n```\n\n**Trade-off:** Shuffle sharding trades determinism for fairness. A flow may experience variable queueing if its hand happens to be congested.\n\n---\n\n## 5. Failure Modes, Limits, and Design Tradeoffs\n\n### 5.1 Overflow Management\n\n**Problem:** Virtual time `R` is a `uint64`. After ~584 CPU-years of continuous operation, it overflows.\n\n**Solution:** Epoch advancement (advanceEpoch, queueset.go:492\u2013522)\n\n```\nTrigger: currentR >= highR (where highR = 2^63 + 2^62)\n\nAction:\n  1. Subtract rDecrement (2^63) from:\n     - currentR\n     - All queue.nextDispatchR\n     - All request.arrivalR (for queued requests)\n\n  2. Log the event with old/new values and integrity check\n\nRationale:\n  - Preserves relative ordering (A - C == (A-D) - (C-D))\n  - Resets counter, extending lifetime to another ~584 years\n  - Idempotent for empty queues (nextDispatchR \u2192 0 if queue unused)\n```\n\n**Risk:** If any adjustment underflows (detected and logged), fairness metrics become unreliable, but scheduling continues.\n\n### 5.2 Seat Accommodation and Borrowing Limits\n\n**Problem:** A request may require more seats than the concurrency limit.\n\n**Handling** (canAccommodateSeatsLocked, queueset.go:748\u2013769):\n\n```\nCase 1: DesiredNumQueues < 0 (exempt from limitation)\n  \u2192 Always return true (admit request immediately)\n\nCase 2: RequestSeats > ConcurrencyLimit\n  \u2192 If totRequestsExecuting == 0: admit (allow one to start)\n  \u2192 Else: reject (wait for current requests to finish)\n\nCase 3: RequestSeats + TotSeatsInUse <= ConcurrencyLimit\n  \u2192 Admit (sufficient capacity)\n\nCase 4: RequestSeats + TotSeatsInUse > ConcurrencyLimit\n  \u2192 Reject (insufficient capacity, must wait)\n```\n\n**Trade-off:** Case 2 allows a single request to exceed the limit, but prevents cascading overflow. This violates strict concurrency control but ensures liveness.\n\n### 5.3 Virtual Dispatch Start (Anti-Windup) Limitations\n\n**Problem:** A queue that alternates between idle and active can accumulate a very early `nextDispatchR`, gaining unfair priority.\n\n**Partial mitigation:** boundNextDispatchLocked bounds `nextDispatchR \u2265 oldestWaitingRequest.arrivalR`.\n\n**Limitation:** This bound is conservative. If a queue is idle for 1 hour, then receives a request, its virtual start is bumped to the current virtual time, but by then global virtual time has advanced ~10^6 seat-seconds. The queue may have gained priority unfairly.\n\n**Why not fully resync?** Setting `nextDispatchR = currentR` for each idle queue would be too aggressive, allowing \"fresh\" queues to cut in line.\n\n### 5.4 Queue Attrition and Configuration Changes\n\n**Problem:** If desired number of queues decreases, we can't immediately remove occupied queues.\n\n**Handling** (setConfiguration, queueset.go:254\u2013287):\n\n```\nIf DesiredNumQueues decreases from 10 to 5:\n  1. Do not remove extra queues immediately\n  2. Only remove queues when they become empty (attrition)\n  3. Other queue parameters (QueueLengthLimit) frozen while removing queues\n```\n\n**Trade-off:** Transient over-provisioning until queues drain. Memory impact is bounded by queue count (each queue is ~200 bytes) and request count.\n\n### 5.5 Queue Length Limits and Rejection\n\n**Rejection criteria** (rejectOrEnqueueToBoundLocked, queueset.go:618\u2013629):\n\n```\nReject if:\n  totSeatsInUse >= ConcurrencyLimit AND\n  curQueueLength >= QueueLengthLimit\n\nRationale:\n  - Prevent unbounded queue growth (memory DOS)\n  - Allow queueing when concurrency is available (latency isolation)\n  - Admit if either constraint has room\n```\n\n**Problem:** This is an AND condition, not OR. If concurrency limit is hit but queue is short, request is admitted. If queue is full but concurrency available, request is admitted. Only when both are maxed is it rejected.\n\n**Justification:** Reduces unfair rejection during transient overload.\n\n### 5.6 Estimation Error and Service Time Feedback\n\n**Problem:** `EstimatedServiceDuration` is fixed (default 3ms) per queueset. Actual service times vary widely.\n\n**Feedback mechanism** (finishRequestLocked, queueset.go:937\u2013938):\n\n```\nqueue.nextDispatchR -= SeatsTimesDuration(req.InitialSeats,\n                                           estimatedServiceDuration - actualServiceDuration)\n```\n\n**Effect:** If actual < estimated, queue's virtual start decreases (gains priority). If actual > estimated, increases (loses priority).\n\n**Limitation:** Feedback is local to queue; a flow's error doesn't propagate to other queues in the hand. Flow-wide fairness can degrade if one queue overestimates.\n\n### 5.7 Metrics Observation Lag\n\n**Race:** Metrics are updated under lock, but observed external to lock. Between observation and action, state may change.\n\n**Accepted:** This is inherent to sampling metrics. APF clients should not act on single observations.\n\n---\n\n## 6. Operational Implications: Performance, Reliability, and Maintainability\n\n### 7.1 Concurrency and Locking Strategy\n\n**Lock design:**\n- Single `sync.Mutex` per queueset protects all mutable state\n- Lock held during:\n  - Initial request ingress (StartRequest)\n  - Virtual time updates (syncTimeLocked)\n  - All dispatch operations\n  - Request finish handling\n\n**Lock release:**\n- Lock released immediately after decision is made\n- Actual request execution runs outside lock\n- Decision delivery (promise.Get) doesn't hold lock\n\n**Performance implications:**\n```\nLock contention = (arrival rate) \u00d7 (average time in critical section)\n\nTypical: ~1\u03bcs per StartRequest + ~100ns per syncTimeLocked\n         With 10k req/s: ~10ms contention per second (acceptable)\n\nWorst case: Large queuesets (100+ queues) with high arrival rates\n            Round-robin dispatch loop scales O(queues) per dispatch\n            With 1000 queues: ~5\u03bcs per dispatch\n```\n\n**Mitigation:** Lock-free data structures for queue peeking (future work).\n\n### 7.2 Memory Footprint\n\nPer queueset:\n- `queueSet` struct: ~500 bytes\n- Per queue: ~200 bytes (FIFO list, sync.Set)\n- Per request: ~256 bytes (context, metadata, promise)\n\n**Example:** 100 queues + 1000 waiting requests = 100\u00d7200 + 1000\u00d7256 = ~280KB\n\n**Scalability:** Memory grows with O(queues + waiting requests). No inherent limits; configured via QueueLengthLimit.\n\n### 7.3 Virtual Time Stability\n\n**Risk:** Overflow, underflow, or monotonicity violations can corrupt fairness.\n\n**Safeguards:**\n1. **Epoch advancement:** Detects overflow, logs warnings if underflow occurs during decrement\n2. **Monotonicity checks:** `boundNextDispatchLocked` asserts `nextDispatchR >= arrivalR`\n3. **Metrics correlation:** Dispatch metrics include virtual time (external monitoring can alert)\n\n**Operational alert:** If `epoch_advance_errors` > 0, investigate virtual time corruption.\n\n### 7.4 Tuning Parameters and Trade-offs\n\n| Parameter | Location | Impact | Notes |\n|-----------|----------|--------|-------|\n| `DesiredNumQueues` | QueuingConfig | Fairness, latency | Higher \u2192 better fairness, higher overhead |\n| `HandSize` | QueuingConfig | Collision, predictability | Higher \u2192 lower collision, higher shuffle cost |\n| `QueueLengthLimit` | QueuingConfig | Memory, rejection rate | Higher \u2192 more memory, less rejection |\n| `ConcurrencyLimit` | DispatchingConfig | Throughput, latency | Sweet spot depends on hardware |\n| `EstimatedServiceDuration` | queueset.go:215 | Seat allocation fairness | Too high \u2192 underutilization; too low \u2192 overload |\n\n### 7.5 Observability Checklist\n\n**Key metrics to monitor:**\n1. **`apiserver_priority_level_request_utilization`**: Executing requests / limit (target: 0.7\u20130.9)\n2. **`apiserver_priority_level_seat_utilization`**: Occupied seats / limit (target: 0.6\u20130.8)\n3. **`apiserver_priority_level_request_wait_duration`**: P50, P99 queueing latency\n4. **`apiserver_request_total` with `code=429`**: Rejected requests (trending up = concern)\n5. **`apiserver_priority_level_queued_requests`**: Requests waiting (should be < QueueLengthLimit \u00d7 DesiredNumQueues / 2)\n6. **`apiserver_priority_level_epoch_advance_total`**: Epoch advancements (OK to see every 1\u201310 seconds under load)\n7. **`apiserver_flowschema_dispatched_requests_total`**: By flow schema (check for fairness across flows)\n\n### 7.6 Debugging and Introspection\n\n**Available methods:**\n- `queueSet.Dump(includeRequestDetails bool)`: Snapshots internal state (lock held during snapshot)\n  - Returns requests in queues, executing requests, virtual times, seat counts\n  - Safe for live queries (not async)\n\n**Limitations:**\n- Dump is point-in-time; # Kubernetes APF QueueSet: Algorithmic Deep-Dive

## Executive Summary

The QueueSet subsystem implements **Fair Queuing for Server Requests (FQSR)**, a specialized fair-queuing algorithm that multiplexes concurrent API server requests across priority levels. It provides weighted fairness guarantees through virtual time scheduling, shuffle-sharded queue assignment, and adaptive dispatch based on estimated work (seat-seconds). This document traces the end-to-end control flow, dissects the fairness algorithm, and discusses operational tradeoffs.

---

## 1. Subsystem Purpose, Boundaries, and Dependencies

### 1.1 Purpose

The QueueSet is the **execution scheduling engine** of API Priority and Fairness (APF). It sits between the APF filter (which classifies requests) and the main request handler (which executes them). Its role:

- **Admission control**: Reject or queue requests when concurrency limits are exceeded
- **Fair dispatch**: Select which queued request executes next, ensuring:
  - Proportional fairness among flows (via virtual time)
  - Starvation prevention (via round-robin queue selection)
  - Work-aware scheduling (via seat-seconds, not just request count)
- **Concurrency accounting**: Track occupied "seats" (a logical concurrency unit) across phases of request execution

**Upstream dependencies:**
- **APF Controller** (`staging/src/k8s.io/apiserver/pkg/util/flowcontrol/apf_controller.go`): Provides `QueuingConfig` (desired number of queues, queue length limit, hand size for shuffle sharding) and `DispatchingConfig` (concurrency limit)
- **Request classifiers**: Extract flow distinguisher and match requests to priority levels
