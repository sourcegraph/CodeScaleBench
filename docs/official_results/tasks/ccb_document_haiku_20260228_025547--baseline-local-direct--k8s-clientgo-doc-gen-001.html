  "content": "# Kubernetes client-go Systems Guide\n\n## Overview\n\n`client-go` is the foundational Go library for programmatic interaction with Kubernetes clusters. It provides multiple abstraction layers to support diverse use cases\u2014from simple CLI tools to complex controllers managing distributed systems. The subsystem is architecturally designed to balance type safety, performance, extensibility, and operational resilience.\n\n## 1. Subsystem Purpose, Boundaries, and Dependencies\n\n### Purpose\n\nclient-go serves as the canonical interface for Go programs to:\n- **Query and mutate** Kubernetes API objects (pods, deployments, custom resources, etc.)\n- **Watch** cluster state changes in real-time\n- **Discover** available APIs and their schemas\n- **Cache** API state locally with automatic synchronization\n- **Coordinate** multi-replica controllers via leader election\n- **Handle authentication**, authorization, and transport layer security\n\n### Subsystem Boundaries\n\n**Upstream (dependencies):**\n- `k8s.io/apimachinery`: Core runtime objects (ObjectMeta, TypeMeta, runtime.Object), watch interfaces, and versioning primitives\n- `k8s.io/api`: Generated type definitions for all Kubernetes API resources (Pod, Deployment, etc.)\n- `k8s.io/utils`, `k8s.io/klog`: Utility libraries (time management, logging, sets)\n- Standard library: `net/http`, `encoding/json`, `crypto/tls`\n\n**Downstream (consumers):**\n- **Controllers**: Operators, custom controllers that reconcile cluster state\n- **CLI tools**: kubectl (internally uses client-go)\n- **Webhooks**: Admission controllers, mutating/validating webhooks\n- **Monitoring/observability**: Custom metrics collectors, log aggregators\n- **End-user applications**: Any program that needs Kubernetes integration\n\n### Scope and Separation of Concerns\n\nclient-go does **not**:\n- Generate or manage Kubernetes API objects themselves (that's `k8s.io/api` via code-gen)\n- Provide control plane functionality (that's kube-apiserver's job)\n- Handle specific business logic for controller reconciliation patterns (that's the consumer's responsibility)\n- Enforce cluster-wide ordering guarantees (watch events may arrive out-of-order; the consumer must handle this)\n\nclient-go **does**:\n- Manage HTTP transport, authentication, and TLS\n- Serialize/deserialize Kubernetes objects (JSON, Protobuf, CBOR)\n- Maintain an in-process cache synchronized via watch\n- Provide rate limiting and exponential backoff\n- Support both typed (generated code) and untyped (dynamic) access\n\n---\n\n## 2. Key Components and Responsibility Split\n\nclient-go is organized into several tightly integrated subsystems:\n\n### 2.1 REST Foundation (`rest/` package)\n\n**Responsibility:** Low-level HTTP client and configuration management.\n\n**Key Files:**\n- `rest/config.go`: `Config` struct holding credentials, API server URL, TLS settings, rate limits\n- `rest/client.go`: `RESTClient` implementing verb-based request builder pattern (e.g., `.Get().Resource(\"pods\").Into(&pod)`)\n- `rest/request.go`: Request pipeline\u2014constructs URLs, handles serialization, error parsing\n- `rest/transport.go`: HTTP round-tripper composition for authentication, compression, retry logic\n- `rest/with_retry.go`: Exponential backoff with jitter for transient failures\n\n**Design Pattern:**\n- **Builder pattern** for requests: `.Verb(\"GET\").Namespace(\"default\").Resource(\"pods\").Name(\"my-pod\").Do(ctx)` defers I/O until terminal method is invoked\n- **RoundTripper chain**: Authentication, rate limiting, warning handler, and compression are injected as middleware layers\n- **Content negotiation**: Accept headers (JSON, Protobuf, CBOR, metadata-only) allow clients to optimize wire format per request\n\n**Example Request Flow:**\n```\nRESTClient.Get()\n  \u2192 *Request (builder object, no I/O yet)\n  \u2192 .Namespace(\"default\")\n  \u2192 .Resource(\"pods\")\n  \u2192 .Name(\"my-pod\")\n  \u2192 .Do(ctx)  \u2190 actual HTTP call via transport chain\n```\n\n### 2.2 Configuration and Authentication (`rest/config.go`, `tools/clientcmd/`)\n\n**Responsibility:** Load credentials and cluster endpoints from kubeconfig files or in-cluster service accounts.\n\n**Key Abstractions:**\n- `rest.Config`: Immutable configuration object (host, TLS certs, bearer token, rate limits)\n  - `QPS`: Requests per second (default 5)\n  - `Burst`: Burst capacity above sustained QPS (default 10)\n  - `WrapTransport`: Hook for custom middleware\n- `rest.InClusterConfig()`: Reads `/var/run/secrets/kubernetes.io/serviceaccount/{token,ca.crt}` and `KUBERNETES_SERVICE_HOST` environment variables\n- `clientcmd.BuildConfigFromFlags()`: Parses `~/.kube/config`, merges contexts, handles external auth providers (OIDC, exec-based)\n\n**Separation of Concerns:**\n- Configuration **loading** is intentionally separated from **usage**, allowing decoupling of kubeconfig parsing logic from client code\n- Auth providers (exec-based, plugin-based) are pluggable via `AuthProviderConfig` and `ExecConfig`\n- TLS mutual authentication is configured at the REST transport level, not at the Kubernetes object level\n\n### 2.3 Client Models: Typed vs. Dynamic\n\nclient-go provides two complementary client models:\n\n#### 2.3.1 Typed Client (`kubernetes.Clientset`)\n\n**Responsibility:** Compile-time type-safe access to core and built-in API resources.\n\n**Key Files:**\n- `kubernetes/clientset.go`: Aggregates clients for all API groups and versions\n- `kubernetes/typed/core/v1/pod.go` (auto-generated): Interface `PodsGetter` with `Pods(namespace string) PodInterface`\n- Generated via `k8s.io/code-generator`: code-gen tools produce strongly-typed interfaces for each API GroupVersion\n\n**Design:**\n- Each API group version (e.g., `apps/v1`, `batch/v1`) has a generated client\n- Namespace-scoped and cluster-scoped resources are differentiated at the interface level\n- Subresources (`/status`, `/scale`) are exposed as separate methods\n\n**Usage Example:**\n```go\nclientset.CoreV1().Pods(\"default\").Get(ctx, \"my-pod\", metav1.GetOptions{})\nclientset.AppsV1().Deployments(\"default\").Update(ctx, deployment, metav1.UpdateOptions{})\n```\n\n**Limitation:** Requires code generation for new API types; cannot access CRDs without additional code generation\n\n#### 2.3.2 Dynamic Client (`dynamic.DynamicClient`)\n\n**Responsibility:** Runtime-polymorphic access to any API resource (including CRDs) using `unstructured.Unstructured`.\n\n**Key Files:**\n- `dynamic/simple.go`: `DynamicClient` wraps a `rest.Interface`\n- `dynamic/interface.go`: `Resource()` method returns a `ResourceInterface` for any GVR (GroupVersionResource)\n\n**Design:**\n- Works with any API resource without code generation\n- Operates on untyped JSON/CBOR maps: `.Get()` returns `*unstructured.Unstructured`\n- Relies on discovery to determine namespace scope, subresources, etc.\n\n**Usage Example:**\n```go\ndynClient := dynamic.NewForConfig(config)\ngvr := schema.GroupVersionResource{Group: \"example.com\", Version: \"v1\", Resource: \"foos\"}\nobj, err := dynClient.Resource(gvr).Namespace(\"default\").Get(ctx, \"my-foo\", metav1.GetOptions{})\n// obj is *unstructured.Unstructured\n```\n\n**Trade-offs:**\n- **Pro:** Immediate support for any new API type (including CRDs)\n- **Con:** No compile-time type checking; schema validation is deferred to runtime via OpenAPI spec\n\n### 2.4 Discovery (`discovery/` package)\n\n**Responsibility:** Determine which APIs exist on the cluster and their structure.\n\n**Key Files:**\n- `discovery/discovery_client.go`: `DiscoveryClient` queries `/api`, `/apis`, `/openapi/v3`\n- `discovery/cached/disk/cached_discovery.go`: Caches discovery data on disk (e.g., `~/.kube/cache/discovery`)\n- `discovery/aggregated_discovery.go`: Implements newer aggregated discovery format (APIGroupDiscoveryList)\n\n**Three Discovery Mechanisms:**\n1. **Legacy Discovery** (`ServerGroups()`, `ServerResourcesForGroupVersion()`)\n   - Lightweight: returns list of groups and their resources\n   - No schema information; used primarily by `dynamic` client to determine namespace scope\n2. **OpenAPI v2** (deprecated): Full JSON schema via `/openapi/v2`\n3. **OpenAPI v3** (preferred): Modular JSON schemas via `/openapi/v3/{group}/{version}/...`\n\n**Failure Handling:**\n- Caching solves the \"cluster unreachable\" problem for tools that cached discovery data\n- Disk cache allows kubectl to function even when API server is temporarily unavailable\n- Aggregated discovery (v2) streams all information in one request, reducing round-trips\n\n**Usage in Practice:**\n- `dynamic.DynamicClient` calls discovery to determine if a resource is namespaced\n- Tools like kubectl use discovery to populate CLI completion and validate resource types\n- Controllers typically **don't** use discovery at runtime; they use pre-generated, compile-time-known types\n\n### 2.5 Controller Infrastructure (`tools/cache/` package)\n\n**Responsibility:** Provide an event-driven, cached model for watching cluster state changes.\n\n**Key Components:**\n\n#### 2.5.1 Reflector (`tools/cache/reflector.go`)\n\n**Responsibility:** Synchronize a local cache with the API server's authoritative state.\n\n**Design:**\n1. Performs a **LIST** to snapshot all current objects (identified by `resourceVersion`)\n2. Starts a **WATCH** from that `resourceVersion` to receive subsequent changes\n3. Handles the \"too old resourceVersion\" error by re-listing with exponential backoff\n4. Consumes watch bookmarks to advance the restart point without re-listing\n\n**Key Parameters:**\n- `minWatchTimeout` (default 5min): Minimum timeout for watch requests to spread load\n- `resyncPeriod`: Periodic re-list frequency (e.g., 10 minutes for informers)\n- `WatchListPageSize`: Request pagination during initial list and re-list\n\n**Failure Modes:**\n- **\"Too old resourceVersion\"**: Watch stream is too stale; Reflector must re-list. Bookmarks reduce cost.\n- **Network partition**: Watch connection dropped; exponential backoff before retry\n- **API server restarts**: Watch stream terminates; Reflector automatically resumes\n\n#### 2.5.2 DeltaFIFO (`tools/cache/delta_fifo.go`)\n\n**Responsibility:** Queue of object deltas that preserves the sequence of changes for each object.\n\n**Key Insight:**\n- Unlike a simple FIFO (which stores objects), DeltaFIFO stores `Deltas` (slice of changes)\n- Example: `pod1: [Added, Modified, Modified, Deleted]`\n- Allows the consumer to replay the entire history of changes without losing intermediate states\n\n**Delta Types:**\n- `Added`: Object appeared in LIST or WATCH ADDED event\n- `Updated`: Object was mutated (WATCH MODIFIED event)\n- `Deleted`: Object was removed\n- `Replaced`: Result of a LIST operation (if `EmitDeltaTypeReplaced` is true)\n- `Sync`: Resync operation (periodic re-processing)\n\n**Usage:**\n- Reflector **pushes** object changes into DeltaFIFO\n- Controller **pops** deltas and processes them sequentially\n- Deduplication: If last delta is `Deleted` and new delta is also `Deleted`, the FIFO doesn't grow\n\n#### 2.5.3 Indexer (Thread-safe Cache) (`tools/cache/store.go`, `tools/cache/index.go`)\n\n**Responsibility:** In-memory, thread-safe store of objects with secondary indices for efficient queries.\n\n**Key Files:**\n- `tools/cache/store.go`: `Store` interface and `cache` implementation\n- `tools/cache/index.go`: `Indexer` interface with `Index()`, `IndexKeys()`, `ByIndex()` methods\n\n**Features:**\n- **Thread-safe**: RWMutex guards reads and writes\n- **Secondary indices**: Index by namespace, owner references, labels, or custom functions\n- **Efficient queries**: `ByIndex(\"namespace\", \"default\")` returns all objects in that namespace without full scan\n\n**Example:**\n```go\nindexer.Add(pod)  // pod added to cache\nindexer.ByIndex(\"namespace\", \"default\")  // O(1) lookup via index\n```\n\n#### 2.5.4 Event Handlers and Informer Pattern\n\n**Responsibility:** Notify consumers of object changes and manage event ordering.\n\n**Key Files:**\n- `tools/cache/shared_informer.go`: `SharedInformer` aggregates event listeners\n- `tools/cache/controller.go`: `Config` and `Run()` orchestrating the full flow\n\n**Event Handler Pattern:**\n```go\nhandler := cache.ResourceEventHandlerFuncs{\n  AddFunc: func(obj interface{}) { workQueue.Add(obj) },\n  UpdateFunc: func(oldObj, newObj interface{}) { workQueue.Add(newObj) },\n  DeleteFunc: func(obj interface{}) { workQueue.Add(obj) },\n}\ninformer.AddEventHandler(handler)\n```\n\n**Design Principle:** Event handlers **must not perform expensive operations**. They add work items to a queue; the controller processes items from the queue with retries and backoff.\n\n### 2.6 Work Queue (`util/workqueue/`)\n\n**Responsibility:** Decouple event detection from business logic processing.\n\n**Key Files:**\n- `util/workqueue/queue.go`: `Interface` (Add, Get, Done, Len)\n- `util/workqueue/delaying_queue.go`: `DelayingInterface` (AddAfter)\n- `util/workqueue/rate_limiting_queue.go`: `RateLimitingInterface` (AddRateLimited)\n- `util/workqueue/default_rate_limiters.go`: Built-in rate limiters (exponential backoff, max retries)\n\n**Responsibilities Separated:**\n1. **Detection**: Informer handler adds a work item (e.g., \"default/my-pod\")\n2. **Queueing**: Item waits in the work queue\n3. **Processing**: Controller worker pops item, processes it, calls `Done()` on success or `AddRateLimited()` on failure\n4. **Rate Limiting**: Failed items are re-queued with exponential backoff\n\n**Design:**\n- Work queue stores **keys** (e.g., \"namespace/name\"), not full objects\n- Controller fetches the object from cache (via Lister) when processing the key\n- If object is not in cache (deleted), controller handles the \"not found\" gracefully\n\n**Example Retry Flow:**\n```\nItem \"default/pod1\" fails to reconcile\n\u2192 workqueue.AddRateLimited(\"default/pod1\")  // exponential backoff\n\u2192 2 seconds later, item re-queued\n\u2192 4 seconds later, if failed again\n\u2192 8 seconds later, if failed again\n\u2192 Max retries reached (or max duration), item dropped\n```\n\n### 2.7 Listers (`listers/` and `informers/` packages)\n\n**Responsibility:** Provide read-only, type-safe access to the cache for controllers.\n\n**Key Files:**\n- `listers/core/v1/pod_lister.go` (auto-generated): Interface with methods like `Pods(namespace string) PodNamespaceLister`\n- `informers/core/v1/pod.go` (auto-generated): `PodInformer` with `Lister()` method returning a `PodLister`\n\n**Design:**\n- Generated from API types via code-gen\n- Lister implements `ByIndex()`, `List()` with field selectors\n- Controller never calls `.Get()` directly on the client; it uses the Lister for cached access\n\n**Usage:**\n```go\npodLister := podInformer.Lister()\npod, err := podLister.Pods(\"default\").Get(\"my-pod\")  // cache lookup, no API call\npods, err := podLister.Pods(\"default\").List(selector)  // filtered cache lookup\n```\n\n### 2.8 Server-Side Apply (`applyconfigurations/`, `util/apply/`)\n\n**Responsibility:** Provide a declarative, merge-aware API for object mutation that allows multiple controllers to co-manage the same object.\n\n**Key Files:**\n- `applyconfigurations/apps/v1/deployment_applyconfiguration.go` (auto-generated): Type-safe builder API\n- `util/apply/apply.go`: Helper functions for constructing apply patches\n\n**Contrast with Traditional Update:**\n- **Traditional GET-MODIFY-UPDATE**: Retrieve object \u2192 mutate fields \u2192 UPDATE. Requires **exclusive ownership**.\n- **Server-Side Apply (kubectl apply -f)**: Send desired state + fieldManager \u2192 server merges with other managers. Allows **shared ownership**.\n\n**Example:**\n```go\n// Traditional: risky if someone else modified the object\ndeployment, _ := clientset.AppsV1().Deployments(\"default\").Get(ctx, \"my-app\", metav1.GetOptions{})\ndeployment.Spec.Replicas = ptr.To(int32(3))\nclientset.AppsV1().Deployments(\"default\").Update(ctx, deployment, metav1.UpdateOptions{})\n\n// Server-Side Apply: safe for shared ownership\ndeploymentAC := appsv1ac.Deployment(\"my-app\", \"default\").\n  WithSpec(appsv1ac.DeploymentSpec().WithReplicas(3))\nclientset.AppsV1().Deployments(\"default\").Apply(ctx, deploymentAC, metav1.ApplyOptions{FieldManager: \"my-controller\"})\n```\n\n### 2.9 Leader Election (`tools/leaderelection/`)\n\n**Responsibility:** Ensure only one controller replica actively reconciles at a time.\n\n**Key Files:**\n- `tools/leaderelection/leaderelection.go`: `LeaderElector` with `Run(ctx)` method\n- `tools/leaderelection/resourcelock/lease.go`: Uses `coordination.k8s.io/v1/Lease` objects for locking\n\n**Design:**\n- All replicas compete for a lock on a shared `Lease` object\n- Lock holder periodically renews; if renewal fails, lock is released and another replica can acquire\n- OnNewLeader and OnStoppedLeading callbacks let controller start/stop reconciliation\n\n**Example Flow:**\n```\nReplica-1, Replica-2, Replica-3 all running\n\u2192 All try to acquire lock on \"my-controller-lock\" Lease\n\u2192 Replica-2 succeeds: becomes leader\n\u2192 Replica-2 renews lock every `renewDeadline`\n\u2192 Replica-2 crashes\n\u2192 Lock expires after `leaseDuration`\n\u2192 Replica-3 acquires lock: becomes new leader\n```\n\n---\n\n## 3. Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### 4.1.1 Watch Stream Goes Stale (\"too old resourceVersion\")\n\n**Symptom:**\n```\nHTTP 410 Gone: the object has been deleted because the watch stream fell too far behind\n```\n\n**Root Cause:**\n- Reflector was paused (e.g., due to slow event handler goroutine)\n- Watch stream disconnected; reconnect attempted with stale resourceVersion\n- API server's watch cache (default ~5min) has already discarded that version\n\n**Mitigation:**\n1. **Bookmarks** (KEP-1904): API server sends BOOKMARK events with fresh resourceVersion\n   - Reflector consumes bookmarks, advancing restart point without full re-list\n   - Reduces cost of recovery when watch is merely slow, not stale\n2. **Automatic re-list**: Reflector detects 410 error, performs full LIST, resumes WATCH\n3. **User-controlled resyncPeriod**: Periodic re-list (e.g., every 10 minutes) ensures catch-up even if watch somehow falls behind\n\n**Evidence:** `tools/cache/reflector.go:220-240` (isExpiredError, LIST recovery)\n\n#### 4.1.2 API Server Becomes Unreachable\n\n**Symptom:**\n- All requests fail with connection timeouts\n- Informer watches stuck (cannot reconnect)\n\n**Mitigation:**\n1. **Client-side rate limiting**: RESTClient returns errors immediately when API is unreachable; exponential backoff in controller's retry loop prevents thundering herd\n2. **Discovery caching**: Tools like kubectl can still work if they cached discovery data (e.g., in `~/.kube/cache/discovery`)\n3. **Controller's work queue**: Decouples watch (which is stuck) from reconciliation logic; items remain queued during outage, retried after recovery\n\n**Limitation:** In-process cache becomes stale; no updates for duration of outage. Controllers must tolerate eventual consistency.\n\n#### 4.1.3 Concurrent Modification Conflicts\n\n**Scenario:**\nTwo controllers both call `Update()` on the same object without coordination.\n\n**Symptom:**\n```\nHTTP 409 Conflict: the object has been modified; refresh and retry\n```\n\n**Mitigation:**\n1. **Traditional approach**: Use Server-Side Apply instead of Update (allows shared ownership)\n2. **Leader election**: Only one controller manages the object at a time\n3. **Optimistic locking**: Check `metadata.resourceVersion` before updating; if mismatch, retry\n\n**Evidence:** `tools/leaderelection/leaderelection.go` (Lease-based lock), `applyconfigurations/` (server-side apply pattern)\n\n#### 4.1.4 Memory Pressure from Large Cache\n\n**Scenario:**\nController watching millions of objects; Indexer cache consumes significant memory.\n\n**Symptom:**\n- OOMKilled pod (if deployed in Kubernetes)\n- Slow GC pauses due to large heap\n\n**Mitigation:**\n1. **Selective watching**: Only add event handler for objects in specific namespace or label selector\n   - Reflector respects `FieldSelector` and `LabelSelector` in ListOptions\n   - API server returns fewer objects; cache is smaller\n2. **Resync period tuning**: Reduce resyncPeriod if periodic reprocessing is expensive\n3. **Custom indexing**: Only index on critical fields (namespace, owner references); avoid indexing large label maps\n\n**Design Tradeoff:** In-process caching trades memory for latency (cache lookups are O(1) vs. API calls which are O(100ms+))\n\n#### 4.1.5 Event Handler Panic\n\n**Scenario:**\nUser's event handler panics.\n\n**Symptom:**\n- Goroutine crashes\n- Event stream stops for that informer\n\n**Mitigation:**\n- Informers recover panics internally; panics are logged but don't crash the watch\n- Work items are not added if handler panics, so reconciliation doesn't happen\n- Solution: **Wrap your handler in defer + recover** if you need to ensure other handlers run\n\n**Evidence:** `tools/cache/shared_informer.go:490-510` (processorListener handles panics)\n\n### 4.2 Limits and Constraints\n\n#### 4.2.1 Rate Limiting\n\n**Client-Side (rest.Config):**\n- Default QPS: 5, Burst: 10\n- Enforced per RESTClient instance via token bucket\n- Does NOT enforce cluster-wide limits\n- Can be disabled by setting QPS < 0 and RateLimiter = nil\n\n**Interaction with Server-Side Limits:**\n- API Priority and Fairness (APF): Server has separate rate limiting by priority level, flow schema\n- Client QPS + Burst settings coordinate with server's APF; they are two independent rate limits\n\n**Lesson:** Setting client QPS=1000 does not mean requests arrive at 1000/sec; APF may throttle them further\n\n#### 4.2.2 Watch Stream Timeouts\n\n**Default minWatchTimeout:** 5 minutes (randomized to [5m, 10m) to spread load)\n\n**Implication:**\n- Watch streams are deliberately short-lived to allow graceful API server scaling (e.g., restarting kube-apiserver)\n- Controllers must be prepared for frequent watch reconnects\n- Exponential backoff prevents reconnect thundering\n\n**Tuning:** `Config.MinWatchTimeout` can be increased for slow clients, but values < 5min are ignored (enforced server-side)\n\n#### 4.2.3 List Pagination\n\n**Default page size:** 500 (controlled by API server)\n\n**Impact:**\n- Large LIST operations are chunked; client must handle continue tokens\n- `Reflector.ListAndWatch()` internally handles pagination; users typically don't see it\n- Exception: `DynamicClient.List()` with large result sets may require manual pagination\n\n**Code Location:** `tools/cache/listwatch.go:93` (NewListWatchFromClient handles pagination)\n\n#### 4.2.4 Cache Resync Skew\n\n**Scenario:**\nMultiple SharedInformerFactories with different resyncPeriods create thundering herd during resync windows.\n\n**Mitigation:**\n- Each Reflector adds random jitter to resync interval (\u00b110%)\n- Stagger initial resync times to avoid all informers re-listing simultaneously\n\n**Evidence:** `tools/cache/reflector.go:162-170` (resyncPeriod + jitter)\n\n### 4.3 Design Tradeoffs\n\n#### 4.3.1 Eventual Consistency vs. Strong Consistency\n\n**client-go Design:** Eventual consistency\n\n- Objects in the cache may lag API server state by seconds (watch propagation time) or minutes (if watch stream is stale)\n- Reads from cache are fast (O(1)) but potentially stale\n- Writes go through API server (always authoritative)\n\n**Implication:**\n- Controllers **cannot assume** that a recently-updated object is visible in the cache\n- Must add defensive checks: \"object I just created is not in cache? Retry.\"\n- Acceptable for most Kubernetes controllers (reconciliation loops are idempotent)\n\n#### 4.3.2 Single-Writer (Leader Election) vs. Multi-Writer (Server-Side Apply)\n\n| Pattern | Use Case | Tradeoff |\n|---------|----------|----------|\n| Leader Election (`tools/leaderelection/`) | Only one controller active at a time | Simple, but single point of failure during leadership transfer |\n| Server-Side Apply (`applyconfigurations/`) | Multiple controllers manage different fields | Requires schema awareness of field ownership, more complex coordination |\n\n**Evidence:**\n- Leader election: `tools/leaderelection/leaderelection.go`\n- Server-Side Apply: `applyconfigurations/apps/v1/deployment_applyconfiguration.go` (generated)\n\n#### 4.3.3 Typed Client (compile-time) vs. Dynamic Client (runtime)\n\n| Model | Use Case | Tradeoff |\n|-------|----------|----------|\n| **Typed** (`kubernetes.Clientset`) | Built-in Kubernetes APIs | Requires code generation for new types; type-safe |\n| **Dynamic** (`dynamic.DynamicClient`) | CRDs, extensible APIs | No code generation needed; discover at runtime; no type safety |\n\n**Performance Implication:**\n- Typed: No discovery overhead (types known at compile time)\n- Dynamic: Must call discovery on first access to determine namespace scope; future accesses cached\n\n#### 4.3.4 Caching Model: DeltaFIFO vs. Simple Store\n\n**DeltaFIFO:**\n- Accumulates per-object history (Deltas)\n- Allows consumers to see the full sequence of changes (Added \u2192 Modified \u2192 Modified \u2192 Deleted)\n- Higher memory overhead (stores all deltas until popped)\n\n**Simple Store (List-Watch without DeltaFIFO):**\n- Stores only current state\n- Lower memory, but loses intermediate events\n\n**Design Decision:** client-go uses DeltaFIFO by default because controllers often need to reconcile state correctly after a burst of rapid changes. Storing intermediate deltas allows controllers to avoid missing state.\n\n---\n\n## 5. Operational Implications\n\n### 6.1 Performance Characteristics\n\n#### Memory\n\n**Typical Consumption:**\n- ~1 KB per small object (Pod) in cache\n- 100,000 Pods \u2248 100 MB cache\n- Add 20-30% overhead for indices and synchronization primitives\n\n**Optimization:**\n- Reduce watch scope (namespace selectors, label selectors)\n- Tune resyncPeriod (less frequent = smaller periodic LIST overhead)\n- Custom indexing on essential fields only\n\n**Monitoring:**\n```\ngo_memstats_heap_alloc_bytes  # total heap memory\ninformer_cache_populate_duration_seconds  # LIST/WATCH latency\n```\n\n#### Latency\n\n**Object Update Latency (from API server to cache):**\n- Watch propagation: typically 10-100 ms\n- Plus event handler delay + work queue processing\n- Total: 100-500 ms typical for simple controllers\n\n**Optimization:**\n- Minimize event handler work (add to queue only)\n- Use rate limiting only for retries, not success path\n- Profile slow handlers with pprof\n\n**Monitoring:**\n```\nworkqueue_adds_total  # new items entering queue\nworkqueue_depth  # queue length (backlog indicator)\nworkqueue_work_duration_seconds  # actual reconciliation time\n```\n\n#### Network\n\n**Bandwidth per controller:**\n- LIST operation: O(number of objects)\n- WATCH: small per event (delta only, not full object)\n- Resync: full LIST again (avoid if possible)\n\n**Optimization:**\n- Use FieldSelector to reduce LIST results (server-side filtering)\n- Reduce resyncPeriod if not needed\n- Use dynamic client sparingly (adds discovery overhead)\n\n### 6.2 Reliability and Recovery\n\n#### Graceful Shutdown\n\n```go\nstopCh := make(chan struct{})\nfactory.Start(stopCh)\n\n// On SIGTERM\nsigCh := make(chan os.Signal, 1)\nsignal.Notify(sigCh, syscall.SIGTERM)\n<-sigCh\nclose(stopCh)\n\n// Wait for informers to stop (max 30 sec grace period)\ntime.Sleep(30 * time.Second)\n```\n\n**Key Point:** Closing stopCh does not immediately stop all goroutines; give them time to finish draining work queue before exiting.\n\n#### Crash Recovery\n\n**What happens when controller pod is deleted/restarted:**\n1. All in-memory cache is lost\n2. Lease lock (for leader election) is released (after leaseDuration expires or actively renounced)\n3. New instance restarts with cold cache\n4. Reflector re-LISTs to repopulate\n5. Controller resumes processing work queue items\n\n**Implication:** Reconciliation must be idempotent (applying same desired state twice yields same result).\n\n#### Watch Stream Stale Handling\n\n**Automatic recovery:**\n```\nHTTP 410 error (watch too old)\n\u2192 Reflector backoffs with exponential jitter\n\u2192 Re-LIST with fresh resourceVersion\n\u2192 Resume WATCH\n\u2192 No events lost during re-LIST window (already accumulated in work queue)\n```\n\n**Monitoring:**\n```\nreflector_lists_total  # count of LIST operations (high = frequent staleness recovery)\nreflector_list_duration_seconds  # LIST latency\n```\n\n### 6.3 Observability and Debugging\n\n#### Key Metrics\n\n1. **Reflector health:**\n   - `reflector_lists_total`: Incrementing slowly (healthy) vs. rapidly (watch is stale)\n   - `reflector_list_duration_seconds`: Latency of LIST operations\n   - `reflector_watch_duration_seconds`: Watch stream length (longer = healthier)\n\n2. **Work queue health:**\n   - `workqueue_depth`: Should be low (0-10) in steady state; high = backed up\n   - `workqueue_adds_total`: Event rate\n   - `workqueue_work_duration_seconds`: Reconciliation latency\n   - `workqueue_retries_total`: Failed reconciliations (should be low)\n\n3. **Informer cache health:**\n   - `informer_cache_size`: Number # Kubernetes client-go Systems Guide

## Overview

`client-go` is the foundational Go library for programmatic interaction with Kubernetes clusters. It provides multiple abstraction layers to support diverse use cases—from simple CLI tools to complex controllers managing distributed systems. The subsystem is architecturally designed to balance type safety, performance, extensibility, and operational resilience.

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose

client-go serves as the canonical interface for Go programs to:
- **Query and mutate** Kubernetes API objects (pods, deployments, custom resources, etc.)
- **Watch** cluster state changes in real-time
- **Discover** available APIs and their schemas
- **Cache** API state locally with automatic synchronization
- **Coordinate** multi-replica controllers via leader election
- **Handle authentication**, authorization, and transport layer security

### Subsystem Boundaries

**Upstream (dependencies):**
- `k8s.io/apimachinery`: Core runtime objects (ObjectMeta, TypeMeta, runtime.Object), watch interfaces, and versioning primitives
- `k8s.io/api`: Generated type definitions for all Kubernetes API resources (Pod, Deployment, etc.)
- `k8s.io/utils`, `k8s.io/klog`: Utility libraries (time management, logging, sets)
- Standard library: `net/http`, `encoding/json`, `crypto/tls`

**Downstream (consumers):**
- **Controllers**: Operators, custom controllers that reconcile cluster state
- **CLI tools**: kubectl (internally uses client-go)
- **Webhooks**: Admission controllers, mutating/validating webhooks
- **Monitoring/observability**: Custom metrics collectors, log aggregators
- **End-user applications**: Any program that needs Kubernetes integration
