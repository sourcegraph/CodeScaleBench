### Duplicate Message Processing

**Symptom**: Same record processed multiple times

**Potential Causes**:

1. **Offset not committed before crash**
   ```
   Consumer processes records but crashes before commitSync()
   On restart, reads same records again

   Solution: Enable manual commits AFTER processing
   consumer.commitSync();  // After successful processing
   ```

2. **Auto-commit interval vs processing time**
   ```
   Auto-commit commits every 5 seconds
   Processing takes 10 seconds
   Crash occurs during processing → reprocess

   Solution: Manual commits or increase auto.commit.interval.ms
   ```

### CommitFailedException

**Symptom**: commitSync() throws CommitFailedException

**Cause**: Rebalance occurred; consumer lost group membership

**Solution**:
```java
try {
    consumer.commitSync();
} catch (CommitFailedException e) {
    logger.error("Lost group membership, stopping processing", e);
    // Don't retry - consumer will rejoin on next poll()
    break;  // Exit processing loop
}
```

All network I/O, metadata updates, and callbacks occur within poll().\n\n### `poll(Duration timeout)`\n\n```java\npublic ConsumerRecords<K, V> poll(Duration timeout);\n```\n\nFetch records within the specified timeout.\n\n**Behavioral Notes:**\n\n1. **Blocking Semantics:**\n   - If records are available in client buffer, returns immediately\n   - Otherwise, blocks waiting for broker response (up to timeout duration)\n   - If rebalance occurs during poll, timeout may be exceeded\n   - **GOTCHA**: poll() can block longer than timeout if rebalance callbacks are slow\n\n2. **What Happens During poll():**\n   - Sends heartbeats to group coordinator (keep-alive)\n   - Fetches new records from brokers\n   - Handles metadata updates\n   - Executes rebalance callbacks (onPartitionsRevoked, onPartitionsAssigned)\n   - Commits offsets if auto-commit is enabled\n   - Filters messages based on isolation level\n\n3. **Return Value:**\n   - `ConsumerRecords<K, V>` collection (never null, may be empty)\n   - Records are ordered within each partition\n   - No ordering guarantees across partitions\n   - Records are buffered locally; multiple polls may return buffered records\n\n4. **Exceptions:**\n   - `WakeupException` - if `wakeup()` was called from another thread\n   - `InterruptException` - if thread was interrupted\n   - `SerializationException` - if key/value deserialization fails\n   - `OffsetOutOfRangeException` - if committed offset no longer exists\n   - `NoOffsetForPartitionException` - if auto.offset.reset=none and no committed offset\n\n**Example:**\n```java\nwhile (true) {\n    try {\n        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n\n        for (ConsumerRecord<String, String> record : records) {\n            // Process record\n            System.out.println(\"Topic: \" + record.topic() +\n                             \", Partition: \" + record.partition() +\n                             \", Offset: \" + record.offset() +\n                             \", Key: \" + record.key() +\n                             \", Value: \" + record.value());\n        }\n\n        // Explicit offset commit after batch processing\n        consumer.commitSync();\n    } catch (WakeupException e) {\n        // Consumer was woken up, exit gracefully\n        break;\n    }\n}\n```\n\n### Timeout Configuration\n\nThe `timeout` parameter controls how long poll() waits for broker responses:\n\n```java\n// Short timeout - returns quickly if no data available\nconsumer.poll(Duration.ofMillis(100));\n\n// Long timeout - waits longer for records\nconsumer.poll(Duration.ofMillis(1000));\n\n// Zero timeout - return immediately if no buffered records\nconsumer.poll(Duration.ZERO);\n```\n\n**Real-world Tuning:**\n- High-latency networks: Longer timeouts to accommodate network delays\n- Low-latency requirements: Shorter timeouts for responsiveness\n- **Default (300ms)**: Balances latency and throughput\n\n### `pollForRecords(long timeoutMs)` [Deprecated]\n\nLegacy method; Management\n\n### Understanding Rebalancing\n\nA **rebalance** redistributes partition ownership among group members. Triggered by:\n1. New member joins (via subscribe() on first poll())\n2. Member leaves (close() or timeout)\n3. Topic subscribed via pattern gains new matching topics\n4. Partition count changes for subscribed topics\n\n### Rebalance Lifecycle\n\n```\nPoll detects rebalance needed\n    \u2193\nMember joins group with current assignment\n    \u2193\nGroup enters \"Rebalancing\" state\n    \u2193\nCoordinator assigns new leader\n    \u2193\nLeader computes new assignment (based on configured strategy)\n    \u2193\nAll members receive new assignment\n    \u2193\nonPartitionsRevoked(old_partitions) executed\n    \u2193\nMember revokes partitions, sends commit offsets\n    \u2193\nonPartitionsAssigned(new_partitions) executed\n    \u2193\nMember can seek to custom positions\n    \u2193\nGroup is \"Stable\", polling resumes\n```\n\n### ConsumerRebalanceListener Callbacks\n\n#### `onPartitionsRevoked(Collection<TopicPartition> partitions)`\n\nCalled DURING poll() when partitions are being revoked.\n\n**Key Points:**\n- **This is your last chance to commit offsets** for revoked partitions\n- Called with partitions being removed from this consumer\n- Blocks poll() until callback returns\n- Exceptions thrown here cause rebalance failure\n\n```java\nconsumer.subscribe(Arrays.asList(\"my-topic\"),\n    new ConsumerRebalanceListener() {\n        @Override\n        public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n            System.out.println(\"Revoking: \" + partitions);\n\n            // CRITICAL: Commit offsets before losing partitions\n            try {\n                consumer.commitSync();\n                System.out.println(\"Offsets committed for revoked partitions\");\n            } catch (Exception e) {\n                logger.error(\"Failed to commit during rebalance\", e);\n                // Don't throw - let rebalance continue\n                // But log it so you know you may have lost progress\n            }\n        }\n\n        @Override\n        public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n            System.out.println(\"Assigned: \" + partitions);\n\n            // Optionally seek to custom positions\n            for (TopicPartition tp : partitions) {\n                // Seek to beginning of newly assigned partitions\n                consumer.seekToBeginning(Arrays.asList(tp));\n            }\n        }\n    }\n);\n```\n\n#### `onPartitionsAssigned(Collection<TopicPartition> partitions)`\n\nCalled DURING poll() when new partitions are assigned.\n\n**Key Points:**\n- First chance to set initial position for new partitions\n- If not called, position defaults to committed offset or auto.offset.reset\n- Blocks poll() until callback returns\n- Called AFTER partitions have been revoked from other members\n\n**Default Behavior (if no callback or callback does nothing):**\n```\nFor each assigned partition:\n  if committed offset exists:\n    seek to committed offset\n  else if auto.offset.reset == \"earliest\":\n    seek to beginning\n  else if auto.offset.reset == \"latest\":\n    seek to end\n  else:\n    next poll() throws NoOffsetForPartitionException\n```\n\n#### Rebalance Callback Exceptions\n\nIf callback throws exception:\n- to timeout for broker response\n2. **During rebalance**: Blocks indefinitely in callbacks (timeout ignored)\n3. **Metadata update**: May block fetching metadata from brokers\n4. **Network latency**: Added on top of timeout\n\n**Key Implication**:\n```\nTimeout parameter is NOT a hard guarantee.\npoll() can block longer than timeout if rebalance occurs.\n\nExample:\npoll(Duration.ofMillis(100))  // Timeout: 100ms\n\u2190 rebalance triggered during poll\n\u2190 onPartitionsRevoked() called (blocks indefinitely)\n\u2190 onPartitionsAssigned() called (blocks indefinitely)\n\u2190 poll() can block for many seconds\n```\n\n**Recommendation:**\n```\nUse reasonable timeouts (100-1000ms) but don't rely on exact timing.\nIf you need precise timing, implement a deadline check outside poll():\n\nlong deadline = System.currentTimeMillis() + 60000;\nwhile (System.currentTimeMillis() < deadline) {\n    ConsumerRecords<String, String> records =\n        consumer.poll(Duration.ofMillis(100));\n    // Process...\n    if (shouldShutdown) break;\n}\n```\n\n### Offset Commit Semantics\n\n#### Synchronous Commit (`commitSync()`)\n\n```\nApplication calls commitSync()\n    \u2193\nConsumer builds commit request with current position for all partitions\n    \u2193\nSends to group coordinator\n    \u2193\nCoordinator stores offsets in __consumer_offsets topic\n    \u2193\nReturns success or CommitFailedException\n    \u2193\nApplication resumes (offset is now durable)\n```\n\n**Exception During Commit:**\n- Offset NOT committed\n- Group membership may be lost\n- Next poll() attempts rejoin\n- **Do not retry the same commit** - wait for rejoin\n\n#### Asynchronous Commit (`commitAsync()`)\n\n```\nApplication calls commitAsync(callback)\n    \u2193\nConsumer queues commit request\n    \u2193\ncommitAsync() returns immediately\n    \u2193\nDuring next poll(): commit request sent\n    \u2193\nCoordinator processes and stores offsets\n    \u2193\nDuring later poll(): callback invoked with success/failure\n    \u2193\nApplication handles response in callback\n```\n\n**Key Difference**:\n- Application doesn't block\n- Callback may be invoked many polls later\n- Multiple outstanding commits can cause ordering issues\n\n### Group Membership and Rebalancing\n\nWhen does a consumer stay in the group?\n\n```\nConsumer stays in group IF:\n1. poll() is called frequently enough (at least every max.poll.interval.ms)\n   AND\n2. Heartbeats are sent (automatically during poll())\n```\n\nIf consumer stops calling poll() for max.poll.interval.ms:\n```\nCoordinator removes consumer from group\n    \u2193\nRebalance triggered for remaining consumers\n    \u2193\nEvicted consumer's partitions reassigned\n    \u2193\nWhen evicted consumer calls poll() again:\n    \u2193\nConsumer rejoins group (potentially with different partitions)\n    \u2193\nAnother rebalance occurs\n```\n\n### Static Group Membership\n\nEnable with `group.instance.id` to preserve partition assignment across restarts:\n\n```\nConsumer 1 (instance-1) - assigned [partition-0, partition-1]\nConsumer 1 restarts\n    \u2193\nConsumer rejoins with same instance-1 ID\n    \u2193\nPartitions are reassigned to same instance without rebalance\n```\n\nUse cases:\n- Minimizing rebalance time during rolling restarts\n- Preserving local caches per instance\n\n### Message Filtering with read_committed\n\nWith `isolation.level=read_committed`:\n\n```\nProducer writes messages in transaction:\n- Offset 0: msg1\n- Offset 1: msg2 (uncommitted)\n- Offset 2: msg3 (uncommitted)\n\nConsumer sees Last Stable Offset (LSO) = 0 (before uncommitted messages)\n\nConsumer poll() only returns msg at offset 0\nNext poll() may return more if transaction commits\n\nThis prevents consuming partially committed transactions\n```\n\n---\n\n## Troubleshooting Guide\n\n### Consumer Not Receiving Messages\n\n**Symptom**: Consumer calls poll() but receives no records\n\n**Potential Causes**:\n\n1. **Partition assignment incomplete**\n   ```java\n   // Check assignment\n   System.out.println(\"Assigned: \" + consumer.assignment());\n\n   // Solution: Ensure subscription/assign and initial poll completed\n   consumer.subscribe(Arrays.asList(\"my-topic\"));\n   consumer.poll(Duration.ZERO); Management\n\n### Understanding Rebalancing\n\nA **rebalance** redistributes partition ownership among group members. Triggered by:\n1. New member joins (via subscribe() on first poll())\n2. Member leaves (close() or timeout)\n3. Topic subscribed via pattern gains new matching topics\n4. Partition count changes for subscribed topics\n\n### Rebalance Lifecycle\n\n```\nPoll detects rebalance needed\n    \u2193\nMember joins group with current assignment\n    \u2193\nGroup enters \"Rebalancing\" state\n    \u2193\nCoordinator assigns new leader\n    \u2193\nLeader computes new assignment (based on configured strategy)\n    \u2193\nAll members receive new assignment\n    \u2193\nonPartitionsRevoked(old_partitions) executed\n    \u2193\nMember revokes partitions, sends commit offsets\n    \u2193\nonPartitionsAssigned(new_partitions) executed\n    \u2193\nMember can seek to custom positions\n    \u2193\nGroup is \"Stable\", polling resumes\n```\n\n### ConsumerRebalanceListener Callbacks\n\n#### `onPartitionsRevoked(Collection<TopicPartition> partitions)`\n\nCalled DURING poll() when partitions are being revoked.\n\n**Key Points:**\n- **This is your last chance to commit offsets** for revoked partitions\n- Called with partitions being removed from this consumer\n- Blocks poll() until callback returns\n- Exceptions thrown here cause rebalance failure\n\n```java\nconsumer.subscribe(Arrays.asList(\"my-topic\"),\n    new ConsumerRebalanceListener() {\n        @Override\n        public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n            System.out.println(\"Revoking: \" + partitions);\n\n            // CRITICAL: Commit offsets before losing partitions\n            try {\n                consumer.commitSync();\n                System.out.println(\"Offsets committed for revoked partitions\");\n            } catch (Exception e) {\n                logger.error(\"Failed to commit during rebalance\", e);\n                // Don't throw - let rebalance continue\n                // But log it so you know you may have lost progress\n            }\n        }\n\n        @Override\n        public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n            System.out.println(\"Assigned: \" + partitions);\n\n            // Optionally seek to custom positions\n            for (TopicPartition tp : partitions) {\n                // Seek to beginning of newly assigned partitions\n                consumer.seekToBeginning(Arrays.asList(tp));\n            }\n        }\n    }\n);\n```\n\n#### `onPartitionsAssigned(Collection<TopicPartition> partitions)`\n\nCalled DURING poll() when new partitions are assigned.\n\n**Key Points:**\n- First chance to set initial position for new partitions\n- If not called, position defaults to committed offset or auto.offset.reset\n- Blocks poll() until callback returns\n- Called AFTER partitions have been revoked from other members\n\n**Default Behavior (if no callback or callback does nothing):**\n```\nFor each assigned partition:\n  if committed offset exists:\n    seek to committed offset\n  else if auto.offset.reset == \"earliest\":\n    seek to beginning\n  else if auto.offset.reset == \"latest\":\n    seek to end\n  else:\n    next poll() throws NoOffsetForPartitionException\n```\n\n#### Rebalance Callback Exceptions\n\nIf callback throws exception:\n- Rebalance fails\n- Consumer remains unassigned\n- Next poll() attempts rebalance again\n- **Avoid throwing exceptions** from callbacks; log and handle gracefully\n\n### Group Management Methods\n\n#### `groupMetadata()`\n\n```java\npublic ConsumerGroupMetadata groupMetadata();\n```\n\nReturns metadata about this consumer's group membership.\n\n```java\nConsumerGroupMetadata metadata = consumer.groupMetadata();\nSystem.out.println(\"Generation ID: \" + metadata.generationId());\nSystem.out.println(\"Member ID: \" + metadata.memberId());\n```\n\n#### `leave()`\n\n```java\npublic void leave();\n```\n\nProactively leave the consumer group.\n\n**Behavioral Notes:**\n- Revokes all assigned partitions\n- Triggers `onPartitionsRevoked()` callbacks\n- Consumer becomes unassigned\n- Next poll() on same consumer rejoins the group\n- Thread-unsafe; call from consumer thread\n\n```java\n// Later when shutting down\nconsumer.leave();\nconsumer.close();\n```\n\n#### Assignment Strategies\n\nConfigured via `partition.assignment.strategy`:\n\n1. **RangeAssignor** (default)\n   - Assigns contiguous ranges of partitions to consumers\n   - Topic: [0, 1, 2, 3, 4, 5], 2 consumers\n   - Consumer 1: [0, 1, 2], Consumer 2: [3, 4, 5]\n   - Issue: Imbalanced load distribution in multi-topic scenarios\n\n2. **RoundRobinAssignor**\n   - Assigns partitions in round-robin across consumers\n   - Topic: [0, 1, 2, 3, 4, 5], 2 consumers\n   - Consumer 1: [0, 2, 4], Consumer 2: [1, 3, 5]\n   - Better load balancing across topics\n\n3. **StickyAssignor** (Kafka 0.11+)\n   - Minimizes partition movement during rebalance\n   - Assigns new partitions while keeping existing assignments stable\n   - Best for minimizing state transfer on rebalance\n\n4. **CooperativeStickyAssignor** (Kafka 2.4+)\n   - Like StickyAssignor but uses cooperative rebalance protocol\n   - Reduces stop-the-world pause time during rebalance\n\n---\n\n## Metadata and
