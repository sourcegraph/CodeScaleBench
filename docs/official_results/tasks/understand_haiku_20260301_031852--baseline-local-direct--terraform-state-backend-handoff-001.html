Failure Modes\n\n### State Locking Failures\n\n**Lock Contention**:\n- When lock already held, `Lock()` returns `*statemgr.LockError` with existing lock info\n- `LockWithContext()` implements exponential backoff retry (1s \u2192 2s \u2192 4s ... \u2192 16s max)\n- If context timeout/cancellation occurs during retry, returns last LockError\n\n**Common Issues**:\n```\nError: Error acquiring the state lock\n\nError: error in state lock info: \"...\"\n```\n\nCauses:\n1. Another user running `terraform apply` (legitimate contention)\n2. Stale lock from crashed process (lock not released due to ungraceful termination)\n3. Network issues preventing lock service communication\n4. Insufficient permissions to access lock storage\n\n**Stale Lock Resolution**:\n- S3: Manually delete DynamoDB lock table entry\n- Consul: Manually delete lock key\n- Filesystem: Remove `.terraform.lock.hcl` file\n- Azure: Force-release blob lease\n- Command: `terraform force-unlock <LOCK_ID>`\n\n### Storage Unavailability\n\n**Remote Service Down**:\n- S3/Azure/GCS: Network errors during Get/Put/Delete\n- HTTP backend: 5xx responses\n- Consul/Kubernetes: Service discovery failure\n\nErrors from `RefreshState()` and `PersistState()`:\n- Propagate to terraform operations\n- Commands fail gracefully with error messages\n- No automatic fallback (safety-first: don't proceed with stale state)\n\n### State Corruption Scenarios\n\n**Conflict Detection via Lineage & Serial**:\n\n```go\ntype SnapshotMeta struct {\n    Lineage string      // Unique state ID (doesn't change unless force-push)\n    Serial  uint64      // Version counter (incremented each persist)\n}\n```\n\n**Scenarios**:\n\n1. **Concurrent Writes** (should not happen with locking):\n   - Client A reads state (serial=5, lineage=abc)\n   - Client B reads state (serial=5, lineage=abc)\n   - Client B writes (serial=6, lineage=abc)\n   - Client A writes (serial=6, lineage=abc) \u2192 CONFLICT\n   - Terraform detects serial/lineage mismatch, refuses to write\n   - User must manually resolve (rerun plan, re-apply)\n\n2. **Lineage Mismatch**:\n   - Happens when states are from different runs\n   - `terraform state pull | terraform state push` with wrong state\n   - Causes: `serial` alone doesn't uniquely identify state sequence\n\n3. **Migration Corruption**:\n   - Moving state between backends\n   - `/workspace/internal/command/meta_backend_migrate.go` handles migration\n   - Reads from old backend, writes to new backend\n   - Lockfiles managed carefully to prevent races\n\n**File**: `/workspace/internal/states/statemgr/persistent.go` (193 lines)\n- `Persister.PersistState()` must detect and avoid destroying concurrent changes\n- Implementations use serial/lineage or storage-level atomicity\n\n### Configuration Errors\n\n**Invalid Configuration**:\n- `PrepareConfig()` returns diagnostics with detailed attribute paths\n- Errors prevent `Configure()` from being called\n- Example S3 backend errors:\n  - Missing `bucket` (required)\n  - Invalid AWS region format\n  - Insufficient IAM permissions\n\n**Detection**:\n- Schema validation via `configschema.CoerceValue()`\n- Backend-specific validation in `Configure()`:\n  - Connection tests (can we reach S3?)\n  - Permission checks (can we read/write?)\n  - Resource existence checks (does bucket exist?)\n\n### Workspace Errors\n\n**Unsupported Operations**:\n```go\nErrWorkspacesNotSupported  // Backend doesn't support workspaces\nErrDefaultWorkspaceNotSupported  // Backend requires non-default workspace\n```\n\n**File Deletion Prevention**:\n- `DeleteWorkspace()` cannot delete default workspace\n- Cannot delete workspace if state manager holds lock on it (caller must release lock first)\n\n### Type System Safety\n\n**File**: `/workspace/internal/states/statemgr/statemgr.go` (compile-time check)\n\nAll state manager implementations must verify interface satisfaction:\n```go\nvar _ statemgr.Full = (*YourType)(nil)\n```\n\nFound in implementations:\n- `/workspace/internal/states/statemgr/filesystem.go:67`\n- `/workspace/internal/states/remote/state.go:51-53`\n\nCatch interface drift at compile time.\n\n---\n\n## 5. # Terraform State Backend Subsystem - Onboarding Document

## 1. Purpose

The state backend subsystem is responsible for managing Terraform's state files, which contain the critical mapping between configured resources and real-world infrastructure. This subsystem solves several fundamental problems:

### Core Responsibilities

**State Persistence**: Backends provide pluggable storage mechanisms for state snapshots. Rather than hardcoding local filesystem storage, Terraform abstracts state storage through backends, enabling teams to store state in various systems (S3, Azure Blob Storage, Consul, Kubernetes, HTTP endpoints, etc.).

**Concurrency Management**: Multiple Terraform processes must safely access and modify state. Backends implement state locking mechanisms to prevent concurrent modifications from corrupting state. When one user runs `terraform apply`, the state lock prevents other users from modifying the same state simultaneously.

**Workspace Support**: Most backends support multiple named workspaces (e.g., "prod", "staging", "dev"), allowing a single configuration to manage multiple environments. Each workspace has its own independent state.

**Remote Operations**: Some backends (Remote, Local) can orchestrate Terraform operations (plan, apply, refresh) themselves, while others only provide state storage. This distinction separates concerns: storage backends vs. operational backends.
