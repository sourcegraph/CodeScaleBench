**Bug ID**: 53441
**Summary**: hda_intel: sound stops working after suspend to ram
**Regression**: Yes (works on v3.6.11, broken since v3.7.1)

On a fresh boot the sound card works fine. After a few suspends to ram (it doesn't necessarily happen after just one) the sound card stops working well. The sound comes out choppy, as a series of clicks separated by 2 second or so silences.

I see these messages in the kernel log right after resume:
```
hda-intel: azx_get_response timeout, switching to polling mode: last cmd=0x00170503
hda-intel: No response from codec, disabling MSI: last cmd=0x00170503
hda_intel: azx_get_response timeout, switching to single_cmd mode: last cmd=0x00170503
```

The only way to fix the sound seems to be a reboot. I don't have this problem with v3.6.11, it appeared with kernel v3.7.1 (I have not tested v3.7). Right now I'm running 3.7.6 and the problem is still there.

## Summary

**Buggy File:** `sound/pci/hda/hda_intel.c`

**Buggy Functions:** 
- `azx_suspend` (lines 2513-2537)
- `azx_resume` (lines 2539-2566)  
- `azx_runtime_suspend` (lines 2570-2578)

**Root Cause:**
The HDA Intel driver implements recovery mechanisms for codec communication timeouts by setting two flags:
1. `polling_mode` - set to 1 at line 912 when normal interrupt-based communication fails
2. `single_cmd` - set to 1 at line 951 as a last resort when polling mode also fails

**The Critical Bug:**
These flags are **never reset** during suspend/resume cycles. This causes:
- Hardware is fully reset during suspend/resume (new PCI state, interrupt handlers reset)
- But the software flags `polling_mode` and `single_cmd` remain set to 1
- During resume, `azx_init_chip()` at line 1222 skips CORB/RIRB reinitialization if `single_cmd=1`
- This mismatch between hardware state and software state prevents normal codec communication
- Result: Repeated timeouts, codec communication failures, and choppy audio

**Why It's a Regression:**
The bug report indicates this works in v3.6.11 but fails since v3.7.1. The symptom matches perfectly: sound works fine initially, but fails after suspend/resume cycles because the hardware is reset but the workaround flags persist.
