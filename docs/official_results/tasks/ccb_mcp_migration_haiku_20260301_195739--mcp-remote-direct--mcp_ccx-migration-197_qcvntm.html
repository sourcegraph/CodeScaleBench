### Version Migration Details
Unlike explicit v1/v2 API URLs, Kafka Connect implements versioning through **schema evolution** in internal Kafka backing topics:

**Schema Version Changes:**
- **TARGET_STATE_V0** (legacy): Contains only `state` field
- **TARGET_STATE_V1** (new): Adds optional `state.v2` field

**Key Difference:** The new schema supports the STOPPED state while maintaining backward compatibility. For the new STOPPED state, it stores `state=PAUSED` (for old workers) and `state.v2=STOPPED` (for new workers).

### Task Status and Error Responses
- **Task Status**: Uses `ConnectorStateInfo.TaskState` with id, state, worker_id, trace, and version fields
- **Error Responses**: Uses `ErrorMessage` with error_code and message fields (unchanged between versions)

The migration strategy ensures seamless upgrades where newer versions remain compatible with older cluster members through careful schema evolution in the internal backing topics.
