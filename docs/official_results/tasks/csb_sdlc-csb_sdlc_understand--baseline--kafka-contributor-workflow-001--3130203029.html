# Kafka Code Contribution Process

## JIRA Ticket Management

First, locate an existing JIRA ticket matching your proposed change. The guidelines state: "Find the existing Kafka JIRA ticket that the change pertains to." If no ticket exists for non-trivial changes, create one with a descriptive title, detailed description, and appropriate type/priority fields. Assign the ticket to yourself to avoid conflicts.

## Pull Request Workflow

**Initial Setup:**
Fork the Apache Kafka repository and create a feature branch. Ensure your code follows the project's coding guidelines and includes necessary tests and documentation updates.

**Opening the PR:**
Submit your pull request against the trunk branch with a title format like "KAFKA-xxxx: Description." For work-in-progress PRs, add `[WIP]` after the JIRA ID. Include a statement confirming the contribution is your original work licensed under the project's open source license.

**Continuous Integration:**
The project automatically runs CI tests on Linux AMD64 and ARM64 nodes via Apache Jenkins. Results appear in the PR's checks section with links to detailed test results.

## Review and Approval

Reviewers will comment on changes and suggest modifications. Address feedback through additional commits rather than amending existing ones. When ready, tag reviewers for re-review. 

The approval process includes: "Reviewers can indicate that a change looks suitable for merging by approving it via GitHub's review interface." This represents the strongest technical sign-off and indicates the reviewer takes ownership of the patch.

## Closure

Once approved and merged, the pull request and associated JIRA automatically close, with credit assigned to the primary contributor.
