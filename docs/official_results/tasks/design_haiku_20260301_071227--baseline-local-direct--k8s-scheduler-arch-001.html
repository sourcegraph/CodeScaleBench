## Key Findings

### **Architecture Overview**
The Kubernetes scheduler implements a sophisticated **two-phase commit pattern**:
1. **Scheduling Cycle** (synchronous) - Finding the best node for a pod
2. **Binding Cycle** (asynchronous) - Confirming the binding in the API server

### **14 Extension Points Across Two Phases**

**Scheduling Cycle:**
- PreEnqueue, QueueSort, PreFilter, Filter, PostFilter, PreScore, Score, Reserve, Permit

**Binding Cycle:**
- WaitOnPermit, PreBind, Bind, PostBind

### **Core Components**

1. **SchedulingQueue** - Manages 3 pod queues (active, backoff, unschedulable) with sophisticated state transitions
2. **Cache** - Optimistically assumes pod placement and tracks pod state (Initial → Assumed → Added → Expired/Deleted)
3. **CycleState** - Thread-safe per-cycle state storage for plugin data sharing
4. **Framework Runtime** - Loads plugins and orchestrates their execution at extension points
5. **Parallelizer** - Evaluates nodes in parallel during filtering and scoring phases

### **Design Patterns**
- **Optimistic Locking**: Assume immediately, bind asynchronously
- **Plugin Framework**: 14 extension points with well-defined interface contracts
- **Parallelization**: Concurrent node evaluation with early exit optimization
- **Backoff Strategy**: Exponential backoff for unschedulable pods
- **Snapshot Consistency**: Single cluster view per scheduling cycle
