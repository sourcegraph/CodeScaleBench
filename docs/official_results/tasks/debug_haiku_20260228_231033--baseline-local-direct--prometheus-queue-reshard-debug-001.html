## Summary of Findings

I've completed a comprehensive investigation of the remote-write queue resharding failure in the Prometheus repository. Here's what was discovered:

### **Three Critical Issues Identified:**

1. **Premature Metric Reset (Line 1241)** - The `pendingSamples` metric is set to 0 when new shards start, even if old shards haven't finished flushing, causing loss of visibility into truly pending samples.

2. **Shard Destination Recalculation (Line 1315)** - During the resharding window, samples that fail to enqueue due to shutdown get retried after new shards start, causing them to map to different shard destinations via the modulo calculation, potentially violating ordering guarantees.

3. **Hard Shutdown Data Loss (Lines 1563-1578)** - When flush deadline timeout expires, samples are dropped without proper accounting, leading to metric inconsistencies.

### **Why It's Intermittent:**
The race condition is non-deterministic and depends on:
- When samples are retried relative to `start()` being called
- Whether old shards complete flushing before hard shutdown timeout
- Network and system latency at the time of resharding

### **Key Diagnostic Indicators:**
- `prometheus_remote_storage_samples_pending` unexpectedly drops to 0
- Spikes in `enqueue_retries_total` during resharding
- Uneven shard load distribution post-resharding
- "Dropped on hard shutdown" errors in logs

The report includes detailed code references, evidence, affected components, and recommendations for both root fixes and short-term mitigations.
