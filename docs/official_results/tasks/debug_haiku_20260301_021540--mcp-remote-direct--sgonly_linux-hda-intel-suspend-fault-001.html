**Root Cause Analysis:**

The bug occurs because:
1. When a codec timeout occurs before suspend, `chip->polling_mode` is set to 1 (line 912) or `chip->single_cmd` is set to 1 (line 951)
2. During suspend, `azx_stop_chip()` is called which sets `chip->initialized = 0` but does NOT reset `polling_mode` or `single_cmd`
3. During resume, `azx_init_chip()` is called but has an early return if `chip->initialized` is already 1 (line 1211)
4. More critically, `azx_init_chip()` only reinitializes the CORB/RIRB if `chip->single_cmd` is 0 (line 1222-1223)
5. If `single_cmd` is still 1 from previous error recovery, the codec command I/O never gets reinitialized
6. This causes codec communication to fail immediately after resume

The fix is to reset `chip->polling_mode` to 0 in `azx_init_chip()` when a full reset is requested.

### Root Cause:

The `azx_init_chip()` function fails to reset error recovery state variables (`chip->polling_mode` and `chip->poll_count`) when performing a full reset during resume operations.

### Problem Flow:

1. **Before suspend**: A codec timeout triggers error recovery that sets `polling_mode=1`
2. **During suspend**: `azx_stop_chip()` resets `initialized=0` but **NOT** `polling_mode` or `poll_count`
3. **During resume**: `azx_init_chip(chip, 1)` is called, but `polling_mode` remains 1
4. **Result**: All subsequent codec communications are forced through the polling/error recovery path instead of normal operation, causing immediate timeouts and the observed error messages

### Why This Appeared in v3.7.1+:

The regression appeared between v3.6.11 (working) and v3.7.1 (broken), suggesting changes to the resume/reset handling code path removed or missed resetting these error recovery state flags.

### Required Fix:

Reset `chip->polling_mode=0` and `chip->poll_count=0` in `azx_init_chip()` when `full_reset` is requested, before line 1215.
