Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n**Authentication Failure (401 Unauthorized)**\n- Invalid or missing credentials\n- Authenticator returns error\n- Response includes audit event with `user.anonymous`\n- Connection remains open (client can retry with valid credentials)\n- Impact: Client must resubmit with valid auth\n\n**Authorization Failure (403 Forbidden)**\n- User authenticated but lacks permission for verb/resource\n- Request fully processed (authenticated, admitted) before authz check\n- Impact: Confirms existence of resource (information leak - design tradeoff)\n\n**Admission Rejection (400 Bad Request or Custom Status)**\n- Mutating webhook rejects due to policy violation\n- Validating webhook rejects after mutation or storage write\n- Audit event includes rejection reason\n- Impact: Object may exist in storage but client sees rejection\n\n**Storage Failure (500 Internal Server Error)**\n- Etcd unreachable or timeout\n- Serialization error\n- Quota exceeded\n- Request marked as failed in audit\n- Impact: Request incomplete, may be partially persisted (depends on storage semantics)\n\n**Rate Limiting (429 Too Many Requests)**\n- Request queue full in Priority & Fairness controller\n- Response includes `Retry-After` header\n- Request NOT processed, no admission or storage mutation\n- Impact: Request is dropped at gateway before reaching handlers\n\n**Timeout (504 Gateway Timeout or 408 Request Timeout)**\n- Request context deadline exceeded\n- Non-long-running requests: timeout after `RequestTimeout` config\n- Watch requests: can run indefinitely\n- If mutating operation: may partially complete\n- Impact: Connection closed, client must retry (potential duplication)\n\n**Panic (500 Internal Server Error)**\n- Handler panic caught by `WithPanicRecovery`\n- Stack trace logged, response set to 500\n- Recovery prevents cascade failures\n- Impact: Minimal - single request fails, server continues\n\n### 4.2 Resource Limits\n\n**Request Body Size**\n- `maxRequestBodyBytes` config limits request body\n- Default: unlimited (0)\n- Exceeded: 413 Request Entity Too Large\n- Applied at handler level (before unmarshaling)\n\n**In-Flight Request Limits**\n- `MaxRequestsInFlight`: max concurrent requests\n- `MaxMutatingRequestsInFlight`: max concurrent mutating requests\n- Exceeded: 429 Too Many Requests (new requests queued)\n- Design: Prevents resource exhaustion\n\n**Timeout Limits**\n- `minRequestTimeout`: minimum allowed client timeout\n- `requestTimeoutUpperBound`: max timeout for context\n- Set: 34 seconds default (unusual number to catch errors)\n- Design: Prevents clients hanging indefinitely, but allows long operations\n\n**Watch Streaming**\n- Watches are long-running (exempt from timeouts)\n- Streaming responses sent incremental to client\n- Client can cancel with TCP FIN or timeout\n\n**Memory Limits**\n- No built-in limits on number of objects\n- Relies on etcd/storage backend limits\n- No garbage collection by apiserver\n\n### 4.3 Design Tradeoffs\n\n**Tradeoff: Admission Ordering vs. Flexibility**\n\n*Decision*: Admission controllers execute in fixed order (mutating \u2192 validating)\n\n*Benefit*: Predictable behavior, validators see final state\n*Cost*: Can't inject conditional validators, can't skip controllers\n\n*Evidence*: `pkg/admission/chain.go` hardcodes mutating vs validating phases\n\n**Tradeoff: Handler Chain Wrapping vs. Plugin System**\n\n*Decision*: Filter middleware wraps HTTP handler (not plugin system)\n\n*Benefit*: Type-safe composition, no reflection, clear execution order\n*Cost*: Must recompile to add filters, can't dynamically load middleware\n\n*Evidence*: `pkg/server/config.go` DefaultBuildHandlerChain returns wrapped handler\n\n**Tradeoff: Synchronous Admission vs. Async Webhooks**\n\n*Decision*: Admission runs synchronously in request handler\n\n*Benefit*: Client receives immediate feedback (success/failure)\n*Cost*: Webhook latency directly impacts API latency (many orgs add load balancer timeout)\n*Evidence*: Many deployments disable validating webhooks on high-volume resources\n\n**Tradeoff: Storage Interface Flexibility vs. Consistency**\n\n*Decision*: Storage interface minimal, no transaction support\n\n*Benefit*: Simple, works with any backend\n*Cost*: No ACID transactions across multiple resources (leads to eventual consistency patterns)\n*Evidence*: `pkg/registry/rest/rest.go` defines simple Create/Update/Delete\n\n**Tradeoff: Audit After Storage vs. Before**\n\n*Decision*: Audit runs after handlers complete\n\n*Benefit*: Audit logs reflect actual stored state\n*Cost*: Audit lag (asynchronous in many backends), audit webhook latency impacts request latency\n*Evidence*: `pkg/endpoints/filters/audit.go` wraps handlers, not called before\n\n**Tradeoff: RequestInfo Parsing Overhead**\n\n*Decision*: All requests parsed to extract namespace/resource/verb\n\n*Benefit*: Consistent across all handlers, enables RBAC\n*Cost*: CPU overhead on every request\n*Evidence*: `WithRequestInfo` (line 1091) called for all requests including 404s\n\n**Tradeoff: Director Pattern (GoRestful + NonGoRestfulMux)**\n\n*Decision*: Dual-handler architecture with director dispatch\n\n*Benefit*: Enables OpenAPI generation (go-restful) while supporting delegation\n*Cost*: Complexity, potential for routing bugs, go-restful performance overhead\n*Evidence*: `pkg/server/handler.go` director type, lines 121-154\n*Comment*: This is widely recognized as a wart; maintainers acknowledge go-restful dependency should be removed\n\n**Tradeoff: Imperative vs. Declarative Request Processing**\n\n*Decision*: Handlers are imperative (hand-written, not declarative schemas)\n\n*Benefit*: Flexibility (can implement non-CRUD operations), precise control\n*Cost*: High complexity for standard CRUD, duplicated logic across handlers\n*Evidence*: `pkg/endpoints/handlers/create.go` (10KB) vs. similar in other frameworks\n\n---\n\n## 5. Extension Points, Customization Hooks, and Risks\n\n### 5.1 Primary Extension Points\n\n#### 5.1.1 Handler Chain Customization\n\n**Extension Point**: `Config.BuildHandlerChainFunc`\n\n```go\ntype Operational Implications\n\n### 6.1 Performance Characteristics\n\n**Request Latency Breakdown** (typical values):\n\n1. **Handler chain overhead**: 1-2ms\n   - WithPanicRecovery, WithRequestInfo, context setup\n2. **Authentication**: 5-50ms\n   - Token validation, webhook calls\n3. **Authorization**: 10-100ms\n   - RBAC policy evaluation (worse with large role sets)\n4. **Admission**: 50-500ms\n   - Webhook roundtrips (worst offender in most deployments)\n5. **Storage read**: 5-50ms (etcd 100ms+ over WAN)\n6. **Encoding/serialization**: 1-10ms\n\n**Total (median create request)**: 75-300ms\n**P99 (with webhook retries, lock contention)**: 1-5s\n\n**Optimization Opportunities**:\n- Move webhook validation to separate component\n- Cache authorization decisions (10s TTL typical)\n- Batch etcd requests for list operations\n- Use protobuf instead of JSON (3-5x faster)\n\n### 6.2 Memory Usage\n\n**Per-Server Baseline**: 100-300MB\n- go-restful reflection for registered routes: ~50MB\n- OpenAPI schema: ~50MB\n- Admission controller state: variable\n\n**Per-Request Memory**: 1-10MB\n- Request deserialization buffer\n- Audit event structures\n- Watch connection metadata (long-lived)\n\n**Watch Connections**: 10KB-1MB each\n- 10,000 watch connections = 100-1000MB RAM\n- Long-lived TCP connections from clients\n\n**Total Capacity**: Typically 1000-10,000 concurrent requests on small cluster, scales with memory limit\n\n### 6.3 Network I/O\n\n**Request Size Limits**:\n- Body: configurable (default unlimited)\n- Large requests (>100MB) not recommended\n- Streaming watch responses sent incrementally\n\n**Backend Communication**:\n- etcd: Roundtrip 50-200ms over LAN\n- Webhook: Roundtrip 100-1000ms common\n- WAN deployments: add 50-500ms latency\n\n### 6.4 Storage Interaction\n\n**Write Amplification**:\n- Single create request = 1 etcd write + 1 audit write\n- With webhooks: multiple reads (policy fetch) + writes\n- Strategic merge patch: read + merge + write\n\n**Consistency Model**:\n- Per-key: atomic (optimistic locking via resourceVersion)\n- Cross-key: eventual consistency (no transactions)\n- Watch: \"best effort\" ordering (may miss intermediate states in high-load scenarios)\n\n**Garbage Collection**:\n- No automatic cleanup by apiserver\n- Depends on storage implementation (etcd has compaction)\n- Finalizers pattern used for controlled deletion\n\n### 6.5 Reliability Patterns\n\n**Graceful Shutdown** (`pkg/server/secure_serving.go`):\n\n1. Signal received (SIGTERM)\n2. Stop accepting NEW requests (return 429 or 503)\n3. Wait for in-flight requests to complete (timeout configurable)\n4. Close connections\n5. Shutdown hooks execute\n6. Exit\n\n**Configuration**:\n- `ShutdownTimeout`: how long to wait (typical 30s)\n- `ShutdownDelayDuration`: how long to return 503 before draining (typical 5-10s)\n\n**Best Practice**: Configure load balancer to health check /readyz during shutdown drain period\n\n**Watch Connection Draining** (lines 1072-1074):\n\nDuring shutdown, watch connections get graceful termination notice:\n- Server sends `WatchClose` event to clients\n- Clients reconnect to new server\n- Prevents data loss for watch-based controllers\n\n### 6.6 Debugging and Troubleshooting\n\n**Available Endpoints** (when enabled in config):\n\n- `/debug/pprof` - CPU profiling, memory profiling\n- `/metrics` - Prometheus metrics\n- `/debug/flags` - View/modify runtime flags\n- `/version` - Server version info\n- `/healthz`, `/readyz`, `/livez` - Health checks with details\n\n**Key Metrics to Monitor**:\n\n```\napiserver_request_duration_seconds - p99 > 1s indicates slow handlers\napiserver_request_total - sudden drop indicates auth/admission failure\napiserver_authorized_decision_total - track authz denial rates\napiserver_admission_decision_total - track webhook rejection rates\napiserver_request_body_size_bytes - detect large requests\nstorage_operation_duration_seconds - etcd performance\n```\n\n**Common Issues**:\n\n1. **High latency**: Check webhook performance, etcd latency, admission controller logs\n2. **High memory**: Check watch connection count, audit event buffering\n3. **Timeouts**: Increase RequestTimeout, check backend connectivity\n4. **Auth failures**: Check authenticator configuration, token validation\n5. **Cascading failures**: Check health check configuration, avoid external dependencies\n\n### 6.7 Multi-Server Coordination\n\n**Etcd Serves as Coordination Layer**:\n- Leader election via etcd leases (not in apiserver library)\n- Configuration distribution via etcd watch\n- Storage versioning coordination\n\n**Apiserver Library Does NOT Handle**:\n- Cluster membership (delegate to etcd or external)\n- Consensus (delegate to etcd)\n- Configuration agreement (each server has independent config)\n\n**Design**: Stateless, coordinate via shared storage\n\n---\n\n## 7. Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n**Authentication Failure (401 Unauthorized)**\n- Invalid or missing credentials\n- Authenticator returns error\n- Response includes audit event with `user.anonymous`\n- Connection remains open (client can retry with valid credentials)\n- Impact: Client must resubmit with valid auth\n\n**Authorization Failure (403 Forbidden)**\n- User authenticated but lacks permission for verb/resource\n- Request fully processed (authenticated, admitted) before authz check\n- Impact: Confirms existence of resource (information leak - design tradeoff)\n\n**Admission Rejection (400 Bad Request or Custom Status)**\n- Mutating webhook rejects due to policy violation\n- Validating webhook rejects after mutation or storage write\n- Audit event includes rejection reason\n- Impact: Object may exist in storage but client sees rejection\n\n**Storage Failure (500 Internal Server Error)**\n- Etcd unreachable or timeout\n- Serialization error\n- Quota exceeded\n- Request marked as failed in audit\n- Impact: Request incomplete, may be partially persisted (depends on storage semantics)\n\n**Rate Limiting (429 Too Many Requests)**\n- Request queue full in Priority & Fairness controller\n- Response includes `Retry-After` header\n- Request NOT processed, no admission or storage mutation\n- Impact: Request is dropped at gateway before reaching handlers\n\n**Timeout (504 Gateway Timeout or 408 Request Timeout)**\n- Request context deadline exceeded\n- Non-long-running requests: timeout after `RequestTimeout` config\n- Watch requests: can run indefinitely\n- If mutating operation: may partially complete\n- Impact: Connection closed, client must retry (potential duplication)\n\n**Panic (500 Internal Server Error)**\n- Handler panic caught by `WithPanicRecovery`\n- Stack trace logged, response set to 500\n- Recovery prevents cascade failures\n- Impact: Minimal - single request fails, server continues\n\n### 4.2 Resource Limits\n\n**Request Body Size**\n- `maxRequestBodyBytes` config limits request body\n- Default: unlimited (0)\n- Exceeded: 413 Request Entity Too Large\n- Applied at handler level (before unmarshaling)\n\n**In-Flight Request Limits**\n- `MaxRequestsInFlight`: max concurrent requests\n- `MaxMutatingRequestsInFlight`: max concurrent mutating requests\n- Exceeded: 429 Too Many Requests (new requests queued)\n- Design: Prevents resource exhaustion\n\n**Timeout Limits**\n- `minRequestTimeout`: minimum allowed client timeout\n- `requestTimeoutUpperBound`: max timeout for context\n- Set: 34 seconds default (unusual number to catch errors)\n- Design: Prevents clients hanging indefinitely, but allows long operations\n\n**Watch Streaming**\n- Watches are long-running (exempt from timeouts)\n- Streaming responses sent incremental to client\n- Client can cancel with TCP FIN or timeout\n\n**Memory Limits**\n- No built-in limits on number of objects\n- Relies on etcd/storage backend limits\n- No garbage collection by apiserver\n\n### 4.3 Design Tradeoffs\n\n**Tradeoff: Admission Ordering vs. Flexibility**\n\n*Decision*: Admission controllers execute in fixed order (mutating \u2192 validating)\n\n*Benefit*: Predictable behavior, validators see final state\n*Cost*: Can't inject conditional validators, can't skip controllers\n\n*Evidence*: `pkg/admission/chain.go` hardcodes mutating vs validating phases\n\n**Tradeoff: Handler Chain Wrapping vs. Plugin System**\n\n*Decision*: Filter middleware wraps HTTP handler (not plugin system)\n\n*Benefit*: Type-safe composition, no reflection, clear execution order\n*Cost*: Must recompile to add filters, can't dynamically load middleware\n\n*Evidence*: `pkg/server/config.go` DefaultBuildHandlerChain returns wrapped handler\n\n**Tradeoff: Synchronous Admission vs. Async Webhooks**\n\n*Decision*: Admission runs synchronously in request handler\n\n*Benefit*: Client receives immediate feedback (success/failure)\n*Cost*: Webhook latency directly impacts API latency (many orgs add load balancer timeout)\n*Evidence*: Many deployments disable validating webhooks on high-volume resources\n\n**Tradeoff: Storage Interface Flexibility vs. Consistency**\n\n*Decision*: Storage interface minimal, no transaction support\n\n*Benefit*: Simple, works with any backend\n*Cost*: No ACID transactions across multiple resources (leads to eventual consistency patterns)\n*Evidence*: `pkg/registry/rest/rest.go` defines simple Create/Update/Delete\n\n**Tradeoff: Audit After Storage vs. Before**\n\n*Decision*: Audit runs after handlers complete\n\n*Benefit*: Audit logs reflect actual stored state\n*Cost*: Audit lag (asynchronous in many backends), audit webhook latency impacts request latency\n*Evidence*: `pkg/endpoints/filters/audit.go` wraps handlers, not called before\n\n**Tradeoff: RequestInfo Parsing Overhead**\n\n*Decision*: All requests parsed to extract namespace/resource/verb\n\n*Benefit*: Consistent across all handlers, enables RBAC\n*Cost*: CPU overhead on every request\n*Evidence*: `WithRequestInfo` (line 1091) called for all requests including 404s\n\n**Tradeoff: Director Pattern (GoRestful + NonGoRestfulMux)**\n\n*Decision*: Dual-handler architecture with director dispatch\n\n*Benefit*: Enables OpenAPI generation (go-restful) while supporting delegation\n*Cost*: Complexity, potential for routing bugs, go-restful performance overhead\n*Evidence*: `pkg/server/handler.go` director type, lines 121-154\n*Comment*: This is widely recognized as a wart; maintainers acknowledge go-restful dependency should be removed\n\n**Tradeoff: Imperative vs. Declarative Request Processing**\n\n*Decision*: Handlers are imperative (hand-written, not declarative schemas)\n\n*Benefit*: Flexibility (can implement non-CRUD operations), precise control\n*Cost*: High complexity for standard CRUD, duplicated logic across handlers\n*Evidence*: `pkg/endpoints/handlers/create.go` (10KB) vs. similar in other frameworks\n\n---\n\n## 5. Extension Points, Customization Hooks, and Risks\n\n### 5.1 Primary Extension Points\n\n#### 5.1.1 Handler Chain Customization\n\n**Extension Point**: `Config.BuildHandlerChainFunc`\n\n```go\ntype Operational Implications\n\n### 6.1 Performance Characteristics\n\n**Request Latency Breakdown** (typical values):\n\n1. **Handler chain overhead**: 1-2ms\n   - WithPanicRecovery, WithRequestInfo, context setup\n2. **Authentication**: 5-50ms\n   - Token validation, webhook calls\n3. **Authorization**: 10-100ms\n   - RBAC policy evaluation (worse with large role sets)\n4. **Admission**: 50-500ms\n   - Webhook roundtrips (worst offender in most deployments)\n5. **Storage read**: 5-50ms (etcd 100ms+ over WAN)\n6. **Encoding/serialization**: 1-10ms\n\n**Total (median create request)**: 75-300ms\n**P99 (with webhook retries, lock contention)**: 1-5s\n\n**Optimization Opportunities**:\n- Move webhook validation to separate component\n- Cache authorization decisions (10s TTL typical)\n- Batch etcd requests for list operations\n- Use protobuf instead of JSON (3-5x faster)\n\n### 6.2 Memory Usage\n\n**Per-Server Baseline**: 100-300MB\n- go-restful reflection for registered routes: ~50MB\n- OpenAPI schema: ~50MB\n- Admission controller state: variable\n\n**Per-Request Memory**: 1-10MB\n- Request deserialization buffer\n- Audit event structures\n- Watch connection metadata (long-lived)\n\n**Watch Connections**: 10KB-1MB each\n- 10,000 watch connections = 100-1000MB RAM\n- Long-lived TCP connections from clients\n\n**Total Capacity**: Typically 1000-10,000 concurrent requests on small cluster, scales with memory limit\n\n### 6.3 Network I/O\n\n**Request Size Limits**:\n- Body: configurable (default unlimited)\n- Large requests (>100MB) not recommended\n- Streaming watch responses sent incrementally\n\n**Backend Communication**:\n- etcd: Roundtrip 50-200ms over LAN\n- Webhook: Roundtrip 100-1000ms common\n- WAN deployments: add 50-500ms latency\n\n### 6.4 Storage Interaction\n\n**Write Amplification**:\n- Single create request = 1 etcd write + 1 audit write\n- With webhooks: multiple reads (policy fetch) + writes\n- Strategic merge patch: read + merge + write\n\n**Consistency Model**:\n- Per-key: atomic (optimistic locking via resourceVersion)\n- Cross-key: eventual consistency (no transactions)\n- Watch: \"best effort\" ordering (may miss intermediate states in high-load scenarios)\n\n**Garbage Collection**:\n- No automatic cleanup by apiserver\n- Depends on storage implementation (etcd has compaction)\n- Finalizers pattern used for controlled deletion\n\n### 6.5 Reliability Patterns\n\n**Graceful Shutdown** (`pkg/server/secure_serving.go`):\n\n1. Signal received (SIGTERM)\n2. Stop accepting NEW requests (return 429 or 503)\n3. Wait for in-flight requests to complete (timeout configurable)\n4. Close connections\n5. Shutdown hooks execute\n6. Exit\n\n**Configuration**:\n- `ShutdownTimeout`: how long to wait (typical 30s)\n- `ShutdownDelayDuration`: how long to return 503 before draining (typical 5-10s)\n\n**Best Practice**: Configure load balancer to health check /readyz during shutdown drain period\n\n**Watch Connection Draining** (lines 1072-1074):\n\nDuring shutdown, watch connections get graceful termination notice:\n- Server sends `WatchClose` event to clients\n- Clients reconnect to new server\n- Prevents data loss for watch-based controllers\n\n### 6.6 Debugging and Troubleshooting\n\n**Available Endpoints** (when enabled in config):\n\n- `/debug/pprof` - CPU profiling, memory profiling\n- `/metrics` - Prometheus metrics\n- `/debug/flags` - View/modify runtime flags\n- `/version` - Server version info\n- `/healthz`, `/readyz`, `/livez` - Health checks with details\n\n**Key Metrics to Monitor**:\n\n```\napiserver_request_duration_seconds - p99 > 1s indicates slow handlers\napiserver_request_total - sudden drop indicates auth/admission failure\napiserver_authorized_decision_total - track authz denial rates\napiserver_admission_decision_total - track webhook rejection rates\napiserver_request_body_size_bytes - detect large requests\nstorage_operation_duration_seconds - etcd performance\n```\n\n**Common Issues**:\n\n1. **High latency**: Check webhook performance, etcd latency, admission controller logs\n2. **High memory**: Check watch connection count, audit event buffering\n3. **Timeouts**: Increase RequestTimeout, check backend connectivity\n4. **Auth failures**: Check authenticator configuration, token validation\n5. **Cascading failures**: Check health check configuration, avoid external dependencies\n\n### 6.7 Multi-Server Coordination\n\n**Etcd Serves as Coordination Layer**:\n- Leader election via etcd leases (not in apiserver library)\n- Configuration distribution via etcd watch\n- Storage versioning coordination\n\n**Apiserver Library Does NOT Handle**:\n- Cluster membership (delegate to etcd or external)\n- Consensus (delegate to etcd)\n- Configuration agreement (each server has independent config)\n\n**Design**: Stateless, coordinate via shared storage\n\n---\n\n## 7. # Kubernetes API Server Architecture and Extension Guide

## Executive Summary

The Kubernetes apiserver library (`staging/src/k8s.io/apiserver/`) is a foundational subsystem that implements the core request handling pipeline for API servers. It provides the HTTP handler chain, resource installation, authentication/authorization, admission control, and lifecycle management. The architecture emphasizes layered responsibility separation, with each layer addressing distinct concerns (security, rate limiting, request validation, storage) while maintaining extension points for customization.

---

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose
The apiserver library provides a generic, reusable foundation for building Kubernetes-compatible REST API servers. Its primary responsibilities are:
- **Request routing and handling** via an HTTP handler chain
- **Resource lifecycle management** (CRUD operations + watch)
- **Security enforcement** (authentication, authorization, admission control)
- **Request management** (rate limiting, timeout, request tracking)
- **Lifecycle hooks** (startup, shutdown, health checks)
- **API discovery and OpenAPI generation**

### Boundaries
- **Upstream dependencies**: Uses apimachinery (type system, serialization, runtime schema) and client-go (client configuration)
- **Downstream consumers**: Used by kube-apiserver, kube-aggregator, and custom API servers built on top
- **Storage abstraction**: The library defines `rest.Storage` interface but does NOT implement storage; consumers provide storage implementations
- **Configuration**: Does NOT manage configuration files; consumers provide `Config` struct with pre-configured components
