Returns consumer to \"no subscription\" state.\n\n### Manual Partition Assignment\n\n```java\nvoid assign(Collection<TopicPartition> partitions)\n```\n\n**Characteristics**:\n- Disables group coordination entirely\n- No rebalancing; partitions only change via explicit `assign()` calls\n- Consumer acts independently\n- Still uses `group.id` for offset storage, but not for coordination\n- Suitable for standalone consumers or externally-coordinated systems\n\n**Example**:\n\n```java\nTopicPartition partition0 = new TopicPartition(\"topic\", 0);\nTopicPartition partition1 = new TopicPartition(\"topic\", 1);\nconsumer.assign(Arrays.asList(partition0, partition1));\n```\n\n**Use Cases**:\n- State machines with local state per partition\n- External orchestration (e.g., YARN, Kubernetes)\n- Distributed stream processing frameworks\n- Partition-level failure recovery\n\n---\n\n## Polling and Data Fetching\n\n### Poll Semantics\n\n```java\nConsumerRecords<K, V> poll(Duration timeout)\n```\n\n**Critical Behavior**:\n\n1. **Immediate Return**: If records are available or position advances past control/aborted records, returns immediately\n2. **Blocking Wait**: Otherwise waits up to `timeout` for data\n3. **May Block Longer**: Poll can block longer than `timeout` during:\n   - Group rebalances (running `ConsumerRebalanceListener` callbacks)\n   - Metadata refresh\n   - Fetching stable offsets\n4. **Rebalance Execution**: Only during `poll()` are rebalances detected and executed\n5. **Position Advancement**: Automatically updates consumer position as records are returned\n\n**Implementation Pattern**:\n\n```java\nwhile (true) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n    for (ConsumerRecord<String, String> record : records) {\n        processRecord(record);\n    }\n}\n```\n\n### Offset Behavior During Poll\n\n**Initial State**: Consumer position initialized from one of:\n1. `seek()` calls\n2. Committed offset from `committed()`\n3. Offset reset policy (`auto.offset.reset`)\n\n**Position Tracking**:\n- Position advances **after** records are returned to application\n- Returned records: offsets [position, position + count)\n- After poll returns, position = position + count\n\n**Example**:\n\n```\nInitial position: 100\nPoll returns records with offsets: 100, 101, 102\nNew position: 103\nNext poll returns: 103+\n```\n\n### Empty Records\n\nIf no data is available within the timeout:\n\n```java\nConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\nassert records.isEmpty(); // true\n```\n\n---\n\n## Offset Management\n\n### Offset Concepts\n\n**Position**: Offset of the next record to be fetched. Automatically maintained and advanced by `poll()`.\n\n**Committed Offset**: Last offset whose processing is complete. Used to resume consumption on failure/restart.\n\n**Offset Semantics**: The committed offset should represent the **next offset to consume**, not the last consumed offset.\n\nExample:\n```\nConsumed: [0, 1, 2, 3]\nPosition: 4\nCommit: 4  (not 3!)\n```\n\n### Automatic Offset Commits\n\nWhen `enable.auto.commit=true`:\n\n- Commits offsets periodically (controlled by `auto.commit.interval.ms`)\n- Commit happens asynchronously during `poll()`\n- Commits the current position for all assigned partitions\n\n**Risk**: If application crashes between `poll()` returning records and those records being processed, committed offset may be ahead of actual processing.\n\n```java\n// \u274c RISKY: Auto-commit with sync processing\nconsumer.poll(...);  // Offsets committed after this\n// ... if we crash here, records are lost\nprocessRecords();\n```\n\n### Manual Offset Commits (Synchronous)\n\n```java\nvoid commitSync()                                    // All assigned partitions\nvoid commitSync(Duration timeout)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets)\nvoid commitSync(Map<TopicPartition, OffsetAndMetadata> offsets, Duration timeout)\n```\n\n**Behavior**:\n- Blocks until commit succeeds or fails\n- Commits offsets to Kafka's `__consumer_offsets` topic\n- Uses `default.api.timeout.ms` as default timeout\n- Subsequent async commits guaranteed to complete before `commitSync()` returns\n\n**Exceptions**:\n- `CommitFailedException`: Unrecoverable; consumer lost partitions during rebalance\n- `RebalanceInProgressException`: Transient; retry after `poll()`\n- `WakeupException`: `wakeup()` called from another thread\n- `TimeoutException`: Timeout expired\n\n**Pattern: Commit After Processing**\n\n```java\nconsumer.subscribe(Arrays.asList(\"topic\"));\nwhile (true) {\n    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));\n    for Buffering**\n\n```java\nconsumer.subscribe(Arrays.asList(\"topic\"));\nList<ConsumerRecord<String, String>> buffer = new ArrayList<>();\nint batchSize = 1000;\n\nwhile (true) {\n Membership\n\n**Automatic**: When using `subscribe()`, consumer automatically joins group when first `poll()` is called.\n\n**Heartbeats**: Consumer sends periodic heartbeats to coordinator to signal liveness.\n\n**Session Timeout**: If heartbeats not received for `session.timeout.ms`, broker considers consumer dead:\n- Triggers rebalance\n- Partitions reassigned to healthy consumers\n- Consumer that timed out may not know it's been evicted\n\n### Liveness Detection: Max Poll Interval\n\n**Problem**: Heartbeats only prove network connectivity, not processing progress.\n\n**Solution**: `max.poll.interval.ms` - if consumer doesn't call `poll()` frequently enough, it's considered hung:\n\n```java\n// If gap between polls exceeds max.poll.interval.ms...\nsleep(max_poll_interval_ms + 1000);\nconsumer.poll(Duration.ofSeconds(1));  // CommitFailedException!\n```\n\n**Behavior**:\n- Consumer proactively leaves group to allow partition reassignment\n- Any subsequent `commitSync()` fails with `CommitFailedException`\n- Application must call `poll()` to rejoin group\n\n**Tuning**:\n- Increase `max.poll.interval.ms` for slow processing\n- Reduce `max.poll.records` to return fewer records per poll\n- Move processing to separate thread (see patterns below)\n\n### Static Group Membership\n\n**Feature**: Static group instance IDs allow consumers to rejoin without triggering rebalance.\n\n**Configuration**:\n\n```properties\ngroup.instance.id=consumer-1\n```\n\n**Benefit**: Rebalance doesn't occur when consumer restarts if shutdown is brief.\n\n### Group Metadata\n\n```java\nConsumerGroupMetadata groupMetadata()\n```\n\nReturns metadata about consumer's group membership (generational ID, member ID, etc.).\n\n---\n\n## Rebalancing and Callbacks\n\n### Rebalance Triggers\n\nRebalances occur when:\n\n1. **Group membership changes**: Consumer joins/leaves/fails\n2. **Topic subscription changes**: Via `subscribe()` call\n3. **Pattern matches new topics**: Via pattern matching\n4. **Partition count increases**: New partitions added to subscribed topic\n5. **Manual enforcement**: `enforceRebalance()` called\n\n### Rebalance Timeline\n\n1. **Detection**: Broker detects group change\n2. **Initiation**: During `poll()`, consumer receives revoke signal\n3. **Callback - onPartitionsRevoked()**: Application commits offsets, cleans up state\n4. **Partition Assignment**: Assignor runs; partitions reassigned\n5. **Callback - onPartitionsAssigned()**: Application initializes state for new partitions\n6. **Resume Fetching**: `poll()` returns normally\n\n### Callback Ordering\n\n**Standard Path** (eager rebalancing):\n1. `onPartitionsRevoked()` called with all assigned partitions\n2. Rebalance completes\n3. `onPartitionsAssigned()` called with all assigned partitions (even if same)\n\n**Cooperative Rebalancing**:\n1. `onPartitionsRevoked()` called only with partitions actually revoked\n2. Rebalance completes\n3. `onPartitionsAssigned()` called only with new partitions\n\n**Exception Path** (session timeout):\n1. `onPartitionsLost()` called instead of `onPartitionsRevoked()`\n2. Cannot safely commit offsets (partitions already owned by others)\n\n### ConsumerRebalanceListener Implementation\n\n```java\nclass SaveOffsetsOnRebalance implements ConsumerRebalanceListener {\n    private final Consumer<?, ?> consumer;\n    private final OffsetStore store;\n\n    SaveOffsetsOnRebalance(Consumer<?, ?> consumer, OffsetStore store) {\n        this.consumer = consumer;\n        this.store = store;\n    }\n\n    @Override\n    public void onPartitionsRevoked(Collection<TopicPartition> partitions) {\n        // Commit offsets before losing partitions\n        for (TopicPartition partition : partitions) {\n            long offset = consumer.position(partition);\n            store.saveOffset(partition, offset);\n        }\n    }\n\n    @Override\n    public void onPartitionsAssigned(Collection<TopicPartition> partitions) {\n        // Restore offsets for new partitions\n        for (TopicPartition partition : partitions) {\n            long offset = store.loadOffset(partition);\n            consumer.seek(partition, offset);\n        }\n    }\n\n    @Override\n    public void onPartitionsLost(Collection<TopicPartition> partitions) {\n        // Partitions already lost; don't try to commit\n    }\n}\n```\n\n### Enforce Rebalance\n\n```java\nvoid enforceRebalance()\nvoid enforceRebalance(String reason)\n```\n\nForces consumer to rejoin group on next `poll()`, triggering rebalance.\n\n**Use Case**: Application state changes warrant partition reassignment based on `ConsumerPartitionAssignor` user data.\n\n---\n\n## Metadata and Monitoring\n\n### Partition Information\n\n```java\nList<PartitionInfo> partitionsFor(String topic)\nList<PartitionInfo> partitionsFor(String topic, Duration timeout)\nMap<String, List<PartitionInfo>> listTopics()\nMap<String, List<PartitionInfo>> listTopics(Duration timeout)\n```\n\n**Behavior**:\n- Fetches metadata from broker (blocking)\n- Does not require assignment or subscription\n- Useful for discovering available topics/partitions\n\n### Metrics\n\n```java\nMap<MetricName, ? extends Metric> metrics()\n```\n\nReturns JMX metrics exposed by consumer. `max.poll.interval.ms` exceeded; consumer proactively left group.\n\n**Recovery**: Not recoverable; indicates poll loop is blocked too long.\n\n```java\ntry {\n    consumer.commitSync();\n} catch (CommitFailedException e) {\n    // Consumer lost group membership\n    // Exit poll loop and rejoin\n    break;\n}\n```\n\n### RetriableCommitFailedException\n\n**When**: Transient commit failure; can retry.\n\n**Causes**:\n- Broker temporarily unavailable\n- Network timeout\n- Rebalance in progress\n\n**Recovery**: Retry or wait for transient condition to pass.\n\n```java\ntry {\n    consumer.commitAsync(offsets, (offsets, ex) -> {\n        if (ex instanceof RetriableCommitFailedException) {\n            // Retry later\n        }\n    });\n} catch (RetriableCommitFailedException e) {\n    // Retry in next poll\n}\n```\n\n### OffsetOutOfRangeException\n\n**When**: Seeking to invalid offset (not in broker log).\n\n**Causes**:\n- Offset smaller than log start (old messages deleted)\n- Offset larger than log end\n- Partition doesn't exist\n\n**Recovery**: Use `auto.offset.reset` policy or explicitly seek to valid range.\n\n```java\ntry {\n    consumer.poll(Duration.ofSeconds(1));\n} catch (OffsetOutOfRangeException e) {\n    // Handle invalid offset\n    consumer.seekToBeginning(consumer.assignment());\n}\n```\n\n### NoOffsetForPartitionException\n\n**When**: No offset available for partition (never committed, offset expired).\n\n**Recovery**: Use `auto.offset.reset` to determine fallback behavior.\n\n### InvalidOffsetException\n\n**When**: Offset initialization fails; unrecoverable.\n\n**Indicates**: Configuration or state issue; usually fatal.\n\n### WakeupException\n\n**When**: Another thread called `wakeup()`.\n\n**Behavior**: Interrupts blocking operation (poll, commit, etc.).\n\n**Pattern**:\n\n```java\ntry {\n    consumer.poll(Duration.ofSeconds(1));\n} catch (WakeupException e) {\n    if (shutdownRequested) {\n        // Expected; exit gracefully\n        break;\n    } else {\n        // Unexpected; rethrow\n        throw e;\n    }\n}\n```\n\n### InterruptException\n\n**When**: Consumer thread was interrupted (via `Thread.interrupt()`).\n\n**Note**: `WakeupException` is preferred; `InterruptException` supported for compatibility.\n\n### AuthenticationException\n\n**When**: Authentication credentials invalid or expired.\n\n**Recoverable**: No; indicates configuration issue.\n\n### AuthorizationException\n\n**When**: Consumer lacks permissions for topics/group.\n\n**Recoverable**: No; requires permission changes.\n\n### RebalanceInProgressException\n\n**When**: Sync commit fails because rebalance in progress.\n\n**Recovery**: Retry after `poll()` completes rebalance.\n\n```java\nwhile (true) {\n    try {\n        consumer.commitSync();\n        break;\n    } catch (RebalanceInProgressException e) {\n        consumer.poll(Duration.ofSeconds(1));  // Complete rebalance\n        // Try commit again\n    }\n}\n```\n\n### FencedInstanceIdException\n\n**When**: Another consumer with same `group.instance.id` joined group.\n\n**Cause**: Two instances with same static member ID.\n\n**Recovery**: Ensure unique instance IDs or wait for fencing timeout.\n\n---\n\n## Configuration-Driven Behaviors\n\n### Offset ## Overview

The `KafkaConsumer<K, V>` is a Java client for consuming records from Apache Kafka clusters. It provides a flexible API supporting multiple consumption patterns:

- **Group-managed consumption**: Multiple consumer instances automatically partition work through group coordination
- **Standalone consumption**: Manual partition assignment for independent consumers
- **Dynamic topic subscription**: Subscribe to topics and automatically discover new partitions
- **Manual offset management**: Either automatic or explicit offset commits
