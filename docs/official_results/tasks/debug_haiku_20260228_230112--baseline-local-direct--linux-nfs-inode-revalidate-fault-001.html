## Bug Report

**Bug ID**: 117651
**Summary**: Root NFS and autofs - mount disappears due to inode revalidate failed
**Regression**: Yes (works on 3.14.12, broken on 4.1.15)

After upgrading from 3.14.12 to 4.1.15 I've got strange behaviour with autofs.

It happens on diskless machine with root on NFS. There is autofs indirect mount on /storage, where NFS volumes are mounted on demand.

Everything mounts normally, but after mount expiration (during unmount) autofs mount disappears - that is, not only /storage/<volume> unmounts, but /storage is also unmounted. This happens quite often - usually on the very first expiration.

automount logs umount error in such cases.

Without autofs, mount/umount of the same NFS volume works correctly.

With NFS debug enabled, I've got following errors:

```
NFS reply getattr: -512
nfs_revalidate_inode: (0:15/6291480) getattr failed, error=-512
NFS: nfs_lookup_revalidate(/storage) is invalid
```

After inode revalidate failure all mounts on this inode are unmounted and so autofs mount disappears, while automount daemon itself continue to run.

tcpdump shows, that server replies to getattr without error, but client doesn't see this reply, and returning error instead.

I am ready to supply additional information.

## Summary

I have successfully identified the root cause of the NFS mount disappearance bug during autofs mount expiration. The issue is a **race condition in signal handling between the RPC layer and the NFS layer**.

### Key Findings:

**Buggy Files:**
1. `net/sunrpc/sched.c` - RPC scheduler task execution
2. `fs/nfs/inode.c` - NFS inode revalidation

**Buggy Functions:**
1. `__rpc_execute()` (net/sunrpc/sched.c:780-792) - Exits with -ERESTARTSYS when signal is detected during reply wait
2. `__nfs_revalidate_inode()` (fs/nfs/inode.c:960-970) - Treats -ERESTARTSYS as a permanent error

### Root Cause:
When NFS revalidates an inode (e.g., during autofs unmount), it initiates an RPC getattr call. The RPC scheduler waits for the server's reply. If a signal (SIGKILL) is delivered during this wait - which can happen during autofs unmount/shutdown processing - the scheduler immediately exits with `-ERESTARTSYS` (error code -512) even though the server may have sent or be about to send the reply. The NFS layer treats this as a permanent error, marking the inode as invalid, causing the mount point to become inaccessible and subsequently unmounted.

- Detailed code inspection of RPC scheduler signal handling

- Understanding of the autofs mount expiration scenario
