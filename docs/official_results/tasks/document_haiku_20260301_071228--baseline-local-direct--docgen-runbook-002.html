# Envoy Connection Pool Management Troubleshooting Runbook

## 1. Architecture Overview

### Connection Pool Organization

Envoy maintains **per-cluster, per-thread connection pools** to manage upstream connections efficiently. Each Envoy worker thread has its own connection pool for every configured cluster, avoiding synchronization overhead.

**Pool Structure:**
```
Cluster
  ├─ Thread 0 Pool
  │   ├─ Ready Clients (state: Ready/ReadyForEarlyData)
  │   ├─ Busy Clients (state: Busy - at concurrent stream limit)
  │   ├─ Connecting Clients (state: Connecting - awaiting connection establish)
  │   ├─ Early Data Clients (HTTP/3 QUIC connections)
  │   └─ Pending Streams Queue (waiting for available connection)
  ├─ Thread 1 Pool
  │   └─ ...
  └─ Thread N Pool
```

**Key Metrics Tracked per Pool:**
- `connecting_stream_capacity_`: Available capacity from connecting connections
- `connecting_and_connected_stream_capacity_`: Total capacity from connecting + ready connections
- `pending_streams_`: Queued requests awaiting connection availability
- `num_active_streams_`: Currently active streams across all connections

### Connection Lifecycle States

Each active connection in the pool transitions through states:

1. **Connecting** (initial state)
   - TCP handshake in progress
   - `connect_timeout` timer active (cluster config)
   - Connection duration timer started

2. **Ready** or **ReadyForEarlyData** (HTTP/3 only)
   - Handshake completed, accepts new streams
   - For HTTP/2: can accept up to `max_concurrent_streams` streams
   - For HTTP/1.1: can only accept 1 stream at a time

3. **Busy**
   - At concurrent stream limit
   - No new streams assigned until capacity opens
   - Contributes to `upstream_cx_active` counter

4. **Draining**
   - No new streams accepted
