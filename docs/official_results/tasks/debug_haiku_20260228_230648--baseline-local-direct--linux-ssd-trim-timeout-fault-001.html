## Summary

I have successfully localized the fault causing the Samsung 860 EVO queued TRIM issues in the Linux kernel (4.14.114). Here are my findings:

### **Root Cause**
The Samsung 860 EVO SSD is **missing from the ATA device blacklist**, causing it to attempt queued TRIM operations that it cannot handle properly, resulting in 20-30 second lockups during fstrim.

### **Buggy File**
- **`drivers/ata/libata-core.c`** - Contains the ATA device blacklist with workarounds for problematic storage devices

### **Buggy Data Structure**
- **`ata_device_blacklist[]`** - A static array starting at line 4393 that maps device models to horkage flags (workarounds)

### **The Problem**
The `ata_device_blacklist` array includes entries for:
- Samsung SSD 840* (lines 4562-4563)
- Samsung SSD 850* (lines 4564-4565)

But **Samsung SSD 860* is missing**. Both the 840 and 850 have the `ATA_HORKAGE_NO_NCQ_TRIM` flag set, which disables queued TRIM support. The 860 EVO needs the same workaround.

### **How the Fix Works**
1. When a device is probed, `ata_dev_blacklisted()` (line 4620) searches the blacklist table
2. The matching horkage flags are applied (line 2477): `dev->horkage |= ata_dev_blacklisted(dev)`
3. Later, `ata_dev_config_ncq_send_recv()` (line 2197) checks for `ATA_HORKAGE_NO_NCQ_TRIM`
4. If the flag is set, queued TRIM is disabled (lines 2218-2221), preventing the device from using the problematic feature

### **Required Fix**
Add a new entry to the blacklist array after line 4565:
```c
{ "Samsung SSD 860*",		NULL,	ATA_HORKAGE_NO_NCQ_TRIM |
					ATA_HORKAGE_ZERO_AFTER_TRIM, },
```
