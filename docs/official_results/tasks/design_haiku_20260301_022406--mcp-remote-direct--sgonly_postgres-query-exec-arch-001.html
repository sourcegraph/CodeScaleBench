Calls its children to get tuples (demand-pull)\n  2. Applies its operation (scan, join, aggregate, etc.)\n  3. Returns the result tuple\n- This is implemented via function pointers in PlanState nodes\n- Allows for:\n  - Pipelined execution (no need to materialize all intermediate results)\n  - Partial/LIMIT execution (stop when limit is reached)\n  - Streaming joins and aggregations\n\n#### 4. **Parallel Read-Only Data Structures (Plan Tree)**\n- **Plan Tree** (created by planner) is read-only throughout execution\n- **PlanState Tree** (created by executor) is the mutable execution state\n- This separation allows plan reuse without modification\n\n#### 5. **Context-Based Memory Management**\n- Per-query context: Holds query plan and long-lived data\n- Per-tuple context: Holds temporary per-tuple expression evaluation results\n- Contexts are freed hierarchically to avoid memory leaks\n\n#### 6. **Hook-Based Extensibility**\nPostgreSQL allows plugins to intercept execution at multiple points:\n- `planner_hook` - Plugin planner implementation\n- `ExecutorStart_hook`, `ExecutorRun_hook`, `ExecutorEnd_hook` - Plugin executor hooks\n- `post_parse_analyze_hook` - Post-analysis hook\n- This enables extensions, query logging, and custom planning strategies\n\n### Component
