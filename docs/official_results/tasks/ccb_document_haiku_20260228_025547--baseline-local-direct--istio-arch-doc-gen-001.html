moves on\n- Otherwise, generates the requested resources and sends `DiscoveryResponse`\n\n### 2.4 Push Delivery (`pushConnection()`)\n\nWhen a configuration change triggers a fleet-wide push:\n\n```go\nfunc (s *DiscoveryServer) pushConnection(con *Connection, pushEv *Event) error {\n    pushRequest := pushEv.pushRequest\n\n    if pushRequest.Full {\n        // Update proxy state (service targets, sidecar scope, etc.)\n        s.computeProxyState(con.proxy, pushRequest)\n    }\n\n    // Check if this proxy actually needs updates\n    pushRequest, needsPush := s.ProxyNeedsPush(con.proxy, pushRequest)\n    if !needsPush {\n        return nil  // Skip push for this proxy\n    }\n\n    // Send all watched resource types in order: CDS \u2192 EDS \u2192 LDS \u2192 RDS \u2192 SDS \u2192 ...\n    wrl := con.watchedResourcesByOrder()\n    for _, w := range wrl {\n        if err := s.pushXds(con, w, pushRequest); err != nil {\n            return err\n        }\n    }\n    return nil\n}\n```\n\n**Push Ordering:** `PushOrder` defines the order xDS resources are sent:\n1. **CDS** (Clusters): Service discovery, load balancing endpoints\n2. **EDS** (Endpoints): Which instances (IPs/ports) are available\n3. **LDS** (Listeners): Which ports to listen on, filter chains\n4. **RDS** (Routes): HTTP routing rules\n5. **SDS** (Secrets): TLS certificates\n6. **Address, Workload, WorkloadAuthorization** (for ambient mesh)\n\nThis order is critical: Envoy requires CDS before EDS, and LDS before RDS.\n\n### 2.5 StartPush \u2014 Broadcast to All Proxies\n\n```go\nfunc (s *DiscoveryServer) StartPush(req *model.PushRequest) {\n    req.Start = time.Now()\n    for _, p := range s.AllClients() {\n        s.pushQueue.Enqueue(p, req)\n    }\n}\n```\n\nWhen a configuration change occurs, `StartPush()` enqueues the same `PushRequest` for every connected proxy. The `PushQueue` then distributes pushes, respecting concurrency limits.\n\n---\n\n## 3. PushContext \u2014 Configuration Snapshot and Translation\n\n**Location:** `pilot/pkg/model/push_context.go`\n\nThe `PushContext` is an immutable snapshot of the entire mesh configuration at a specific point in time. It is created when configuration changes and used during the push phase to generate xDS resources for all proxies.\n\n### 3.1 PushContext Data Structure\n\n```go\ntype PushContext struct {\n    // Service index\n    ServiceIndex serviceIndex\n\n    // Configuration indexes\n    virtualServiceIndex virtualServiceIndex\n    destinationRuleIndex destinationRuleIndex\n    sidecarIndex sidecarIndex\n    gatewayIndex gatewayIndex\n\n    // Policy indexes\n    AuthnPolicies *AuthenticationPolicies\n    AuthzPolicies *AuthorizationPolicies\n    Telemetry *Telemetries\n    ProxyConfigs *ProxyConfigs\n\n    // Mesh configuration\n    Mesh *meshconfig.MeshConfig\n    Networks *meshconfig.MeshNetworks\n\n    // Push metadata\n    PushVersion string  // Unique identifier for this push context\n\n    // Status tracking\n    ProxyStatus map[string]map[string]ProxyPushStatus\n}\n```\n\n**Purpose:** Rather than each resource generator re-querying the config store, they access pre-indexed data from `PushContext`. This:\n- Improves performance (avoid repeated lookups)\n- Ensures consistency (all generators see the same config snapshot)\n- Enables efficient scoping (e.g., which VirtualServices apply to which proxies)\n\n### 3.2 Context Initialization (`InitContext()`)\n\nWhen a configuration change occurs, a new `PushContext` is created and initialized:\n\n```go\nfunc (ps *PushContext) InitContext(env *Environment, oldPushContext *PushContext, pushReq *PushRequest) {\n    ps.Mesh = env.Mesh()\n    ps.Networks = env.MeshNetworks()\n\n    ps.initDefaultExportMaps()\n\n    // Full rebuild or incremental update?\n    if pushReq == nil || oldPushContext == nil || !oldPushContext.InitDone.Load() || pushReq.Forced {\n        ps.createNewContext(env)  // Full rebuild\n    } else {\n        ps.updateContext(env, oldPushContext, pushReq)  // Incremental\n    }\n\n    ps.InitDone.Store(true)\n}\n```\n\n**Full Rebuild (`createNewContext`):**\nRebuilds all indexes from scratch. Called when:\n- Initial push context creation\n- Full push triggered (e.g., due to mesh configuration change)\n- Forced push\n\n```go\nfunc (ps *PushContext) createNewContext(env *Environment) {\n    ps.initServiceRegistry(env, nil)      // Load services from K8s + ServiceEntry\n    ps.initVirtualServices(env)           // Index VirtualServices\n    ps.initDestinationRules(env)          // Index DestinationRules\n    ps.initAuthnPolicies(env)             // Authentication policies\n    ps.initAuthorizationPolicies(env)     // Authorization policies\n    ps.initTelemetry(env)                 // Telemetry configuration\n    ps.initWasmPlugins(env)               // WASM plugin configuration\n    ps.initEnvoyFilters(env, nil, nil)   // Envoy filters\n    ps.initGateways(env)                  // Gateway configurations\n    ps.initSidecarScopes(env)             // Sidecar scopes (MUST be last)\n}\n```\n\n**Incremental Update (`updateContext`):**\nOnly rebuilds indexes for changed resource types. Example:\n\n```go\nfunc (ps *PushContext) updateContext(env *Environment, oldPushContext *PushContext, pushReq *PushRequest) {\n    // Check which types changed\n    for conf := range pushReq.ConfigsUpdated {\n        switch conf.Kind {\n        case kind.ServiceEntry, kind.DNSName:\n            servicesChanged = true\n        case kind.VirtualService:\n            virtualServicesChanged = true\n        case kind.DestinationRule:\n            destinationRulesChanged = true\n        // ... etc\n        }\n    }\n\n    // Only rebuild affected indexes\n    if servicesChanged {\n        ps.initServiceRegistry(env, pushReq.ConfigsUpdated)\n    } else {\n        ps.ServiceIndex = oldPushContext.ServiceIndex  // Reuse\n    }\n\n    if virtualServicesChanged {\n        ps.initVirtualServices(env)\n    } else {\n        ps.virtualServiceIndex = oldPushContext.virtualServiceIndex  // Reuse\n    }\n    // ... similar for other types\n}\n```\n\n### 3.3 Configuration Translation: From Istio Config to xDS\n\nThe `PushContext` contains pre-indexed Istio configuration. **Generators** use this during xDS generation to translate Istio config \u2192 Envoy config:\n\n**Example: RouteConfiguration (RDS) Generation**\n- Input: `PushContext.virtualServiceIndex` (VirtualServices) + `PushContext.destinationRuleIndex` (DestinationRules)\n- Process: For each proxy, select applicable VirtualServices (matching hostname, gateway) and merge DestinationRules\n- Output: Envoy `RouteConfiguration` with routes, weighted clusters, traffic policies\n\n**Example: Cluster (CDS) Generation**\n- Input: `PushContext.ServiceIndex` (Services from K8s + ServiceEntry)\n- Process: For each service, create Envoy Cluster with load balancing settings from DestinationRules\n- Output: Envoy `Cluster` with endpoint discovery (EDS), outlier detection, connection pooling\n\n**Example: Endpoint (EDS) Generation**\n- Input: `PushContext.ServiceIndex` + current endpoints from service registry\n- Process: Fetch endpoints for requested cluster, apply subset selection from DestinationRules\n- Output: Envoy `ClusterLoadAssignment` with endpoint list and health information\n\n### 3.4 Service Registry Index (`ServiceIndex`)\n\n```go\ntype serviceIndex struct {\n    // Services scoped to single namespace (exportTo \".\")\n    privateByNamespace map[string][]*Service\n\n    // Services visible mesh-wide (exportTo \"*\")\n    public []*Service\n\n    // Services exported to specific namespaces\n    exportedToNamespace map[string][]*Service\n\n    // Lookup table: hostname + namespace \u2192 service\n    HostnameAndNamespace map[host.Name]map[string]*Service\n\n    // Cache of endpoints per port\n    instancesByPort services matching a hostname\n- EDS generator calls `GetAggregatedInstances()` to fetch endpoints for a service\n\n### 3.5 Metrics and Error Tracking\n\n```go\ntype PushContext struct {\n Data Flow: Configuration Change to xDS Push\n\n### 5.1 Full Push Flow: Config Change \u2192 Envoy\n\n```\n1. CONFIG CHANGE\n   \u251c\u2500 User creates/updates VirtualService\n   \u251c\u2500 Or K8s applies ServiceEntry\n   \u2514\u2500 Pilot's config store watcher detects change\n\n2. WATCH EVENT\n   \u2514\u2500 `ConfigUpdate(req *model.PushRequest)` called\n      \u251c\u2500 ConfigsUpdated = {VirtualService/my-vs}\n      \u2514\u2500 Full = true (rebuild context)\n\n3. DEBOUNCE (in handleUpdates goroutine)\n   \u251c\u2500 Event added to pushChannel\n   \u251c\u2500 Waits DebounceAfter (e.g., 100ms)\n   \u251c\u2500 If no new events, proceeds to next step\n   \u251c\u2500 If new events, resets timer (up to debounceMax of ~10s)\n   \u2514\u2500 Multiple events merged into single PushRequest\n\n4. PUSH CONTEXT INIT\n   \u2514\u2500 `initPushContext(req)` called\n      \u251c\u2500 Creates new PushContext\n      \u251c\u2500 Calls `InitContext(env, oldContext, req)`\n         \u251c\u2500 Determines if full or incremental rebuild\n         \u251c\u2500 If incremental: checks ConfigsUpdated kinds\n         \u2502  \u251c\u2500 VirtualService changed? rebuild virtualServiceIndex\n         \u2502  \u251c\u2500 ServiceEntry changed? rebuild ServiceIndex\n         \u2502  \u2514\u2500 Reuse other indexes from oldContext\n         \u251c\u2500 If full: rebuilds all indexes\n         \u2514\u2500 Sets InitDone=true\n      \u251c\u2500 Updates global PushContext (stored in Environment)\n      \u2514\u2500 Clears cache entries related to changed configs\n\n5. FLUSH QUEUE (sendPushes goroutine)\n   \u2514\u2500 For each connected proxy (from Clients() list):\n      \u251c\u2500 Enqueue(proxy, pushRequest) into PushQueue\n      \u2514\u2500 Semaphore respects concurrent push limit\n\n6. GENERATE XDS (doSendPushes goroutine for each proxy)\n   \u251c\u2500 Read from PushQueue\n   \u251c\u2500 Call `pushConnection(con, event)`\n   \u2502  \u251c\u2500 Update proxy state: computeProxyState()\n   \u2502  \u2502  \u251c\u2500 Set service targets (which services affect this proxy)\n   \u2502  \u2502  \u251c\u2500 Compute sidecar scope\n   \u2502  \u2502  \u2514\u2500 Set workload labels/locality\n   \u2502  \u251c\u2500 Check ProxyNeedsPush() \u2014 skip if no relevant changes\n   \u2502  \u2514\u2500 For each watched resource type (CDS, LDS, RDS, EDS, SDS):\n   \u2502     \u2514\u2500 Call `pushXds(con, watchedResource, pushRequest)`\n   \u2502        \u251c\u2500 Find generator for resource type\n   \u2502        \u251c\u2500 Generator.Generate(proxy, watchedResource, pushRequest)\n   \u2502        \u2502  \u2514\u2500 Uses PushContext to build xDS resource\n   \u2502        \u2502     \u251c\u2500 CDS: Queries ServiceIndex, applies DestinationRules\n   \u2502        \u2502     \u251c\u2500 LDS: Queries SidecarScope, Gateways\n   \u2502        \u2502     \u251c\u2500 RDS: Queries VirtualServices + DestinationRules\n   \u2502        \u2502     \u2514\u2500 EDS: Queries ServiceIndex for endpoints\n   \u2502        \u251c\u2500 Construct DiscoveryResponse\n   \u2502        \u251c\u2500 Send via gRPC stream: xds.Send(con, resp)\n   \u2502        \u2514\u2500 Log: \"CDS: PUSH for proxy-id resources:5 size:2.3KB\"\n   \u2514\u2500 Mark push done, release semaphore\n\n7. ENVOY RECEIVES & ACKs\n   \u251c\u2500 Envoy receives DiscoveryResponse(TypeUrl=CDS, VersionInfo=v42)\n   \u251c\u2500 Validates clusters\n   \u251c\u2500 Sends back DiscoveryRequest(TypeUrl=CDS, VersionInfo=v42, ResponseNonce=nonce42)\n   \u2514\u2500 Pilot logs: \"CDS: ACK from proxy-id\"\n```\n\n### 5.2 Incremental Push Flow: Endpoint Change\n\n```\n1. ENDPOINT CHANGE\n   \u2514\u2500 Pod scales from 2 \u2192 3 replicas\n      \u251c\u2500 K8s updates Endpoint/EndpointSlice\n      \u2514\u2500 Kube controller detects change\n\n2. EDS UPDATE\n   \u2514\u2500 `EDSUpdate(shard=K8s/cluster-1, hostname=reviews, endpoints=[...])`\n      \u251c\u2500 Updates endpoint cache\n      \u251c\u2500 Calls `ConfigUpdate(&model.PushRequest{Full: false})`\n      \u2514\u2500 Reason = EDS\n\n3. DEBOUNCE (if EDS debounce enabled)\n   \u2514\u2500 Similar to config change, but Full=false\n\n4. NO CONTEXT REBUILD\n   \u2514\u2500 Full=false, so skip `initPushContext()`\n      \u251c\u2500 Reuse global PushContext\n      \u2514\u2500 Only clear EDS cache\n\n5. INCREMENTAL PUSH\n   \u2514\u2500 `AdsPushAll(req)` with Full=false\n      \u2514\u2500 For each proxy:\n         \u251c\u2500 Enqueue(proxy, req)\n         \u2514\u2500 In `pushConnection()`:\n            \u251c\u2500 Skip `computeProxyState()` (Full=false)\n            \u251c\u2500 Only push EDS (skipped by generators: cdsNeedsPush, rdsNeedsPush, etc.)\n            \u2514\u2500 `pushXds(con, EDS)`\n               \u2514\u2500 Send only EDS response with new endpoints\n\n6. ENVOY UPDATES LOAD BALANCER\n   \u2514\u2500 Receives EDS update\n      \u251c\u2500 Adds new endpoint\n      \u2514\u2500 Immediately available for traffic\n```\n\n---\n\n## 6. # Architecture Document: Istio Pilot's Discovery Service

## Overview

Istio Pilot's Discovery Service is the core xDS (Envoy Discovery Service) implementation that translates high-level Istio mesh configuration into low-level Envoy proxy configuration. It continuously watches for changes in the mesh (service registries, Istio configuration resources, endpoint updates) and pushes the translated configuration to all connected proxies via gRPC xDS streams.

The Discovery Service follows a server-push model: rather than waiting for clients to request updates, Pilot proactively sends configuration changes to all connected proxies. This ensures low-latency propagation of configuration changes across the mesh.
