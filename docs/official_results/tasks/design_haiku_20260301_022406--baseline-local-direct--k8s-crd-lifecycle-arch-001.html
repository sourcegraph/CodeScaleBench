3. **apimachinery** — Runtime scheme, GVK/GVR types, ObjectMeta, Unstructured representation
4. **api** — Internal hub types and group registration

## Summary of Analysis

I traced the complete CRD lifecycle through the Kubernetes ecosystem:

**Foundation (apimachinery):**
- **Scheme**: Central type registry mapping GVK → Go struct types
- **GVK/GVR**: Type and resource identifiers enabling routing and versioning
- **TypeMeta/ObjectMeta**: Standard metadata on all Kubernetes objects
- **Unstructured**: The critical generic wrapper (`map[string]interface{}`) that enables dynamic CRD handling

**Type Definition & Server-Side (apiextensions-apiserver):**
- Internal hub types and external v1/v1beta1 API versions for CustomResourceDefinition
- Validation layer enforcing OpenAPI schemas, defaults, and pruning
- Dynamic HTTP handler (customresource_handler.go) routing requests by GVR
- etcd storage using Unstructured objects, persisting custom resources without pre-registration

**Client-Side Access (client-go):**
- **DynamicClient**: REST-based access for any Unstructured resource
- **SharedInformerFactory**: Watches and caches any resource via GenericInformer
- **DynamicSharedInformerFactory**: Specialized informers for CRDs
- **Typed clients**: Generated for built-in resources via code-generator
- **Listers**: Query cached data without network overhead

**Key Insight:** The checkpoint between server and client is the **watch stream**—etcd persists Unstructured objects, broadcasts changes via HTTP watch, and the client informer receives events to update its local cache, enabling eventual consistency across the cluster.
