## Investigation Complete

I've analyzed the Grafana codebase and identified the root cause of the regression. The investigation report has been written to `/logs/agent/investigation.md` with the following findings:

### Key Discovery

The **V38 schema migration** in `/workspace/apps/dashboard/pkg/migration/schemaversion/v38.go` has a critical bug: it skips processing field overrides for table panels that don't have an explicit `fieldConfig.defaults.custom` object.

### Root Cause Details

The problematic code flow (lines 115-135 in v38.go):
- Checks if `fieldConfig.defaults.custom` exists
- **If it doesn't exist**, the code uses `continue` to skip to the next panel entirely
- This means `migrateOverrides()` is **never called** for panels without explicit custom config
- Field overrides with `custom.displayMode` properties remain unmigrated, causing loss of table formatting

### Why This Happens

1. **Missing test coverage** - All v38 tests include explicit `defaults.custom`, so this scenario was never tested
2. **Frontend mismatch** - The frontend implementation always processes overrides regardless of `defaults.custom` existence
3. **Incorrect guard pattern** - Uses early `continue` statements instead of the defensive unpacking pattern used in v35.go

### Affected Scenarios

- **Only affects** dashboards saved in v10.3 or earlier that have:
  - Table panels with field overrides (`custom.displayMode` properties)
  - NO explicit `fieldConfig.defaults.custom` in the saved JSON
- **Unaffected** are dashboards that already had explicit `defaults.custom` defined
