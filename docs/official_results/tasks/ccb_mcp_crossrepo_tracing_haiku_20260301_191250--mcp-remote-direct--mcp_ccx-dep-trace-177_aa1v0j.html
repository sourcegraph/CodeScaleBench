## Summary of Findings

**Filter Factory Registry**: 
- The `NamedHttpFilterConfigFactory` interface (envoy/server/filter_config.h) is the core abstraction
- Convenience template base classes (`FactoryBase`, `CommonFactoryBase`) in source/extensions/filters/http/common/factory_base.h reduce boilerplate
- Factories register via macro and are looked up by name

**Per-Connection Filter Chain Initialization**:
- `HttpConnectionManagerConfig` processes configured filters during setup (config.cc:709-714)
- `FilterChainHelper::processFilters()` iterates through configured filters
- Each factory's `createFilterFactoryFromProto()` is called, returning a `FilterFactoryCb` callback
- These callbacks are stored for later use

**Per-Request Filter Chain Creation**:
- When a new HTTP stream arrives, `ConnectionManagerImpl::ActiveStream::decodeHeaders()` is called
- This calls `FilterManager::createFilterChain()` which delegates to `HttpConnectionManagerConfig::createFilterChain()`
- `FilterChainUtility::createFilterChainForFactories()` iterates through stored factory callbacks
- Each callback is invoked with a `FilterChainFactoryCallbacksImpl` manager

**Per-Request Filter Instantiation**:
- The factory callbacks create actual filter instances and register them via the callbacks manager
- `ActiveStreamDecoderFilter` and `ActiveStreamEncoderFilter` wrappers wrap the filters
- These wrappers handle filter lifecycle, callbacks, and iteration through the filter chain
