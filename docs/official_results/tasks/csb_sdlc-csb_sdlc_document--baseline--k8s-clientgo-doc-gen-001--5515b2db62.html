     1→# `client-go` Architecture
     2→
     3→This document explains the internal architecture of `client-go` for contributors. It describes the
     4→major components, how they interact, and the key design decisions that shape the library.
     5→
     6→## Client Configuration
     7→
     8→There is an architectural separation between loading client configuration and using it. The
     9→`rest.Config` object is the in-memory representation of this configuration. The
    10→`tools/clientcmd` package is the standard factory for producing it. `clientcmd` handles the
    11→complex logic of parsing `kubeconfig` files, merging contexts, and handling external
    12→authentication providers (e.g., OIDC).
    13→
    14→## REST Client
    15→
    16→The `rest.Client` is the foundational HTTP client that underpins all other clients. It separates
    17→the low-level concerns of HTTP transport, serialization, and error handling from higher-level,
    18→Kubernetes-specific object logic.
    19→
    20→The `rest.Config` object is used to build the underlying HTTP transport, which is typically a
    21→chain of `http.RoundTripper` objects. Each element in the chain is responsible for a specific
    22→task, such as adding an `Authorization` header. This is the mechanism by which all authentication
    23→is injected into requests.
    24→
    25→The client uses a builder pattern for requests (e.g., `.Verb()`, `.Resource()`), deferring
    26→response processing until a method like `.Into(&pod)` is called. This separation is key to
    27→supporting different client models from a common base.

(`tools/cache/store.go`, `tools/cache/index.go`)\n\n**Responsibility:** In-memory, thread-safe store of objects with secondary indices for efficient queries.\n\n**Key Files:**\n- `tools/cache/store.go`: `Store` interface and `cache` implementation\n- `tools/cache/index.go`: `Indexer` interface with `Index()`, `IndexKeys()`, `ByIndex()` methods\n\n**Features:**\n- **Thread-safe**: RWMutex guards reads and writes\n- **Secondary indices**: Index by namespace, owner references, labels, or custom functions\n- **Efficient queries**: `ByIndex(\"namespace\", \"default\")` returns all objects in that namespace without full scan\n\n**Example:**\n```go\nindexer.Add(pod)  // pod added to cache\nindexer.ByIndex(\"namespace\", \"default\")  // O(1) lookup via index\n```\n\n#### 2.5.4 Event Handlers and Informer Pattern\n\n**Responsibility:** Notify consumers of object changes and manage event ordering.\n\n**Key Files:**\n- `tools/cache/shared_informer.go`: `SharedInformer` aggregates event listeners\n- `tools/cache/controller.go`: `Config` and `Run()` orchestrating the full flow\n\n**Event Handler Pattern:**\n```go\nhandler := cache.ResourceEventHandlerFuncs{\n  AddFunc: func(obj interface{}) { workQueue.Add(obj) },\n  UpdateFunc: func(oldObj, newObj interface{}) { workQueue.Add(newObj) },\n  DeleteFunc: func(obj interface{}) { workQueue.Add(obj) },\n}\ninformer.AddEventHandler(handler)\n```\n\n**Design Principle:** Event handlers **must not perform expensive operations**. They add work items to a queue; the controller processes items from the queue with retries and backoff.\n\n### 2.6 Work Queue (`util/workqueue/`)\n\n**Responsibility:** Decouple event detection from business logic processing.\n\n**Key Files:**\n- `util/workqueue/queue.go`: `Interface` (Add, Get, Done, Len)\n- `util/workqueue/delaying_queue.go`: `DelayingInterface` (AddAfter)\n- `util/workqueue/rate_limiting_queue.go`: `RateLimitingInterface` (AddRateLimited)\n- `util/workqueue/default_rate_limiters.go`: Built-in rate limiters (exponential backoff, max retries)\n\n**Responsibilities Separated:**\n1. **Detection**: Informer handler adds a work item (e.g., \"default/my-pod\")\n2. **Queueing**: Item waits in the work queue\n3. **Processing**: Controller worker pops item, processes it, calls `Done()` on success or `AddRateLimited()` on failure\n4. **Rate Limiting**: Failed items are re-queued with exponential backoff\n\n**Design:**\n- Work queue stores **keys** (e.g., \"namespace/name\"), not full objects\n- Controller fetches the object from cache (via Lister) when processing the key\n- If object is not in cache (deleted), controller handles the \"not found\" gracefully\n\n**Example Retry Flow:**\n```\nItem \"default/pod1\" fails to reconcile\n\u2192 workqueue.AddRateLimited(\"default/pod1\")  // exponential backoff\n\u2192 2 seconds later, item re-queued\n\u2192 4 seconds later, if failed again\n\u2192 8 seconds later, if failed again\n\u2192 Max retries reached (or max duration), item dropped\n```\n\n### 2.7 Listers (`listers/` and `informers/` packages)\n\n**Responsibility:** Provide read-only, type-safe access to the cache for controllers.\n\n**Key Files:**\n- `listers/core/v1/pod_lister.go` (auto-generated): API for object mutation that allows multiple controllers to co-manage the same object.\n\n**Key Files:**\n- `applyconfigurations/apps/v1/deployment_applyconfiguration.go` (auto-generated): Type-safe fields \u2192 UPDATE. Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### 4.1.1 Watch Stream Goes Stale (\"too old resourceVersion\")\n\n**Symptom:**\n```\nHTTP 410 Gone: the object has been deleted because the watch stream fell too far behind\n```\n\n**Root Cause:**\n- Reflector was paused (e.g., due to slow event handler goroutine)\n- Watch stream disconnected; reconnect attempted with stale resourceVersion\n- API server's watch cache (default ~5min) has already discarded that version\n\n**Mitigation:**\n1. **Bookmarks** (KEP-1904): API server sends BOOKMARK events with fresh resourceVersion\n   - Reflector consumes bookmarks, advancing restart point without full re-list\n   - Reduces cost of recovery when watch is merely slow, not stale\n2. **Automatic re-list**: Reflector detects 410 error, performs full LIST, resumes WATCH\n3. **User-controlled resyncPeriod**: Periodic re-list (e.g., every 10 minutes) ensures catch-up even if watch somehow falls behind\n\n**Evidence:** `tools/cache/reflector.go:220-240` (isExpiredError, LIST recovery)\n\n#### 4.1.2 API Server Becomes Unreachable\n\n**Symptom:**\n- All requests fail with connection timeouts\n- Informer watches stuck (cannot reconnect)\n\n**Mitigation:**\n1. **Client-side rate limiting**: RESTClient returns errors immediately when API is unreachable; exponential backoff in controller's retry loop prevents thundering herd\n2. **Discovery caching**: Tools like kubectl can still work if they cached discovery data (e.g., in `~/.kube/cache/discovery`)\n3. **Controller's work queue**: Decouples watch (which is stuck) from reconciliation logic; items remain queued during outage, retried after recovery\n\n**Limitation:** In-process cache becomes stale; no updates for duration of outage. Controllers must tolerate eventual consistency.\n\n#### 4.1.3 Concurrent Modification Conflicts\n\n**Scenario:**\nTwo controllers both call `Update()` on the same object without coordination.\n\n**Symptom:**\n```\nHTTP 409 Conflict: the object has been modified; refresh and retry\n```\n\n**Mitigation:**\n1. **Traditional approach**: Use Server-Side Apply instead of Update (allows shared ownership)\n2. **Leader election**: Only one controller manages the object at a time\n3. **Optimistic locking**: Check `metadata.resourceVersion` before updating; if mismatch, retry\n\n**Evidence:** `tools/leaderelection/leaderelection.go` (Lease-based lock), `applyconfigurations/` (server-side apply pattern)\n\n#### 4.1.4 Memory Pressure from Large Cache\n\n**Scenario:**\nController watching millions of objects; Indexer cache consumes significant memory.\n\n**Symptom:**\n- OOMKilled pod (if deployed in Kubernetes)\n- Slow GC pauses due to large heap\n\n**Mitigation:**\n1. **Selective watching**: Only add event handler for objects in specific namespace or label selector\n   - Reflector respects `FieldSelector` and `LabelSelector` in ListOptions\n   - API server returns fewer objects; cache is smaller\n2. **Resync period tuning**: Reduce resyncPeriod if periodic reprocessing is expensive\n3. **Custom indexing**: Only index on critical fields (namespace, owner references); avoid indexing large label maps\n\n**Design Tradeoff:** In-process caching trades memory for latency (cache lookups are O(1) vs. API calls which are O(100ms+))\n\n#### 4.1.5 Event Handler Panic\n\n**Scenario:**\nUser's event handler panics.\n\n**Symptom:**\n- Goroutine crashes\n- Event stream stops for that informer\n\n**Mitigation:**\n- Informers recover panics internally; panics are logged but don't crash the watch\n- Work items are not added if handler panics, so reconciliation doesn't happen\n- Solution: **Wrap your handler in defer + recover** if you need to ensure other handlers run\n\n**Evidence:** `tools/cache/shared_informer.go:490-510` (processorListener handles panics)\n\n### 4.2 Limits and Constraints\n\n#### 4.2.1 Rate Limiting\n\n**Client-Side (rest.Config):**\n- Default QPS: 5, Burst: 10\n- Enforced per RESTClient instance via token bucket\n- Does NOT enforce cluster-wide limits\n- Can be disabled by setting QPS < 0 and RateLimiter = nil\n\n**Interaction with Server-Side Limits:**\n- API Priority and Fairness (APF): Server has separate rate limiting by priority level, flow schema\n- Client QPS + Burst settings coordinate with server's APF; they are two independent rate limits\n\n**Lesson:** Setting client QPS=1000 does not mean requests arrive at 1000/sec; APF may throttle them further\n\n#### 4.2.2 Watch Stream Timeouts\n\n**Default minWatchTimeout:** 5 minutes (randomized to [5m, 10m) to spread load)\n\n**Implication:**\n- Watch streams are deliberately short-lived to allow graceful API server scaling (e.g., restarting kube-apiserver)\n- Controllers must be prepared for frequent watch reconnects\n- Exponential backoff prevents reconnect thundering\n\n**Tuning:** `Config.MinWatchTimeout` can be increased for slow clients, but values < 5min are ignored (enforced server-side)\n\n#### 4.2.3 List Pagination\n\n**Default page size:** 500 (controlled by API server)\n\n**Impact:**\n- Large LIST operations are chunked; client must handle continue tokens\n- `Reflector.ListAndWatch()` internally handles pagination; users typically don't see it\n- Exception: `DynamicClient.List()` with large result sets may require manual pagination\n\n**Code Location:** `tools/cache/listwatch.go:93` (NewListWatchFromClient handles pagination)\n\n#### 4.2.4 Cache Resync Skew\n\n**Scenario:**\nMultiple SharedInformerFactories with different resyncPeriods create thundering herd during resync windows.\n\n**Mitigation:**\n- Each Reflector adds random jitter to resync interval (\u00b110%)\n- Stagger initial resync times to avoid all informers re-listing simultaneously\n\n**Evidence:** `tools/cache/reflector.go:162-170` (resyncPeriod + jitter)\n\n### 4.3 Design Tradeoffs\n\n#### 4.3.1 Eventual Consistency vs. Strong Consistency\n\n**client-go Design:** Eventual consistency\n\n- Objects in the cache may lag API server state by seconds (watch propagation time) or minutes (if watch stream is stale)\n- Reads from cache are fast (O(1)) but potentially stale\n- Writes go through API server (always authoritative)\n\n**Implication:**\n- Controllers **cannot assume** that a recently-updated object is visible in the cache\n- Must add defensive checks: \"object I just created is not in cache? Retry.\"\n- Acceptable for most Kubernetes controllers (reconciliation loops are idempotent)\n\n#### 4.3.2 Single-Writer (Leader Election) vs. Multi-Writer (Server-Side Apply)\n\n| Pattern | Use Case | Tradeoff |\n|---------|----------|----------|\n| Leader Election (`tools/leaderelection/`) | Only one controller active at a time | Simple, but single point of failure during leadership transfer |\n| Server-Side Apply (`applyconfigurations/`) | Multiple controllers manage different fields | Requires schema awareness of field ownership, more complex coordination |\n\n**Evidence:**\n- Leader election: `tools/leaderelection/leaderelection.go`\n- Server-Side Apply: `applyconfigurations/apps/v1/deployment_applyconfiguration.go` (generated)\n\n#### 4.3.3 Typed Client (compile-time) vs. Dynamic Client (runtime)\n\n| Model | Use Case | Tradeoff |\n|-------|----------|----------|\n| **Typed** (`kubernetes.Clientset`) | Built-in Kubernetes APIs | Requires code generation for new types; type-safe |\n| **Dynamic** (`dynamic.DynamicClient`) | CRDs, extensible APIs | No code generation needed; discover at runtime; no type safety |\n\n**Performance Implication:**\n- Typed: No discovery overhead (types known at compile time)\n- Dynamic: Must call discovery on first access to determine namespace scope; future accesses cached\n\n#### 4.3.4 Caching Model: DeltaFIFO vs. Simple Store\n\n**DeltaFIFO:**\n- Accumulates per-object history (Deltas)\n- Allows consumers to see the full sequence of changes (Added \u2192 Modified \u2192 Modified \u2192 Deleted)\n- Higher memory overhead (stores all deltas until popped)\n\n**Simple Store (List-Watch without DeltaFIFO):**\n- Stores only current state\n- Lower memory, but loses intermediate events\n\n**Design Decision:** client-go uses DeltaFIFO by default because controllers often need to reconcile state correctly after a burst of rapid changes. Storing intermediate deltas allows controllers to avoid missing state.\n\n---\n\n## 5. interface{}) { workQueue.Add(obj) },\n}\ninformer.AddEventHandler(handler)\n```\n\n**Design Principle:** Event handlers **must not perform expensive operations**. They add work items to a queue; the controller processes items from the queue with retries and backoff.\n\n### 2.6 Work Queue (`util/workqueue/`)\n\n**Responsibility:** Decouple event detection from business logic processing.\n\n**Key Files:**\n- `util/workqueue/queue.go`: `Interface` (Add, Get, Done, Len)\n- `util/workqueue/delaying_queue.go`: `DelayingInterface` (AddAfter)\n- `util/workqueue/rate_limiting_queue.go`: `RateLimitingInterface` (AddRateLimited)\n- `util/workqueue/default_rate_limiters.go`: Built-in rate limiters (exponential backoff, max retries)\n\n**Responsibilities Separated:**\n1. **Detection**: Informer handler adds a work item (e.g., \"default/my-pod\")\n2. **Queueing**: Item waits in the work queue\n3. **Processing**: Controller worker pops item, processes it, calls `Done()` on success or `AddRateLimited()` on failure\n4. **Rate Limiting**: Failed items are re-queued with exponential backoff\n\n**Design:**\n- Work queue stores **keys** (e.g., \"namespace/name\"), not full objects\n- Controller fetches the object from cache (via Lister) when processing the key\n- If object is not in cache (deleted), controller handles the \"not found\" gracefully\n\n**Example Retry Flow:**\n```\nItem \"default/pod1\" fails to reconcile\n\u2192 workqueue.AddRateLimited(\"default/pod1\")  // exponential backoff\n\u2192 2 seconds later, item re-queued\n\u2192 4 seconds later, if failed again\n\u2192 8 seconds later, if failed again\n\u2192 Max retries reached (or max duration), item dropped\n```\n\n### 2.7 Listers (`listers/` and `informers/` packages)\n\n**Responsibility:** Provide read-only, type-safe access to the cache for controllers.\n\n**Key Files:**\n- `listers/core/v1/pod_lister.go` (auto-generated): API for object mutation that allows multiple controllers to co-manage the same object.\n\n**Key Files:**\n- `applyconfigurations/apps/v1/deployment_applyconfiguration.go` (auto-generated): Type-safe fields \u2192 UPDATE. Requires **exclusive ownership**.\n- **Server-Side Apply (kubectl apply -f)**: Send desired state + fieldManager \u2192 server merges with other managers. Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### 4.1.1 Watch Stream Goes Stale (\"too old resourceVersion\")\n\n**Symptom:**\n```\nHTTP 410 Gone: the object has been deleted because the watch stream fell too far behind\n```\n\n**Root Cause:**\n- Reflector was paused (e.g., due to slow event handler goroutine)\n- Watch stream disconnected; reconnect attempted with stale resourceVersion\n- API server's watch cache (default ~5min) has already discarded that version\n\n**Mitigation:**\n1. **Bookmarks** (KEP-1904): API server sends BOOKMARK events with fresh resourceVersion\n   - Reflector consumes bookmarks, advancing restart point without full re-list\n   - Reduces cost of recovery when watch is merely slow, not stale\n2. **Automatic re-list**: Reflector detects 410 error, performs full LIST, resumes WATCH\n3. **User-controlled resyncPeriod**: Periodic re-list (e.g., every 10 minutes) ensures catch-up even if watch somehow falls behind\n\n**Evidence:** `tools/cache/reflector.go:220-240` (isExpiredError, LIST recovery)\n\n#### 4.1.2 API Server Becomes Unreachable\n\n**Symptom:**\n- All requests fail with connection timeouts\n- Informer watches stuck (cannot reconnect)\n\n**Mitigation:**\n1. **Client-side rate limiting**: RESTClient returns errors immediately when API is unreachable; exponential backoff in controller's retry loop prevents thundering herd\n2. **Discovery caching**: Tools like kubectl can still work if they cached discovery data (e.g., in `~/.kube/cache/discovery`)\n3. **Controller's work queue**: Decouples watch (which is stuck) from reconciliation logic; items remain queued during outage, retried after recovery\n\n**Limitation:** In-process cache becomes stale; no updates for duration of outage. Controllers must tolerate eventual consistency.\n\n#### 4.1.3 Concurrent Modification Conflicts\n\n**Scenario:**\nTwo controllers both call `Update()` on the same object without coordination.\n\n**Symptom:**\n```\nHTTP 409 Conflict: the object has been modified; refresh and retry\n```\n\n**Mitigation:**\n1. **Traditional approach**: Use Server-Side Apply instead of Update (allows shared ownership)\n2. **Leader election**: Only one controller manages the object at a time\n3. **Optimistic locking**: Check `metadata.resourceVersion` before updating; if mismatch, retry\n\n**Evidence:** `tools/leaderelection/leaderelection.go` (Lease-based lock), `applyconfigurations/` (server-side apply pattern)\n\n#### 4.1.4 Memory Pressure from Large Cache\n\n**Scenario:**\nController watching millions of objects; Indexer cache consumes significant memory.\n\n**Symptom:**\n- OOMKilled pod (if deployed in Kubernetes)\n- Slow GC pauses due to large heap\n\n**Mitigation:**\n1. **Selective watching**: Only add event handler for objects in specific namespace or label selector\n   - Reflector respects `FieldSelector` and `LabelSelector` in ListOptions\n   - API server returns fewer objects; cache is smaller\n2. **Resync period tuning**: Reduce resyncPeriod if periodic reprocessing is expensive\n3. **Custom indexing**: Only index on critical fields (namespace, owner references); avoid indexing large label maps\n\n**Design Tradeoff:** In-process caching trades memory for latency (cache lookups are O(1) vs. API calls which are O(100ms+))\n\n#### 4.1.5 Event Handler Panic\n\n**Scenario:**\nUser's event handler panics.\n\n**Symptom:**\n- Goroutine crashes\n- Event stream stops for that informer\n\n**Mitigation:**\n- Informers recover panics internally; panics are logged but don't crash the watch\n- Work items are not added if handler panics, so reconciliation doesn't happen\n- Solution: **Wrap your handler in defer + recover** if you need to ensure other handlers run\n\n**Evidence:** `tools/cache/shared_informer.go:490-510` (processorListener handles panics)\n\n### 4.2 Limits and Constraints\n\n#### 4.2.1 Rate Limiting\n\n**Client-Side (rest.Config):**\n- Default QPS: 5, Burst: 10\n- Enforced per RESTClient instance via token bucket\n- Does NOT enforce cluster-wide limits\n- Can be disabled by setting QPS < 0 and RateLimiter = nil\n\n**Interaction with Server-Side Limits:**\n- API Priority and Fairness (APF): Server has separate rate limiting by priority level, flow schema\n- Client QPS + Burst settings coordinate with server's APF; they are two independent rate limits\n\n**Lesson:** Setting client QPS=1000 does not mean requests arrive at 1000/sec; APF may throttle them further\n\n#### 4.2.2 Watch Stream Timeouts\n\n**Default minWatchTimeout:** 5 minutes (randomized to [5m, 10m) to spread load)\n\n**Implication:**\n- Watch streams are deliberately short-lived to allow graceful API server scaling (e.g., restarting kube-apiserver)\n- Controllers must be prepared for frequent watch reconnects\n- Exponential backoff prevents reconnect thundering\n\n**Tuning:** `Config.MinWatchTimeout` can be increased for slow clients, but values < 5min are ignored (enforced server-side)\n\n#### 4.2.3 List Pagination\n\n**Default page size:** 500 (controlled by API server)\n\n**Impact:**\n- Large LIST operations are chunked; client must handle continue tokens\n- `Reflector.ListAndWatch()` internally handles pagination; users typically don't see it\n- Exception: `DynamicClient.List()` with large result sets may require manual pagination\n\n**Code Location:** `tools/cache/listwatch.go:93` (NewListWatchFromClient handles pagination)\n\n#### 4.2.4 Cache Resync Skew\n\n**Scenario:**\nMultiple SharedInformerFactories with different resyncPeriods create thundering herd during resync windows.\n\n**Mitigation:**\n- Each Reflector adds random jitter to resync interval (\u00b110%)\n- Stagger initial resync times to avoid all informers re-listing simultaneously\n\n**Evidence:** `tools/cache/reflector.go:162-170` (resyncPeriod + jitter)\n\n### 4.3 Design Tradeoffs\n\n#### 4.3.1 Eventual Consistency vs. Strong Consistency\n\n**client-go Design:** Eventual consistency\n\n- Objects in the cache may lag API server state by seconds (watch propagation time) or minutes (if watch stream is stale)\n- Reads from cache are fast (O(1)) but potentially stale\n- Writes go through API server (always authoritative)\n\n**Implication:**\n- Controllers **cannot assume** that a recently-updated object is visible in the cache\n- Must add defensive checks: \"object I just created is not in cache? Retry.\"\n- Acceptable for most Kubernetes controllers (reconciliation loops are idempotent)\n\n#### 4.3.2 Single-Writer (Leader Election) vs. Multi-Writer (Server-Side Apply)\n\n| Pattern | Use Case | Tradeoff |\n|---------|----------|----------|\n| Leader Election (`tools/leaderelection/`) | Only one controller active at a time | Simple, but single point of failure during leadership transfer |\n| Server-Side Apply (`applyconfigurations/`) | Multiple controllers manage different fields | Requires schema awareness of field ownership, more complex coordination |\n\n**Evidence:**\n- Leader election: `tools/leaderelection/leaderelection.go`\n- Server-Side Apply: `applyconfigurations/apps/v1/deployment_applyconfiguration.go` (generated)\n\n#### 4.3.3 Typed Client (compile-time) vs. Dynamic Client (runtime)\n\n| Model | Use Case | Tradeoff |\n|-------|----------|----------|\n| **Typed** (`kubernetes.Clientset`) | Built-in Kubernetes APIs | Requires code generation for new types; type-safe |\n| **Dynamic** (`dynamic.DynamicClient`) | CRDs, extensible APIs | No code generation needed; discover at runtime; no type safety |\n\n**Performance Implication:**\n- Typed: No discovery overhead (types known at compile time)\n- Dynamic: Must call discovery on first access to determine namespace scope; future accesses cached\n\n#### 4.3.4 Caching Model: DeltaFIFO vs. Simple Store\n\n**DeltaFIFO:**\n- Accumulates per-object history (Deltas)\n- Allows consumers to see the full sequence of changes (Added \u2192 Modified \u2192 Modified \u2192 Deleted)\n- Higher memory overhead (stores all deltas until popped)\n\n**Simple Store (List-Watch without DeltaFIFO):**\n- Stores only current state\n- Lower memory, but loses intermediate events\n\n**Design Decision:** client-go uses DeltaFIFO by default because controllers often need to reconcile state correctly after a burst of rapid changes. Storing intermediate deltas allows controllers to avoid missing state.\n\n---\n\n## 5. # Kubernetes client-go Systems Guide

## Overview

`client-go` is the foundational Go library for programmatic interaction with Kubernetes clusters. It provides multiple abstraction layers to support diverse use cases—from simple CLI tools to complex controllers managing distributed systems. The subsystem is architecturally designed to balance type safety, performance, extensibility, and operational resilience.

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose

client-go serves as the canonical interface for Go programs to:
- **Query and mutate** Kubernetes API objects (pods, deployments, custom resources, etc.)
- **Watch** cluster state changes in real-time
- **Discover** available APIs and their schemas
- **Cache** API state locally with automatic synchronization
- **Coordinate** multi-replica controllers via leader election
- **Handle authentication**, authorization, and transport layer security

### Subsystem Boundaries

**Upstream (dependencies):**
- `k8s.io/apimachinery`: Core runtime objects (ObjectMeta, TypeMeta, runtime.Object), watch interfaces, and versioning primitives
- `k8s.io/api`: Generated type definitions for all Kubernetes API resources (Pod, Deployment, etc.)
- `k8s.io/utils`, `k8s.io/klog`: Utility libraries (time management, logging, sets)
- Standard library: `net/http`, `encoding/json`, `crypto/tls`

**Downstream (consumers):**
- **Controllers**: Operators, custom controllers that reconcile cluster state
- **CLI tools**: kubectl (internally uses client-go)
- **Webhooks**: Admission controllers, mutating/validating webhooks
- **Monitoring/observability**: Custom metrics collectors, log aggregators
- **End-user applications**: Any program that needs Kubernetes integration
