Failure Modes\n\n### eBPF Program Compilation Failures\n\n**Causes:**\n- Unsupported kernel features (e.g., calling a helper function not available in the kernel)\n- eBPF verifier rejects the program (memory safety violations)\n- Clang compilation errors\n- Invalid C syntax in generated headers\n\n**Handling:**\n- **File**: `pkg/datapath/loader/compile.go:compileDatapath()`\n- **Mechanism**: Captures clang stderr/stdout, returns error with diagnostic info\n- **User Impact**: Endpoint fails to become ready; error logged with compilation output\n- **Recovery**: Fix kernel version, disable incompatible features via config, or update code\n\n**Diagnostics:**\n```bash\n# Check compilation output\nkubectl logs -n kube-system cilium-xxxx --tail=100 | grep \"BPF\"\n# Check verifier logs\n# Check kernel version compatibility\nuname -r\n```\n\n### eBPF Verifier Failures\n\n**Causes:**\n- **Uninitialized variables**: Using variables without setting all bytes\n- **Unbounded loops**: eBPF doesn't allow loops that verifier can't prove bounded\n- **Out-of-bounds memory access**: Accessing packet/map data beyond bounds\n- **Invalid pointer arithmetic**: Pointer math that could point outside valid ranges\n- **Stack overflow**: Local variables exceed 512-byte stack limit\n- **Tail call limits**: Exceeded max tail call depth (typically 32-33 levels)\n\n**Handling:**\n- **File**: `pkg/datapath/loader/netlink.go:loadDatapath()`\n- **Mechanism**: cilium/ebpf library captures verifier output\n- **Detection**: `ebpf.VerifierError` type\n- **User Impact**: Program fails to load; verbose verifier error logged\n- **Workaround**: Refactor code to avoid verifier issues\n\n**Example Error:**\n```\nverifier error:\n  ... invalid memory access for read from stack R1 offset=-8 size=8\n```\n\n### eBPF Map Failures\n\n**Causes:**\n- **Map full**: Map max_entries exceeded (pre-allocated max size)\n- **Map incompatibility**: Pinned map has different structure than current code\n- **Permission denied**: Lack of CAP_SYS_ADMIN or MAP_CREATE capability\n- **Type mismatch**: Trying to update map with wrong key/value size\n\n**Handling:**\n- **Files**:\n  - `pkg/datapath/loader/netlink.go:loadDatapath()` - detects incompatibilities\n  - `pkg/maps/` - individual map implementations\n- **Mechanism**: Maps are pinned to `/sys/fs/bpf/tc/` and reloaded across restarts\n- **Type checking**: Signature mismatch detected and maps recreated\n- **Policy maps**: Automatically enlarged in `pkg/maps/policymap/`\n\n**Recovery:**\n```bash\n# Check map status\nbpftool map list\n# Check policy map size\nbpftool map show id <MAP_ID>\n# Remove stale maps (careful!)\nrm /sys/fs/bpf/tc/globals/cilium_policy_*\n```\n\n### Kernel Compatibility Issues\n\n**Categories:**\n\n1. **Missing Kernel Features**: Helper function not available\n2. **Behavioral Changes**: Different kernel version changes semantics\n3. **API Changes**: Deprecated system calls or structures\n\n**Detection:**\n- **Files**: `pkg/datapath/linux/probes/`\n- **Mechanism**: Loads test programs to probe kernel capabilities\n- **Key Function**: `probes.ProbeManager.GetProbes()` compiles features.h header with detected capabilities\n\n**Example Probes:**\n```go\n// In pkg/datapath/linux/probes/probes.go\nHaveFibIfindex bool         // FIB lookup support\nHaveSessionCookie bool      // Session cookie available\nHaveIfindex bool            // Interface index in context\n```\n\n**Handling:**\n- Conditional compilation via `#ifdef` in eBPF C code\n- Config headers (`node_config.h`, `ep_config.h`) control features\n- Features disabled if not supported\n\n### Configuration Errors\n\n**Common Issues:**\n\n1. **Invalid IPv4/IPv6 CIDR blocks**: IPcache configuration\n2. **Invalid MAC addresses**: Used in ARP/neighbor lookup\n3. **Invalid endpoint IDs**: Out of range (0-65535 for standard endpoints)\n4. **Inconsistent policy configuration**: Policy verdict mismatches\n\n**Detection:**\n- **File**: `pkg/datapath/loader/` - config generation and validation\n- **Mechanisms**:\n  - Template endpoint (ID 65535) used for validation\n  - Config validation before header generation\n  - Endpoint hash changes trigger recompilation\n\n**Recovery:**\n```bash\n# Inspect configuration\ncilium config\n# Check template compilation\nls -la /var/run/cilium/state/templates/\n```\n\n### eBPF Program Crashes (Kernel Panics)\n\n**Causes:**\n- Invalid memory access (prevented by verifier in theory)\n- Null pointer dereference\n- Integer overflow in loops\n- Corrupted map structures\n\n**Prevention:**\n- Verifier prevents most memory access violations\n- Type-safe map access via cilium/ebpf library\n- Bounds checking in C code\n\n**Debugging:**\n```bash\n# Kernel logs\ndmesg | tail -50\n# Check for BPF-related panics\njournalctl -xe | grep -i bpf\n```\n\n### Header File Generation Failures\n\n**Causes:**\n- Invalid endpoint configuration passed from Go\n- IOError writing config files\n- Filesystem issues (permission denied, disk full)\n\n**Handling:**\n- **File**: `pkg/datapath/loader/base.go`\n- **Functions**:\n  - `writeNetdevHeader()` - writes `netdev_config.h`\n  - `writeNodeConfigHeader()` - writes `node_config.h`\n  - `writePreFilterHeader()` - writes `filter_config.h`\n\n**Recovery:**\n```bash\n# Check state directory permissions\nls -la /var/run/cilium/state/\n# Check disk space\ndf -h\n```\n\n---\n\n## 5. # Cilium eBPF Datapath Subsystem Handoff Guide

## Overview

This document provides a comprehensive guide to the Cilium eBPF datapath subsystem—the core of Cilium's networking and security enforcement capabilities. The datapath is responsible for loading eBPF programs into the kernel, enforcing network policies, performing NAT/load balancing, and implementing service mesh functionality.

---

## 1. Purpose

### Problem Statement

The eBPF datapath solves a fundamental networking problem: **how to enforce complex network policies and perform sophisticated packet manipulation at kernel level with minimal overhead**.

Traditional approaches (iptables, userspace proxies) have significant limitations:
- **iptables**: Hard-coded rule format, poor scalability, limited context awareness
- **Userspace networking**: High CPU overhead from kernel-userspace context switches, packet copies, and increased latency
- **Traditional kernel modules**: Complex maintenance, versioning issues, reduced stability

### Why eBPF?

Cilium uses eBPF because it provides:

1. **Kernel-Native Execution**: Programs run directly in the kernel with minimal overhead
2. **Dynamic Programmability**: Policies can be changed without recompiling the kernel or restarting services
3. **Efficient Packet Processing**: Direct memory access allows fine-grained packet inspection and modification
4. **Atomic Policy Updates**: Maps and tail calls allow coordinated policy changes across hooks
5. **Rich Context**: Access to connection state, routing tables, socket information, and custom maps
6. **Security Isolation**: Kernel verifier ensures programs cannot crash the kernel or access arbitrary memory
