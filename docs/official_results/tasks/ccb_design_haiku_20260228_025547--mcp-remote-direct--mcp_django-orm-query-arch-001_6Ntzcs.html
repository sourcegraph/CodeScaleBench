## Key Findings

**Files Examined (12 core files):**
- Manager/QuerySet: `manager.py`, `query.py`
- Query representation: `sql/query.py`, `sql/compiler.py`, `sql/where.py`
- Expression system: `expressions.py`, `lookups.py`
- Backend dispatch: `backends/base/operations.py`, vendor-specific compilers

**Dependency Chain (7 stages):**
1. **Manager.get_queryset()** — Entry point that creates QuerySet
2. **Query Building** — Filter/exclude/order_by accumulate conditions (lazy, no SQL yet)
3. **Trigger Evaluation** — When result accessed, calls get_compiler()
4. **Compiler Instantiation** — Vendor dispatch selects appropriate compiler class
5. **SQL Compilation** — `as_sql()` protocol recursively generates SQL
6. **Expression Compilation** — Vendor-specific `as_{vendor}()` methods with fallback
7. **Execution** — `execute_sql()` runs query and returns results

**Key Design Patterns:**
- **Lazy Evaluation** — QuerySet methods chain without executing
- **as_sql() Protocol** — Every expression implements `as_sql(compiler, connection) → (sql_string, params)`
- **Vendor Dispatch** — `as_postgresql()`, `as_mysql()`, etc. override for database-specific SQL
- **Recursive Expression Trees** — Lookups and Expressions nest and compile recursively
- **WhereNode Tree** — WHERE/HAVING clauses as tree structure with AND/OR/XOR connectors
- **Compiler Context** — Compiler object provides state and helper methods during compilation
