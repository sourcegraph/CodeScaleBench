# Task: Generate Operational Runbook for Prometheus TSDB Compaction

**Repository:** prometheus/prometheus

## Objective

Produce an operational runbook for Prometheus's TSDB (Time Series Database) compaction process. TSDB compaction merges overlapping time blocks, removes tombstoned series, and maintains query performance over time.

## Scope

Explore the TSDB compaction implementation in:
- `tsdb/compact.go` — core compaction logic
- `tsdb/head_read.go` and `tsdb/head.go` — head block behavior
- `tsdb/db.go` — database lifecycle and compaction triggering

## Required Sections

Your runbook at `/workspace/documentation.md` must include:

### 1. Overview
- What compaction does and why it is needed
- Compaction levels and block structure
- When compaction is triggered (time-based, size-based)

### 2. Monitoring Indicators
- Key Prometheus metrics to watch during compaction (e.g., `prometheus_tsdb_compactions_total`, `prometheus_tsdb_compaction_duration_seconds`)
- Alert thresholds and what they indicate
- How to distinguish healthy vs. stuck compaction

### 3. Failure Modes
- At least 4 failure modes with symptoms and causes:
  - Disk space exhaustion
  - Compaction stuck / not progressing
  - Block corruption
  - OOM during compaction
- For each: symptom, likely cause, and diagnostic steps

### 4. Recovery Procedures
- Step-by-step recovery for each failure mode
- When to restart Prometheus vs. manual block manipulation
- How to use `promtool tsdb` commands for inspection/repair

### 5. Code Reference
- Key functions in the codebase relevant to each failure mode

The process is critical for maintaining query performance and disk space efficiency as data ages.\n\n### Why Compaction is Needed\n\n1. **Memory Efficiency**: Head block data is persisted to disk blocks, freeing memory\n2. **Query Performance**: Fewer, larger blocks improve query speed by reducing I/O and index lookups\n3. **Disk Optimization**: Removes tombstoned series and consolidates data\n4. **Space Reclamation**: Unused chunks are dropped during merging\n5. **Overlap Resolution**: Vertical compaction merges blocks with overlapping time ranges\n\n### Compaction Levels and Block Structure\n\nPrometheus uses **leveled compaction** with exponential block ranges:\n\n```\nLevel 0: 2h blocks (HEAD block, freshly persisted)\nLevel 1: 4h blocks (1 compaction, 2 Level-0 blocks merged)\nLevel 2: 16h blocks (2 compactions, 2 Level-1 blocks merged)\nLevel 3: 64h blocks (3 compactions, 2 Level-2 blocks merged)\nLevel 4: 256h blocks (4 compactions, 2 Level-3 blocks merged)\n...and so on\n```\n\nEach block is identified by a ULID (Universally Unique Lexicographically Sortable Identifier) and contains:\n- `meta.json` - metadata (ULID, time range, sample count, compaction level, sources)\n- `index` - label index and postings lists\n- `chunks/` - time-series chunk data\n- `tombstones` - deletion markers for series\n\n### When Compaction is Triggered\n\nCompaction is triggered by three mechanisms:\n\n1. **Time-Based Triggering (Primary)**\n   - Triggered after HEAD block reaches `MinBlockDuration` (default: 2 hours)\n   - Checked on every `Appender.Commit()` call\n   - Set via `db.head.compactable()` condition\n\n2. **Background Compaction Loop**\n   - Runs in a dedicated goroutine (`db.compactLoop()`)\n   - Periodically attempts to compact eligible on-disk blocks\n   - Respects `autoCompact` flag (can be disabled)\n\n3. **Size-Based Triggering**\n   - Triggered if tombstone ratio exceeds 5% of series in large blocks\n   - Triggered if a block is entirely deleted (all series tombstoned)\n\n**Code References:**\n- Triggering: `tsdb/db.go:1120-1126` (dbAppender.Commit)\n- Compaction loop: `tsdb/db.go:1008-1027` (db.compactLoop)\n- Eligibility check: `tsdb/head.go` (head.compactable)\n\n---\n\n## 2. Failure Modes\n\n### Failure Mode 1: Disk Space Exhaustion\n\n**Symptoms:**\n- Prometheus logs: `no space left on device`\n- `prometheus_tsdb_compaction_populating_block` stuck at 1\n- Disk utilization at 100%\n- Queries may be slow or fail\n- TSDB cannot persist new blocks\n\n**Likely Causes:**\n- Retention duration or max bytes setting too high\n- Compaction failing silently, blocks not merged\n- WAL (Write-Ahead Log) accumulation without cleanup\n- Monitoring metrics storage exhausting disk\n\n**Diagnostic Steps:**\n\n1. Check disk usage and available space:\n   ```bash\n   df -h /prometheus-data\n   du -sh /prometheus-data/*\n   du -sh /prometheus-data/wal /prometheus-data/block*\n   ```\n\n2. List blocks and their sizes:\n   ```bash\n   promtool tsdb list -r /prometheus-data\n   # Look for:\n   # - Unusually large blocks\n   # - Blocks outside retention window\n   # - Duplicate time ranges (compaction failed?)\n   ```\n\n3. Check TSDB metrics:\n   ```bash\n   curl -s http://localhost:9090/metrics | grep -E \"prometheus_tsdb_(storage_blocks|retention|blocks_loaded)\"\n   ```\n\n4. Inspect Prometheus logs for compaction errors:\n   ```bash\n   tail -f /var/log/prometheus.log | grep -iE \"compaction|disk|space\"\n   ```\n\n**Code References:**\n- Block writing: `tsdb/compact.go:607-718` (write method)\n- Compaction triggering: `tsdb/db.go:1133-1213` (Compact method)\n\n### Failure Mode 2: Compaction Stuck / Not Progressing\n\n**Symptoms:**\n- `prometheus_tsdb_compactions_total` not incrementing for hours\n- `prometheus_tsdb_compaction_populating_block` metric stuck at 1\n- Prometheus CPU usage elevated but not responding to queries normally\n- Log messages: \"Head compaction took longer than the block time range\"\n- Compaction duration continuously > block range\n- `prometheus_tsdb_blocks_loaded` growing unboundedly\n\n**Likely Causes:**\n- Extremely high cardinality metrics (millions of series)\n- Memory pressure (GC pauses, OOM)\n- I/O contention on disk\n- Corrupted block preventing compaction\n- Context cancellation during shutdown attempting to gracefully stop compaction\n- System resource limits (file descriptors, memory cgroups)\n\n**Diagnostic Steps:**\n\n1. Check if Prometheus is in compaction:\n   ```bash\n   curl -s http://localhost:9090/metrics | grep prometheus_tsdb_compaction_populating_block\n   # Value of 1 means currently compacting\n   ```\n\n2. Monitor compaction duration:\n   ```bash\n   curl -s http://localhost:9090/metrics | grep prometheus_tsdb_compaction_duration_seconds_bucket | tail -5\n   # Check if last bucket shows compactions taking > block_range\n   ```\n\n3. Inspect the HEAD block for compactability:\n   ```bash\n   # Check logs for: \"msg=\"compact head\" duration=X\"\n   # If duration > MinBlockDuration (2h), compaction is slow\n   ```\n\n4. Check system resources:\n   ```bash\n   top -p $(pidof prometheus) # Check CPU, memory, threads\n   lsof -p $(pidof prometheus) | wc -l # Check open file descriptors\n   cat /proc/sys/fs/file-max # System file descriptor limit\n   ```\n\n5. Check for high cardinality:\n   ```bash\n   promtool query instant http://localhost:9090 'count(count by (__name__) ({__name__=~\".+\"}))' 2>/dev/null | tail -3\n   # If > 5 million, this is likely the cause\n   ```\n\n6. Check for corrupted blocks:\n   ```bash\n   # Try to open each block's index\n   for block in /prometheus-data/[0-9a-z]*; do\n     echo \"Checking $block...\"\n     promtool tsdb analyze \"$block\" 2>&1 | grep -i error\n   done\n   ```\n\n**Code References:**\n- Compaction check: `tsdb/db.go:1162-1164` (if !db.head.compactable())\n- Duration warning: `tsdb/db.go:1197-1202` (compaction duration > block range warning)\n- Block population: `tsdb/compact.go:729-875` (PopulateBlock method)\n\n### Failure Mode 3: Block Corruption\n\n**Symptoms:**\n- Queries fail with errors about corrupted indices or chunks\n- Compaction fails when trying to compact specific blocks\n- Log messages: `open index reader`, `chunk`, `series` errors\n- `promtool tsdb analyze` shows errors\n- `prometheus_tsdb_compactions_failed_total` incrementing\n\n**Likely Causes:**\n- Unclean shutdown during block write (power loss, SIGKILL)\n- Corrupted meta.json file\n- Corrupted index or chunks file\n- Incomplete block rename operation\n- filesystem corruption\n\n**Diagnostic Steps:**\n\n1. Identify which block is corrupted:\n   ```bash\n   # Check all blocks\n   for block in /prometheus-data/[0-9a-z]*; do\n     if ! promtool tsdb analyze \"$block\" > /dev/null 2>&1; then\n       echo \"Corrupted: $block\"\n     fi\n   done\n   ```\n\n2. Inspect meta.json:\n   ```bash\n   cat /prometheus-data/BLOCK_ULID/meta.json | jq .\n   # Check for: ULID, MinTime, MaxTime, Stats, Compaction.Level\n   ```\n\n3. Check if block is marked Deletable:\n   ```bash\n   cat /prometheus-data/BLOCK_ULID/meta.json | jq '.Compaction.Deletable'\n   # true = block is marked for deletion\n   ```\n\n4. Try to read block index directly:\n   ```bash\n   # This will fail if index is corrupted\n   promtool tsdb analyze --limit=1 /prometheus-data/BLOCK_ULID\n   ```\n\n**Code References:**\n- Block opening: `tsdb/block.go` (OpenBlock function)\n- Meta file reading: `tsdb/db.go` (readMetaFile function)\n- Repair function: `tsdb/repair.go:30-124` (repairBadIndexVersion)\n\n### Failure Mode 4: Out-of-Memory (OOM) During Compaction\n\n**Symptoms:**\n- Prometheus process killed with signal 9\n- System logs show \"Out of memory: Kill process\"\n- Incomplete blocks left on disk (meta.json.tmp or chunks.tmp)\n- `prometheus_tsdb_compaction_populating_block` stuck at 1\n- Memory usage spikes during compaction window\n\n**Likely Causes:**\n- Very high cardinality metrics (millions of unique series)\n- Large time range blocks with dense data\n- Insufficient memory allocated to Prometheus\n- Memory leak in compaction path\n- Simultaneous compaction of too many series\n\n**Diagnostic Steps:**\n\n1. Check Prometheus memory limits:\n   ```bash\n   # From Kubernetes\n   kubectl describe pod PROMETHEUS_POD | grep -A 5 \"memory:\"\n\n   # From systemd\n   systemctl status prometheus | grep Memory\n\n   # From process\n   ps aux | grep prometheus | grep -v grep\n   cat /proc/PID/limits | grep memory\n   ```\n\n2. Check peak memory during compaction:\n   ```bash\n   # Monitor in real-time\n   watch -n 1 'ps aux | grep prometheus | grep -v grep | awk \"{print \\$6}\"'\n   ```\n\n3. Estimate memory needed:\n   ```bash\n   # Count series in largest block\n   for block in /prometheus-data/[0-9a-z]*; do\n     series=$(jq '.Stats.NumSeries' \"$block/meta.json\")\n     echo \"$block: $series series\"\n   done | sort -k2 -nr | head -3\n   # Rough estimate: 1 million series needs 2-4 GB\n   ```\n\n4. Check if out-of-order (OOO) compaction is enabled:\n   ```bash\n   curl -s http://localhost:9090/metrics | grep -i \"out.of.order\"\n   # OOO compaction uses additional memory\n   ```\n\n**Code References:**\n- Block population (memory intensive): `tsdb/compact.go:729-875` (PopulateBlock)\n- OOO compaction: `tsdb/db.go:1238-1283` (compactOOOHead)\n- Chunk merging: `tsdb/compact.go:819-823` (NewMergeChunkSeriesSet)\n\n---\n\n## 4. Recovery Procedures\n\n### Recovery for Failure Mode 1: Disk Space Exhaustion\n\n**Recovery Steps:**\n\n1. **Immediate Actions (within minutes)**\n   ```bash\n   # Stop any additional writes\n   sudo systemctl stop prometheus\n   # This prevents new blocks being created\n   ```\n\n2. **Identify and Remove Old Blocks**\n   ```bash\n   # List all blocks with human-readable dates\n   promtool tsdb list -r /prometheus-data | head -20\n\n   # Identify blocks outside retention window\n   # Calculate which blocks to delete:\n   # retention_duration = 15 days (default)\n   # current_timestamp = $(date +%s)000  (milliseconds)\n   # block_max_time < (current_timestamp - retention_duration)\n\n   RETENTION_MS=$((15 * 24 * 3600 * 1000))  # 15 days in ms\n   CURRENT_MS=$(date +%s)000\n   CUTOFF=$((CURRENT_MS - RETENTION_MS))\n\n   # Find blocks to delete\n   for block in /prometheus-data/[0-9a-z]*; do\n     max_time=$(jq '.MaxTime' \"$block/meta.json\")\n     if [ \"$max_time\" -lt \"$CUTOFF\" ]; then\n       echo \"DELETE: $block (max_time=$max_time < $CUTOFF)\"\n     fi\n   done\n   ```\n\n3. **Delete Oldest Blocks**\n   ```bash\n   # BE CAREFUL: Manual deletion of blocks\n   # Always backup first\n   tar czf /backup/prometheus-blocks-$(date +%s).tar.gz /prometheus-data\n\n   # Delete by ULID (from step 2 above)\n   rm -rf /prometheus-data/01ARZ3NDEKTSV4RRFFQ69G5FAV  # Verify startup\n   curl -s http://localhost:9090/api/v1/status/config | jq .\n   ```\n\n7. **Monitor Recovery**\n   ```bash\n   # Watch compaction progress\n   watch -n 5 'curl -s http://localhost:9090/metrics | \\\n     grep \"prometheus_tsdb_\\(blocks_loaded\\|storage_blocks_bytes\\|compactions_total\\)\"'\n\n   # Should see:\n   # - storage_blocks_bytes decreasing\n   # - blocks_loaded stable\n   # - compactions_total incrementing\n   ```\n\n**Testing Recovery:**\n```bash\n# After 30 minutes, verify:\n1. Disk space stabilized: df -h /prometheus-data\n2. Queries working: curl http://localhost:9090/api/v1/instant_query?query=up\n3. No new errors in logs: tail -f /var/log/prometheus.log\n```\n\n### Recovery for Failure Mode 2: Compaction Stuck / Not Progressing\n\n**Recovery Steps:**\n\n1. **Assess Current State**\n   ```bash\n   #
