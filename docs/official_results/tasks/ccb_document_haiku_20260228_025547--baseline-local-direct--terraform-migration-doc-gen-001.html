# Task: Terraform v1.9.0 to v1.10.0 Migration Guide

## Objective

Analyze the changes between Terraform v1.9.0 and v1.10.0 and produce a comprehensive migration guide for users upgrading from v1.9.0 to v1.10.0.

## Context

The workspace contains two versions of the Terraform codebase:
- **v1.9.0**: implemented in `/workspace/terraform-v1.9.0/` (commit 7637a92)
- **v1.10.0**: implemented in `/workspace/terraform-v1.10.0/` (commit 24236f4)

Your task is to identify breaking changes and behavioral changes that affect users, then document migration steps.

## Required Analysis

Your migration guide must cover:

1. **S3 Backend Changes**
   - Removal of deprecated IAM role assumption attributes
   - Migration to `assume_role` block syntax
   - New S3 native state locking support

2. **Moved Blocks Syntax Changes**
   - New requirement to prepend `resource.` identifier when referencing resources with type names matching top-level blocks/keywords
   - Impact on existing `moved` blocks in configurations

3. **Sensitive Value Handling**
   - Changes to mark propagation in conditional expressions
   - When sensitive marks are now preserved (previously lost)
   - How to use `nonsensitive()` to override when needed

4. **Ephemeral Resources and Values** (new feature with migration implications)
   - Introduction of ephemeral input variables and outputs
   - New ephemeral resource mode
   - Impact on secret handling in state files

## Expected Output

Write your migration guide to `/workspace/documentation.md` with the following structure:

1. **Overview** - Summary of the upgrade and major themes
2. **Breaking Changes** - Each breaking change with:
   - What changed and why
   - Before/after code examples
   - Step-by-step migration instructions
   - Common pitfalls to avoid
3. **New Features with Migration Impact** - Features that change how users should write Terraform
4. **Testing Your Migration** - How to validate the upgrade was successful
5. **Rollback Guidance** - How to safely rollback if needed

## Evaluation

Your review will be evaluated on detection accuracy and fix quality.

## Tips

- Compare the two version directories to identify changes in specific files
- Look for UPGRADE notes in release documentation
- Examine test files to understand behavioral changes
- Check for deprecation warnings and removed features
- Pay attention to changes in internal/backend/remote-state/s3/, internal/terraform/node_resource_abstract.go, and lang/marks/ directories

When enabled, Terraform uses S3's built-in locking mechanisms instead of (or in addition to) DynamoDB-based locking.\n\n#### Why This Changed\n\nS3-native locking reduces dependencies on DynamoDB and provides better integration with S3's native capabilities. For users with existing DynamoDB locking, both locking mechanisms will be acquired to maintain compatibility.\n\n#### Configuration\n\n```hcl\nterraform {\n  backend \"s3\" {\n    bucket = \"my-terraform-state\"\n    key    = \"terraform.tfstate\"\n    region = \"us-east-1\"\n\n    # Enable S3-native locking (experimental)\n    use_lockfile = true\n\n    # DynamoDB locking can still be used alongside\n    dynamodb_table = \"terraform-locks\"\n  }\n}\n```\n\n#### Behavior\n\n- When `use_lockfile = true`: Terraform uses S3-native locking\n- When using with DynamoDB: Both locking mechanisms are acquired for safety\n- When using S3-native only: Eliminates DynamoDB dependency\n- **Future deprecation notice**: DynamoDB-based locking will be deprecated in a future Terraform release\n\n#### Migration Steps\n\n1. **Review your locking needs** - Moved Blocks: Reserved Keyword Prefix Requirement\n\n#### What Changed\n\nMoved blocks now strictly parse resource addresses and require that resource types matching reserved keywords be prefixed with `resource.`. This prevents ambiguity when a resource type name matches Terraform keywords.\n\n#### Why This Changed\n\nReserved keywords like `count`, `module`, `var`, and `local` can be confused with actual resource types. By requiring explicit `resource.` prefixes, Terraform can unambiguously parse resource addresses in moved blocks.\n\n#### Reserved Keywords\n\nThe following keywords are now reserved and require the `resource.` prefix when used as resource type names in `moved` blocks:\n\n- `count`\n- `each`\n- `local`\n- `module`\n- `path`\n- `self`\n- `terraform`\n- `var`\n- `template`\n- `lazy`\n- `arg`\n\n#### Before (v1.9.0)\n\nIf you had a resource type matching a reserved keyword (less common but possible with custom providers):\n\n```hcl\n# This would work in v1.9.0 with custom providers that use these names\nmoved {\n  from = module.old_module[0]\n  to   = module.new_module[0]\n}\n\n# If a provider had a resource type called \"var\" (unusual but possible)\nmoved {\n  from = var.old\n  to   = var.new\n}\n```\n\n#### After (v1.10.0)\n\nResources with type names matching reserved keywords must now be prefixed:\n\n```hcl\n# Module moves still work the same (module is a keyword)\nmoved {\n  from = module.old_module[0]\n  to   = module.new_module[0]\n}\n\n# But resource types matching keywords need resource. prefix\nmoved {\n  from = resource.var.old\n  to   = resource.var.new\n}\n\n# Similarly for custom provider resources with reserved keyword names\nmoved {\n  from = resource.count.old_name\n  to   = resource.count.new_name\n}\n```\n\n#### How to Migrate\n\n1. **Review existing moved blocks** in your configuration\n2. **Identify resource types that match reserved keywords** - these are rare\n3. **Add `resource.` prefix** to the resource portion of moved block addresses\n4. **Test the configuration** with `terraform validate`\n\n#### Examples\n\n```hcl\n# BEFORE (would fail in v1.10.0)\nmoved {\n  from = var.my_resource\n  to   = var.renamed_resource\n}\n\n# AFTER (works in v1.10.0)\nmoved {\n  from = resource.var.my_resource\n  to   = resource.var.renamed_resource\n}\n\n# Regular resource moves (unchanged)\nmoved {\n  from = aws_instance.example\n  to   = aws_instance.renamed\n}\n\n# The prefix is needed even for qualified resource moves\nmoved {\n  from = resource.module.foo.resource.count.example\n  to   = resource.module.foo.resource.count.renamed\n}\n```\n\n#### Common Pitfalls\n\n- **Module moves are unaffected**: Module moves like `module.foo` don't need `resource.` prefix because `module` is always a module reference, not a resource type\n- **Data sources unaffected**: Data sources can still use the `data.` prefix or no prefix: `data.aws_ami.example` works without requiring `resource.data.`\n- **Custom provider resources**: If you use third-party providers with resource types matching keywords, you'll need to update your moved blocks\n- **Error messages**: If you get an error saying a keyword is reserved, check if you need to add the `resource.` prefix\n\n---\n\n### 4. Sensitive Value Handling: Improved Mark Propagation in Conditional Expressions\n\n#### What Changed\n\nTerraform v1.10.0 improves how sensitive marks are propagated through conditional expressions and for expressions. Previously, sensitive marks could be lost in certain combinations of conditional expressions and unknown values. Now, marks are combined from all values within the expression to ensure consistency.\n\n#### Why This Changed\n\nThe previous behavior could hide sensitive information or cause inconsistencies where values were marked as sensitive during apply but not during plan. The new behavior ensures sensitive marks are preserved consistently, which is more secure and predictable.\n\n#### Impact\n\n- **More values marked sensitive**: Some values that previously lost their sensitive marks will now correctly be marked as sensitive\n- **Better security**: Sensitive information is now less likely to leak due to mark propagation issues\n- **Potential output changes**: State files and logs may show different sensitive redaction patterns\n\n#### Before (v1.9.0)\n\n```hcl\nlocals {\n  password = \"secret123\"\n  is_prod  = true\n}\n\noutput \"value\" {\n  value = local.is_prod ? sensitive(local.password) : \"default\"\n  # The output could lose the sensitive mark in certain cases\n}\n```\n\n#### After (v1.10.0)\n\n```hcl\nlocals {\n  password = \"secret123\"\n  is_prod  = true\n}\n\noutput \"value\" {\n  value = local.is_prod ? sensitive(local.password) : \"default\"\n  # The output now correctly preserves the sensitive mark\n  # Output is always marked as sensitive due to mark propagation\n}\n```\n\n#### How Marks Are Combined\n\nIn v1.10.0, marks are combined as follows:\n\n1. **Conditional expressions**: If any branch of a conditional can produce a sensitive value, the result is marked sensitive\n2. **For expressions**: Sensitive marks are preserved through iteration\n3. **Object/list construction**: If any element is sensitive, the entire structure can be marked sensitive (depending on usage)\n\n#### Example: Conditional Expressions\n\n```hcl\n# All three branches can produce sensitive output, so result is marked sensitive\noutput \"secure_value\" {\n  value = var.use_secret ? var.secret :\n          var.use_default_secret ? sensitive(\"default-secret\") :\n          \"fallback\"\n  # Result is marked sensitive because branches 1 and 2 can produce sensitive values\n}\n\n# Mix of sensitive and non-sensitive branches\noutput \"mixed_value\" {\n  value = var.is_admin ? var.admin_key : var.user_key\n  # If var.admin_key is sensitive, output is marked sensitive\n}\n```\n\n#### Using `nonsensitive()` to Override\n\nIf you have legitimate reasons to remove sensitive marks, use the `nonsensitive()` function:\n\n```hcl\noutput \"declassified_value\" {\n  value = nonsensitive(var.secret_that_should_be_visible)\n  # Explicitly removes the sensitive mark\n  # Be cautious: this value will appear in plain text in logs and state\n}\n\noutput \"conditional_declassified\" {\n  value = var.show_secret ? nonsensitive(var.secret) : \"hidden\"\n  # Only declassified when condition is true\n}\n```\n\n#### Migration Steps\n\n1. **Test your configuration** with `terraform plan` in v1.10.0\n2. **Review output changes** - note which values are now marked as sensitive\n3. **Evaluate if the behavior is correct** for your use case\n4. **Use `nonsensitive()`** only if you have legitimate reasons to declassify values\n5. **Update tests and validations** if you have checks that verify sensitive status\n\n#### Common Issues and Solutions\n\n**Issue**: Outputs that worked in v1.9.0 now show as sensitive in v1.10.0\n\n```hcl\n# v1.9.0 - might not mark as sensitive\noutput \"value\" {\n  value = var.condition ? sensitive(local.secret) : \"default\"\n}\n\n# v1.10.0 - marked as sensitive due to conditional branch\n# Solution 1: Accept the more secure behavior\n# Solution 2: Use nonsensitive() if truly not a secret\noutput \"value\" {\n  value = nonsensitive(\n    var.condition ? sensitive(local.secret) : \"default\"\n  )\n}\n```\n\n**Issue**: Plan shows sensitive values but apply shows them in plain text\n\n```hcl\n# This indicates inconsistent mark propagation in v1.9.0\n# In v1.10.0, it will be consistent\n# Solution: Ensure all code paths that handle secrets explicitly mark them\n\noutput \"config_value\" {\n  # GOOD: Consistent marking\n  value = var.is_secret ? sensitive(data.external.config.result.value) : \"default\"\n\n  # AVOID: Inconsistent marking\n  value = var.is_secret ? data.external.config.result.value : \"default\"\n}\n```\n\n---\n\n## New Features with Migration Impact\n\n### Ephemeral Resources and Values\n\nTerraform v1.10.0 introduces ephemeral resources and values - data that exists only in memory during a single phase and cannot be persisted to state files or plan files.\n\n#### What Are Ephemeral Values?\n\nEphemeral values are:\n- Only available during Terraform evaluation\n- Not persisted to state files\n- Not persisted to saved plan files\n- Useful for handling secrets that should never be stored\n- Automatically lost between phases\n\n#### Ephemeral Input Variables\n\nYou can now mark input variables as ephemeral:\n\n```hcl\nvariable \"api_key\" {\n  type        = string\n  sensitive   = true\n  ephemeral   = true\n  description = \"API key that should not be stored in state\"\n}\n\nresource \"example_api\" \"example\" {\n  api_key = var.api_key\n  # # Terraform v1.9.0 to v1.10.0 Migration Guide

## Overview

Terraform v1.10.0 introduces several significant changes that require attention when upgrading from v1.9.0. The most impactful changes involve:

1. **S3 Backend Configuration** - Removal of deprecated IAM role attributes and migration to `assume_role` block syntax
2. **Moved Blocks Syntax** - New requirement to prepend `resource.` when referencing resources with type names matching reserved keywords
3. **Sensitive Value Handling** - Improved mark propagation in conditional expressions that may result in more values being marked as sensitive
4. **Ephemeral Resources and Values** - New feature introducing ephemeral input variables, outputs, and resources that don't persist to state files

This guide walks you through each breaking change with detailed migration instructions and examples.

---

## Breaking Changes

### 1. S3 Backend: Removal of Deprecated IAM Role Attributes

#### What Changed

Terraform v1.10.0 removes deprecated IAM role assumption attributes from the S3 backend configuration. These attributes have been replaced with the `assume_role` block syntax. The deprecated attributes are:

- `role_arn`
- `session_name`
- `external_id`
