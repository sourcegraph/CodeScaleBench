HTTP service container for registered REST endpoints (go-restful)\n- `NonGoRestfulMux`: Fallthrough mux for non-API paths (health checks, metrics, webhooks)\n- `Director`: Smart dispatcher determining which handler to use\n\nThe Director introspects the request path to determine if it matches a go-restful route. If yes, delegates to `GoRestfulContainer`; otherwise, delegates to `NonGoRestfulMux`. This avoids forcing all paths through the heavyweight go-restful machinery.\n\n### 2.4 APIGroupInfo & APIInstaller (`pkg/server/config.go`, `pkg/endpoints/installer.go`)\n\n**Responsibility**: Register API groups' resources as HTTP endpoints and enable discovery.\n\n**APIGroupInfo** (line 71 in config.go) bundles:\n- `PrioritizedVersions`: Ordered list of supported API versions (e.g., `[v1, v1beta1]`)\n- `VersionedResourcesStorageMap`: Nested map `{version \u2192 {resource \u2192 rest.Storage}}`\n- `Scheme`: Type registry for serialization, conversion, defaulting\n- `NegotiatedSerializer`: Codec factory for JSON, YAML, Protocol Buffers\n\n**Installation Process** (line 876 in genericapiserver.go):\n1. `InstallAPIGroups()` validates group/version presence\n2. `getOpenAPIModels()` introspects storage and types to generate OpenAPI schema\n3. `installAPIResources()` uses `APIInstaller` to create REST routes for each resource\n4. Others are not consulted.\n\n**User Info** includes:\n- Name, UID\n- Groups (for RBAC)\n- Extra fields (for custom attributes)\n\n### 2.6 Authorization (`pkg/authorization/`)\n\n**Responsibility**: Decide if an authenticated user is allowed to perform an action.\n\n**Pluggable Authorizers** (union pattern):\n- `rbac`: Role-Based Access Control (core authorization model)\n- `node`: Special rules for kubelet system requests\n- `webhook`: Custom authorization logic via webhook\n- `always-allow`: Unconditional allow (insecure, testing only)\n- `always-deny`: Unconditional deny\n\n**Key Property**: Authorizers vote on requests. First authorizer to **allow** \u2192 request allowed. First to **deny** \u2192 request denied. If all abstain (no opinion) \u2192 deny by default.\n\n### 2.7 Admission Control (`pkg/admission/`)\n\n**Responsibility**: Validate or mutate request objects before storage, enforce policies.\n\n**Two Phases**:\n1. **Mutating Admission Plugins**: Can modify the object (e.g., inject sidecar containers, fill defaults)\n2. **Validating Admission Plugins**: Can only reject (e.g., enforce Pod Security Standards)\n\n**Plugins**:\n- Built-in: PodSecurity, ResourceQuota, PodDisruptionBudget\n- Dynamic webhooks: ValidatingAdmissionWebhook, MutatingAdmissionWebhook\n- Dynamic policies (CEL-based): ValidatingAdmissionPolicy\n\n**Key Property**: Object deserialization happens *after* auth/authz/admission initialization but *before* validation (line 113 in ARCHITECTURE.md states \"request body is not deserialized until it has passed authentication and authorization\"). This prevents parsing attacks from expensive resources.\n\n### 2.8 Storage Interface & Registry (`pkg/registry/`, `pkg/storage/`)\n\n**Responsibility**: Abstract data persistence layer; enable string, obj, out runtime.Object, ttl *uint64) error`\n- `Delete(ctx, key string, obj runtime.Object, preconditions *Preconditions) error`\n- `Watch(ctx, key string, opts WatchOptions) (watch.Interface, error)`\n- `List(ctx, key string, opts ListOptions) (*List, error)`\n- `GuaranteedUpdate(ctx, key string, ptrToType runtime.Object, ignoreNotFound bool, preconditions *Preconditions, cache, reducing etcd load\n- Automatically refreshes from backend when consistency required\n- Uses bookmarks to track progress for unchanged objects\n\n### 2.9 REST Storage Strategy (`pkg/registry/generic/`)\n\n**Responsibility**: Resource-specific business logic for validation, versioning, and object lifecycle.\n\n**Strategy Interface** (in rest package):\n- `New()`: Allocate (immutable).\n\n### 2.10 Server Chain Delegation\n\nThe production kube-apiserver is **not a single server** but a **chain of three GenericAPIServer instances**:\n\n```\nRequest \u2192 End-to-End Request Flow with Code Evidence\n\n### Scenario: Client creates a Deployment via kube-apiserver\n\n**Step 1: Request Arrives at FullHandlerChain**\n- Client: `POST /apis/apps/v1/namespaces/default/deployments`\n- Handler: `APIServerHandler.FullHandlerChain` (line 45 in handler.go)\n\n**Step 2: Request Parsing & Request Info**\n- Filter: `WithRequestInfo` (endpoints/filters/requestinfo.go) parses path \u2192 extracts resource, verb, namespace\n- Result: `apirequest.RequestInfo` attached to context\n\n**Step 3: Authentication**\n- Filter: `WithAuthentication` (endpoints/filters/authentication.go)\n- Authenticator: Union of x509, bearer-token, OIDC, webhook\n- Result: `user.Info` attached to context if successful; Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Resource Version Conflicts\n\n**Scenario**: Two clients update same Deployment concurrently.\n\n**Mechanism** (optimistic locking):\n- Client A reads Deployment `resourceVersion: 12345`\n- Client B reads Deployment `resourceVersion: 12345`\n- Client B updates \u2192 etcd revision advances to 12346\n- Client A attempts update with `resourceVersion: 12345` (now stale)\n- etcd compares: stored version (12346) \u2260 provided version (12345)\n- Result: `409 Conflict` returned; Client A must retry (read fresh, merge, resubmit)\n\n**Tradeoff**: Optimistic concurrency is simple and lock-free but requires clients to handle conflicts. This is acceptable because Kubernetes expects controllers to retry on 409.\n\n### 4.2 Watch Cache Staleness Bounds\n\n**Problem**: Watch cache may lag behind etcd if object unchanged but other objects changed.\n\n**Solution** (bookmarks, line 213 in ARCHITECTURE.md):\n- Cacher periodically sends bookmark events (`type: BOOKMARK`) for unchanged objects\n- Bookmark includes current ResourceVersion, signaling to clients that cache is fresh as of that version\n- Prevents expensive relist operations when cache position is old but no events for specific resource\n\n**Tradeoff**: Bookmarks add message overhead (more events) but reduce thundering-herd relist storms when watchers are many.\n\n### 4.3 Request Timeout & Long-Running Requests\n\n**Problem**: Generic timeout logic unsuitable for watches, port-forward, exec which stream indefinitely.\n\n**Solution** (line 1067 in config.go):\n- `LongRunningFunc` identifies requests by verb (watch, exec, attach, proxy)\n- Short timeout applied to CRUD operations (default 60s)\n- No timeout for identified long-running requests\n- Separate goroutine handles timeout logic; original request can keep writing response\n\n**Tradeoff**: Requires explicit classification of long-running verbs; custom servers must extend `LongRunningFunc` if adding new streaming endpoints.\n\n### 4.4 Memory Exhaustion from Large Objects\n\n**Limit**: `MaxRequestBodyBytes` (line 466 in config.go, default 3 MiB).\n\n**Mechanism**: Request body size checked before deserialization (prevents parsing large payloads).\n\n**Tradeoff**: Hardcoded limit prevents DOS but may block legitimate large bulk operations (e.g., large ConfigMap). Mitigated by allowing configuration; extensions can adjust for their workloads.\n\n### 4.5 Thundering Herd in Watch Cache\n\n**Problem**: Large resource lists with rapid updates cause all watchers to receive events, overwhelming client bandwidth.\n\n**Mitigation** (in cacher):\n- Bookmark events space out ResourceVersion updates, preventing repeated sends of same version\n- Rate limiting on event emission prevents queue explosion\n- Old events dropped from buffer once new events arrive\n\n**Tradeoff**: Watchers may miss intermediate states if buffer overflows, but most applications only care about final state anyway.\n\n### 4.6 Admission Plugin Ordering & Re-invocation\n\n**Problem**: Mutating plugins may change objects in ways that trigger validating plugin denials; order matters.\n\n**Solution** (line 34 in ARCHITECTURE.md, \"re-invocation policy\"):\n- Mutating plugins run first; validating plugins run second\n- If mutating plugin changes object, validating plugins re-run on modified object\n- Circular mutations prevented by re-invocation control (a plugin's mutations don't trigger other plugins' mutations indefinitely)\n\n**Tradeoff**: Re-invocation adds complexity and latency; plugins must declare which fields they mutate.\n\n### 4.7 Discovery & OpenAPI Latency\n\n**Problem**: Generating OpenAPI spec from all installed resources is expensive (reflection + JSON serialization).\n\n**Solution** (line 239 in ARCHITECTURE.md):\n- `openapi-gen` tool pre-generates Go functions that build spec once at startup\n- Spec cached and served statically thereafter; no runtime generation\n\n**Tradeoff**: Spec is stale if new CRD installed (solved by cache invalidation on CRD add/delete, but adds latency).\n\n---\n\n## 5. Extension Points, Customization Hooks, and Associated Risks\n\n### 5.1 Custom Admission Plugins\n\n**Extension Point**: `admission.Interface` (pkg/admission/interfaces.go)\n\n**How**:\n1. Implement `Admit(ctx context.Context, attrs Attributes, o ObjectInterfaces) error`\n2. Inspect attributes (user, resource, object)\n3. Mutate object (mutating) or return error (validating)\n4. Register plugin via `admission.Registry`\n\n**Example Use Case**: Custom sidecar injector for security policies.\n\n**Risks**:\n- **Latency**: Plugin runs on every write; slow plugins block all mutations\n- **Denial of Service**: Plugin can unconditionally reject all writes\n- **Object Leakage**: Plugin receives full object including secrets; must not log confidential fields\n- **Cascading Failures**: If plugin crashes, admission control chain halts; other plugins not consulted\n\n**Mitigation**:\n- Webhooks (dynamic, can be disabled without restart) preferred over compiled plugins\n- Timeout per plugin (if supported) prevents indefinite hangs\n- Audit plugins' object access to detect data leaks\n\n### 5.2 Custom Authorization Plugins\n\n**Extension Point**: `authorizer.Authorizer` (pkg/authorization/authorizer/interfaces.go)\n\n**How**:\n1. Implement `Authorize(ctx context.Context, a Attributes) (Decision, Reason, Error)`\n2. Return `Allow`, `Deny`, or `NoOpinion`\n3. Register via `authorizer.NewUnion(...)`\n\n**Example Use Case**: Custom ABAC (Attribute-Based Access Control) for dynamic policies.\n\n**Risks**:\n- **Privilege Escalation**: Incorrect policy grants unintended access\n- **Latency**: Authorization check on every request; slow authorizers block all operations\n- **Policy Inconsistency**: Mismatch between local authorization and webhook authorization\n- **Cascading Denial**: If all authorizers abstain, request denied; client can't determine why\n\n**Mitigation**:\n- Test authorization rules against audit logs (see what decisions were made)\n- Webhook authorizers allow disabling without restart\n- Explicit deny policies prevent accidental privilege escalation\n\n### 5.3 Custom Storage Backends\n\n**Extension Point**: `rest.Storage` interface (pkg/registry/rest/)\n\n**How**:\n1. Implement CRUD operations: `Create()`, `Get()`, `Update()`, `Delete()`, `List()`, `Watch()`\n2. Pass implementation in `APIGroupInfo.VersionedResourcesStorageMap`\n3. `InstallAPIGroup()` registers the storage as HTTP endpoint handler\n\n**Example Use Case**: Custom database (PostgreSQL, Cassandra) instead of etcd.\n\n**Risks**:\n- **Watch Semantics Deviation**: Custom backends must respect Kubernetes watch semantics (incremental events, resource versions) or clients fail\n- **Consistency Violations**: If custom backend doesn't support transactions, updates may partially succeed\n- **ResourceVersion Mapping**: Custom backend must map internal versions to Kubernetes resource versions; mismatch breaks optimistic locking\n- **List Continuation**: Pagination via `continue` token requires backend understanding of list semantics\n\n**Mitigation**:\n- Use `genericregistry.Store` wrapper which handles versioning and watch semantics\n- Test custom storage against Kubernetes client conformance tests\n\n### 5.4 Custom Handler Chain\n\n**Extension Point**: `Config.BuildHandlerChainFunc` (line 186 in config.go)\n\n**How**:\n```go\nconfig.BuildHandlerChainFunc = func(apiHandler http.Handler, c *Config) http.Handler {\n    handler := apiHandler\n    handler = myCustomFilter(handler)  // Add custom filter\n    handler = DefaultBuildHandlerChain(handler, c)  // Wrap with standard chain\n    return handler\n}\n```\n\n**Example Use Case**: Custom audit filter, request mutation, or telemetry.\n\n**Risks**:\n- **Filter Order Violation**: Custom filter placed incorrectly may see unauthenticated requests or unresolved resource names\n- **Context Mutation**: If custom filter modifies request context, downstream handlers see corrupted state\n- **Panic Recovery**: If custom filter panics, bypasses panic recovery if placed after `WithPanicRecovery`\n\n**Mitigation**:\n- Document filter ordering expectations in comments\n- Use standard `http.Handler` middleware patterns (defer cleanup, check context)\n- Test with malformed requests to ensure robustness\n\n### 5.5 PostStartHooks & PreShutdownHooks\n\n**Extension Point**: `GenericAPIServer.AddPostStartHook()` (genericapiserver.go)\n\n**How**:\n```go\nserver.AddPostStartHook(\"my-initializer\", func(ctx PostStartHookContext) error {\n    // Initialize custom components after server starts listening\n    return nil\n})\n```\n\n**Lifecycle**:\n- `PostStartHooks` run sequentially after server begins listening on secure port\n- `PreShutdownHooks` run sequentially when server receives shutdown signal (SIGTERM)\n- If hook returns error, server panics (halts startup/shutdown)\n\n**Example Use Cases**: Initialize custom controllers, health checks, cleanup connections.\n\n**Risks**:\n- **Deadlock**: Hook waits for resource that requires server (e.g., listing Deployments); creates circular dependency\n- **Blocking Startup**: Long-running hook delays server readiness; clients fail with connection refused\n- **Lost State**: If hook fails halfway, partial initialization leaves inconsistent state\n\n**Mitigation**:\n- Use timeout for hooks (context includes deadline)\n- Return errors immediately without waiting for resources\n- Idempotent cleanup in PreShutdownHooks (may be called multiple times)\n\n### 5.6 REST Routes & Subroutes\n\n**Extension Point**: `GenericAPIServer.NonGoRestfulMux` (handler.go, line 51)\n\n**How**:\n```go\nserver.Handler.NonGoRestfulMux.HandleFunc(\"/custom/path\", http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {\n    // Custom handler\n}))\n```\n\n**Example Use Case**: Webhooks, health checks, debug endpoints.\n\n**Risks**:\n- **Path Collision**: Custom path may shadow built-in paths (e.g., /healthz)\n- **Missing Auth**: Custom handlers not protected by authentication/authorization filters\n- **Inconsistent Response**: Custom handlers may not respect content negotiation, error formatting\n\n**Mitigation**:\n- Use unique paths to avoid collisions (e.g., `/custom/v1/...`)\n- Manually apply auth filters if sensitive\n- Document custom endpoint contracts for clients\n\n---\n\n## 6. Operational Implications\n\n### 6.1 Request Timeout Tuning\n\n**Default**: 60 seconds for non-long-running requests (line 445 in config.go).\n\n**Operational Concern**: Timeout applies to entire request lifecycle including:\n- Authentication check (may call webhooks)\n- Authorization check (may call webhooks)\n- Admission control (may call webhooks, mutation plugins)\n- Storage I/O (etcd latency)\n\n**Long Webhooks Impact**: If auth webhook takes 30s, only 30s remains for remaining stages; admission webhook fails with timeout.\n\n**Mitigation**:\n- Monitor webhook latencies (metrics: `apiserver_authorization_decision_latency`, `apiserver_admission_webhook_latency`)\n- Increase timeout if webhooks are slow, but accept increased user latency\n- Implement webhook caching to reduce roundtrips\n\n### 6.2 Watch Cache Sizing\n\n**Default**: In-memory cache per resource, size not explicitly limited by apiserver (depends on object count and size).\n\n**Operational Concern**: Large clusters with many resources (e.g., 1M Pods) can cause memory exhaustion. Watch cache holds all objects in memory.\n\n**Example**: 1M Pods \u00d7 1 KiB per Pod = ~1 GiB memory; with overhead, may reach 2-3 GiB.\n\n**Mitigation**:\n- Monitor memory usage (metric: `apiserver_watch_cache_capacity`)\n- Reduce watch cache scope (disable for large resources via storage config)\n- Use etcd backend's watch streams directly (bypass cache) if memory critical\n\n### 6.3 Admission Webhook Cascading Failures\n\n**Scenario**: Mutating webhook fails (e.g., sidecar injector crashes).\n\n**Impact**: All Pod creations fail with 500 Internal Server Error (if webhook deadline exceeded) or 403 Forbidden (if webhook returns deny).\n\n**Operational Concern**: **Cascading failure**: If webhook is misconfigured, entire cluster becomes unable to schedule new Pods.\n\n**Mitigation**:\n- Webhook failurePolicy: `Fail` (deny on failure, safe) vs `Ignore` (allow on failure, unsafe)\n- Set webhook timeout low (e.g., 5s) so cascading failures are transient\n- Monitor webhook response times; alert if 95th percentile latency grows\n- Disable webhook if necessary: `kubectl delete validatingwebhookconfigurations <name>`\n\n### 6.4 Audit Logging Volume\n\n**Default**: Audit policy controls verbosity (line 249 in ARCHITECTURE.md).\n\n**Operational Concern**: High audit level (`RequestResponse`) logs full request/response bodies; can generate 1 GiB/hour in busy clusters.\n\n**Example**: Each request logged includes entire object, which for bulk operations (apply 1000 objects) multiplies storage.\n\n**Mitigation**:\n- Use audit level `Metadata` (log metadata only, not bodies) for most requests\n- Use `RequestResponse` for sensitive resources (Secrets, ServiceAccounts) only\n- Configure log rotation/retention (e.g., 30 days)\n- Stream audit logs to centralized system (SIEM) for retention compliance\n\n### 6.5 Concurrency Limits & Pressure Shedding\n\n**Default**: `MaxRequestsInFlight: 400`, `MaxMutatingRequestsInFlight: 200` (line 443-444 in config.go).\n\n**Behavior**: Requests exceeding limits are queued (if Priority & Fairness enabled) or rejected with 429 Too Many Requests.\n\n**Operational Concern**: If limit is too low, legitimate traffic rejected. If too high, server memory exhausted by buffered requests.\n\n**Mitigation**:\n- Monitor request rates (metric: `apiserver_request_total`)\n- Adjust limits based on cluster size and workload (e.g., larger clusters need higher limits)\n- Implement client-side retry backoff (exponential with jitter) to avoid thundering herd on 429\n\n### 6.6 Discovery & OpenAPI Serving Latency\n\n**Default**: OpenAPI spec served from static cache (line 243 in ARCHITECTURE.md).\n\n**Operational Concern**: First client to call `/openapi/v2` triggers spec generation (if not cached), causing latency spike.\n\n**Mitigation**:\n- Pre-warm cache during server startup by explicitly building spec\n- Monitor endpoint latency (metric: `apiserver_request_duration_seconds` with endpoint=/openapi/v2)\n- Cache spec response in client-side libraries (client-go does this)\n\n---\n\n## 7. # Kubernetes API Server Library: Architecture and Extension Guide

## Executive Summary

The Kubernetes `apiserver` library (`staging/src/k8s.io/apiserver`) is a reusable framework for building Kubernetes-compatible API servers. It abstracts the common machinery required for REST API serving, request filtering, admission control, storage abstraction, and watch caching. The library is the foundation of the official kube-apiserver and is designed to enable custom API servers to provide first-class Kubernetes experiences without reimplementing core infrastructure.

---

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose

The apiserver library solves the problem of building production-grade REST APIs that behave consistently with Kubernetes. Rather than requiring each API server to implement authentication, authorization, admission control, audit logging, storage coordination, and conflict resolution independently, the library provides:

- **Unified HTTP request handling pipeline** (authentication → authorization → admission → storage)
- **Abstract storage backend** enabling multiple data store implementations (primarily etcd)
- **Request rate limiting and concurrency management** via Priority and Fairness subsystem
- **Watch caching layer** reducing backend storage load
- **Pluggable extension points** for custom business logic via admission control and storage strategies

- HTTP handler chain construction and filter ordering
