## Summary

I've analyzed the HDA Intel sound driver bug that manifests as choppy audio after suspend to RAM in Linux kernel v3.7.6. Here's what I found:

### Root Cause

The bug is a **state persistence issue** across suspend/resume cycles in `sound/pci/hda/hda_intel.c`:

1. **Initial Problem**: When a codec communication timeout occurs (which can happen during normal initialization or under certain hardware conditions), the error recovery code in `azx_rirb_get_response()` progressively tries fallback mechanisms:
   - Polling mode
   - Disabling MSI
   - Finally switching to "single command mode" (line 951) by setting `chip->single_cmd = 1`

2. **The Bug**: Once `chip->single_cmd` is set to 1, it **never gets reset** on subsequent suspend/resume cycles. 

3. **Impact on Resume**: In `azx_resume()`, the function calls `azx_init_chip()`. However, `azx_init_chip()` contains this check at line 1222:
   ```c
   if (!chip->single_cmd)
       azx_init_cmd_io(chip);
   ```
   Since `single_cmd` is still 1 from the previous error, **CORB/RIRB are not reinitialized**, leaving the hardware in an inconsistent state.

4. **Result**: Codec communication fails with the timeout errors and error recovery code keeps triggering, resulting in choppy audio with 2-second silences.

### Files and Functions Involved

- **`sound/pci/hda/hda_intel.c`**:
  - `azx_rirb_get_response()` (line 863): Sets `single_cmd = 1` on error
  - `azx_resume()` (line 2539): Doesn't reset `single_cmd` before reinitializing
  - `azx_init_chip()` (line 1209): Skips CORB/RIRB init if `single_cmd` is true
