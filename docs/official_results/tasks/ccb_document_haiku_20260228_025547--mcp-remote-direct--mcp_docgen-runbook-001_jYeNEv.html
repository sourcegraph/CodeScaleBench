# Prometheus TSDB Compaction Operational Runbook

## 1. Overview

### What is TSDB Compaction?

TSDB compaction is a background process in Prometheus that merges overlapping time-series data blocks, removes deleted series (tombstones), and optimizes the database structure for efficient storage and querying. Compaction is essential for:

- **Reducing disk usage** by merging blocks and consolidating tombstones
- **Improving query performance** by maintaining reasonable block sizes
- **Enabling storage retention** by building exponential block hierarchies
- **Cleaning up WAL** by flushing persisted data from write-ahead logs

### Block Structure and Compaction Levels

Prometheus TSDB uses a **leveled compaction** strategy with exponential block ranges:

- **Level 1**: Smallest blocks (default 2 hours) created directly from the head block
- **Level 2**: Merged from 3+ Level 1 blocks (6 hours by default)
- **Level 3**: Merged from 3+ Level 2 blocks (18 hours by default)
- And so on...

Each block is identified by a ULID (Universally Unique Lexicographically Sortable Identifier) in the format: `blocks/<ULID>/meta.json`

Block structure:
```
<ULID>/
├── meta.json           # Block metadata, compaction level, stats
├── index               # Series index file
├── chunks/            # Compressed time-series data
├── tombstones         # Deleted series markers
└── .tmp-for-creation  # Transient directory during creation (atomic rename)
```

1. **Time-based trigger** (primary): Runs every 1 minute (db.go:995), checking if head block is compactable
