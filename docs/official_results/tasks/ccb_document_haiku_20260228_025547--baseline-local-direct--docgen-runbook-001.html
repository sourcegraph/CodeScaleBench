# Operational Runbook: Prometheus TSDB Compaction

## 1. Overview

### What is TSDB Compaction?

TSDB compaction is the process of merging overlapping time-series blocks, removing tombstoned (deleted) series, and optimizing the on-disk structure of the Prometheus time-series database. The process is critical for maintaining query performance and disk space efficiency as data ages.

### Why Compaction is Needed

1. **Memory Efficiency**: Head block data is persisted to disk blocks, freeing memory
2. **Query Performance**: Fewer, larger blocks improve query speed by reducing I/O and index lookups
3. **Disk Optimization**: Removes tombstoned series and consolidates data
4. **Space Reclamation**: Unused chunks are dropped during merging
5. **Overlap Resolution**: Vertical compaction merges blocks with overlapping time ranges

### Compaction Levels and Block Structure

Prometheus uses **leveled compaction** with exponential block ranges:

```
Level 0: 2h blocks (HEAD block, freshly persisted)
Level 1: 4h blocks (1 compaction, 2 Level-0 blocks merged)
Level 2: 16h blocks (2 compactions, 2 Level-1 blocks merged)
Level 3: 64h blocks (3 compactions, 2 Level-2 blocks merged)
Level 4: 256h blocks (4 compactions, 2 Level-3 blocks merged)
...and so on
```

Each block is identified by a ULID (Universally Unique Lexicographically Sortable Identifier) and contains:
- `meta.json` - metadata (ULID, time range, sample count, compaction level, sources)
- `index` - label index and postings lists
- `chunks/` - time-series chunk data
- `tombstones` - deletion markers for series
