# big-code-kafka-sec-001: Kafka SASL Authentication Flow Security Analysis

## Task

Trace the Apache Kafka SASL authentication flow from channel creation through mechanism negotiation (PLAIN, SCRAM-SHA-256, OAUTHBEARER, GSSAPI), the challenge-response exchange, principal extraction, to ACL-based authorization. Identify all entry points where untrusted client data enters the SASL subsystem, map the data flow through each authentication mechanism, and analyze the security properties of the authentication and authorization chain.

## Context

- **Repository**: apache/kafka (Java, ~1.2M LOC)
- **Category**: Security Analysis
- **Difficulty**: hard
- **Subsystem Focus**: `clients/src/main/java/org/apache/kafka/common/security/` (authenticator, plain, scram, oauthbearer packages) and `clients/src/main/java/org/apache/kafka/common/network/` (channel builders)

## Requirements

1. Identify all entry points where untrusted data enters the SASL subsystem (client connections, SASL tokens, handshake requests)
2. Trace the channel builder architecture: how `ChannelBuilders` creates `SaslChannelBuilder` which wires up `SaslServerAuthenticator`
3. Map the SASL handshake protocol: `ApiVersionsRequest` -> `SaslHandshakeRequest` -> mechanism-specific exchange -> authenticated principal
4. Trace at least two mechanism implementations (PLAIN and SCRAM) through their callback handlers to credential verification
5. Identify how `KafkaPrincipal` is extracted post-authentication and how `Authorizer.authorize()` enforces ACLs
6. Document existing mitigations and where they may be insufficient

Only broker-configured mechanisms accepted.\n\n**Gap:** If mechanism name parsing is case-sensitive but lookup is not (or vice versa), case confusion attacks could bypass checks. Current code uses direct string comparison; should be robust if config is consistent.\n\n**Attack Scenario:** If broker enables \"PLAIN\" and attacker requests \"plain\" (lowercase), check fails \u2192 good. But if multiple mechanisms differ only in case, configuration could be confused.\n\n**Mitigation Status:** \u2713 Sufficient (string comparison is case-sensitive; config must be consistent)\n\n---\n\n#### 2. **PLAIN Token Parsing: Unbounded Username/Password Handling**\n\n**Issue:** `PlainSaslServer.evaluateResponse()` (line 85) blindly converts client bytes to UTF-8 string without size validation:\n```java\nString response = new String(responseBytes, StandardCharsets.UTF_8);\n```\n\n**Vulnerability:**\n- If client sends 1MB of random bytes, UTF-8 decoding creates a 1MB string\n- `extractTokens()` parses the string by scanning for NUL; linear time, but unbounded\n- No explicit size limit per token (authzid, username, password)\n\n**Attack Scenario:**\n1. Client sends `PLAIN` token with 256KB username field\n2. `PlainServerCallbackHandler.authenticate()` looks up `user_<256KB_string>` in JAAS config\n3. JAAS config lookup is O(n) in config size; could trigger DoS via memory exhaustion or CPU spike\n4. Even if lookup fails, temporary string objects are created\n\n**Mitigation in place:**\n- `SASL_SERVER_MAX_RECEIVE_SIZE` limits total message size to 1MB by default (line 195-197)\n- After UTF-8 decode, `extractTokens()` validates `tokens.size() == 3` (line 129)\n- Empty token check for username (line 91) and password (line 94)\n\n**Mitigations are insufficient:**\n- Per-token size limits not enforced (3 tokens could be [256KB, 256KB, 256KB])\n- JAAS config lookup performance not bounded; no caching strategy if config is large\n- Callback handler does not validate token contents (no regex on username, no special char rejection)\n\n**Remediation:**\n- Add per-token size limits (e.g., max 255 bytes per RFC 4616)\n- Validate username format (e.g., alphanumeric + select punctuation)\n- Implement credential cache with pre-computed hashes to avoid repeated JAAS lookups\n\n---\n\n#### 3. **SCRAM Username Handling: SASLname Decoding**\n\n**Issue:** `ScramSaslServer` extracts username via `ScramFormatter.username(saslName)` (line 109).\n\n**SCRAM RFC 5802 SASLname:** Username is UTF-8 encoded and may contain escaped NUL bytes (`=3D`, `=2C`, etc.)\n\n**Vulnerability:**\n- SCRAM allows usernames with arbitrary UTF-8 characters\n- Upon decoding, username could contain characters that confuse downstream systems\n- Credential cache lookup: `credentialCache.get(username)` (ScramServerCallbackHandler line 67)\n  - If cache is backed by a map, any string is valid key; risk is low\n  - If cache involves external lookup (LDAP, database), injection possible\n\n**Attack Scenario:**\n1. Attacker requests SCRAM auth with username `admin\\0admin` (null byte injection)\n2. If credential cache performs string operations without null-byte handling, could authenticate as wrong user\n3. In Java, strings are null-byte safe, so this is low risk; but plugin developers might use unsafe C libraries\n\n**Mitigation in place:**\n- CredentialCache is in-memory Java map; null-byte safe\n- Username validation depends on callback handler implementation\n\n**Mitigations are insufficient:**\n- Default cache does not validate username format\n- No protection against plugin callback handlers that might be vulnerable to special characters\n- SCRAM extensions (e.g., SCRAM-PLUS channel binding) are parsed but validation is incomplete\n\n**Remediation:**\n- Whitelist allowed characters in usernames (e.g., `[a-zA-Z0-9._-]`)\n- Document security requirements for custom callback handler implementations\n- Validate channel binding claim against SSL transport layer state strictly\n\n---\n\n#### 4. **OAUTHBEARER: JWT Validation Gaps**\n\n**Issue:** OAuth Bearer token validation delegated to callback handler.\n\n**Risk 1: Unsecured Validator**\n- `OAuthBearerUnsecuredValidatorCallbackHandler` (line 331) is default for brokers\n- If misconfigured, allows unsigned JWT tokens\n- Principal extracted directly from JWT claims without signature verification\n- Attacker can forge JWT with arbitrary principal name\n\n**Attack Scenario:**\n1. Broker configured with `sasl.oauthbearer.token.endpoint.url` = invalid (defaults to unsecured)\n2. Attacker sends JWT `{\"principal\": \"admin\", ...}` (no signature)\n3. Token accepted as-is; principal becomes \"admin\" without verification\n\n**Mitigation in place:**\n- Configuration should enforce JWT signature validation for production\n- `DefaultJwtValidator` checks signature if configured\n- Documentation warns against unsecured validator\n\n**Mitigations are insufficient:**\n- Default configuration is insecure if not explicitly set\n- No compile-time enforcement of validation\n\n**Risk 2: Principal Name Injection**\n- JWT `principal_claim` (default: `sub`) value becomes KafkaPrincipal.name\n- If JWT is forged/modified, attacker controls principal name\n- ACL system trusts principal name \u2192 authorization bypass\n\n**Remediation:**\n- Require signed JWKS configuration\n- Add principal name whitelist validation\n- Implement JWT claim value sanitization\n\n---\n\n#### 5. **Principal Name to Authorization Mapping**\n\n**Issue:** `DefaultKafkaPrincipalBuilder.build()` (line 69) extracts principal and creates `KafkaPrincipal`:\n\nFor SASL:\n```java\nSaslServer saslServer = ((SaslAuthenticationContext) context).server();\nif (SaslConfigs.GSSAPI_MECHANISM.equals(saslServer.getMechanismName()))\n    return applyKerberosShortNamer(saslServer.getAuthorizationID());\nelse\n    return new KafkaPrincipal(KafkaPrincipal.USER_TYPE, saslServer.getAuthorizationID());\n```\n\n**Vulnerability: Principal Derivation Has No Injection Protection**\n\nThe `saslServer.getAuthorizationID()` is used directly as principal name for non-GSSAPI mechanisms.\n\n**For PLAIN:**\n- Username comes from client SASL token\n- No further validation in principal builder\n- Callback handler validates credentials, but principal name is not sanitized\n\n**Attack Scenario (if auth succeeds but principal is malformed):**\n1. Broker accepts SCRAM auth with username `admin/backup` (contains `/`)\n2. ACL system might misinterpret principal as `admin` + `/backup` resource\n3. Depending on ACL implementation, could cause scope confusion\n\n**For GSSAPI:**\n```java\nreturn applyKerberosShortNamer(saslServer.getAuthorizationID());\n```\n- Principal parsed with `KerberosName.parse()` (line 91)\n- `parse()` could throw if format is invalid (but GSSAPI already validated)\n- `KerberosShortNamer.shortName()` applies regex rules; rules could have injection if user-controllable\n\n**Mitigation in place:**\n- For PLAIN/SCRAM: Direct credential verification ensures username is known/stored\n- For GSSAPI: KDC validation ensures principal is legitimate\n- For OAuth: JWT signature validation (if configured)\n\n**Mitigations are insufficient:**\n- Principal name passed to Authorizer without further validation\n- If Authorizer uses principal name in file paths or shell commands, injection possible\n- Custom Authorizer implementations may not handle all special characters\n\n**Remediation:**\n- Implement strict principal name validation in builder (whitelist: `[a-zA-Z0-9._-]`)\n- Sanitize principal names before passing to Authorizer\n- Document expectations for Authorizer implementations regarding principal format\n\n---\n\n#### 6. **Authorizer Invocation & ACL Enforcement**\n\n**Issue:** `Authorizer.authorize(AuthorizableRequestContext context, List<Action> actions)` is called post-authentication.\n\n**Vulnerability: Implicit Trust in Principal**\n\nOnce authentication succeeds, `context.principal()` is used for all authorization decisions. There is no re-validation of principal validity.\n\n**Attack Scenario:**\n1. Attacker authenticates via PLAIN with username `attacker`\n2. During request processing, suppose connection is hijacked or principal is spoofed (post-auth)\n3. Authorizer checks ACLs for `attacker` \u2192 access denied (expected)\n4. But if Authorizer is stateless and doesn't verify principal on every request, could be exploited\n\n**Mitigation in place:**\n- Principal is bound to KafkaChannel; re-used for all requests on that channel\n- Authorizer is invoked for every request; checks ACLs against principal\n\n**Mitigations are sufficient for normal attacks:**\n- Once channel is authenticated, principal cannot be changed mid-connection\n- Authorization is checked per-request\n\n**Potential gap:**\n- If Authorizer is misconfigured with deny-all rules, legitimate users are blocked (not a security issue)\n- If Authorizer has bugs (e.g., null principal handling), could bypass checks\n\n**Remediation:**\n- Authorizer implementations should validate principal is non-null before checking ACLs\n- Implement logging of all authorization decisions for audit trail\n\n---\n\n#### 7. **Credential Storage: SCRAM in CredentialCache**\n\n**Issue:** `ScramServerCallbackHandler` retrieves credentials from `CredentialCache` (line 67):\n```java\nScramCredentialCallback sc = (ScramCredentialCallback) callback;\nsc.scramCredential(credentialCache.get(username));\n```\n\n**Vulnerability: Credential Cache Key Extraction**\n\n- Cache is indexed by username\n- If credential lookup is not found, `credentialCache.get(username)` returns `null`\n- Server sends `ScramException` \u2192 information leak\n\n**Attack Scenario:**\n1. Attacker tries SCRAM auth with username `admin`\n2. If username not in cache: exception thrown \u2192 client learns \"admin\" doesn't exist\n3. Attacker can enumerate valid usernames by timing responses or error messages\n\n**Mitigation in place:**\n- ScramException is caught and converted to generic error (line 127-128)\n- Error message: \"Authentication failed: Invalid user credentials\" (generic)\n\n**Mitigation is sufficient:**\n- Exception message does not reveal whether username was found\n- Timing might still leak info (credential lookup vs. missing user)\n\n**Potential gap:**\n- If cache lookup is slow for non-existent users (e.g., database round-trip), timing attack possible\n- No rate limiting on auth attempts per username\n\n**Remediation:**\n- Implement constant-time credential lookup (e.g., always check against dummy credential if user not found)\n- Add rate limiting per username + source IP\n- Implement authentication attempt logging for intrusion detection\n\n---\n\n#### 8. **Re-authentication: Mechanism Switching**\n\n**Issue:** `SaslServerAuthenticator.reauthenticate()` (line 343) handles connection re-auth:\n\n```java\nif (!reauthInfo.reauthenticating() || reauthInfo.saslMechanismUnchanged(clientMechanism)) {\n    createSaslServer(clientMechanism);\n    setSaslState(SaslState.AUTHENTICATE);\n}\n```\n\nIf mechanism is unchanged, new SaslServer is created. Request re-auth with PLAIN (weak)\n3.
