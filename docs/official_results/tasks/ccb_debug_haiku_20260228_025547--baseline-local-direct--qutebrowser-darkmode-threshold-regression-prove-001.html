# Bug Investigation: Dark Mode Text Threshold Setting Has No Effect on Qt 6.4

**Repository:** qutebrowser/qutebrowser
**Task Type:** Find and Prove (write a regression test)

## Reported Issue

When users configure the dark mode text brightness threshold setting (`colors.webpage.darkmode.threshold.text`) and qutebrowser is running with Qt version 6.4, the setting has no effect on dark mode rendering.

Users report that setting the threshold to any value (for example, `100`) does not change the dark mode behavior. The exact same configuration works correctly on Qt 6.3 and earlier versions.

This only affects the text/foreground threshold setting. Other dark mode settings (algorithm, grayscale, image policy) work correctly on the same Qt version.

## Investigation Summary

### Bug Root Cause

The dark mode text threshold setting has no effect on Qt 6.4 because:

1. **Chromium 99.0.4785.0 (Qt 6.4)** renamed the setting from `TextBrightnessThreshold` to `ForegroundBrightnessThreshold`
2. **The current qutebrowser code does NOT distinguish between Qt 6.3 and 6.4** - both use the same `qt_63` variant
3. Therefore, Qt 6.4 still sends the old (incorrect) key name `TextBrightnessThreshold`, which Chromium no longer recognizes

### Evidence

- **Qt 6.3**: Correctly uses `TextBrightnessThreshold` ✅
- **Qt 6.4**: Should use `ForegroundBrightnessThreshold` but uses `TextBrightnessThreshold` ❌

## Regression Test

I've created a self-contained regression test at `/workspace/regression_test.py` with 6 test cases that:

1. **test_qt64_variant_exists** - Verifies qt_64 variant should exist (FAILS on buggy code)
2. **test_qt64_uses_different_definition** - Checks qt_64 differs from qt_63 (SKIPS on buggy code)
3. **test_variant_detection_qt64** - Confirms 6.4 falls back to qt_63 (PASSES on buggy code)
4. **test_variant_detection_qt63** - Confirms 6.3 returns qt_63 (PASSES on buggy code)
5. **test_threshold_setting_key_names** - Checks threshold key names (FAILS on buggy code)
6. **test_correct_variant_routing** - Verifies variant routing is different for 6.3 vs 6.4 (FAILS on buggy code)

# Dark Mode Text Threshold Bug - Regression Test Summary

## Bug Description

The dark mode text brightness threshold setting (`colors.webpage.darkmode.threshold.text`) has no effect on Qt 6.4 due to a Chromium rename of the underlying setting.

### Root Cause

In Chromium 99.0.4785.0 (which corresponds to Qt 6.4), the Chromium setting was renamed from `TextBrightnessThreshold` to `ForegroundBrightnessThreshold`. However, the qutebrowser code does not have a separate variant definition for Qt 6.4 and still uses the old key name.

### Issue Details

- **Qt 6.3**: Uses `TextBrightnessThreshold` ✅ (correct)
- **Qt 6.4**: Should use `ForegroundBrightnessThreshold` ❌ (but still uses `TextBrightnessThreshold`)

The problem is that both Qt 6.3 and 6.4 use the same variant (`qt_63`) definition, so Qt 6.4 is not distinguished from Qt 6.3.

## Regression Test

The regression test is located at `/workspace/regression_test.py` and contains 6 test cases:
