# Envoy Connection Pool Management Troubleshooting Runbook

## 1. Architecture Overview

### Connection Pool Organization

Envoy manages connection pools per cluster, per thread, and per host. Each cluster maintains a separate pool per upstream host, and each pool operates in a thread-local manner to avoid contention.

**Pool Components:**
- **ready_clients_**: List of active connections ready to accept new streams
- **busy_clients_**: List of connections at their concurrent stream limit
- **connecting_clients_**: List of connections in the establishment phase
- **pending_streams_**: Queue of requests waiting for a connection to become available

### Connection Lifecycle

Each connection in the pool transitions through the following states:

```
Connecting → Ready → [Busy ↔ Ready] → Draining → Closed
```

**State Definitions:**
- **Connecting**: Connection establishment in progress; no streams can be assigned
- **Ready**: Connection established; accepts new streams (has stream capacity)
- **Busy**: Connection at concurrent stream limit; no new streams can be assigned
- **ReadyForEarlyData**: (HTTP/3) Can accept early data streams before TLS handshake completes
- **Draining**: No new streams assigned; existing streams complete before closure
- **Closed**: Connection closed; object queued for deferred deletion

**Key Timings:**
- `connect_timeout`: Default 10 seconds (configurable via cluster.connect_timeout)
- `connection_duration_timeout`: Max time a connection can remain active (if configured)
- Per-connection idle timeout (TCP only): Closes idle TCP connections after threshold
