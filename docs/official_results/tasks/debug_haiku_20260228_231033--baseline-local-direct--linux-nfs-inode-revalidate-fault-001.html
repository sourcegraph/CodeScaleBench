## Bug Report

**Bug ID**: 117651
**Summary**: Root NFS and autofs - mount disappears due to inode revalidate failed
**Regression**: Yes (works on 3.14.12, broken on 4.1.15)

After upgrading from 3.14.12 to 4.1.15 I've got strange behaviour with autofs.

It happens on diskless machine with root on NFS. There is autofs indirect mount on /storage, where NFS volumes are mounted on demand.

Everything mounts normally, but after mount expiration (during unmount) autofs mount disappears - that is, not only /storage/<volume> unmounts, but /storage is also unmounted. This happens quite often - usually on the very first expiration.

automount logs umount error in such cases.

Without autofs, mount/umount of the same NFS volume works correctly.

With NFS debug enabled, I've got following errors:

```
NFS reply getattr: -512
nfs_revalidate_inode: (0:15/6291480) getattr failed, error=-512
NFS: nfs_lookup_revalidate(/storage) is invalid
```

After inode revalidate failure all mounts on this inode are unmounted and so autofs mount disappears, while automount daemon itself continue to run.

tcpdump shows, that server replies to getattr without error, but client doesn't see this reply, and returning error instead.

I am ready to supply additional information.

## Summary

I've successfully completed the fault localization analysis for the NFS mount disappearance bug in Linux kernel 4.1.15. Here are my findings:

### **Root Cause**
The bug is a **race condition in the RPC scheduler** (`/workspace/net/sunrpc/sched.c`, function `__rpc_execute()`, lines 783-792) where a fatal signal can cause a synchronous RPC call to return `-ERESTARTSYS` (-512) even when the RPC has completed successfully with a server reply.

### **The Race Condition**
1. An NFS `getattr` RPC call is made to revalidate an inode
2. The sync task waits for `RPC_TASK_QUEUED` to be cleared via `out_of_line_wait_on_bit()`
3. **Simultaneously**: The server responds (clearing the `RPC_TASK_QUEUED` bit) AND a fatal signal arrives
4. The `rpc_wait_bit_killable()` action returns `-ERESTARTSYS`
5. **The bug**: The code unconditionally calls `rpc_exit(task, -ERESTARTSYS)` without checking if the RPC actually completed
6. This overwrites the successful RPC response with error `-512`, causing NFS to mark the inode as invalid

### **Result**
The fault localization result has been written to `/workspace/fault_localization_result.json`:
- **Buggy file**: `net/sunrpc/sched.c`
- **Buggy function**: `__rpc_execute()`
- **Confidence**: 0.90 (90%)

### **The Fix**
The code at line 783-792 should check if `RPC_IS_QUEUED(task)` is still true before calling `rpc_exit()` with `-ERESTARTSYS`. If the task is no longer queued (meaning the RPC completed and the reply was received), the loop should continue to process the actual response instead of overwriting it with the signal error.

If the task is no longer queued, the RPC has completed and the loop should continue to process the actual response rather than overwriting it with -ERESTARTSYS."
