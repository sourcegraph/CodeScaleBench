## Summary of Findings

I've identified the key Go source files in prometheus/prometheus that handle the critical TSDB head block functionality:

### **Series Limit Enforcement**
- **tsdb/head.go**: Contains the `Head` struct with `numSeries` atomic counter tracking active series, incremented during series creation in `getOrCreateWithOptionalID()`

### **Out-of-Bounds & Out-of-Order Handling**
- **tsdb/head_append.go**: The primary implementation file containing:
  - **Fast-path out-of-bounds checks** in `Append()` (float samples, lines 426-430) and `AppendHistogram()` (histogram samples, lines 808-812)
  - **Detailed validation methods**:
    - `memSeries.appendable()` for float samples
    - `memSeries.appendableHistogram()` for histogram samples  
    - `memSeries.appendableFloatHistogram()` for float histogram samples
  - These methods determine whether samples are in-order, out-of-order, or out-of-bounds

- **tsdb/ooo_head.go**: Manages out-of-order sample storage with the `OOOChunk` structure, which maintains time-ascending order and prevents duplicate timestamps

- **storage/interface.go**: Defines the error types (`ErrOutOfBounds`, `ErrOutOfOrderSample`, `ErrTooOldSample`)

The crash with 'out of bound samples' at high cardinality occurs because the system becomes memory-constrained, and the out-of-bounds rejection mechanism may fail when dealing with large numbers of series combined with timestamp-variance in ingested data.
