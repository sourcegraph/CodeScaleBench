## Bug Report

**Bug ID**: 117651
**Summary**: Root NFS and autofs - mount disappears due to inode revalidate failed
**Regression**: Yes (works on 3.14.12, broken on 4.1.15)

After upgrading from 3.14.12 to 4.1.15 I've got strange behaviour with autofs.

It happens on diskless machine with root on NFS. There is autofs indirect mount on /storage, where NFS volumes are mounted on demand.

Everything mounts normally, but after mount expiration (during unmount) autofs mount disappears - that is, not only /storage/<volume> unmounts, but /storage is also unmounted. This happens quite often - usually on the very first expiration.

automount logs umount error in such cases.

Without autofs, mount/umount of the same NFS volume works correctly.

With NFS debug enabled, I've got following errors:

```
NFS reply getattr: -512
nfs_revalidate_inode: (0:15/6291480) getattr failed, error=-512
NFS: nfs_lookup_revalidate(/storage) is invalid
```

After inode revalidate failure all mounts on this inode are unmounted and so autofs mount disappears, while automount daemon itself continue to run.

tcpdump shows, that server replies to getattr without error, but client doesn't see this reply, and returning error instead.

I am ready to supply additional information.

## Fault Localization Summary

**Bug**: NFS mount disappears due to inode revalidate failure (regression from 4.1.15)

### Root Cause Identified

The bug is caused by **improper handling of transient RPC errors** in the NFS client code. Specifically:

**1. In `fs/nfs/nfs3proc.c` (nfs3_rpc_wrapper function, lines 30-42):**
- When the NFSv3 RPC layer retries after an EJUKEBOX error and gets interrupted by a signal, it returns `-ERESTARTSYS` (error code 512)
- This is a **transient, signal-based error** that should be handled specially, not treated as a permanent inode failure

**2. In `fs/nfs/inode.c` (__nfs_revalidate_inode function, lines 960-971):**
- Only `-ESTALE` errors get special handling (cache invalidation)
- **All other errors, including transient `-ERESTARTSYS`, are treated as permanent inode failures**
- This causes the inode to be invalidated and the dentry to be marked as invalid

### Impact

When autofs indirect mount expires and tries to unmount:
1. The NFS client needs to revalidate the `/storage` directory inode
2. The getattr RPC encounters a signal-interrupted EJUKEBOX retry → returns `-ERESTARTSYS`
3. `__nfs_revalidate_inode()` treats this as a permanent failure
4. The inode cache is invalidated
5. The parent directory's cache is zapped via `nfs_lookup_revalidate()` → `out_zap_parent`
6. The entire autofs mount point `/storage` disappears

### Files Requiring Fix

- `fs/nfs/nfs3proc.c` - Error handling in NFSv3 RPC wrapper
- `fs/nfs/inode.c` - Error categorization in inode revalidation

1. Retried internally without propagating to higher layers
2. Converted back to retriable conditions
