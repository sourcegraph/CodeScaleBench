You are reviewing a recently merged pull request that adds a `DisplayName<TValue>` component to Blazor. ## Context

1. **`src/Components/Web/src/Forms/DisplayName.cs`** — Component class: implements `IComponent`, accepts a `For` expression parameter, resolves the display name via `ExpressionMemberAccessor`, and renders it.

## Defects Identified

### 1. **Missing Render Optimization** (DisplayName.cs, line 39) — Severity: HIGH
The component always re-renders on every `SetParametersAsync` call without checking if the `For` parameter changed. This violates Blazor component conventions for render optimization. The unused `_previousFieldAccessor` field (line 20) was clearly intended for this optimization but was never implemented.

**Fix**: Check if `For` has changed before re-computing and re-rendering, following the pattern used in ValidationMessage.cs.

### 2. **Incomplete Hot Reload Cache Invalidation** (ExpressionMemberAccessor.cs, line 88) — Severity: CRITICAL
The `ClearCache()` method clears `_memberInfoCache` but not `_displayNameCache`. When hot reload applies metadata updates, cached display names become stale, causing the component to show outdated information even after attributes change during development.

**Fix**: Clear both caches in the `ClearCache()` method.

### 3. **Incorrect Attribute Precedence** (ExpressionMemberAccessor.cs, line 59) — Severity: HIGH
The code checks `[DisplayName]` before `[Display]`, contradicting the documentation which specifies "reading from `DisplayAttribute` or `DisplayNameAttribute`". The `Display` attribute is newer and more comprehensive, so it should take precedence when both are present.

**Fix**: Reorder the checks to evaluate `DisplayAttribute` first, then fall back to `DisplayNameAttribute`.

All proposed fixes are provided as unified diffs in the JSON report.
