## Task

Produce an architecture document for **Istio Pilot's Discovery Service** subsystem. The document must explain how Pilot serves xDS (Envoy Discovery Service) configuration to proxies, translates Istio's high-level configuration into Envoy-specific resources, integrates with service registries, and supports multi-cluster deployments. Do not simply list APIs — explain the **design**, **data flow**, and **extension points**.

## Scope

Your document must cover these four components and how they work together:

### 1. DiscoveryServer — The xDS gRPC Server
The central component implementing Pilot's gRPC interface for Envoy's xDS APIs. Explain:
- How it implements Envoy's xDS v3 protocol (ADS, incremental/state-of-the-world modes)
- The role of `adsClients` in tracking active proxy connections
- How configuration pushes are coordinated: debouncing, rate limiting, concurrent push limits
- Cache management for xDS resources (CDS, LDS, RDS, EDS)
- The lifecycle: initialization, handling new connections, shutdown

### 2. ADS and the Connection Model
The Aggregated Discovery Service stream handler. Explain:
- How `Stream()` establishes a bidirectional gRPC stream with an Envoy proxy
- The `Connection` type: wrapping the stream with proxy metadata and state
- Request processing: how `processRequest()` determines what configuration to send
- Push coordination: how `pushConnection()` transmits updates to a single proxy
- The `StartPush()` broadcast mechanism for fleet-wide updates

### 3. PushContext — Configuration Snapshot and Translation
The immutable configuration snapshot used during a push. Explain:
- What `PushContext` holds: services, destination rules, virtual services, sidecars, gateways
- How `InitContext()` rebuilds the push context when Istio config changes (CRD watches)
- Config translation: how Pilot converts Istio's `VirtualService` and `DestinationRule` into Envoy's `RouteConfiguration` and `Cluster` resources
- The relationship between push triggers (config changes, service registry updates) and push context regeneration
- Metrics and error tracking during push generation

### 4. Service Registry and Multi-Cluster Support
The service discovery layer feeding the DiscoveryServer. Explain:
- How `ServiceRegistry` (Kubernetes controller) watches K8s Services, Endpoints, and Pods
- The aggregate controller pattern for multi-registry support (K8s services + `ServiceEntry` resources)
- Multi-cluster support: how Pilot aggregates services from multiple clusters
- How endpoint updates (pod scale-up, health changes) trigger EDS pushes
- The relationship between the service registry and xDS: when does a service change trigger CDS vs. EDS?

DiscoveryServer \u2014 The xDS gRPC Server\n\n**Location:** `pilot/pkg/xds/discovery.go`\n\nThe `DiscoveryServer` is Pilot's central gRPC server that implements Envoy's xDS APIs (v3). It is the main coordinator for:\n- Managing active proxy connections\n- Debouncing and throttling configuration updates\n- Creating immutable snapshots (PushContext) when configuration changes\n- Pushing those snapshots to all connected proxies\n\n### Core Responsibilities\n\n#### 1.1 Connection Management\n\n```go\ntype DiscoveryServer struct {\n    Env *model.Environment\n\n    // Track all active xDS client connections\n    adsClients      map[string]*Connection\n    adsClientsMutex sync.RWMutex\n\n    // Rate limiting for new connection requests\n    RequestRateLimit *rate.Limiter\n\n    // Limit concurrent pushes to avoid overwhelming Pilot\n    concurrentPushLimit chan struct{}\n\n    // ... other fields\n}\n```\n\n**Key Operations:**\n- `addCon()` / `removeCon()` \u2014 Register/unregister proxy connections (tracked in `adsClients` map)\n- `Clients()` \u2014 Returns all fully initialized connections\n- `AllClients()` \u2014 Returns all connections (including uninitialized)\n- `WaitForRequestLimit()` \u2014 Enforces rate limiting on new connection requests (prevents thundering herd)\n\n#### 1.2 Configuration Push Coordination\n\nThe Discovery Server coordinates pushes through a multi-stage pipeline:\n\n1. **Event Collection (`pushChannel`):** Configuration updates from watchers (K8s controllers, config CRD watchers) arrive on `pushChannel`\n\n2. **Debouncing (`handleUpdates`):** The `debounce()` function prevents thrashing from rapid successive updates:\n   - `DebounceAfter`: Waits this long before processing (default ~100ms)\n   - `debounceMax`: Maximum wait (default ~10s); forces push if changes keep arriving\n   - Merges multiple update requests received while debouncing into a single push\n   - EDS (endpoint) updates can bypass debouncing with `enableEDSDebounce` flag\n\n3. **Push Context Initialization (`initPushContext`):** Once debounced, a full push generates a new immutable `PushContext`:\n   ```go\n   push := model.NewPushContext()\n   push.InitContext(s.Env, oldPushContext, req)\n   s.Env.SetPushContext(push)\n   ```\n\n4. **Queue and Send (`pushQueue`):** All pending connections are enqueued in `PushQueue`, which:\n   - Respects `concurrentPushLimit` (semaphore) to avoid overwhelming Pilot\n   - Calls `pushConnection()` for each proxy to generate and send xDS resources\n\n#### 1.3 xDS Protocol Implementation\n\nThe Discovery Server implements three xDS modes:\n\n- **Aggregated Discovery Service (ADS):** Bidirectional gRPC stream where both client and server can send messages\n  - Implemented in `ads.go` as `StreamAggregatedResources()`\n  - Single stream carries all resource types (CDS, LDS, RDS, EDS, SDS, etc.)\n  - Envoy can request specific resources or resource updates\n\n- **State-of-the-World (SotW):** Full resource set sent on each update (default)\n  - All matching resources for a type sent in one response\n  - Higher overhead but simpler\n\n- **Incremental/Delta:** Only changed resources sent\n  - Implemented in `delta.go` as `DeltaAggregatedResources()`\n  - Newer, more efficient protocol\n  - Uses the `Delta` field in `PushRequest` to track subscriptions\n\n#### 1.4 Cache Management\n\n```go\nCache model.XdsCache\n```\n\nThe DiscoveryServer maintains a cache of generated xDS resources:\n- **Purpose:** Avoid recomputing the same xDS resources for multiple proxies\n- **Invalidation:** On config changes, `dropCacheForRequest()` clears cache entries\n- **Lazy computation:** Generators compute resources on-demand and cache the results\n\n---\n\n## 2. ADS and the Connection Model\n\n**Location:** `pilot/pkg/xds/ads.go`\n\nThe ADS module handles the bidirectional gRPC stream between Pilot and each Envoy proxy, including request processing and push delivery.\n\n### 2.1 Stream Establishment (`Stream()` / `StreamAggregatedResources()`)\n\nWhen an Envoy proxy connects:\n\n```go\nfunc (s *DiscoveryServer) Stream(stream DiscoveryStream) error {\n    // 1. Rate limit check\n    if err := s.WaitForRequestLimit(stream.Context()); err != nil {\n        return status.Errorf(codes.ResourceExhausted, \"...\")\n    }\n\n    // 2. Authentication (mTLS identity verification)\n    ids, err := s.authenticate(ctx)\n\n    // 3. Ensure push context is ready\n    s.globalPushContext().InitContext(s.Env, nil, nil)\n\n    // 4. Create connection wrapper\n    con := newConnection(peerAddr, stream)\n    con.ids = ids\n    con.s = s\n\n    // 5. Delegate to xds.Stream for handling the connection lifecycle\n    return xds.Stream(con)\n}\n```\n\nThe connection lifecycle:\n1. **Initialize:** `initConnection()` extracts proxy metadata from the first request\n2. **Track:** `addCon()` registers the connection in `adsClients`\n3. **Serve:** `xds.Stream()` handles bidirectional message exchange\n4. **Close:** `removeCon()` cleans up when stream ends\n\n### 2.2 Connection Type and Proxy State\n\n```go\ntype Connection struct {\n    xds.Connection              // Base gRPC stream wrapper\n\n    node *core.Node             // Original Envoy node metadata\n    proxy *model.Proxy          // Parsed proxy identity and configuration\n\n    deltaStream DeltaDiscoveryStream  // For delta xDS (if used)\n    deltaReqChan chan *discovery.DeltaDiscoveryRequest\n\n    s *DiscoveryServer\n    ids []string                // Authenticated identities (from mTLS)\n}\n```\n\n**Proxy State:** Each connection manages a `model.Proxy` object containing:\n- Proxy metadata: ID, cluster, namespace, IP addresses\n- Service targets: Which services this proxy can access\n- Workload labels: Pod labels, locality information\n- Watched resources: Which xDS resource types this proxy is subscribed to\n- Cached push context: Last configuration version sent to this proxy\n\n### 2.3 Request Processing (`processRequest()`)\n\nWhen Envoy sends an xDS request:\n\n```go\nfunc (s *DiscoveryServer) processRequest(req *discovery.DiscoveryRequest, con *Connection) error {\n    stype := v3.GetShortType(req.TypeUrl)\n    log.Debugf(\"ADS:%s: REQ %s resources:%d\", stype, con.ID(), len(req.ResourceNames))\n\n    // Update proxy's watched resource for this type\n    shouldRespond, delta := xds.ShouldRespond(con.proxy, con.ID(), req)\n    if !shouldRespond {\n        return nil  // ACK'd by proxy, no new data needed\n    }\n\n    // Create a push request from this client request\n    request := &model.PushRequest{\n        Full:   true,\n        Push:   con.proxy.LastPushContext,\n        Reason: model.NewReasonStats(model.ProxyRequest),\n        Delta:  delta,  // Resource subscription delta\n        Forced: true,\n    }\n\n    // Generate and send the requested resource type\n    return s.pushXds(con, con.proxy.GetWatchedResource(req.TypeUrl), request)\n}\n```\n\n**Request Handling:**\n- Envoy sends `DiscoveryRequest` with: `TypeUrl` (resource type), `ResourceNames` (which resources), `VersionInfo` (last version ACK'd), `ResponseNonce` (request number)\n- Pilot checks if Envoy already has latest version; if so, ACKs and moves on\n- Otherwise, generates the requested resources and sends `DiscoveryResponse`\n\n### 2.4 Push Delivery (`pushConnection()`)\n\nWhen a configuration change triggers a fleet-wide push:\n\n```go\nfunc (s *DiscoveryServer) pushConnection(con *Connection, pushEv *Event) error {\n    pushRequest := pushEv.pushRequest\n\n    if pushRequest.Full {\n        // Update proxy state (service targets, sidecar scope, etc.)\n        s.computeProxyState(con.proxy, pushRequest)\n    }\n\n    // Check if this proxy actually needs updates\n    pushRequest, needsPush := s.ProxyNeedsPush(con.proxy, pushRequest)\n    if !needsPush {\n        return nil  // Skip push for this proxy\n    }\n\n    // Send all watched resource types in order: CDS \u2192 EDS \u2192 LDS \u2192 RDS \u2192 SDS \u2192 ...\n    wrl := con.watchedResourcesByOrder()\n    for _, w := range wrl {\n        if err := s.pushXds(con, w, pushRequest); err != nil {\n            return err\n        }\n    }\n    return nil\n}\n```\n\n**Push Ordering:** `PushOrder` defines the order xDS resources are sent:\n1. **CDS** (Clusters): Service discovery, load balancing endpoints\n2. **EDS** (Endpoints): Which instances (IPs/ports) are available\n3. **LDS** (Listeners): Which ports to listen on, filter chains\n4. **RDS** (Routes): HTTP routing rules\n5. **SDS** (Secrets): TLS certificates\n6. **Address, Workload, WorkloadAuthorization** (for ambient mesh)\n\nThis order is critical: Envoy requires CDS before EDS, and LDS before RDS.\n\n### 2.5 StartPush \u2014 Broadcast to All Proxies\n\n```go\nfunc (s *DiscoveryServer) StartPush(req *model.PushRequest) {\n    req.Start = time.Now()\n    for _, p := range s.AllClients() {\n        s.pushQueue.Enqueue(p, req)\n    }\n}\n```\n\nWhen a configuration change occurs, `StartPush()` enqueues the same `PushRequest` for every connected proxy. The `PushQueue` then distributes pushes, respecting concurrency limits.\n\n---\n\n## 3. PushContext \u2014 Configuration Snapshot and Translation\n\n**Location:** `pilot/pkg/model/push_context.go`\n\nThe `PushContext` is an immutable snapshot of the entire mesh configuration at a specific point in time. It is created when configuration changes and used during the push phase to generate xDS resources for all proxies.\n\n### 3.1 PushContext Data Structure\n\n```go\ntype PushContext struct {\n    // Service index\n    ServiceIndex serviceIndex\n\n    // Configuration indexes\n    virtualServiceIndex virtualServiceIndex\n    destinationRuleIndex destinationRuleIndex\n    sidecarIndex sidecarIndex\n    gatewayIndex gatewayIndex\n\n    // Policy indexes\n    AuthnPolicies *AuthenticationPolicies\n    AuthzPolicies *AuthorizationPolicies\n    Telemetry *Telemetries\n    ProxyConfigs *ProxyConfigs\n\n    // Mesh configuration\n    Mesh *meshconfig.MeshConfig\n    Networks *meshconfig.MeshNetworks\n\n    // Push metadata\n    PushVersion string  // Unique identifier for this push context\n\n    // Status tracking\n    ProxyStatus map[string]map[string]ProxyPushStatus\n}\n```\n\n**Purpose:** Rather than each resource generator re-querying the config store, they access pre-indexed data from `PushContext`. virtualServicesChanged = true\n        case kind.DestinationRule:\n            destinationRulesChanged = true\n        // ... etc\n        }\n    }\n\n    // Only rebuild affected indexes\n    if DiscoveryServer \u2014 The xDS gRPC Server\n\n**Location:** `pilot/pkg/xds/discovery.go`\n\nThe `DiscoveryServer` is Pilot's central gRPC server that implements Envoy's xDS APIs (v3). It is the main coordinator for:\n- Managing active proxy connections\n- Debouncing and throttling configuration updates\n- Creating immutable snapshots (PushContext) when configuration changes\n- Pushing those snapshots to all connected proxies\n\n### Core Responsibilities\n\n#### 1.1 Connection Management\n\n```go\ntype DiscoveryServer struct {\n    Env *model.Environment\n\n    // Track all active xDS client connections\n    adsClients      map[string]*Connection\n    adsClientsMutex sync.RWMutex\n\n    // Rate limiting for new connection requests\n    RequestRateLimit *rate.Limiter\n\n    // Limit concurrent pushes to avoid overwhelming Pilot\n    concurrentPushLimit chan struct{}\n\n    // ... other fields\n}\n```\n\n**Key Operations:**\n- `addCon()` / `removeCon()` \u2014 Register/unregister proxy connections (tracked in `adsClients` map)\n- `Clients()` \u2014 Returns all fully initialized connections\n- `AllClients()` \u2014 Returns all connections (including uninitialized)\n- `WaitForRequestLimit()` \u2014 Enforces rate limiting on new connection requests (prevents thundering herd)\n\n#### 1.2 Configuration Push Coordination\n\nThe Discovery Server coordinates pushes through a multi-stage pipeline:\n\n1. **Event Collection (`pushChannel`):** Configuration updates from watchers (K8s controllers, config CRD watchers) arrive on `pushChannel`\n\n2. **Debouncing (`handleUpdates`):** The `debounce()` function prevents thrashing from rapid successive updates:\n   - `DebounceAfter`: Waits this long before processing (default ~100ms)\n   - `debounceMax`: Maximum wait (default ~10s); forces push if changes keep arriving\n   - Merges multiple update requests received while debouncing into a single push\n   - EDS (endpoint) updates can bypass debouncing with `enableEDSDebounce` flag\n\n3. **Push Context Initialization (`initPushContext`):** Once debounced, a full push generates a new immutable `PushContext`:\n   ```go\n   push := model.NewPushContext()\n   push.InitContext(s.Env, oldPushContext, req)\n   s.Env.SetPushContext(push)\n   ```\n\n4. **Queue and Send (`pushQueue`):** All pending connections are enqueued in `PushQueue`, which:\n   - Respects `concurrentPushLimit` (semaphore) to avoid overwhelming Pilot\n   - Calls `pushConnection()` for each proxy to generate and send xDS resources\n\n#### 1.3 xDS Protocol Implementation\n\nThe Discovery Server implements three xDS modes:\n\n- **Aggregated Discovery Service (ADS):** Bidirectional gRPC stream where both client and server can send messages\n  - Implemented in `ads.go` as `StreamAggregatedResources()`\n  - Single stream carries all resource types (CDS, LDS, RDS, EDS, SDS, etc.)\n  - Envoy can request specific resources or resource updates\n\n- **State-of-the-World (SotW):** Full resource set sent on each update (default)\n  - All matching resources for a type sent in one response\n  - Higher overhead but simpler\n\n- **Incremental/Delta:** Only changed resources sent\n  - Implemented in `delta.go` as `DeltaAggregatedResources()`\n  - Newer, more efficient protocol\n  - Uses the `Delta` field in `PushRequest` to track subscriptions\n\n#### 1.4 Cache Management\n\n```go\nCache model.XdsCache\n```\n\nThe DiscoveryServer maintains a cache of generated xDS resources:\n- **Purpose:** Avoid recomputing the same xDS resources for multiple proxies\n- **Invalidation:** On config changes, `dropCacheForRequest()` clears cache entries\n- **Lazy computation:** Generators compute resources on-demand and cache the results\n\n---\n\n## 2. ADS and the Connection Model\n\n**Location:** `pilot/pkg/xds/ads.go`\n\nThe ADS module handles the bidirectional gRPC stream between Pilot and each Envoy proxy, including request processing and push delivery.\n\n### 2.1 Stream Establishment (`Stream()` / `StreamAggregatedResources()`)\n\nWhen an Envoy proxy connects:\n\n```go\nfunc (s *DiscoveryServer) Stream(stream DiscoveryStream) error {\n    // 1. Rate limit check\n    if err := s.WaitForRequestLimit(stream.Context()); err != nil {\n        return status.Errorf(codes.ResourceExhausted, \"...\")\n    }\n\n    // 2. Authentication (mTLS identity verification)\n    ids, err := s.authenticate(ctx)\n\n    // 3. Ensure push context is ready\n    s.globalPushContext().InitContext(s.Env, nil, nil)\n\n    // 4. Create connection wrapper\n    con := newConnection(peerAddr, stream)\n    con.ids = ids\n    con.s = s\n\n    // 5. Delegate to xds.Stream for handling the connection lifecycle\n    return xds.Stream(con)\n}\n```\n\nThe connection lifecycle:\n1. **Initialize:** `initConnection()` extracts proxy metadata from the first request\n2. **Track:** `addCon()` registers the connection in `adsClients`\n3. **Serve:** `xds.Stream()` handles bidirectional message exchange\n4. **Close:** `removeCon()` cleans up when stream ends\n\n### 2.2 Connection Type and Proxy State\n\n```go\ntype Connection struct {\n    xds.Connection              // Base gRPC stream wrapper\n\n    node *core.Node             // Original Envoy node metadata\n    proxy *model.Proxy          // Parsed proxy identity and configuration\n\n    deltaStream DeltaDiscoveryStream  // For delta xDS (if used)\n    deltaReqChan chan *discovery.DeltaDiscoveryRequest\n\n    s *DiscoveryServer\n    ids []string                // Authenticated identities (from mTLS)\n}\n```\n\n**Proxy State:** Each connection manages a `model.Proxy` object containing:\n- Proxy metadata: ID, cluster, namespace, IP addresses\n- Service targets: Which services this proxy can access\n- Workload labels: Pod labels, locality information\n- Watched resources: Which xDS resource types this proxy is subscribed to\n- Cached push context: Last configuration version sent to this proxy\n\n### 2.3 Request Processing (`processRequest()`)\n\nWhen Envoy sends an xDS request:\n\n```go\nfunc (s *DiscoveryServer) processRequest(req *discovery.DiscoveryRequest, con *Connection) error {\n    stype := v3.GetShortType(req.TypeUrl)\n    log.Debugf(\"ADS:%s: REQ %s resources:%d\", stype, con.ID(), len(req.ResourceNames))\n\n    // Update proxy's watched resource for this type\n    shouldRespond, delta := xds.ShouldRespond(con.proxy, con.ID(), req)\n    if !shouldRespond {\n        return nil  // ACK'd by proxy, no new data needed\n    }\n\n    // Create a push request from this client request\n    request := &model.PushRequest{\n        Full:   true,\n        Push:   con.proxy.LastPushContext,\n        Reason: model.NewReasonStats(model.ProxyRequest),\n        Delta:  delta,  // Resource subscription delta\n        Forced: true,\n    }\n\n    // Generate and send the requested resource type\n    return s.pushXds(con, con.proxy.GetWatchedResource(req.TypeUrl), request)\n}\n```\n\n**Request Handling:**\n- Envoy sends `DiscoveryRequest` with: `TypeUrl` (resource type), `ResourceNames` (which resources), `VersionInfo` (last version ACK'd), `ResponseNonce` (request number)\n- Pilot checks if Envoy already has latest version; if so, ACKs and moves on\n- Otherwise, generates the requested resources and sends `DiscoveryResponse`\n\n### 2.4 Push Delivery (`pushConnection()`)\n\nWhen a configuration change triggers a fleet-wide push:\n\n```go\nfunc (s *DiscoveryServer) pushConnection(con *Connection, pushEv *Event) error {\n    pushRequest := pushEv.pushRequest\n\n    if pushRequest.Full {\n        // Update proxy state (service targets, sidecar scope, etc.)\n        s.computeProxyState(con.proxy, pushRequest)\n    }\n\n    // Check if this proxy actually needs updates\n    pushRequest, needsPush := s.ProxyNeedsPush(con.proxy, pushRequest)\n    if !needsPush {\n        return nil  // Skip push for this proxy\n    }\n\n    // Send all watched resource types in order: CDS \u2192 EDS \u2192 LDS \u2192 RDS \u2192 SDS \u2192 ...\n    wrl := con.watchedResourcesByOrder()\n    for _, w := range wrl {\n        if err := s.pushXds(con, w, pushRequest); err != nil {\n            return err\n        }\n    }\n    return nil\n}\n```\n\n**Push Ordering:** `PushOrder` defines the order xDS resources are sent:\n1. **CDS** (Clusters): Service discovery, load balancing endpoints\n2. **EDS** (Endpoints): Which instances (IPs/ports) are available\n3. **LDS** (Listeners): Which ports to listen on, filter chains\n4. **RDS** (Routes): HTTP routing rules\n5. **SDS** (Secrets): TLS certificates\n6. **Address, Workload, WorkloadAuthorization** (for ambient mesh)\n\nThis order is critical: Envoy requires CDS before EDS, and LDS before RDS.\n\n### 2.5 StartPush \u2014 Broadcast to All Proxies\n\n```go\nfunc (s *DiscoveryServer) StartPush(req *model.PushRequest) {\n    req.Start = time.Now()\n    for _, p := range s.AllClients() {\n        s.pushQueue.Enqueue(p, req)\n    }\n}\n```\n\nWhen a configuration change occurs, `StartPush()` enqueues the same `PushRequest` for every connected proxy. The `PushQueue` then distributes pushes, respecting concurrency limits.\n\n---\n\n## 3. PushContext \u2014 Configuration Snapshot and Translation\n\n**Location:** `pilot/pkg/model/push_context.go`\n\nThe `PushContext` is an immutable snapshot of the entire mesh configuration at a specific point in time. It is created when configuration changes and used during the push phase to generate xDS resources for all proxies.\n\n### 3.1 PushContext Data Structure\n\n```go\ntype PushContext struct {\n    // Service index\n    ServiceIndex serviceIndex\n\n    // Configuration indexes\n    virtualServiceIndex virtualServiceIndex\n    destinationRuleIndex destinationRuleIndex\n    sidecarIndex sidecarIndex\n    gatewayIndex gatewayIndex\n\n    // Policy indexes\n    AuthnPolicies *AuthenticationPolicies\n    AuthzPolicies *AuthorizationPolicies\n    Telemetry *Telemetries\n    ProxyConfigs *ProxyConfigs\n\n    // Mesh configuration\n    Mesh *meshconfig.MeshConfig\n    Networks *meshconfig.MeshNetworks\n\n    // Push metadata\n    PushVersion string  // Unique identifier for this push context\n\n    // Status tracking\n    ProxyStatus map[string]map[string]ProxyPushStatus\n}\n```\n\n**Purpose:** Rather than each resource generator re-querying the config store, they access pre-indexed data from `PushContext`. Called when:\n- Initial push context creation\n- Full push triggered (e.g., due to mesh configuration change)\n- Forced push\n\n```go\nfunc (ps *PushContext) createNewContext(env *Environment) {\n    ps.initServiceRegistry(env, nil)      // Load services from K8s + ServiceEntry\n    ps.initVirtualServices(env)           // Index VirtualServices\n    ps.initDestinationRules(env)          // Index DestinationRules\n    ps.initAuthnPolicies(env)             // Authentication policies\n    ps.initAuthorizationPolicies(env)     // Authorization policies\n    ps.initTelemetry(env)                 // Telemetry configuration\n    ps.initWasmPlugins(env)               // WASM plugin configuration\n    ps.initEnvoyFilters(env, nil, nil)   // Envoy filters\n    ps.initGateways(env)                  // Gateway configurations\n    ps.initSidecarScopes(env)             // Sidecar scopes (MUST be last)\n}\n```\n\n**Incremental Update (`updateContext`):**\nOnly rebuilds indexes for changed resource types. virtualServicesChanged = true\n        case kind.DestinationRule:\n            destinationRulesChanged = true\n        // ... etc\n        }\n    }\n\n    // Only rebuild affected indexes\n    if # Architecture Document: Istio Pilot's Discovery Service

## Overview

Istio Pilot's Discovery Service is the core xDS (Envoy Discovery Service) implementation that translates high-level Istio mesh configuration into low-level Envoy proxy configuration. It continuously watches for changes in the mesh (service registries, Istio configuration resources, endpoint updates) and pushes the translated configuration to all connected proxies via gRPC xDS streams.

The Discovery Service follows a server-push model: rather than waiting for clients to request updates, Pilot proactively sends configuration changes to all connected proxies. This ensures low-latency propagation of configuration changes across the mesh.
