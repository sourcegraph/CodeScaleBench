Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Conflict Scenarios and Resolution\n\n**Scenario 1: Non-Force Conflict**\n\n```\nLive object (in etcd):\n- spec.replicas = 5 (owned by \"scaling-controller\")\n\nUser tries to apply:\n- spec.replicas = 3 (as \"kubectl\")\n\nResult:\n- Error: conflict with \"scaling-controller\" on field \"spec.replicas\"\n- User must either:\n  a) Use --force-conflicts (not recommended; overwrites other controller)\n  b) Contact scaling-controller owner to release field\n  c) Coordinate with other controller\n```\n\n**Scenario 2: Force Conflict Resolution**\n\n```\nSame setup, but user applies with force=true:\n\nResult:\n- Merge succeeds\n- \"scaling-controller\" loses ownership of spec.replicas\n- \"kubectl\" now owns it\n- scaling-controller sees object changed; may react unexpectedly (race condition risk)\n```\n\n**Scenario 3: Allowed Conflict (LastAppliedManager Bridge)**\n\n```\nLive object (created with kubectl client-side apply, not SSA):\n- spec.replicas = 5\n- kubectl.kubernetes.io/last-applied-configuration annotation contains\n  metadata: spec.replicas: 3\n\nManager \"kubectl\" tries to apply:\n- spec.replicas = 2\n\nWithout LastAppliedManager:\n- Error: conflict (no managedFields entry, so treated as unowned)\n\nWith LastAppliedManager:\n- LastAppliedManager detects:\n  - Field is in last-applied annotation\n  - Field is unchanged from when last-applied was set\n  - Therefore \"owned\" by CSA\n- Force-applies anyway\n- Seamless upgrade path for resources created with CSA\n```\n\n### 4.2 Design Limits\n\n#### **Manager Entry Limit: DefaultMaxUpdateManagers = 10**\n- **Location**: `staging/src/k8s.io/apimachinery/pkg/util/managedfields/internal/fieldmanager.go:35`\n- **Purpose**: Prevent unbounded growth of managedFields\n- **Behavior**: When entry count exceeds 10, oldest entries are merged into newest entry of same type\n- **Implication**: If more than 10 different managers update via Update (non-Apply), oldest are dropped\n- **Risk**: Very distributed systems with many different CRDs/operators may exceed this; mergence loses granular history\n\n#### **Field Granularity**\n- **Limitation**: SSA tracks ownership at field level, not sub-field\n  - Can track \"spec.containers[0]\" via key merge\n  - Cannot track individual container properties separately\n- **Implication**: If two managers manage the same array element, they conflict even on different properties\n- **Mitigation**: Use different array elements or coordinate via merge keys\n\n#### **No Cross-Version Field Ownership**\n- **Limitation**: Managed fields track APIVersion; different API versions have separate entries\n- **Implication**: If object stored as v1 in live, converted to v1beta1 for apply, causes version mismatch\n- **Risk**: Version conversion bugs can lose field ownership\n- **Mitigation**: API server handles version conversion carefully\n\n### 4.3 Performance Characteristics\n\n#### **Apply Operation Cost**\n- **Type Conversion** (O(n) in object size): Converting runtime object \u2194 typed representation\n- **Merge Computation** (O(n) in field count): structured-merge-diff algorithm\n- **managedFields Encoding** (O(m) in manager count): Encoding field ownership trie\n- **Impact**: Large objects (>1MB) or objects with many fields can slow applies\n- **Optimization**: Cache typed representations; reuse schemas\n\n#### **Field Extraction Cost**\n- **Decoding managedFields** (O(m) in manager count and field count)\n- **Typed value extraction** (O(n) in total fields)\n- **Result marshalling** (O(n))\n- **Impact**: Extracting then reapplying large Deployments expensive; use in-place updates when possible\n\n### 4.4 Design Tradeoffs\n\n| Tradeoff | Choice | Rationale |\n|----------|--------|-----------|\n| **Server-side vs Client-side Merge** | Server-side | Server has authority; prevents distributed merge conflicts. Client would need to coordinate. |\n| **Granularity: Field vs Type** | Field-level | Allows multiple managers same object; type-level would prevent coexistence. Downside: more metadata. |\n| **Pointer-based ApplyConfig** | All fields pointers | Distinguishes \"not set\" from \"set to zero\". Trade: less intuitive API. |\n| **Managed Fields Storage** | Object metadata | Simplifies schema; no separate storage layer. Trade: bloats object size; immutable during admission. |\n| **Conflict Resolution: Auto-merge vs Error** | Error (with force flag) | Safe default; prevents silent overwrites. Force flag available for intentional takeover. |\n| **CSA Migration Strategy** | Decorated manager (LastAppliedManager) | Smooth upgrade without manual migration. Trade: special case logic. |\n| **structured-merge-diff Dependency** | External library | Shared logic across client/server; independent from Kubernetes release cycle. Trade: tight coupling. |\n\n---\n\n## 5. Server-Side Apply (SSA) and ApplyConfigurations represent a fundamental shift in how Kubernetes manages declarative configuration and multi-actor field ownership.
