## 1. Architecture Overview

### 1.1 Connection Pool Organization

Envoy's connection pool management is organized around **clusters and upstream hosts**. Each upstream host maintains its own set of connection pools, segregated by HTTP protocol version and resource priority level. The pool manages the lifecycle of all connections to that host.

#### Per-Cluster and Per-Thread Model
- Connection pools are created **per-cluster, per-upstream-host, per-thread** (thread-local)
- Each Envoy worker thread maintains independent pools to avoid synchronization overhead
- Cluster manager (`source/common/upstream/cluster_manager_impl.cc`) allocates pools via `allocateConnPool()` and `allocateTcpConnPool()`

#### Key Components
1. **HTTP Connection Pools** (`source/common/http/conn_pool_base.cc`):

   - HTTP/1.1 implementation: `source/common/http/http1/conn_pool.cc`
   - HTTP/2 implementation: `source/common/http/http2/conn_pool.cc`
