## Key Findings

**Root Cause:** The v38 dashboard migration silently drops table panel field override configuration when panels lack explicit `fieldConfig.defaults.custom` objects.

**Primary Bug Location:** `apps/dashboard/pkg/migration/schemaversion/v38.go`, line 116
- The conditional `if defaults, ok := fieldConfig["defaults"].(map[string]interface{}); ok` checks if defaults exists and is a map
- When defaults is missing or malformed, this check fails, and while `migrateOverrides()` is still called, the fieldConfig structure is left in an invalid state

**Secondary Bug Location:** `apps/dashboard/pkg/migration/frontend_defaults.go`, line 611  
- Uses the same flawed conditional pattern
- When a panel originally had `fieldConfig.defaults.custom`, the cleanup code tries to preserve it via `_originallyHadFieldConfigCustom` flag
- If the conditional fails (defaults missing), the code falls through to `delete(panel, prop)`, **permanently removing the entire fieldConfig**
- This causes loss of column widths, text alignment, and other field override configurations

**The Flawed Conditional Expression:**
```go
if defaults, ok := fieldConfig["defaults"].(map[string]interface{}); ok {
```

This fails silently when `fieldConfig["defaults"]` is absent or not a map, causing preservation logic to be skipped and the entire fieldConfig to be deleted.
