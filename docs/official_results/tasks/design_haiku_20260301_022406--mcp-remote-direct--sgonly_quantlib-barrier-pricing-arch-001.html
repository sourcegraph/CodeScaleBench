## Summary of Findings

The analysis traces the complete pricing chain for barrier options in QuantLib:

1. **Entry Point**: `BarrierOption.NPV()` → `Instrument::NPV()` → `Instrument::calculate()`

2. **Lazy Evaluation Chain**: 
   - `Instrument::calculate()` → `LazyObject::calculate()` → `performCalculations()`
   - This implements lazy evaluation with result caching and automatic invalidation via observer pattern

3. **Engine Dispatch**:
   - `Instrument::performCalculations()` calls `engine->calculate()`, delegating to either:
     - **AnalyticBarrierEngine**: Closed-form Black-Scholes style pricing
     - **MCBarrierEngine**: Monte Carlo simulation with Brownian bridge correction

4. **Monte Carlo Pipeline**:
   - `McSimulation::calculate()` manages the sampling loop
   - `PathGenerator` evolves `GeneralizedBlackScholesProcess` through time grid using:
     - Gaussian sequence generators
     - Drift/diffusion from term structures (rates and volatility)
     - Brownian bridge for variance reduction
   - `BarrierPathPricer` prices each path by checking barrier crossing and computing option payoff

5. **Term Structure Integration**:
   - `GeneralizedBlackScholesProcess` holds handles to `YieldTermStructure` and `BlackVolTermStructure`
   - During Monte Carlo path evolution, each step queries term structures for forward rates and implied volatilities
   - Changes in market data automatically invalidate cached calculations via observer notifications

The document details the design patterns (LazyObject, Strategy, Template Method, Observer), component responsibilities, and data flow throughout the system
