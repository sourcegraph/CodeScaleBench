## Bug Report

**Bug ID**: 207835
**Summary**: ACPI video backlight brightness control does not work on Acer TravelMate 5735Z by default

On the Intel GM45/GMA 4500MHD laptop with Debian Sid/unstable and Linux 5.6.7 and GNOME Shell 3.36.2, trying to change the brightness of the internal display using the function keys, GNOME shows the OSD, which seems to have five levels (from acer_wmi?), but the actual brightness does *not* change. There is only `/sys/class/backlight/intel_backlight/brightness` though, and writing values to it seems to work.

Booting the system with `acpi_backlight=vendor`, exposes `/sys/devices/platform/acer-wmi/backlight/acer-wmi/brightness`, but the behavior is the same.

Booting the system with `acpi_backlight=native` or `acpi_backlight=native`, changing the brightness with the function keys works, and there now seem to be 15 (or 16) levels.

## Summary of Fault Localization

I've analyzed the ACPI video backlight control failure on the Acer TravelMate 5735Z and identified the root cause:

### **Buggy File**: `drivers/platform/x86/acer-wmi.c`

### **Buggy Data Structure**: `video_vendor_dmi_table` (lines 584-637)

### **Root Cause Analysis**:

1. **Problem**: The Acer TravelMate 5735Z is **missing** from the `video_vendor_dmi_table` in acer-wmi.c
   
2. **Why this breaks brightness control**:
   - The `video_vendor_dmi_table` contains DMI entries for Acer machines that should use native/ACPI video backlight instead of acer-wmi's interface
   - When a device is not in this table, acer-wmi doesn't call `video_set_backlight_video_vendor` to disable brightness capability
   - The device falls back to `quirk_unknown` which has `brightness = 0` (default)
   - Since `brightness >= 0` (line 926), the `ACER_CAP_BRIGHTNESS` capability is enabled (line 927)
   - acer-wmi registers a brightness backlight device
   - This interferes with the native intel_backlight driver

3. **Evidence**:
   - User reports `acpi_backlight=native` parameter **works perfectly** with 15-16 brightness levels
   - Similar Acer TravelMate models (4750, 5760, etc.) are already in the `video_vendor_dmi_table`
   - The pattern clearly shows these models need the native backlight handler

4. **The Fix**:
   - Add an entry for TravelMate 5735Z to `video_vendor_dmi_table` with callback `video_set_backlight_video_vendor`
   - This will disable `ACER_CAP_BRIGHTNESS` capability for this model
   - acer-wmi won't interfere with native backlight
   - System will properly use intel_backlight/ACPI video for brightness control
