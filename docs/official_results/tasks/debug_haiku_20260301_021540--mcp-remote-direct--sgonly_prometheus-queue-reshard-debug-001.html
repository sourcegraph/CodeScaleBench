## Investigation Summary

**Root Cause Found**: A metrics update race condition during remote-write queue resharding.

### The Core Issue

When Prometheus reshards queues (due to target discovery changes), the system cancels the context to force shutdown of old shards. However, there's a mismatch between the code's intent and implementation:

1. **The intent** (line 1790-1791): "When there is resharding, we cancel the context for this queue, which means the data is not sent. So we exit early to **not update the metrics**."

2. **The reality** (line 1648): The calling function `sendSamples()` **always calls `updateMetrics()`** regardless of whether the context was cancelled.

3. **The consequence**: Batches that were in-flight when hard shutdown occurred are **never sent** to the remote endpoint, but the pending sample counters are still decremented. This causes `prometheus_remote_storage_samples_pending` to get stuck at inaccurate values.

### Why It's Intermittent

The bug only manifests when:
- Resharding occurs during active batch sends
- The flush deadline expires before pending batches can complete
- Network latency causes batches to be in-flight at the exact moment of resharding

### Key Evidence

- **File**: `storage/remote/queue_manager.go`
- **Functions**: `sendSamplesWithBackoff()`, `sendSamples()`, `updateMetrics()`
- **Mechanism**: Context cancellation during `reshardLoop()` → `stop()` → hard shutdown
- **Affected Metrics**: All `pending` and potentially `failed` counters

The full investigation report has been written to `/logs/agent/investigation.md` with detailed code references, timing analysis, diagnostic steps, and recommended fixes.

# Investigation Report: Remote-Write Queue Resharding Failure

## Summary

During remote-write queue resharding triggered by target discovery changes, samples pending in flight batches are lost when the resharding hard shutdown cancels the context mid-send. However, metric update logic still executes, incorrectly decrementing `prometheus_remote_storage_samples_pending` despite the samples never reaching the remote endpoint. This causes pending counters to become inaccurate and manifests as shards stalling with samples stuck in pending state.

---

## Root Cause

**File:** `storage/remote/queue_manager.go`
**Mechanism:** Context-cancellation race condition during batch sending in resharding

### The Bug

1. **Lines 1789-1792** in `sendSamplesWithBackoff()`: When the context is cancelled during resharding, the function returns early with `context.Canceled` error:
