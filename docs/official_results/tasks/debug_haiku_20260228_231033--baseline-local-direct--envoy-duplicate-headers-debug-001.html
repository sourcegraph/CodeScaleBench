# Investigation: Duplicate Response Headers in Envoy Filter Pipeline

**Repository:** envoyproxy/envoy

## Scenario

An Envoy proxy operator reports that response headers configured via `response_headers_to_add` in route configuration are being **duplicated** in certain responses. Specifically, when the router filter generates a local reply (e.g., upstream timeout, connection failure, or request too large), custom response headers like `x-request-id-echo` appear twice in the HTTP response.

The issue is intermittent — it only affects responses where the router itself generates the reply (local replies) rather than forwarding an upstream response. Normal proxied responses have headers added exactly once.

Access log snippet showing the problem (using `%RESPONSE_CODE_DETAILS%` formatter):

```
[2025-08-15T10:23:45.001Z] "GET /api/v1/data HTTP/1.1" 504 UT
response_code_details=upstream_response_timeout
x-custom-trace: abc123
x-custom-trace: abc123
```

The header `x-custom-trace` appears twice. The route config has:

```yaml
response_headers_to_add:
  - header:
      key: "x-custom-trace"
      value: "%REQ(x-request-id)%"
    append_action: OVERWRITE_IF_EXISTS_OR_ADD
```

Despite using `OVERWRITE_IF_EXISTS_OR_ADD`, the header is duplicated on local replies but not on proxied upstream responses.

## Key Findings

**Root Cause**: The `finalizeResponseHeaders()` method is called **twice** when generating local replies:

1. **First call**: Explicit invocation in filter_manager.cc's local reply construction (lines 1123-1129, 1072-1078, 1156-1162)
2. **Second call**: Implicit invocation when the `modify_headers_` callback is executed, which already contains its own `finalizeResponseHeaders()` call embedded by the router filter (router.cc:453)

**Why it doesn't affect upstream responses**: For upstream responses, the router filter calls `modify_headers_(*headers)` directly without a wrapper that adds another finalization step. But for local replies, the filter manager creates a lambda that explicitly finalizes headers AND then calls `modify_headers_`, which finalizes again.

**The consequence**: Depending on the `append_action` configuration:
- `APPEND_IF_EXISTS_OR_ADD` → Headers are appended on the second call, creating duplicates
- `OVERWRITE_IF_EXISTS_OR_ADD` → Headers are overwritten (less visible but still inefficient)

# Investigation Report: Duplicate Response Headers in Envoy Filter Pipeline

## Summary

Response headers configured via `response_headers_to_add` in route configuration are duplicated in local replies generated by the router filter. This occurs because `finalizeResponseHeaders()` is called twice in the local reply code path—once explicitly in the filter manager and again implicitly when the `modify_headers_` callback (which itself contains a `finalizeResponseHeaders()` call) is invoked.

## Root Cause

The root cause is the **double invocation of `finalizeResponseHeaders()`** in the local reply handling path. This happens because:

1. **In router.cc (lines 444-461)**: The router filter creates a `modify_headers_` callback that includes an explicit call to `finalizeResponseHeaders()`:
   ```cpp
   modify_headers_ = [this](Http::ResponseHeaderMap& headers) {
     // ... other code ...
     route_entry_->finalizeResponseHeaders(headers, callbacks_->streamInfo());
   };
   ```

2. **In filter_manager.cc (lines 1123-1129, 1072-1078, 1156-1162)**: When constructing the local reply, the filter manager creates a lambda that calls `finalizeResponseHeaders()` directly AND then calls the `modify_headers_` callback:
   ```cpp
   [this, modify_headers](ResponseHeaderMap& headers) -> void {
     if (streamInfo().route() && streamInfo().route()->routeEntry()) {
       streamInfo().route()->routeEntry()->finalizeResponseHeaders(headers, streamInfo());  // FIRST
     }
     if (modify_headers) {
       modify_headers(headers);  // SECOND (contains finalizeResponseHeaders internally)
     }
   }
   ```

3. **In utility.cc (line 719)**: This lambda is invoked during the local reply preparation, causing both calls to `finalizeResponseHeaders()` to execute on the same response headers.
