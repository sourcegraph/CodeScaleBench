### Core Implementation
The middleware system uses an **onion-like stacking pattern**:
1. **Construction**: `BaseHandler.load_middleware()` iterates through `settings.MIDDLEWARE` in **reverse order**, creating a nested callable chain where the first middleware becomes the outermost layer
2. **Request Flow**: Requests pass inward (top-to-bottom of MIDDLEWARE list) with each middleware able to execute `process_request` logic
3. **Response Flow**: Responses travel outward (bottom-to-top) with each middleware executing `process_response` logic
4. **Exception Safety**: Each middleware is wrapped with `convert_exception_to_response()` to immediately convert exceptions to HTTP responses

### Comparison with WSGI
Django's MIDDLEWARE differs from traditional WSGI middleware stacking:
- **Short-circuiting**: Only outer middleware see the response when a middleware short-circuits
- **Exception handling**: Exceptions are immediately converted and passed as responses, not propagated
- **Execution pattern**: True onion pattern ensures request/response phases are symmetric
