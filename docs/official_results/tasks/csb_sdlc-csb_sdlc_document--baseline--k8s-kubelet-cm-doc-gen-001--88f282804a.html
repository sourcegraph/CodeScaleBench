CPU Pinning and Latency\n\n**Impact**: Static CPU policy (pinning) reduces latency variance for latency-critical workloads\n\n**Behavior**:\n- Guaranteed pods get exclusive CPU cores (no sharing with other pods)\n- Cores reserved via `--reserved-cpus` for system daemons\n- Reconciliation applies pinning every 5s (configurable)\n\n**Operational considerations:**\n- **Overallocation risk**: If too many guaranteed pods, run out of exclusive CPUs \u2192 admission failures\n- **Planning**: Size `--reserved-cpus` to account for kubelet, container runtime, system services\n- **Monitoring**: Track CPU allocation per pod; alert if utilization low (wasted cores)\n\n**Tuning parameters:**\n- `--cpu-manager-policy`: Change from 'none' to 'static' for pinning\n- `--cpu-manager-reconcile-period`: Lower value (faster reconciliation) increases CRI chatter\n- `--reserved-cpus`: Reserve more CPUs for system \u2192 fewer CPUs available to pods\n\n**File**: `cpumanager/cpu_manager.go` manager struct (lines 105-150)\n\n### 2. NUMA Topology Complexity\n\n**Impact**: Memory affinity and device locality decisions must account for NUMA topology\n\n**Behavior**:\n- Topology manager aggregates hints from all managers\n- Single NUMA node preference reduces memory latency (local access faster than remote)\n- Device locality affects throughput (GPUs, NICs attached to specific NUMA nodes)\n\n**Operational considerations:**\n- **Policy selection**:\n  - `none`: Safe default; simplest to understand and troubleshoot\n  - `best-effort`: Improves latency without hard guarantees; recommended for most clusters\n  - `restricted`: Strict resource co-location; only for latency-sensitive workloads\n  - `single-numa-node`: Most restrictive; rarely needed\n- **Asymmetric NUMA**: Some systems have unbalanced NUMA (different node sizes) \u2192 allocation algorithms may struggle\n- **Observability**: NUMA topology not directly visible to users; requires `numactl` or `/proc/cpuinfo` inspection\n\n**Monitoring**:\n- Track pod admissions by NUMA scope (single-node vs multi-node allocations)\n- Monitor memory latency if performance regression observed\n\n**File**: `topologymanager/topology_manager.go` (aggregation and policy enforcement)\n\n### 3. Device Plugin Reliability\n\n**Impact**: Device allocation failures block pod admission; device loss affects running pods\n\n**Failure scenarios:**\n- Plugin crashes \u2192 devices reported as unavailable; kubelet may deallocate and admit new pods\n- Plugin slow to respond \u2192 pod admission latency increases\n- Plugin state loss \u2192 devices appear reallocated incorrectly across restarts\n\n**Operational practices:**\n- **Plugin health monitoring**: Implement plugin-level health checks (heartbeat, resource validation)\n- **Redundancy**: Run plugin replicas with shared state (if plugin design permits)\n- **Graceful degradation**: Device plugin should report partial health if some devices healthy\n- **Resource reservation**: Reserve devices for system/monitoring pods; don't over-commit\n\n**Kubelet health checks**:\n- Kubelet monitors plugin socket availability\n- Unresponsive plugin may be evicted by plugin manager\n- Requires manual plugin restart\n\n**File**: `devicemanager/manager.go` plugin communication (lines 62-100)\n\n### 4. Dynamic Resource Allocation (DRA) Claim Lifecycle\n\n**Impact**: Claim preparation failures prevent pod startup; claim state must survive kubelet restart\n\n**Failure scenarios:**\n- Plugin unavailable during PrepareResources \u2192 pod stuck preparing indefinitely\n- Claim preparation succeeds but plugin crashes \u2192 claim state in limbo\n- Multiple kubelet instances claim same resource \u2192 conflict\n\n**Operational practices:**\n- **Claim ownership**: Resource claims should be namespaced to node; claims cannot be shared between nodes\n- **Plugin idempotency**: Plugins must accept re-preparation of same claim with same parameters\n- **Timeout enforcement**: Consider external watchdog if plugin hangs during preparation\n- **Health status**: Monitor claim health status field (set by plugin) to detect degradation\n\n**Checkpointing**:\n- Claim info and prepared state checkpointed in `dra_manager_state` file\n- Survive kubelet crashes; claims in \"prepared\" state on restart\n\n**File**: `dra/manager.go` claim preparation (lines 350-450)\n\n### 5. Cgroup Enforcement and Resource Limits\n\n**Impact**: Resource limits applied at kernel level; hard enforcement prevents workload expansion\n\n**Behavior**:\n- CPU quota (CFS) limits CPU time per period; kernel throttles exceeding processes\n- Memory limits trigger OOM killer if exceeded\n- Memory.min (cgroup v2) reserves memory; memory.high (soft) suggests limit\n\n**Operational considerations:**\n- **CPU throttling**: Containers hitting CPU quota experience latency spikes; monitor throttle counts\n- **Memory OOM**: OOM killer selection is unpredictable; prefer eviction-based pod removal via kubelet\n- **Swap**: `--fail-swap-on` prevents swap usage (recommended); without it, memory pressure causes swap thrashing\n- **Overcommit**: Pod memory requests < limits allows overcommit; requires careful monitoring\n\n**Tuning**:\n- Pod QoS: Guaranteed pods get priority in eviction; Best Effort pods evicted first\n- `--enforce-node-allocatable`: Enable to enforce node allocatable limits at node level (via kubepods cgroup)\n- `--memory-reservation`: Reserve memory for system via qos-reserved flag\n\n**Monitoring**:\n- Track CPU throttle events per container (via cgroup metrics)\n- Monitor memory pressure via PSI (Pressure Stall Information) metrics\n- Alert on OOM killer events in kernel logs\n\n**File**: `qos_container_manager_linux.go` (QoS enforcement, lines 84-140)\n\n### 6. pod admission rejected\n- **Root cause**: All reserved CPUs already allocated to guaranteed pods\n- **Mitigation**: Ensure `--reserved-cpus` or `--kube-reserved` not over-subscribed\n- **File evidence**: `cpumanager/policy_static.go` allocation logic\n\n**Scenario 2: Device allocation fails**\n- **Symptom**: Device plugin's `Allocate()` RPC fails or times out\n- **Root cause**: Plugin crash, insufficient healthy devices, plugin version mismatch\n- **Outcome**: Pod fails to admit; kubelet retries admission\n- **Mitigation**: Monitor device plugin health via healthz endpoints, implement plugin redundancy\n- **File evidence**: `devicemanager/manager.go:Allocate()` plugin RPC invocation\n\n**Scenario 3: Topology constraints unsatisfiable**\n- **Symptom**: Topology manager rejects pod under 'restricted' or 'single-numa-node' policy\n- **Root cause**: Pod's resource requirements span multiple NUMA nodes but policy forbids it\n- **Outcome**: Pod pending; will not schedule until resources become available\n- **Tradeoff**: Strict policies reduce latency variance but increase scheduling complexity\n\n### Cgroup System Limits\n\n**Cgroup v1 limitations (deprecated but still supported):**\n- Separate CPU controllers for CPU shares and CPU quota (inconsistency risk)\n- Per-subsystem hierarchies can diverge, complicating management\n- Memory controller lacks fine-grained control (no memory.min in v1)\n- **Mitigation**: Migrate to cgroup v2 (see `CgroupV1DeprecatedWarning` in `container_manager.go:52`)\n\n**Cgroup v2 requirements:**\n- Requires kernel \u2265 5.7 for full features\n- Root `cpu.stat` requires kernel \u2265 5.8 (see `CgroupV2KernelWarning` in `container_manager.go:54`)\n- Single unified hierarchy eliminates v1 inconsistencies\n\n### Memory Overcommit Risks\n\n**Scenario**: Pod memory requests < memory limits; kernel OOM killer activates\n\n**Behavior:**\n- Kubelet's eviction manager may not react fast enough\n- Kernel OOM killer selects victim processes (often unpredictable)\n- Pod may terminate abruptly without graceful shutdown\n\n**Tradeoff:**\n- Kubelet's `--eviction-hard` thresholds let kubelet proactively evict pods before OOM\n- Requires buffer between allocatable and pod requests\n- `--qos-reserved` reserves memory for burstable/besteffort pod protection\n\n**Design choice in `pod_container_manager_linux.go:92-96`:**\n```go\n// Pod cgroup memory.high set only if MemoryQoS enabled (cgroup v2)\n// Without this, burstable pod memory can eat into node reserved memory\n```\n\n### Device Plugin State Consistency\n\n**Challenge**: Plugin crashes mid-allocation; CPU Pinning and Latency\n\n**Impact**: Static CPU policy (pinning) reduces latency variance for latency-critical workloads\n\n**Behavior**:\n- Guaranteed pods get exclusive CPU cores (no sharing with other pods)\n- Cores reserved via `--reserved-cpus` for system daemons\n- Reconciliation applies pinning every 5s (configurable)\n\n**Operational considerations:**\n- **Overallocation risk**: If too many guaranteed pods, run out of exclusive CPUs \u2192 admission failures\n- **Planning**: Size `--reserved-cpus` to account for kubelet, container runtime, system services\n- **Monitoring**: Track CPU allocation per pod; alert if utilization low (wasted cores)\n\n**Tuning parameters:**\n- `--cpu-manager-policy`: Change from 'none' to 'static' for pinning\n- `--cpu-manager-reconcile-period`: Lower value (faster reconciliation) increases CRI chatter\n- `--reserved-cpus`: Reserve more CPUs for system \u2192 fewer CPUs available to pods\n\n**File**: `cpumanager/cpu_manager.go` manager struct (lines 105-150)\n\n### 2. NUMA Topology Complexity\n\n**Impact**: Memory affinity and device locality decisions must account for NUMA topology\n\n**Behavior**:\n- Topology manager aggregates hints from all managers\n- Single NUMA node preference reduces memory latency (local access faster than remote)\n- Device locality affects throughput (GPUs, NICs attached to specific NUMA nodes)\n\n**Operational considerations:**\n- **Policy selection**:\n  - `none`: Safe default; simplest to understand and troubleshoot\n  - `best-effort`: Improves latency without hard guarantees; recommended for most clusters\n  - `restricted`: Strict resource co-location; only for latency-sensitive workloads\n  - `single-numa-node`: Most restrictive; rarely needed\n- **Asymmetric NUMA**: Some systems have unbalanced NUMA (different node sizes) \u2192 allocation algorithms may struggle\n- **Observability**: NUMA topology not directly visible to users; requires `numactl` or `/proc/cpuinfo` inspection\n\n**Monitoring**:\n- Track pod admissions by NUMA scope (single-node vs multi-node allocations)\n- Monitor memory latency if performance regression observed\n\n**File**: `topologymanager/topology_manager.go` (aggregation and policy enforcement)\n\n### 3. Device Plugin Reliability\n\n**Impact**: Device allocation failures block pod admission; device loss affects running pods\n\n**Failure scenarios:**\n- Plugin crashes \u2192 devices reported as unavailable; kubelet may deallocate and admit new pods\n- Plugin slow to respond \u2192 pod admission latency increases\n- Plugin state loss \u2192 devices appear reallocated incorrectly across restarts\n\n**Operational practices:**\n- **Plugin health monitoring**: Implement plugin-level health checks (heartbeat, resource validation)\n- **Redundancy**: Run plugin replicas with shared state (if plugin design permits)\n- **Graceful degradation**: Device plugin should report partial health if some devices healthy\n- **Resource reservation**: Reserve devices for system/monitoring pods; don't over-commit\n\n**Kubelet health checks**:\n- Kubelet monitors plugin socket availability\n- Unresponsive plugin may be evicted by plugin manager\n- Requires manual plugin restart\n\n**File**: `devicemanager/manager.go` plugin communication (lines 62-100)\n\n### 4. Dynamic Resource Allocation (DRA) Claim Lifecycle\n\n**Impact**: Claim preparation failures prevent pod startup; claim state must survive kubelet restart\n\n**Failure scenarios:**\n- Plugin unavailable during PrepareResources \u2192 pod stuck preparing indefinitely\n- Claim preparation succeeds but plugin crashes \u2192 claim state in limbo\n- Multiple kubelet instances claim same resource \u2192 conflict\n\n**Operational practices:**\n- **Claim ownership**: Resource claims should be namespaced to node; claims cannot be shared between nodes\n- **Plugin idempotency**: Plugins must accept re-preparation of same claim with same parameters\n- **Timeout enforcement**: Consider external watchdog if plugin hangs during preparation\n- **Health status**: Monitor claim health status field (set by plugin) to detect degradation\n\n**Checkpointing**:\n- Claim info and prepared state checkpointed in `dra_manager_state` file\n- Survive kubelet crashes; claims in \"prepared\" state on restart\n\n**File**: `dra/manager.go` claim preparation (lines 350-450)\n\n### 5. Cgroup Enforcement and Resource Limits\n\n**Impact**: Resource limits applied at kernel level; hard enforcement prevents workload expansion\n\n**Behavior**:\n- CPU quota (CFS) limits CPU time per period; kernel throttles exceeding processes\n- Memory limits trigger OOM killer if exceeded\n- Memory.min (cgroup v2) reserves memory; memory.high (soft) suggests limit\n\n**Operational considerations:**\n- **CPU throttling**: Containers hitting CPU quota experience latency spikes; monitor throttle counts\n- **Memory OOM**: OOM killer selection is unpredictable; prefer eviction-based pod removal via kubelet\n- **Swap**: `--fail-swap-on` prevents swap usage (recommended); without it, memory pressure causes swap thrashing\n- **Overcommit**: Pod memory requests < limits allows overcommit; requires careful monitoring\n\n**Tuning**:\n- Pod QoS: Guaranteed pods get priority in eviction; Best Effort pods evicted first\n- `--enforce-node-allocatable`: Enable to enforce node allocatable limits at node level (via kubepods cgroup)\n- `--memory-reservation`: Reserve memory for system via qos-reserved flag\n\n**Monitoring**:\n- Track CPU throttle events per container (via cgroup metrics)\n- Monitor memory pressure via PSI (Pressure Stall Information) metrics\n- Alert on OOM killer events in kernel logs\n\n**File**: `qos_container_manager_linux.go` (QoS enforcement, lines 84-140)\n\n### 6. Reconciliation Loop Gaps\n\n**Impact**: Delays in reconciliation loops mean resource constraints not immediately enforced\n\n**Behavior**:\n- CPU manager reconciles every `--cpu-manager-reconcile-period` (default 5s)\n- Memory manager similar reconciliation period\n- If pod created and immediately deletes container, reconciliation may miss cleanup\n\n**Operational considerations:**\n- **Latency sensitivity**: High-frequency pod churn (seconds) may not reconcile properly\n- **Reconciliation period tuning**: Lower period (e.g., 1s) increases kubelet load; higher period (e.g., 30s) delays enforcement\n- **Container runtime lag**: If runtime slow to return container state, reconciliation waits\n\n**Observability**:\n- Metrics: Track reconciliation loop duration, number of containers updated per loop\n- Logs: Reconciliation failures logged at INFO level\n\n**File**: `cpumanager/cpu_manager.go` reconciliation (lines 190-220)\n\n### 7. Node Initialization Latency\n\n**Impact**: Container manager initialization blocks kubelet startup; slow initialization delays node readiness\n\n**Initialization steps** (from `container_manager_linux.go:207-391`):\n1. Cgroup system validation (~100ms)\n2. Cadvisor machine info query (~500ms on slow systems)\n3. CPU/Memory manager checkpoint loading (100-500ms depending on pod count)\n4. Device plugin discovery (~1s if plugins slow to register)\n5. DRA manager initialization (if enabled)\n\n**Typical initialization**: 1-3 seconds for typical nodes; 5-10 seconds with many plugins\n\n**Optimization**:\n- Parallelize where possible (CPU/Memory managers can start concurrently)\n- Reduce checkpoint size (prune stale pods periodically)\n- Optimize device plugin registration (avoid RPC serialization overhead)\n\n**Operational**: Allow 10-15 second startup grace period before expecting pod scheduling\n\n**File**: `container_manager_linux.go:NewContainerManager()` (lines 207-391)\n\n### 8. Resource Leak Prevention\n\n**Risk**: Leaked cgroups, checkpoints, or device allocations persist after pod deletion\n\n**Leakage scenarios:**\n- **Cgroup orphans**: Pod deleted but cgroup not destroyed (disk full, permission errors)\n- **Checkpoint entries**: Stale pod entries in cpu_manager_state, memory_manager_state\n- **Device allocations**: Plugin crashes mid-deallocation; allocation persists\n\n**Prevention mechanisms:**\n- Periodic cleanup jobs: CPU manager prunes stale pod entries on reconciliation\n- Cgroup garbage collection: Stale cgroups cleaned up by kernel\n- DRA claim cleanup: ClaimInfo marked \"no longer needed\" after grace period\n\n**Operational**:\n- Monitor `/sys/fs/cgroup` for orphaned cgroups\n- Monitor checkpoint file sizes (should grow with active pods, shrink as pods delete)\n- Check device plugin allocation counts against running pods\n\n**File**: `cpumanager/cpu_manager.go` state pruning\n\n### 9. Multi-Kubelet Scenarios (Edge/Scaling)\n\n**Impact**: Multiple kubelets on different nodes may coordinate device plugins or claim resources\n\n**Constraints**:\n- Each kubelet instance is independent; no distributed state\n- Device plugins must handle multiple kubelet instances (e.g., two kubelets preparing same device)\n- DRA claims are node-local; scheduler ensures pod scheduled to claim's node\n\n**Operational**:\n- Device plugin architecture must be multi-kubelet aware (no shared state assumptions)\n- DRA drivers must handle per-node claim state\n- Ensure unique node names to avoid kubelet state collisions\n\n### 10. Upgrade and Rollback Considerations\n\n**Impact**: Checkpoint format changes during upgrades may require migration; downgrade may lose data\n\n**Checkpoint versioning**:\n- CPU manager checkpoint: Versioned format, supports migration\n- Memory manager checkpoint: Versioned format, supports migration\n- Device allocations: Ephemeral (not checkpointed across plugin restart)\n- DRA manager checkpoint: Versioned, persists claim prepare state\n\n**Upgrade path**:\n1. Stop kubelet\n2. Upgrade kubelet binary\n3. Kubelet startup loads old checkpoint, migrates if needed\n4. New kubelet writes checkpoint in new format\n5. Resume normal operation\n\n**Risks**:\n- Non-idempotent migration logic \u2192 data corruption\n- Downgrade after upgrade \u2192 checkpoint incompatible\n- Checkpoint loss \u2192 all allocations reset (CPU/memory re-allocated, DRA claims re-prepared)\n\n**Operational**:\n- Always backup checkpoint files before upgrade\n- Test upgrade path in staging environment\n- Allow extra time on first kubelet start after upgrade (migration runs)\n\n**File**: CPU manager state (lines 52-53): `cpuManagerStateFileName = \"cpu_manager_state\"`\n\n## Important Files and Directories\n\n### Core Container # Kubelet Container Manager Architecture Guide

## Overview

The kubelet container manager is the foundational subsystem responsible for managing container resources on a Kubernetes node. It orchestrates CPU allocation, memory management, device allocation, topology-aware resource placement, and dynamic resource allocation. The subsystem acts as a central hub that coordinates between the kubelet's pod synchronization loop and multiple resource-specific managers, enforcing kernel-level resource constraints through cgroup hierarchies.

## Subsystem Purpose, Boundaries, and Dependencies

### Purpose

The container manager serves three primary responsibilities:

1. **Resource enforcement**: Allocates and enforces compute resources (CPU, memory, devices) to containers through cgroups and the container runtime
2. **QoS hierarchy management**: Organizes pods into Quality of Service classes (Guaranteed, Burstable, BestEffort) with enforced resource isolation
3. **Node allocatable enforcement**: Reserves resources for system daemons and kubelet components, ensuring pod workloads do not starve node infrastructure
