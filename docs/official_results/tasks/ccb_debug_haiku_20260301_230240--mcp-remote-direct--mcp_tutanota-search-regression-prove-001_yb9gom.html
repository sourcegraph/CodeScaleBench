# Bug Investigation: Non-legacy mail content fails to decrypt

**Repository:** github.com/sg-evals/tutanota--f373ac38 (mirror of github.com/sg-evals)/tutanota--f373ac38 (mirror of tutao/tutanota)
**Task Type:** Find and Prove (write a regression test)

## Reported Issue

Users who have accounts with the new (non-legacy) permission model report that opening certain emails results in blank message bodies, missing reply-to addresses, and missing attachment listings. The browser console shows "Missing decryption key" errors when these mails are opened.

The issue specifically affects mails that use owner-encrypted session keys under the new permission model. Legacy mails continue to work correctly. The problem manifests when:

1. Opening a non-legacy mail in the inbox — the body fails to render
2. Opening a draft with reply-to recipients — the reply-to list cannot be loaded
3. The mail indexer/search encounters non-legacy mails — indexing fails silently with decryption errors

The root cause involves how encrypted session keys are propagated when loading related entities. When the application loads supplementary mail data (body content, draft details), the session key needed for decryption is available from the parent mail object but is not being passed through to the entity loading and caching layers. This means the loaded entities cannot be decrypted even though the necessary key exists.

Additionally, batch loading of multiple entities has the same propagation gap — when loading several related items at once, the per-entity session keys are not forwarded to the decryption step.

1. Investigate the codebase to find the root cause of the missing key propagation

# Non-Legacy Mail Decryption Key Propagation Bug - Analysis Report

## Summary

A critical bug exists in the Tutanota codebase where **encrypted session keys are not propagated when loading supplementary mail data**. This causes non-legacy mails (mails using owner-encrypted session keys) to fail decryption, resulting in blank message bodies, missing reply-to addresses, and missing attachment listings.

## Root Cause

**Location**: `src/api/worker/search/MailIndexer.ts:733-785`

The bug is in the `loadMailDetails` method which loads related mail entities in bulk. When loading `MailDetailsBlob` or `MailDetailsDraft` entities, the method calls:

```typescript
// Line 733 - Buggy code
const mailDetailsBlobs = await this.loadInChunks(MailDetailsBlobTypeRef, listId, ids)
```

The `loadInChunks` method (line 785) then calls:

```typescript
// Line 785 - Missing session keys!
this._entity.loadMultiple(typeRef, listId, chunk)  // No providedOwnerEncSessionKeys!
```

**The problem**: The `loadMultiple` call does NOT pass a `Map<Id, Uint8Array>` of `providedOwnerEncSessionKeys`.

## How It Manifests

1. Mail objects contain an `_ownerEncSessionKey` field that is encrypted with the mail owner group's key
2. When loading child entities (MailDetailsBlob, MailDetailsDraft), this session key needs to be passed to `EntityRestClient.loadMultiple`
3. Without this mapping, `EntityRestClient._handleLoadMultipleResult` cannot inject the `_ownerEncSessionKey` on the loaded entities
4. The entities arrive at decryption with `sessionKey=null`
5. Decryption fails silently, setting `_errors` on the entities
6. Users see blank mail bodies, missing reply-to addresses, etc.
