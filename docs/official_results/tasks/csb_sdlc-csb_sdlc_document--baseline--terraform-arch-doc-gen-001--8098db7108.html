It orchestrates configuration parsing, dependency analysis, state management, and provider interactions to create reproducible infrastructure deployments. The architecture is built around four key components: the graph-based execution model, provider plugin system, state management, and a sophisticated execution engine that drives graph evaluation.\n\n---\n\n## 1. Graph Builder System\n\n### 1.1 Core Graph Architecture\n\nThe dependency graph is Terraform's fundamental execution model. It represents the entire infrastructure as a directed acyclic graph (DAG) where:\n- **Nodes** represent resources, data sources, modules, variables, outputs, and providers\n- **Edges** represent dependencies between nodes\n- **Graph properties** enable parallel execution of independent resources\n\nThe graph builder constructs this DAG in multiple phases:\n\n```\nConfiguration \u2192 Resource Graph Construction \u2192 Graph Transformations \u2192 Optimized Execution Graph\n```\n\n### 1.2 Graph Construction Phase\n\n**Primary Node Types:**\n\n- **Resource Nodes**: Represent managed resources (e.g., `aws_instance.web`). Each resource node encapsulates:\n  - The resource type and name\n  - The provider instance it depends on\n  - The configuration schema\n  - State data for diff computation\n\n- **Data Source Nodes**: Similar to resource nodes but read-only. They fetch external data without managing resources.\n\n- **Module Nodes**: Represent module invocations. They contain sub-graphs for their contained resources and recursively apply transformations.\n\n- **Provider Nodes**: Represent provider configuration blocks. They establish dependencies on provider resources and variable references.\n\n- **Root Variable/Local Nodes**: Represent input variables, locals, and their computed values.\n\n- **Output Nodes**: Represent outputs that depend on resource attributes.\n\n**Dependency Discovery:**\n\nDependencies are discovered through:\n\n1. **Explicit References**: Direct attribute references like `aws_instance.web.id` create edges from referencing nodes to referenced resources.\n\n2. **Provider Dependencies**: Every resource node implicitly depends on its provider node, ensuring providers are initialized before resources are operated on.\n\n3. **Transitive Dependencies**: Module outputs that reference resources create transitive edges through the module hierarchy.\n\n4. **Implicit Ordering**: Some dependencies are implicit (e.g., destroy operations reverse implicit creation order).\n\n### 1.3 Graph Transformers\n\nGraph transformers are middleware components that modify and enhance the graph before execution. They run sequentially and each transformer sees the output of the previous one. Key transformers include:\n\n**Resource Transformers:**\n\n- **Expand Transform**: Expands `count` and `for_each` resources into multiple concrete nodes. For example:\n  ```hcl\n  resource \"aws_instance\" \"server\" {\n    count = 3\n  }\n  ```\n  becomes three separate graph nodes: `aws_instance.server[0]`, `aws_instance.server[1]`, `aws_instance.server[2]`. The expansion is lazy\u2014it occurs once count/for_each values are known.\n\n- **Orphan Resource Transform**: Detects resources that existed in prior state but are no longer in configuration (removed from code). These nodes are added to the graph with special handling to ensure they're destroyed.\n\n- **Provisioner Transform**: Wraps provisioner configurations around resource nodes, creating sub-graphs for provisioner execution.\n\n**Dependency Transformers:**\n\n- **Provider Config Transform**: Ensures provider configuration resources (especially `terraform_data` and similar) are available before regular resources depend on them.\n\n- **Destroy Edge Transform**: For destroy operations, reverses edges so resources are destroyed in reverse dependency order.\n\n- **Targeted Resource Transform**: When using `terraform plan -target`, the graph is pruned to only include the targeted resource and its dependents.\n\n**Validation Transformers:**\n\n- **Cycle Detection**: Detects circular dependencies and reports them as errors before execution begins.\n\n- **Schema Validation**: Ensures resource and provider schemas are available and valid.\n\n### 1.4 Module Graph Expansion\n\nModules create nested graph structures:\n\n1. **Module Call Node**: When a module is invoked, a module call node is created with edges to:\n   - All variable nodes referenced in module arguments\n   - All resources/outputs accessed from the calling context\n\n2. **Internal Module Graph**: Each module call contains a complete sub-graph of:\n   - Module variables (with edges to call arguments)\n   - Module resources\n   - Module outputs\n   - Module locals\n\n3. **Output Referencing**: When code references a module output (e.g., `module.vpc.vpc_id`), an edge is created from the referencing node to the module output node.\n\nThis hierarchical structure maintains logical separation while enabling dependency tracking across module boundaries.\n\n---\n\n## 2. Provider Interface and Lifecycle\n\n### 2.1 Provider Plugin Architecture\n\nTerraform uses a plugin-based architecture where providers are external processes communicating via gRPC. This provides:\n\n- **Process Isolation**: Provider crashes don't crash Terraform\n- **Language Flexibility**: Providers can be written in any language\n- **Version Independence**: Providers version independently from core\n\n**Plugin Discovery and Loading:**\n\n1. **Provider Resolution**: Terraform searches for providers in:\n   - The provider's `GetSchema` method is called once per provider instance\n   - Schema describes available resource types and data sources\n   - Schema is cached to avoid repeated gRPC calls\n   - Contains field types, descriptions, and validation rules\n\n2. **Configuration Application**:\n   - The provider configuration block (e.g., `provider \"aws\" { region = \"us-east-1\" }`) is interpolated\n   - The `Configure` method receives the provider config data\n   - Provider stores configuration internally for subsequent resource operations\n   - Any configuration errors are reported before resource operations begin\n\n3. **Resource Type Mapping**:\n   - Resource types in configuration are matched to provider schemas\n   - Type mapping enables the provider to route operations correctly\n   - Takes: current state and planned change\n  - Returns: new state after operation\n  - Returns: diagnostic errors if operation fails\n\n- **ImportResourceState**: Imports existing infrastructure (when using `terraform import`)\n\n**Read-Only Operations:**\n\n- **ReadDataSource**: For data sources, fetches external data without state management\n\n### 2.4 Provider Concurrency and Instance Management\n\n**Provider Multiplexing:**\n\n- Multiple resource operations can be performed on the same provider concurrently\n- The provider plugin process handles concurrency internally\n- Terraform manages request queuing to respect provider concurrency limits\n\n**Provider Instance Reuse:**\n\n- Each unique provider configuration creates one provider instance\n- Multiple resources can share the same provider instance\n- Configuration changes create new provider instances\n\n**Error Handling in Provider Operations:**\n\nProvider operations can fail with:\n\n1. **Diagnostic Errors**: Non-fatal errors (invalid argument, resource not found)\n   - Lock State (if remote)\n2. Read Current State\n3. Create State Snapshot (for concurrent graph evaluation)\n4. Walk Graph Nodes\n   - Each node reads from snapshot, not live state\n   - Prevents race conditions during concurrent reads\n5. Unlock State (plan doesn't modify state)\n```\n\nThe snapshot is crucial: it provides a consistent view of state during concurrent graph evaluation. It traverses the DAG respecting dependencies while maximizing parallelism.\n\n**Walker Responsibilities:**\n\n1. **Dependency Tracking**: Maintains node completion state\n2. **Ready Node Detection**: Identifies nodes with all dependencies met\n3. **Concurrent Scheduling**: Submits ready nodes to worker pool\n4. **Completion Detection**: Signals when node finishes\n5. **Error Propagation**: Stops dependent nodes on upstream error\n\n**Walker Algorithm:**\n\n```\nInitialize:\n  - Create dependency map\n  - Mark all nodes as pending\n  - Create worker pool (default 10 workers)\n\nMain Loop:\n  While any node pending or running:\n    1. Mark node complete, update state\n    5. If node failed:\n       - Mark dependent nodes as failed\n       - Continue on other branches if not fatal\n    6. Invoke post-node hooks\n\nCompletion:\n  - All nodes processed\n  - Aggregate errors\n  - Report final status\n```\n\n**Concurrency Control:**\n\n- **Default Parallelism**: 10 concurrent operations\n- **Configurable**: `-parallelism=N` flag\n- **Per-Provider Limits**: Some providers have connection limits\n- **State Lock**: Remote backends may serialize writes despite parallelism\n\n### 4.3 Plan vs Apply Execution\n\n**Plan Mode Behavior:**\n\nDuring plan mode, each resource node:\n\n1. **Calls ReadResource**: Gets current state via provider\n2. **Compares Current vs Desired**: Evaluates configuration against current state\n3. **Calls PlanResourceChange**: Provider computes what would change\n4. **Records Diff**: Captures planned change for display and apply verification\n\nPlan never modifies state or actual resources. Invalid HCL syntax\n   - Missing required blocks\n   - Type mismatches\n\n2. **Validation Errors**: Caught before execution\n   - Missing providers\n   - Circular dependencies\n   - Schema violations\n\n3. **Provider Errors**: Caught during execution\n   - # Terraform Plan/Apply Pipeline Architecture

## Overview

The Terraform plan/apply pipeline is the core execution engine that transforms infrastructure code into actual cloud resources. It orchestrates configuration parsing, dependency analysis, state management, and provider interactions to create reproducible infrastructure deployments. The architecture is built around four key components: the graph-based execution model, provider plugin system, state management, and a sophisticated execution engine that drives graph evaluation.

---

## 1. Graph Builder System

### 1.1 Core Graph Architecture

The dependency graph is Terraform's fundamental execution model. It represents the entire infrastructure as a directed acyclic graph (DAG) where:
- **Nodes** represent resources, data sources, modules, variables, outputs, and providers
- **Edges** represent dependencies between nodes
- **Graph properties** enable parallel execution of independent resources

The graph builder constructs this DAG in multiple phases:

```
Configuration → Resource Graph Construction → Graph Transformations → Optimized Execution Graph
```

- **Resource Nodes**: Represent managed resources (e.g., `aws_instance.web`). Each resource node encapsulates:
  - The resource type and name
  - The provider instance it depends on
  - The configuration schema
  - State data for diff computation

- **Data Source Nodes**: Similar to resource nodes but read-only. They fetch external data without managing resources.
