Failure Modes, Limits, and Design Tradeoffs\n\n### 4.1 Failure Modes\n\n#### Cgroup Mount Failures\n**Scenario**: Required cgroup subsystems (cpu, memory, cpuset) not mounted.\n**Impact**: Container manager refuses to start; kubelet fails to initialize.\n**Resolution** (validateSystemRequirements): Early validation at startup prevents silent degradation.\n\n#### Cgroup v1 \u2192 v2 Transition\n**Scenario**: Node transitions from cgroup v1 to cgroup v2.\n**Impact**:\n- Kubelet must be restarted (hot switch not supported)\n- Checkpoint state from v1 (cpu_manager_state) is NOT migrated\n- CPU allocations reset to default; containers with lowest OOM score adj are killed.\n**By Design**: Unlike CPU (CFS shares limit), memory is enforced at cgroup limit level.\n\n### 4.2 Resource Limits\n\n#### CPU Allocation Precision\n- **Unit**: Millicores (1m = 1/1000 CPU)\n- **CFS Quota Period**: Configurable (default 100ms)\n- **Enforcement**: CFS quota (hard limit) only if `--cpu-cfs-quota=true`; otherwise CPU shares (soft limits)\n- **Tradeoff**: Hard limits reduce latency variance but may cause underutilization (quota throttling)\n\n#### Memory Limits\n- **Unit**: Bytes\n- **Enforcement**: Memory.limit_in_bytes (cgroup v1) or memory.max (cgroup v2)\n- **Enforcement**: Hard limit; exceeding causes immediate OOM kill\n- **Single Process OOM Kill** (Linux): If enabled, memory.oom.group=1 kills entire pod on cgroup OOM\n\n#### Device Limits\n- **Per-Device**: Controlled by device plugin\n- **Allocation**: All-or-nothing (cannot share a GPU across pods)\n- **Failure**: Exceeded device capacity \u2192 pod remains pending\n\n#### PID Limits\n- **Per-Pod**: Configurable via podSpec.PidLimits or node-wide defaults\n- **Enforcement**: Cgroup pids.max limit\n- **Limit**: Bounded by node PID capacity (typically 4M on Linux)\n\n### 4.3 Design Tradeoffs\n\n#### Cgroup Granularity: Pod vs Container\n**Tradeoff 1: Pod-level cgroup hierarchy**\n- **Enabled by**: `--cgroups-per-qos` flag\n- **Advantage**: Fine-grained per-pod resource isolation; accurate pod memory accounting\n- **Cost**: Increased cgroup hierarchy depth; more file descriptors; slower cgroup operations\n- **Default**: Enabled in most production setups\n\n**Tradeoff 2: QoS resource isolation**\n- **Guaranteed QoS**: CPU shares=highest, memory=reserved at pod limit\n- **Burstable QoS**: CPU shares=proportional to request, memory=shared pool with limit\n- **BestEffort QoS**: CPU shares=lowest, memory=free resources only\n- **Cost**: Requires cgroup hierarchy maintenance; resource calculations per pod\n- **Benefit**: Prevents noisy neighbor problems; predictable QoS SLAs\n\n#### CPU Affinity Policy Scope\n**Policy: static CPU manager**\n- **Requirement**: Pod must have Requests==Limits for **all** containers\n- **Benefit**: Exclusive CPU cores \u2192 lower latency, better cache locality, deterministic scheduling\n- **Cost**: Fragmentation; underutilization if pod requests exceed actual usage\n- **Design Choice**: Only allocate exclusive CPUs to Guaranteed QoS; avoids thrashing\n\n#### Memory Allocation: Opportunistic vs Enforced\n**Cgroup v1**: Memory is soft-limit (LRU eviction) or hard-limit (OOM kill)\n**Cgroup v2**: Unified memory enforcement\n- **Design**: Pod memory.limit = sum of container limits; exceeded \u2192 OOM\n- **Tradeoff**: No memory overcommit possible; simpler predictability\n- **Alternative (not implemented)**: Allow overcommit with eviction-driven reclamation\n\n#### Topology Manager Coordination\n**Design**: Central topology manager collects hints from all resource managers before finalizing allocation.\n- **Tradeoff**: Increased latency (must wait for all managers to compute hints)\n- **Benefit**: Prevents siloed allocation decisions; respects NUMA topology\n- **Cost**: Complex hint propagation; may delay container start if topology locked\n\n#### State Persistence\n**CPU Manager**: Persists state to checkpoint file (cpu_manager_state)\n- **Benefit**: Survives kubelet restart; # Kubelet Container Manager Subsystem Architecture Guide

## 1. Subsystem Purpose, Boundaries, and Dependencies

### Purpose
The kubelet container manager (CM) subsystem is responsible for managing the lifecycle and resource allocation of containers on a Kubernetes node. It serves as the bridge between the kubelet's pod orchestration logic and the node's resource constraints (CPU, memory, devices), enforcing QoS guarantees and ensuring proper container isolation through cgroups.

### Boundaries
The container manager operates at the **node level**, managing all containers across all pods running on a single machine. It is distinct from:
- **Pod lifecycle management** (handled by pod workers in `pkg/kubelet/pod_workers.go`)
- **Container runtime integration** (handled by kuberuntime in `pkg/kubelet/kuberuntime/`)
- **Image management** (separate subsystem for pulling and caching images)

### Key Dependencies
- **Upstream**: Pod workers (`pkg/kubelet/pod_workers.go`) drive container creation requests; node API provides resource specifications
- **Downstream**:
  - Container runtime (`pkg/kubelet/kuberuntime/`) executes actual container operations via CRI (Container Runtime Interface)
  - cAdvisor (`pkg/kubelet/cadvisor/`) provides node capacity and resource metrics
  - cgroups filesystem or systemd interfaces for enforcing resource limits
- **Horizontal**: Device manager, CPU manager, memory manager, topology manager for resource-specific policies
