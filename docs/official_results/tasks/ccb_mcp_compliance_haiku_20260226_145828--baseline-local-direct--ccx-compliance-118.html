## Summary

I've created `/workspace/answer.json` with a complete audit of Django's admin filter rendering pipeline for ForeignKey filters. The audit identifies:

### **Four Key Files:**

1. **`django/contrib/admin/filters.py`** - Contains both the base `ListFilter` class and the `RelatedFieldListFilter` class for ForeignKey dropdowns
2. **`django/contrib/admin/views/main.py`** - Contains the `ChangeList` class that orchestrates filter collection
3. **`django/contrib/admin/templatetags/admin_list.py`** - Contains the `admin_list_filter` template tag for rendering individual filters
4. **`django/contrib/admin/templates/admin/change_list.html`** - The template that iterates over filters for sidebar display

### **Control Flow:**

The filter visibility pipeline works as follows:

1. **`ChangeList.get_filters()`** (line 175 in views/main.py) instantiates all filter classes and calls `has_output()` on each
2. **`ListFilter.has_output()`** (line 41 in filters.py) is the abstract method that determines filter visibility
3. **`RelatedFieldListFilter.has_output()`** (line 248 in filters.py) returns `True` only if there are lookup choices to display
4. Only filters with `has_output() == True` are added to `filter_specs`
5. **`admin_list_filter` template tag** (line 521 in admin_list.py) renders each filter in the sidebar
6. **`change_list.html` template** (line 72) iterates over `cl.filter_specs` to render them

The answer file includes all relevant symbols, the call chain, and a detailed narrative explanation of how empty related-field filters are excluded from the admin sidebar.
