## Summary

I identified the core extension registry pattern in Envoy, which enables a compile-time extensible architecture for filters and other components. The pattern includes:

**Key Architecture Components:**

1. **Core Registry** (`envoy/registry/registry.h`):
   - `FactoryRegistry<Base>` - Template class managing factories by name using hash maps
   - `RegisterFactory<T, Base>` - Templated class that registers instances at static initialization time
   - `FactoryCategoryRegistry` - Manages factory categories for discovery

2. **Factory Base Classes** (`envoy/config/typed_config.h`, `envoy/server/filter_config.h`):
   - `UntypedFactory` - Base with name() and category() methods
   - `TypedFactory` - Adds protobuf support
   - `NamedHttpFilterConfigFactory` - HTTP filter factory interface
   - `NamedNetworkFilterConfigFactory` - Network filter factory interface

3. **Factory Lookup** (`source/common/config/utility.h`):
   - `Utility::getFactory()` - Type-first lookup with name fallback
   - `Utility::getFactoryByType()` - Lookup by protobuf type URL
   - `Utility::getFactoryByName()` - Direct name-based lookup

**Self-Registration Flow:**
- `REGISTER_FACTORY()` macro expands to static `RegisterFactory<>` instantiation
- Constructor calls `FactoryRegistry::registerFactory()` storing factory reference
- Automatically registers category with `FactoryCategoryRegistry`
- Enables 250+ extension factories to register without central code
