Mechanics\n\n### Group Membership and Coordination\n\n**Group Concept:**\n- Logical aggregation of consumers with the same `group.id`\n- One consumer group can have multiple members\n- Each partition assigned to exactly one member (for load balancing)\n\n**Lifecycle:**\n1. **Init**: Consumer created but not in group\n2. **Join**: First `poll()` triggers join via JoinGroup request\n3. **Sync**: Broker coordinates partition assignment\n4. **Stable**: Member receives partitions and starts fetching\n5. **Rebalancing**: Triggered by membership or subscription changes\n6. **Revocation**: Member gives up partitions (calls `onPartitionsRevoked`)\n7. **Assignment**: Member receives new partitions (calls `onPartitionsAssigned`)\n\n### Rebalancing Details\n\n**Rebalancing is triggered when:**\n- New member joins group\n- Existing member leaves/crashes\n- Subscription changes (new topics match pattern)\n- Topic partitions change\n- Consumer calls `enforceRebalance()`\n\n**Detection and Execution:**\n- Rebalancing is **ONLY** detected during `poll()` calls\n- Once detected, callbacks execute immediately within `poll()`\n- No `poll()` called = no rebalance detection = potential stale assignment\n\n**Rebalance Protocols:**\n\n**Classic Protocol (Eager):**\n- All members revoke partitions simultaneously\n- `onPartitionsRevoked()` called with full assignment\n- All members wait for synchronization\n- Then all receive new assignments\n- Higher stop-the-world pause time\n- Higher latency during rebalance\n\n**Classic Protocol (Cooperative):**\n- Members retain partitions where possible\n- Only revoked partitions are given up\n- `onPartitionsRevoked()` called with minimal set\n- Reduces stop-the-world pause\n- Requires compatible assignors\n\n**Consumer Protocol (New):**\n- Always cooperative rebalancing\n- Incremental partition migration\n- Lowest downtime during rebalance\n\n**Rebalance Callback Guarantees:**\n- `onPartitionsRevoked()` is called before partition move\n- `onPartitionsAssigned()` is called after partition is acquired\n- During normal rebalance: revoked \u2192 new_owner_assigned guaranteed ordering\n- Exception scenarios: partition may be lost without revoke \u2192 `onPartitionsLost()` called\n- `onPartitionsAssigned()` always called (even if empty list)\n\n### Heartbeating and Session Management\n\n**Session Timeout** (`session.timeout.ms`, default: 10 seconds):\n- If broker receives no heartbeat for this duration, consumer is removed\n- Other members detect removal and rebalance\n\n**Max Poll Interval** (`max.poll.interval.ms`, default: 5 minutes):\n- If `poll()` not called for this duration, consumer proactively leaves\n- Prevents livelock: consumer continues heartbeating but makes no progress\n- Subsequent commits fail with `CommitFailedException`\n\n**Heartbeat Thread:**\n- Background thread sends heartbeats periodically\n- Interval controlled by `heartbeat.interval.ms` (default: 3 seconds)\n- Runs independently of main consumer thread\n\n**Practical Implications:**\n```java\n// GOOD: Tight poll loop maintains membership\nwhile (running) {\n    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));  // \u2713 Called frequently\n    for (ConsumerRecord record : records) {\n        processRecord(record);  // Fast processing\n    }\n}\n\n// BAD: Slow processing causes member removal\nwhile (running) {\n    ConsumerRecords records = consumer.poll(Duration.ofSeconds(30));\n    for (ConsumerRecord record : records) {\n        slowProcess(record);  // \u2717 Can exceed max.poll.interval.ms\n    }\n}\n\n// BETTER: Offload processing to thread pool\nwhile (running) {\n    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));\n    for (ConsumerRecord record : records) {\n        executor.submit(() -> processRecord(record));  // Non-blocking\n    }\n}\n```\n\n### Enforcing Rebalance\n\n```java\nvoid enforceRebalance()\nvoid enforceRebalance(String reason)\n```\n\nManually trigger a rebalance on next `poll()`. Useful for:\n- Graceful shutdown with immediate state flush\n- Redistribution of partitions\n- Internal consumer state reconciliation\n\n---\n\n## Flow Control and Position Management\n\n### Pause and Resume\n\n**Methods:**\n```java\nvoid pause(Collection<TopicPartition> partitions)\nvoid resume(Collection<TopicPartition> partitions)\nSet<TopicPartition> paused()\n```\n\n**Behavior:**\n- `pause()`: Excludes partitions from future `poll()` calls\n- Position is NOT reset (unchanged when resumed)\n- Useful for flow control or batching\n- Multiple pauses for different partitions are independent\n- `paused()` returns the current set of paused partitions\n\n**Use Case - Processing Coordination:**\n```java\n// Pause fast partition to let slow partition catch up in join\nfor (TopicPartition tp : consumer.assignment()) {\n    long lag = endOffset - position(tp);\n    if (lag > 100000) {\n        consumer.resume(Collections.singletonList(tp));\n    } else {\n        consumer.pause(Collections.singletonList(tp));\n    }\n}\n```\n\n### Current Lag\n\n```java\nOptionalLong currentLag(TopicPartition partition)\n```\n\nReturns the difference between high water mark and consumer position. part of the group\n- Consumer didn't call `poll()` within `max.poll.interval.ms`\n- Consumer was fenced by another member\n- Group coordinator changed (rare)\n\n**Recovery:**\n- Stop committing and ## Overview

### Threading Model

The Kafka KafkaConsumer is **NOT thread-safe**. All consumer operations must occur within a single consumer thread. The only exception is `wakeup()`, which is explicitly designed to be called from external threads to interrupt blocking operations.

**Key Threading Points:**
- `KafkaConsumer` maintains an internal event loop that processes rebalances, heartbeats, and data fetches
- All callbacks (ConsumerRebalanceListener, OffsetCommitCallback) execute within the user's calling thread
- Concurrent access without synchronization results in `ConcurrentModificationException`
- The `wakeup()` method is the only safe cross-thread operation
