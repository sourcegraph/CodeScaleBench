Conflict (409 Conflict)\n\n**Symptom:** Apply request returns HTTP 409 with structured conflict information\n\n**Root cause:** Client tries to apply fields owned by another manager without `force=true`\n\n**Detection:** In `FieldManager.Apply()` during field ownership check\n\n**Recovery:**\n- Option 1: Use `force=true` to take ownership (dangerous if field affects other managers)\n- Option 2: Coordinate with other manager to release field ownership\n- Option 3: Edit resource directly to remove other manager's entry from managedFields\n\n**Example:**\n```bash\n# Triggers conflict\nkubectl apply -f deployment.yaml\n\n# Resolve by forcing (use carefully!)\nkubectl apply -f deployment.yaml --force-conflicts\n\n# Or check who owns conflicting field\nkubectl get deployment my-deployment -o json | jq '.metadata.managedFields'\n```\n\n#### Invalid Managed Fields After Admission (Warning)\n\n**Symptom:** HTTP 200 with warning header in response\n\n**Root cause:** Mutating admission controller corrupted `.metadata.managedFields`\n\n**Detection:** Wrapper in fieldmanager admission plugin (admission.go line 74\u201380)\n\n**Resolution:**\n- Server reverts managedFields to pre-admission state automatically\n- Admin must fix or disable the offending admission webhook\n- Indicates webhook is not SSA-aware\n\n#### Strict Field Validation Errors (400 Bad Request)\n\n**Symptom:** \"error strict decoding YAML: unknown field 'typo'\"\n\n**Root cause:** Patch YAML contains unknown fields (when `fieldValidation=Strict`)\n\n**When triggered:**\n- Query parameter: `fieldManager=kubectl&fieldValidation=Strict`\n- Unknown fields in patch YAML\n\n**Recovery:**\n```bash\n# Fix typo in patch, or relax validation\nkubectl apply -f patch.yaml --field-validation=warn  # Warn instead of fail\n```\n\n#### Noop Apply Resource Version Bump (Operational)\n\n**Symptom:** `kubectl apply` causes resourceVersion to increment even with no field changes\n\n**Root cause:** ManagedFields timestamps are always updated, triggering storage write\n\n**Mitigation:** Server can avoid this via `KUBE_APISERVER_AVOID_NOOP_SSA_TIMESTAMP_UPDATES` environment variable (managed by field equality transformer)\n\n**Impact:**\n- Increased watch event traffic\n- Unnecessary etcd writes\n- Controller reconciliation triggered on no-op applies\n\n### 4.2 Design Tradeoffs\n\n#### Tradeoff 1: Field Granularity vs. Simplicity\n\n**Decision:** SSA owns fields at arbitrary path depth (e.g., `.spec.template.spec.containers[0].resources.limits.memory`)\n\n**Benefit:**\n- Multiple managers can own different subfields of same object\n- Precise conflict attribution\n\n**Cost:**\n- Complex managed fields structure (nested maps)\n- Higher memory footprint for large objects\n- Conflict errors can be granular but hard to understand\n\n**Alternative considered:** Ownership at resource level only\n- Simpler but two managers can't safely share editing responsibilities\n\n#### Tradeoff 2: Force-Apply Semantics\n\n**Decision:** `force=true` unilaterally takes ownership of conflicting fields\n\n**Benefit:**\n- Allows operators to override when necessary\n- Useful for migrations (e.g., moving management from webhook to controller)\n\n**Cost:**\n- Can silently break other managers if misused\n- No audit trail of why force was needed\n- Tempts users to use it instead of coordinating\n\n**Mitigation:** kubectl warns on force-apply; audit logs include force flag\n\n#### Tradeoff 3: CBOR Support Behind Feature Gate\n\n**Decision:** `application/apply-patch+cbor` only available with `CBORServingAndStorage` feature gate enabled\n\n**Benefit:**\n- More efficient encoding than YAML (binary, smaller size)\n- Reduces bandwidth for large apply patches\n- Can use CBOR's type system for stricter validation\n\n**Cost:**\n- Users must enable feature gate (affects whole cluster)\n- Requires new tooling (CBOR doesn't have universal CLI support like YAML)\n- Debugging harder (not human-readable)\n\n**Why not default:** Stability risk; CBOR parsing bugs affect all apply operations\n\n#### Tradeoff 4: Admission Plugin Side Effects\n\n**Decision:** Mutating admission can modify objects but must not break managedFields\n\n**Benefit:**\n- Allows webhooks to inject defaults, normalize values\n- Doesn't require webhook to understand SSA\n\n**Cost:**\n- If webhook corrupts managedFields, server silently reverts them (warning only)\n- Webhook may not realize its mutations were rejected\n- Operator must monitor warnings to detect bad webhooks\n\n**Justification:** Webhooks predate SSA; breaking them would be incompatible\n\n#### Tradeoff 5: Extract vs. Full Reapply\n\n**Decision:** Clients can extract previously-applied configuration via `ExtractConfigMapFrom(resource, fieldManager, subresource)`\n\n**Benefit:**\n- Enables extract-modify-apply workflow without storing manifest separately\n- Recover what a manager applied even if object has changed\n\n**Cost:**\n- Extract returns only fields *still owned* by manager (not historical)\n- If another manager took ownership, those fields are missing\n- Users must handle incomplete extract gracefully\n\n---\n\n## Part 5: Extension Points and Customization Hooks\n\n### 5.1 Admission array of `managedFields` entries\n\n**Size per manager:**\n- ~200 bytes for metadata (manager name, timestamp, apiVersion, operation)\n- ~2\u201310x the field paths size (depends on object complexity)\n\n**Example:** ConfigMap with 50 keys\n- One manager entry: ~1\u20132 KB\n- Five managers: ~5\u201310 KB added to resource\n\n**Mitigation:** `capmanagers` implementation limits field managers per resource (line 1\u201350 in capmanagers.go)\n\n```go\n// Caps number of managers if list grows too large\n// Default: keep only 10 most recent managers\n```\n\nOlder managers are automatically pruned to prevent unbounded growth.\n\n#### Storage Write Amplification\n\n**Noop applies** (applying unchanged configuration) still write to etcd:\n- ManagedFields timestamp always updates\n- ResourceVersion increments\n- Triggers all watchers\n\n**Mitigation:**\n- `IgnoreManagedFieldsTimestampsTransformer` skips writes if fields unchanged\n- Environment variable: `KUBE_APISERVER_AVOID_NOOP_SSA_TIMESTAMP_UPDATES=true`\n\n**Recommendation:** Enable for clusters with high apply frequency (GitOps controllers).\n\n#### Conflict Detection Latency\n\n**Scenario:** Apply with many fields, multiple managers own various fields\n\n**Latency:** O(num_managers \u00d7 num_fields_in_patch)\n- Structured merge diff library compares field paths\n- Conflict checking is fast (hash lookups) but expensive at scale\n\n**Typical latency:** <50ms per apply request (on standard hardware)\n\n**Worst case:** Apply touching 1000 fields with 100 managers \u2192 possible 5+ second conflict resolution\n\n**Recommendation:** Limit number of managers per resource via capmanagers tuning.\n\n### 6.2 Reliability and Data Correctness\n\n#### Orphaned Managers\n\n**Scenario:** Webhook that managed fields is deleted but its ManagedFieldsEntry remains\n\n**Effect:** Fields appear \"owned\" by non-existent manager; no one can safely modify them\n\n**Detection:** Admin tools Both read current object with resourceVersion V1\n2. Both compute merged objects\n3. First write succeeds, updates resourceVersion to V2\n4. Second write fails (resourceVersion mismatch)\n5. Second client is told to retry\n6. GuaranteedUpdate retry loop handles this transparently\n\n**Outcome:** Both applies eventually succeed (ordering depends on retry luck)\n\n**Note:** Not \"last-write-wins\" like traditional updates; both managers' fields are merged.\n\n#### Field Manager Name Stability\n\n**Risk:** If client changes its field manager name (e.g., \"controller-v1\" \u2192 \"controller-v2\"), old entries persist\n\n**Effect:**\n- Original fields still owned by \"controller-v1\"\n- New apply claims fields under \"controller-v2\"\n- ManagedFields now has both entries\n\n**Best practice:** Field manager names should be stable identifiers (namespace/label + component name)\n\n**Example:**\n```\n# Good: includes namespace and component\nfieldManager=kube-system:deployment-controller\n\n# Bad: includes version (changes break ownership tracking)\nfieldManager=deployment-controller-v2.3.1\n```\n\n### 6.3 Debugging and Observability\n\n#### Managed Fields Inspection\n\n**Tool:** `kubectl get resource # Kubernetes Server-Side Apply Configuration Guide

## Executive Summary

Server-Side Apply (SSA) is a declarative resource management system in Kubernetes that enables safe, concurrent ownership of fields across multiple controllers, tools, and operators. Unlike imperative updates, SSA tracks which client (field manager) owns each field and automatically resolves conflicts when multiple clients attempt to modify the same fields. This guide details the subsystem architecture, component responsibilities, request flow, conflict handling, and operational patterns.

---

## Part 1: Subsystem Purpose and Boundaries

### Purpose

Server-Side Apply solves the multi-actor field ownership problem: In Kubernetes, many clients (human operators, deployment controllers, admission webhooks, operators) need to concurrently modify resources. Without coordination, their changes collide and overwrite each other.

SSA provides:

1. **Field-level ownership tracking** via `managedFields` in resource metadata
2. **Conflict detection** when clients try to modify fields owned by others
3. **Declarative merging** of configurations from multiple managers
4. **Dry-run support** to preview apply operations before committing
5. **Force-apply capability** to override field ownership when necessary

The core philosophy: *Clients declare their desired state; the server merges multiple declarations into a unified object while preserving ownership boundaries.*

- Patch processing for `application/apply-patch+yaml` and `application/apply-patch+cbor` content types
