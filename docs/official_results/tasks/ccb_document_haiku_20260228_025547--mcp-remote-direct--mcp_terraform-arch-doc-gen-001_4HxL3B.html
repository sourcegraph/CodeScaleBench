This document describes the internal architecture that enables Terraform to safely plan and execute infrastructure changes in a declarative, concurrent manner.\n\nThe core insight of Terraform's architecture is the separation of concerns:\n- **Planning**: The plan phase reads configuration and current state, then delegates to providers to determine what changes are needed\n- **Applying**: The apply phase executes the changes described by the plan, updating infrastructure and state\n\nBoth phases rely on a common graph-based execution engine that respects resource dependencies, manages provider lifecycle, and provides extension points for tooling.\n\n---\n\n## 1. Graph Builder System\n\n### 1.1 Overview\n\nTerraform constructs an execution graph for each operation (plan, apply, validate, etc.). The graph represents all resources, data sources, modules, variables, outputs, and their dependencies. Rather than hardcoding the graph structure, Terraform uses a **builder pattern** with composable **transformers** that incrementally construct the graph.\n\n### 1.2 GraphBuilder Interface\n\nThe `GraphBuilder` interface is the entry point for graph construction:\n\n```go\ntype GraphBuilder interface {\n    Build(addrs.ModuleInstance) (*Graph, tfdiags.Diagnostics)\n}\n```\n\nTerraform provides two primary implementations:\n\n1. **PlanGraphBuilder**: Builds graphs for planning operations (plan, validate, refresh)\n2. **ApplyGraphBuilder**: Builds graphs for applying plans\n\nBoth builders use the **BasicGraphBuilder**, which applies a series of transforms in sequence:\n\n```go\ntype BasicGraphBuilder struct {\n    Steps []GraphTransformer\n    Name  string\n}\n```\n\n### 1.3 Graph Transformers\n\nGraph transformers implement a simple interface:\n\n```go\ntype GraphTransformer interface {\n    Transform(*Graph) error\n}\n```\n\nEach transformer takes a graph and modifies it by adding, removing, or connecting vertices. The builders compose these transformers to progressively build the complete execution graph.\n\n#### Key Transformers (in typical build order):\n\n1. **ConfigTransformer**: Creates nodes for resources and modules from the configuration\n2. **RootVariableTransformer**: Creates nodes for root module input variables\n3. **ModuleVariableTransformer**: Creates nodes for module input variables\n4. **LocalTransformer**: Creates nodes for local value declarations\n5. **OutputTransformer**: Creates nodes for output values\n6. **DiffTransformer** (apply only): Creates resource instance nodes from the plan\n7. **StateTransformer**: Attaches state information to resource nodes\n8. **ProviderTransformer**: Creates and connects provider configuration nodes\n9. **ReferenceTransformer**: Creates edges for resource references (e.g., `aws_instance.web.id`)\n10. **DestroyEdgeTransformer**: Adds edges for create-before-destroy behavior\n11. **TargetTransformer**: Prunes the graph to only targeted resources (when `-target` is used)\n12. **PruneTransformer**: Removes unreferenced nodes\n\nThe transformer pattern allows:\n- **Modularity**: Each concern (references, state, providers) is isolated\n- **Testability**: Individual transformers can be tested independently\n- **Extensibility**: Custom transformers can be injected for specialized behavior\n\n### 1.4 Graph Structure\n\nThe resulting graph is a directed acyclic graph (DAG) where:\n\n- **Vertices** represent execution units:\n  - `NodeAbstractResource`: A resource declaration in configuration\n  - `NodeAbstractResourceInstance`: A specific instance of a resource (e.g., `aws_instance.web[0]`)\n  - `NodeAbstractProvider`: A provider configuration\n  - `NodeAbstractVariable`: An input variable\n  - `NodeAbstractLocalValue`: A local value\n  - `NodeAbstractOutput`: An output value\n  - Module expansion nodes\n  - Check nodes (for test/check blocks)\n\n- **Edges** represent dependencies:\n  - **Explicit dependencies**: From `depends_on` declarations or references to other resources\n  - **Implicit dependencies**: From attribute interpolations (e.g., `aws_instance.app.security_groups = [aws_security_group.app.id]`)\n  - **Provider dependencies**: From `provider` meta-argument\n  - **Implicit ordering**: From create-before-destroy (CBDs) or other lifecycle features\n\n### 1.5 Resource Instance Expansion\n\nTerraform supports dynamic instance expansion through `count` and `for_each`. The graph builder handles this through:\n\n1. **Instance Expander**: A component (`instances.Expander`) that tracks which resource instances should be created based on variable evaluation\n2. **Partial Expansion**: During planning, expressions may reference unknown values (e.g., from other resources). The expander handles both:\n   - **Known instances**: Concrete instance addresses (e.g., `aws_instance.web[0]`)\n   - **Unknown instances**: Partially expanded addresses representing multiple possible instances\n\n3. **Deferred Resources**: If expansion depends on unknown values, Terraform can defer planning these resources to a later plan phase\n\n### 1.6 Module Graph Construction\n\nModule expansion follows a similar pattern:\n- Module instance expansion is tracked during graph walking\n- Child module graphs are built dynamically as the parent graph is walked\n- Module variables are connected from parent module outputs\n- The `ModuleInstanceExpander` manages the expansion of module instances from `count`/`for_each`\n\n---\n\n## 2. Provider Interface and Lifecycle\n\n### 2.1 Provider Architecture Overview\n\nTerraform's provider system is plugin-based. Providers are:\n- Loaded as external processes (over gRPC)\n- Initialized lazily (only when needed)\n- Shared across multiple resource instances\n- Cached per module instance to avoid redundant initialization\n\n### 2.2 Provider Plugin Communication\n\nProviders communicate with Terraform Core via the **gRPC provider protocol** (currently protocol v6):\n\n```\nTerraform Core\n      \u2193 (gRPC)\nProvider Process\n      \u2193 (method calls)\nProvider Implementation\n```\n\nThe protocol defines:\n- **Capabilities negotiation**: Provider declares supported resource types, data source types, and function names\n- **Schema retrieval**: Terraform requests the schema for resources/data sources\n- **Plan/Apply delegation**: Terraform sends state and configuration, provider returns planned/applied state\n- **Validation**: Schemas and values are validated against provider-defined schemas\n\n### 2.3 Provider Initialization and Caching\n\nProvider initialization follows this flow:\n\n1. **Discovery Phase** (before graph building):\n   - Terraform locates provider binaries\n   - Validates provider checksums and signatures\n   - Caches provider metadata\n\n2. **Graph Walking Phase** (during execution):\n   - When a provider node is visited, `InitProvider()` is called on the `EvalContext`\n   - The provider process is spawned if not already running\n   - The provider's schema is retrieved and cached\n   - The provider is configured with the HCL configuration block\n\n3. **Caching**:\n   - Providers are cached in the `ContextGraphWalker`:\n     - `providerCache`: Map of initialized providers by address\n     - `providerSchemas`: Map of cached schemas\n   - A single provider instance handles all resources of that type in its configuration\n   - Providers are stopped gracefully at the end of the operation\n\n### 2.4 Provider Configuration Resolution\n\nProvider configuration is resolved through a multi-step process:\n\n1. **Configuration Body Merging** (`buildProviderConfig`):\n   - Explicit configuration from HCL blocks\n   - Merged with user input (from `-var` or interactive prompts)\n   - Result is an `hcl.Body` ready for evaluation\n\n2. **Provider Inheritance**:\n   - Child modules can inherit provider configuration from parent modules\n   - Explicit `provider` meta-argument overrides inheritance\n   - Provider aliases enable multiple configurations of the same provider type\n\n3. **Configuration Evaluation**:\n   - The merged configuration body is evaluated in the graph walk\n   - Variables and locals are interpolated\n   - The result is a `cty.Value` (Terraform's type system)\n\n4. **Configuration Application**:\n   - `ConfigureProvider()` is called with the evaluated configuration\n   - The provider stores this for use when operating on resources\n\n### 2.5 Resource Operation Delegation\n\nWhen a resource instance is applied:\n\n1. **Prior State**: Retrieved from the state store\n2. **Proposed New State**: Computed from configuration + known values\n3. **Provider Plan** (Plan phase):\n   - Terraform calls `provider.PlanResourceChange()`\n   - Provider receives: prior state, proposed new state, configuration\n   - Provider returns: planned new state + planned action (create/update/delete/no-op)\n   - Provider can suggest replacement if certain attributes require it\n\n4. **Provider Apply** (Apply phase):\n   - Terraform calls `provider.ApplyResourceChange()`\n   - Provider receives: prior state, planned state\n   - Graph nodes read a snapshot of state at the start of evaluation\n   - Multiple nodes can read the same snapshot concurrently\n   - Writes create new snapshots that subsequent nodes see\n\n2. **Write-through Updates**:\n   - When a resource is applied, its state is immediately updated\n   - The update is atomic with respect to concurrent reads\n   - Other graph nodes' snapshots are not invalidated (eventual consistency)\n\n3. **Lock Acquisition**:\n   - Only during write operations (when actually modifying state)\n   - Held for minimal duration (just the state mutation)\n   - Does not block reads (readers use snapshots)\n\n### 3.3 State Snapshots and Isolation\n\nThe `RefreshState` and `PrevRunState` in `ContextGraphWalker` serve specific purposes:\n\n- **State**: The current state being modified by the apply (or plan) operation\n- **RefreshState**: A snapshot of state before refresh (during planning)\n- **PrevRunState**: The state file from the previous Terraform run (for change detection)\n\nThis enables:\n- **Deferred updates**: A resource can update state without affecting other in-flight evaluations\n- **Rollback simulation**: If a node fails, other nodes can still read a consistent state snapshot\n- **Plan preservation**: During planning, original state is preserved separately from computed changes\n\n### 3.4 State Persistence\n\nState is persisted to storage backends through the **backend** interface:\n\n```go\ntype Backend interface {\n    StateMgr(workspace string) (statemgr.Full, error)\n    DeleteWorkspace(name string) error\n    Workspaces() ([]string, error)\n    CurrentWorkspace() string\n}\n```\n\nBackends include:\n- **Local**: File-based state in `.terraform/` directory\n- **Remote** (Terraform Cloud): HTTPS API with versioning and locking\n- **S3, Azure Storage, Consul, etc.**: External storage with state locking\n\n### 3.5 State Locking\n\nFor remote backends, Terraform implements distributed locking:\n\n1. **Lock Acquisition** (before plan/apply):\n   - If lock has stale holder info, Terraform can force-unlock\n   - Force-unlock is a manual operation to prevent accidental state corruption\n\n3. **Lock Release** (after operation):\n   - Lock is released even if errors occur (via defer)\n   - If release fails, a warning is logged but execution continues\n\n### 3.6 State Change Tracking\n\nThe `plans.Changes` object tracks all proposed changes during planning:\n\n```go\ntype Changes struct {\n    Resources []*ResourceInstanceChange\n    Outputs   []*OutputChange\n}\n\ntype ResourceInstanceChange struct {\n    Addr      addrs.AbsResourceInstance\n    DeposedKey DeposedKey\n    Change    Change\n    Provider  addrs.Provider\n    // ...\n}\n```\n\nThis structure:\n- Is computed during the plan phase\n- Is compared against configuration during apply to detect drift\n- Is serialized to the plan file for human review\n- Forms the basis for the apply phase operations\n\n---\n\n## 4. # Terraform Plan/Apply Pipeline Architecture

## Overview

Terraform's plan and apply pipeline is built on a sophisticated architecture that combines directed acyclic graphs (DAGs), plugin-based provider management, concurrent state synchronization, and a extensible hook system. This document describes the internal architecture that enables Terraform to safely plan and execute infrastructure changes in a declarative, concurrent manner.

The core insight of Terraform's architecture is the separation of concerns:
- **Planning**: The plan phase reads configuration and current state, then delegates to providers to determine what changes are needed
- **Applying**: The apply phase executes the changes described by the plan, updating infrastructure and state

Both phases rely on a common graph-based execution engine that respects resource dependencies, manages provider lifecycle, and provides extension points for tooling.
