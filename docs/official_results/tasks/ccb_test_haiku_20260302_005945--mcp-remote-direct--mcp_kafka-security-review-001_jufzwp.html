I've completed the security review of the Apache Kafka 3.8.0 code and identified **3 critical/high severity security defects**:

## Summary of Findings

### 1. **Missing Token Expiration Validation** (ScramSaslServer.java:121)
**Severity: CRITICAL**
- After retrieving the token expiry timestamp from the delegation token callback, there is no validation that the token hasn't expired
- This allows authentication with expired tokens, completely bypassing the intended token lifetime controls
- **Impact**: Authentication bypass for expired delegation tokens

### 2. **Incorrect Authorization ID Validation for Token Auth** (ScramSaslServer.java:132)
**Severity: HIGH**
- The authzid validation compares against `username` instead of the actual `authorizationId`
- For token authentication, `username` is the token ID but `authorizationId` is the token owner
- This causes valid authzid values (matching the token owner) to be incorrectly rejected
- **Impact**: Inconsistent behavior for token-based SCRAM authentication with explicit authzid

### 3. **Authorization Bypass via Missing Deny Prefix Check** (AclAuthorizer.scala:377)
**Severity: CRITICAL**
- The `authorizeByResourceType` method returns ALLOWED for the "allow.everyone.if.no.acl.found" config **before** checking deny prefix ACLs
- Requests that should be denied by prefixed ACLs can bypass authorization when this permissive config is enabled
- **Impact**: Authorization bypass allowing unauthorized access to resources protected by prefixed deny ACLs
