## Key Findings

**Files Related to Memory Pressure Eviction:**

1. **eviction_manager.go** - Contains the main `managerImpl` type that orchestrates pod eviction, with the `synchronize()` method as the core control loop
2. **helpers.go** - Implements critical decision functions:
   - `thresholdsMet()` - Evaluates if resource thresholds are exceeded
   - `nodeConditions()` - Maps signals to node conditions (NodeMemoryPressure)
   - `rankMemoryPressure()` - Ranks pods for eviction
3. **api/types.go** - Defines eviction signals:
   - `SignalMemoryAvailable` ("memory.available")
   - `SignalAllocatableMemoryAvailable` ("allocatableMemory.available")
4. **defaults_linux.go** - Defines the default memory threshold: `"memory.available": "100Mi"`
5. **threshold_notifier_linux.go** - Implements kernel-level memory notifications via cgroup events

**Eviction Decision Chain:**

1. **NewManager()** initializes the eviction manager with configured thresholds
2. **Start()** begins the monitoring loop and creates memory threshold notifiers
3. **synchronize()** is the main loop that:
   - Collects memory observations via `makeSignalObservations()`
   - Checks if observations exceed thresholds with `thresholdsMet()`
   - Maps to `NodeMemoryPressure` node condition
   - Ranks pods using `rankMemoryPressure()` based on memory usage and QoS
   - Terminates selected pods via `evictPod()`

The memory threshold of 100Mi on Linux is the trigger point where pods with the lowest priority are evicted to free up memory and resolve the memory pressure condition.
